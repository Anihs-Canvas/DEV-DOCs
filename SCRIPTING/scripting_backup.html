1.1 What is Scripting vs Programming vs Automation?

Fundamentals Critical Beginner

Understanding the distinctions between scripting, programming, and automation in DevOps context

Comprehensive Overview

Scripting involves writing small, task-focused programs (scripts) that automate specific operations, typically interpreted at runtime. Scripts are usually single-file or few-file solutions designed to solve immediate problems without complex architecture.

Programming encompasses building larger, structured applications with proper architecture, design patterns, multiple modules, testing frameworks, and long-term maintainability in mind. Programs often compile to executables or large application bundles.

Automation is the broader practice of using technology (including scripts and programs) to perform tasks without human intervention. It's the goal, while scripting and programming are the means to achieve it.

Aspect Scripting Programming Automation
Purpose Quick task execution Build complex systems Eliminate manual work
Size 10-500 lines typically 1000+ lines, multiple files Varies (pipelines, workflows)
Execution Interpreted (runtime) Often compiled Scheduled/triggered
Languages Bash, Python, PowerShell Python (Django/FastAPI), TypeScript YAML, HCL, DSLs
Development Time Minutes to hours Days to months Hours to weeks

"""
============================================================
Deployer Module - Core Deployment Logic (PROGRAMMING Approach)
============================================================
WHAT THIS IS: A PROGRAM - multiple files, classes, design patterns, testable

WHY PROGRAMMING HERE (instead of a simple script)?
• Used by 50+ developers daily
• Needs to support multiple deployment strategies (Rolling, Blue-Green, Canary)
• Requires proper error handling, logging, and audit trails
• Must be testable (unit tests, integration tests)
• Will be maintained and extended for years

CONCEPTS DEMONSTRATED:
• Strategy Pattern (swappable deployment methods)
• Dependency Injection (testable, loosely coupled)
• Type Hints (better IDE support, documentation)
• Dataclasses (clean data structures)
============================================================
"""
from abc import ABC, abstractmethod      # For creating abstract base classes
from dataclasses import dataclass        # Python 3.7+ for clean data structures
from typing import List, Optional        # Type hints for better code quality
import logging                           # Standard logging (not print statements!)

from .config import DeployConfig         # Configuration from separate module
from .services.git_service import GitService      # Git operations abstracted
from .services.docker_service import DockerService # Docker operations abstracted

# Create a logger named after this module (best practice for tracing)
logger = logging.getLogger(__name__)


@dataclass  # This decorator auto-generates __init__, __repr__, __eq__ methods
class DeploymentResult:
    """
    Immutable result object for deployment operations.
    
    WHY USE A DATACLASS?
    • Cleaner than dictionaries (IDE autocomplete, type checking)
    • Automatic __init__ and __repr__
    • Can be easily serialized to JSON for APIs
    • Self-documenting with type hints
    
    EXAMPLE USAGE:
        result = DeploymentResult(success=True, environment="prod", ...)
        if result.success:
            print(f"Deployed {result.version} in {result.duration_seconds}s")
    """
    success: bool              # Did the deployment succeed?
    environment: str           # Which environment (staging, production)?
    version: str              # What version was deployed (e.g., "1.2.3")?
    duration_seconds: float    # How long did it take?
    errors: List[str]         # List of any errors that occurred


class DeploymentStrategy(ABC):
    """
    Abstract base class for deployment strategies.
    
    THIS IS THE STRATEGY PATTERN:
    • Define a family of algorithms (deployment methods)
    • Make them interchangeable
    • Client code doesn't need to know which strategy is used
    
    WHY USE THIS PATTERN?
    • Different teams might need different deployment approaches
    • Easy to add new strategies without changing existing code
    • Each strategy can be tested in isolation
    
    SUBCLASSES MUST IMPLEMENT: deploy()
    """
    
    @abstractmethod  # Forces subclasses to implement this method
    def deploy(self, config: DeployConfig) -> DeploymentResult:
        """Execute deployment according to this strategy's algorithm."""
        pass  # No implementation here - subclasses provide it


class RollingDeployment(DeploymentStrategy):
    """
    Rolling Deployment Strategy - Zero Downtime Deployments
    
    HOW ROLLING DEPLOYMENT WORKS:
    ┌─────────────────────────────────────────────────────┐
    │ Pod 1: v1.0 ──► v1.1 (updated first)               │
    │ Pod 2: v1.0 (still serving traffic)                │
    │ Pod 3: v1.0 (still serving traffic)                │
    └─────────────────────────────────────────────────────┘
    Then Pod 2, then Pod 3... gradually rolling through
    
    ADVANTAGES:
    • Zero downtime (some pods always running)
    • Easy rollback (stop and some pods still on old version)
    • Gradual traffic shift (can detect issues early)
    
    USE WHEN: Most production deployments
    """
    
    def __init__(self, git: GitService, docker: DockerService):
        """
        Initialize with required services.
        
        DEPENDENCY INJECTION:
        Services are passed IN, not created here. This allows:
        • Unit testing with mock services
        • Swapping implementations without changing this class
        • Loose coupling between components
        """
        self.git = git
        self.docker = docker
    
    def deploy(self, config: DeployConfig) -> DeploymentResult:
        """
        Execute a rolling deployment.
        
        Args:
            config: DeployConfig with environment, version, branch, etc.
            
        Returns:
            DeploymentResult with success status and metadata
            
        Raises:
            GitError: If checkout fails
            DockerError: If build or push fails
        """
        logger.info(f"Starting rolling deployment to {config.environment}")
        
        # Each step is clear, logged, and can fail independently
        self.git.checkout(config.branch)    # Step 1: Get the code
        image_tag = self.docker.build(      # Step 2: Build container image
            config.dockerfile, 
            config.version
        )
        self.docker.push(image_tag)         # Step 3: Push to registry
        
        # Return structured result (much better than True/False)
        return DeploymentResult(
            success=True,
            environment=config.environment,
            version=config.version,
            duration_seconds=45.2,
            errors=[]
        )


class Deployer:
    """
    Main Deployer Class - Orchestrates the Deployment Process
    
    THIS IS THE CONTEXT IN STRATEGY PATTERN:
    • Holds a reference to the current strategy
    • Delegates the actual work to the strategy
    • Can switch strategies at runtime if needed
    
    SEPARATION OF CONCERNS:
    • Deployer: orchestration, logging, pre/post hooks
    • Strategy: actual deployment algorithm
    """
    
    def __init__(self, strategy: DeploymentStrategy):
        """Store the deployment strategy to use."""
        self._strategy = strategy  # Could be Rolling, BlueGreen, Canary, etc.
    
    def execute(self, config: DeployConfig) -> DeploymentResult:
        """Execute deployment using the configured strategy."""
        return self._strategy.deploy(config)

#!/usr/bin/env python3
"""
Deploy Tool - Enterprise-grade deployment automation
Usage: deploy-tool --env production --version 1.2.3
"""
import argparse
import sys
from pathlib import Path

from .config import load_config
from .deployer import Deployer, RollingDeployment
from .services.git_service import GitService
from .services.docker_service import DockerService


def main():
    parser = argparse.ArgumentParser(description='Deploy applications')
    parser.add_argument('--env', required=True, choices=['staging', 'production'])
    parser.add_argument('--version', required=True)
    parser.add_argument('--dry-run', action='store_true')
    args = parser.parse_args()
    
    config = load_config(args.env)
    config.version = args.version
    
    strategy = RollingDeployment(GitService(), DockerService())
    deployer = Deployer(strategy)
    
    result = deployer.execute(config)
    
    if result.success:
        print(f"✅ Deployed {result.version} to {result.environment}")
        sys.exit(0)
    else:
        print(f"❌ Deployment failed: {result.errors}")
        sys.exit(1)


if __name__ == '__main__':
    main()

#!/bin/bash
#============================================================
# restart-unhealthy.sh - Quick Script to Restart Unhealthy Pods
#============================================================
# PYRAMID LEVEL 1: Ad-hoc script for immediate problem solving
#
# SITUATION: It's 3 AM, pods are crashing, you need to fix it NOW
# This script finds and restarts unhealthy pods automatically.
#
# WHY A SCRIPT (not a tool)?
# • Need it working in 5 minutes
# • One-time or rare use
# • Direct kubectl commands are the right tool
# • No complex logic needed
#
# USAGE: ./restart-unhealthy.sh [namespace]
#============================================================

# PARAMETER: Namespace to check (defaults to 'production' if not provided)
# The ${1:-production} syntax means: use $1 if provided, else use 'production'
NAMESPACE=${1:-production}

echo "🔍 Finding unhealthy pods in $NAMESPACE..."

#------------------------------------------------------------
# STEP 1: Find pods that aren't in "Running" state
#------------------------------------------------------------
# kubectl get pods: List all pods
# -o jsonpath: Output only specific fields (not full YAML)
# The JSONPath query finds pods where status.phase != "Running"
# Possible phases: Pending, Running, Succeeded, Failed, Unknown
UNHEALTHY_PODS=$(kubectl get pods -n $NAMESPACE \
    -o jsonpath='{.items[?(@.status.phase!="Running")].metadata.name}')

#------------------------------------------------------------
# STEP 2: Also find pods that are Running but NOT Ready
#------------------------------------------------------------
# A pod can be "Running" but still unhealthy if containers aren't ready
# -o json: Full JSON output (needed for complex filtering)
# jq: JSON processor (filters pods where ready==false)
# This catches pods stuck in CrashLoopBackOff, ImagePullBackOff, etc.
UNHEALTHY_PODS="$UNHEALTHY_PODS $(kubectl get pods -n $NAMESPACE \
    -o json | jq -r '.items[] | select(.status.containerStatuses[]?.ready==false) | .metadata.name')"

#------------------------------------------------------------
# STEP 3: Deduplicate the list
#------------------------------------------------------------
# tr ' ' '\n': Convert spaces to newlines (one pod per line)
# sort -u: Sort and remove duplicates (-u = unique)
UNHEALTHY_PODS=$(echo $UNHEALTHY_PODS | tr ' ' '\n' | sort -u)

#------------------------------------------------------------
# STEP 4: Exit early if everything is healthy
#------------------------------------------------------------
# -z: Test if string is empty
if [ -z "$UNHEALTHY_PODS" ]; then
    echo "✅ All pods are healthy!"
    exit 0  # Exit with success code
fi

# Show what we found (always good to log before taking action!)
echo "⚠️  Found unhealthy pods:"
echo "$UNHEALTHY_PODS"

#------------------------------------------------------------
# STEP 5: Delete each unhealthy pod
#------------------------------------------------------------
# WHY DELETE? Kubernetes Deployment will automatically recreate them
# This is the "turn it off and on again" of Kubernetes
for POD in $UNHEALTHY_PODS; do
    echo "🔄 Restarting: $POD"
    kubectl delete pod $POD -n $NAMESPACE
    # Kubernetes sees the pod count is below desired replicas
    # and immediately schedules a new pod to replace it
done

#------------------------------------------------------------
# STEP 6: Wait for rollout to complete
#------------------------------------------------------------
echo "✅ Restart initiated. Monitoring..."
kubectl rollout status deployment/anihpj-web -n $NAMESPACE
# This command blocks until deployment is healthy or times out

#!/usr/bin/env python3
"""
============================================================
Pod Manager - Using Kubernetes Python SDK
============================================================
PYRAMID LEVEL 2: Direct SDK/library usage for programmatic control

WHY LEVEL 2 (SDK) INSTEAD OF LEVEL 1 (SCRIPT)?
• Need structured data (not just text output)
• Want proper error handling (not just exit codes)
• Building a reusable tool (not one-off)
• Need rich output formatting (tables, colors)
• Integrating with other Python code

LIBRARIES USED:
• kubernetes: Official K8s Python client (talks to API server)
• rich: Beautiful terminal output (tables, colors, progress)
• click: CLI framework (arguments, help text, validation)

WHEN TO USE THIS LEVEL:
• Task is recurring (daily/weekly)
• Need to build on top of it (import as module)
• Require structured error handling
• Want nice CLI with help text
============================================================
"""
from kubernetes import client, config           # Official K8s SDK
from kubernetes.client.rest import ApiException # K8s-specific exceptions
from rich.console import Console                # Pretty terminal output
from rich.table import Table                    # ASCII tables
from dataclasses import dataclass               # Clean data structures
from typing import List                         # Type hints
import click                                    # CLI framework

# Rich console for pretty output (instead of plain print())
console = Console()


@dataclass  # Auto-generates __init__, __repr__, __eq__
class PodHealth:
    """
    Data class to hold pod health information.
    
    WHY A DATACLASS?
    • Clean, typed structure (better than dictionaries)
    • IDE autocomplete works
    • Easy to pass around between functions
    • Can add methods if needed later
    """
    name: str          # Pod name (e.g., "anihpj-web-7d8f9b7c4d-2xkjl")
    namespace: str     # Namespace (e.g., "production")
    phase: str         # Current phase (Running, Pending, Failed, etc.)
    ready: bool        # Are all containers ready?
    restarts: int      # Total container restarts (high = problem!)


class PodManager:
    """
    Kubernetes pod management using official Python client.
    
    THIS CLASS PROVIDES:
    • list_pods(): Get all pods with health status
    • get_unhealthy_pods(): Filter to only unhealthy ones
    • restart_pod(): Delete a pod to trigger recreation
    • display_status(): Pretty table output
    
    PATTERN: Encapsulate K8s operations so callers don't need to know
    the details of the K8s API.
    """
    
    def __init__(self, namespace: str = "production"):
        """
        Initialize the pod manager with Kubernetes configuration.
        
        This automatically detects whether running:
        • Inside a cluster (in-cluster config from service account)
        • Outside a cluster (uses ~/.kube/config file)
        """
        # Try in-cluster config first (when running as a K8s pod)
        try:
            config.load_incluster_config()
        except config.ConfigException:
            # Fall back to local kubeconfig (~/.kube/config)
            config.load_kube_config()
        
        # CoreV1Api handles pods, services, configmaps, etc.
        self.v1 = client.CoreV1Api()
        self.namespace = namespace
    
    def list_pods(self, label_selector: str = None) -> List[PodHealth]:
        """
        List all pods with health status.
        
        Args:
            label_selector: K8s label selector (e.g., "app=anihpj")
                           If None, returns ALL pods in namespace
        
        Returns:
            List of PodHealth objects with status information
        """
        # Call K8s API to get pods
        pods = self.v1.list_namespaced_pod(
            namespace=self.namespace,
            label_selector=label_selector
        )
        
        results = []
        for pod in pods.items:
            # A pod is "ready" only if ALL containers are ready
            # Using all() with generator expression for efficiency
            ready = all(
                cs.ready for cs in (pod.status.container_statuses or [])
            )
            # Total restarts across all containers
            # High restart count = something is crashing
            restarts = sum(
                cs.restart_count for cs in (pod.status.container_statuses or [])
            )
            results.append(PodHealth(
                name=pod.metadata.name,
                namespace=pod.metadata.namespace,
                phase=pod.status.phase,
                ready=ready,
                restarts=restarts
            ))
        return results
    
    def get_unhealthy_pods(self) -> List[PodHealth]:
        """
        Find pods that are not running or not ready.
        
        UNHEALTHY MEANS:
        • Phase is not "Running" (Pending, Failed, Unknown)
        • OR containers are not ready (CrashLoopBackOff, etc.)
        """
        all_pods = self.list_pods()
        # List comprehension to filter unhealthy pods
        return [p for p in all_pods if p.phase != "Running" or not p.ready]
    
    def restart_pod(self, pod_name: str) -> bool:
        """
        Delete a pod to trigger restart (K8s Deployment will recreate it).
        
        WHY DELETE?
        In Kubernetes, you don't "restart" pods directly.
        The Deployment controller sees replica count dropped and creates new pod.
        
        Returns:
            True if deletion succeeded, False otherwise
        """
        try:
            self.v1.delete_namespaced_pod(
                name=pod_name,
                namespace=self.namespace
            )
            console.print(f"[green]✓[/green] Deleted pod: {pod_name}")
            return True
        except ApiException as e:
            # ApiException contains K8s-specific error details
            console.print(f"[red]✗[/red] Failed to delete {pod_name}: {e.reason}")
            return False
    
    def display_status(self):
        """
        Display pod status in a beautiful Rich table.
        
        WHY RICH TABLES?
        • Much more readable than plain text
        • Colors highlight problems
        • Professional-looking output
        """
        pods = self.list_pods(label_selector="app=anihpj")
        
        # Create Rich table with columns
        table = Table(title=f"Pod Status - {self.namespace}")
        table.add_column("Pod Name", style="cyan")
        table.add_column("Phase", style="magenta")
        table.add_column("Ready", justify="center")
        table.add_column("Restarts", justify="right")
        
        for pod in pods:
            ready_icon = "✅" if pod.ready else "❌"
            phase_color = "green" if pod.phase == "Running" else "red"
            table.add_row(
                pod.name,
                f"[{phase_color}]{pod.phase}[/{phase_color}]",
                ready_icon,
                str(pod.restarts)
            )
        
        console.print(table)


# ============================================================
# CLI INTERFACE using Click
# ============================================================
# Click decorators turn this function into a full CLI with:
# • --help automatically generated
# • Argument parsing and validation
# • Type conversion

@click.command()
@click.option('--namespace', '-n', default='production', 
              help='Kubernetes namespace to manage')
@click.option('--restart-unhealthy', is_flag=True, 
              help='Automatically restart unhealthy pods')
def main(namespace: str, restart_unhealthy: bool):
    """
    Pod Manager CLI - Kubernetes SDK Level 2 automation
    
    Examples:
        python pod_manager.py                    # Show status
        python pod_manager.py -n staging         # Different namespace  
        python pod_manager.py --restart-unhealthy  # Auto-fix
    """
    manager = PodManager(namespace)
    
    # Always show current status
    manager.display_status()
    
    # Optionally restart unhealthy pods
    if restart_unhealthy:
        unhealthy = manager.get_unhealthy_pods()
        if unhealthy:
            console.print(f"\n[yellow]Found {len(unhealthy)} unhealthy pods[/yellow]")
            for pod in unhealthy:
                manager.restart_pod(pod.name)
        else:
            console.print("\n[green]All pods are healthy![/green]")


if __name__ == '__main__':
    main()  # Click handles argument parsing and calls main()

# Backup Solution Evolution Through the Pyramid

## Month 1: Level 1 (Script)
"We need backups NOW"

```bash
#!/bin/bash
pg_dump $DATABASE_URL | gzip > /backups/db_$(date +%Y%m%d).sql.gz
```
- Developer runs manually when they remember
- Sometimes forgets for weeks
- No verification, no retention policy


## Month 3: Level 2 (SDK/Library)
"We need more control and error handling"

```python
import boto3
import subprocess
from datetime import datetime

def backup():
    s3 = boto3.client('s3')
    result = subprocess.run(['pg_dump', os.environ['DATABASE_URL']], 
                          capture_output=True, check=True)
    s3.put_object(Bucket='backups', Key=f'db_{datetime.now()}.sql', 
                  Body=result.stdout)
```
- Added S3 upload, error handling
- Still manual execution
- Better but not automated


## Month 6: Level 3 (Tools)  
"We need this to work across all our databases"

```yaml
# ansible/roles/backup/tasks/main.yml
- name: Backup PostgreSQL database
  community.postgresql.postgresql_dump:
    db: "{{ db_name }}"
    dump_extra_args: "--format=custom"
  register: dump_result

- name: Upload to S3
  amazon.aws.s3_object:
    bucket: "{{ backup_bucket }}"
    object: "{{ db_name }}/{{ ansible_date_time.date }}.dump"
    src: "{{ dump_result.path }}"
    mode: put
    
- name: Cleanup old backups
  amazon.aws.s3_object:
    bucket: "{{ backup_bucket }}"
    prefix: "{{ db_name }}/"
    mode: delobj
  when: object.last_modified < (now() - timedelta(days=retention_days))
```
- Works across all databases
- Scheduled via cron/systemd
- Retention policy enforced

#!/bin/bash
#============================================================
# update-nginx.sh - Update Nginx Config Across All Web Servers
#============================================================
# WHY SCRIPTING MATTERS: This task takes 3+ hours manually.
# With this script: ~5 minutes for ALL 20 servers!
#
# WHAT THIS DEMONSTRATES:
# • Parallel execution (5 servers at once)
# • Automatic rollback on error
# • Logging for audit trail
# • Verification after changes
#
# THIS IS WHY DevOps NEEDS SCRIPTING:
# Manual: 200 minutes, error-prone, no audit trail
# Script: 5 minutes, consistent, fully logged
#============================================================

# SAFETY FLAGS - These make bash scripts much safer:
# -e : Exit immediately if any command fails
# -u : Treat unset variables as errors (catches typos!)
# -o pipefail : Pipeline fails if ANY command fails, not just last
set -euo pipefail

#------------------------------------------------------------
# CONFIGURATION - All changeable values at the top
# Makes the script reusable and easy to modify
#------------------------------------------------------------
SERVERS_FILE="./servers.txt"      # List of servers (one per line)
NGINX_PATCH="./nginx-patch.conf"  # Config changes to apply
SSL_CERT="./certs/anihpj.com.pem" # New SSL certificate
SSL_KEY="./certs/anihpj.com.key"  # New SSL private key
LOG_FILE="./logs/nginx-update-$(date +%Y%m%d-%H%M%S).log"  # Timestamped log

# ANSI color codes for readable output
# These make success/failure obvious at a glance
GREEN='\033[0;32m'
RED='\033[0;31m'
NC='\033[0m'  # No Color (reset)

#------------------------------------------------------------
# HELPER FUNCTION: Logging with timestamps
#------------------------------------------------------------
# Outputs to both screen AND log file (tee -a = append)
# Having timestamps in logs is CRITICAL for troubleshooting
log() {
    echo -e "[$(date '+%Y-%m-%d %H:%M:%S')] $1" | tee -a "$LOG_FILE"
}

#------------------------------------------------------------
# MAIN FUNCTION: Update a single server
#------------------------------------------------------------
# This function does EVERYTHING needed for one server.
# It's designed to be run in parallel across many servers.
update_server() {
    local server=$1  # 'local' prevents variable name collisions
    log "Processing: $server"
    
    #--------------------------------------------------------
    # STEP 1: BACKUP before making any changes
    # GOLDEN RULE: Always have a way to rollback!
    #--------------------------------------------------------
    ssh "$server" "sudo cp /etc/nginx/nginx.conf /etc/nginx/nginx.conf.bak.$(date +%Y%m%d)"
    
    #--------------------------------------------------------
    # STEP 2: Copy new configuration to server
    # scp = secure copy (like cp but over SSH)
    #--------------------------------------------------------
    scp "$NGINX_PATCH" "$server:/tmp/nginx-patch.conf"
    
    #--------------------------------------------------------
    # STEP 3: Apply config changes using sed
    # Why sed? It's available on every Linux server
    # First delete old location block, then insert new one
    #--------------------------------------------------------
    ssh "$server" 'sudo sed -i "/location \/static\//,/}/d" /etc/nginx/nginx.conf'
    ssh "$server" 'sudo sed -i "/server {/r /tmp/nginx-patch.conf" /etc/nginx/nginx.conf'
    
    #--------------------------------------------------------
    # STEP 4: Update SSL certificates
    # Note: Private key needs restricted permissions (600)
    #--------------------------------------------------------
    scp "$SSL_CERT" "$server:/tmp/ssl.pem"
    scp "$SSL_KEY" "$server:/tmp/ssl.key"
    ssh "$server" "sudo mv /tmp/ssl.pem /etc/ssl/certs/anihpj.com.pem"
    ssh "$server" "sudo mv /tmp/ssl.key /etc/ssl/private/anihpj.com.key"
    ssh "$server" "sudo chmod 600 /etc/ssl/private/anihpj.com.key"
    
    #--------------------------------------------------------
    # STEP 5: TEST configuration BEFORE reloading
    # nginx -t = test config syntax
    # If invalid, rollback to backup immediately
    #--------------------------------------------------------
    if ssh "$server" "sudo nginx -t" 2>&1 | tee -a "$LOG_FILE"; then
        log "${GREEN}✓${NC} Config valid on $server"
    else
        log "${RED}✗${NC} Config INVALID on $server - rolling back"
        # AUTOMATIC ROLLBACK - this is why we made a backup!
        ssh "$server" "sudo cp /etc/nginx/nginx.conf.bak.$(date +%Y%m%d) /etc/nginx/nginx.conf"
        return 1  # Return error code so we know this server failed
    fi
    
    #--------------------------------------------------------
    # STEP 6: Reload Nginx (graceful reload = no downtime)
    # 'reload' vs 'restart': reload doesn't drop connections
    #--------------------------------------------------------
    ssh "$server" "sudo systemctl reload nginx"
    
    #--------------------------------------------------------
    # STEP 7: VERIFY the changes actually worked
    # Don't just assume - verify!
    #--------------------------------------------------------
    if ssh "$server" "curl -sI http://localhost/static/test.css | grep -q 'Cache-Control'"; then
        log "${GREEN}✓${NC} Verified: Cache headers present on $server"
    else
        log "${RED}✗${NC} Warning: Cache headers not detected on $server"
    fi
}

#------------------------------------------------------------
# MAIN ENTRY POINT
#------------------------------------------------------------
main() {
    log "Starting Nginx update across $(wc -l < $SERVERS_FILE) servers"
    
    #--------------------------------------------------------
    # PARALLEL EXECUTION - the secret to speed!
    # xargs -P 5: Run 5 servers in parallel
    # Without this: 20 servers × 30 sec = 10 minutes
    # With this: 20 servers ÷ 5 parallel × 30 sec = 2 minutes
    #--------------------------------------------------------
    # Export functions/variables so child processes can use them
    export -f update_server log
    export NGINX_PATCH SSL_CERT SSL_KEY LOG_FILE GREEN RED NC
    
    # xargs reads server names and runs update_server in parallel
    cat "$SERVERS_FILE" | xargs -P 5 -I {} bash -c 'update_server "$@"' _ {}
    
    log "Update complete! Check log: $LOG_FILE"
}

# Run main function with all script arguments
main "$@"

# Can be triggered manually OR automatically by PagerDuty webhook

SLACK_WEBHOOK="${SLACK_INCIDENT_WEBHOOK}"

        -H 'Content-type: application/json' \
        -d "{\"text\": \"$1\"}"
}

log() {
    echo "[$(date '+%H:%M:%S')] $1"
    notify_slack "🔧 Incident Response: $1"
}

main() {
    log "🚨 DB Connection Recovery Started"
    
    # Step 1: Immediate - Scale down to reduce connections
    log "Step 1: Scaling deployment from 10 to 3 replicas"
    kubectl scale deployment/$DEPLOYMENT -n $NAMESPACE --replicas=3
    
    # Step 2: Wait for terminating pods to release connections
    log "Step 2: Waiting for pods to terminate..."
    kubectl wait --for=delete pod -l app=anihpj -n $NAMESPACE --timeout=60s || true
    sleep 10

    CURRENT_CONN=$(psql "$DATABASE_URL" -t -c \
        "SELECT count(*) FROM pg_stat_activity WHERE datname='anihpj';")
    log "Step 3: Current DB connections: $CURRENT_CONN"

    if [ "$CURRENT_CONN" -gt 150 ]; then
        log "Step 4: Killing idle connections older than 5 minutes"
        psql "$DATABASE_URL" -c "
            SELECT pg_terminate_backend(pid) 
            FROM pg_stat_activity 
            WHERE datname = 'anihpj' 
            AND state = 'idle' 
            AND state_change < NOW() - INTERVAL '5 minutes';
        "
    fi
    
    # Step 5: Verify application is responding

#!/usr/bin/env python3
"""
cleanup-resources.py - Automated AWS resource cleanup
Runs nightly via CloudWatch Events/Lambda

Typical savings: $2,000-5,000/month for medium-sized infrastructure
"""
import boto3
from datetime import datetime, timedelta
import logging

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)


class ResourceCleaner:
    def __init__(self):
        self.ec2 = boto3.client('ec2')
        self.rds = boto3.client('rds')
        self.ecr = boto3.client('ecr')
        self.savings = 0.0
    
    def cleanup_old_snapshots(self, retention_days: int = 30):
        """Delete EBS snapshots older than retention period"""
        logger.info("Cleaning up old EBS snapshots...")
        cutoff = datetime.now() - timedelta(days=retention_days)
        
        snapshots = self.ec2.describe_snapshots(OwnerIds=['self'])['Snapshots']
        
        for snap in snapshots:
            if snap['StartTime'].replace(tzinfo=None) < cutoff:
                # Estimate: $0.05/GB/month
                size_gb = snap['VolumeSize']
                monthly_cost = size_gb * 0.05
                
                logger.info(f"Deleting snapshot {snap['SnapshotId']} "
                           f"({size_gb}GB, ~${monthly_cost:.2f}/mo)")
                self.ec2.delete_snapshot(SnapshotId=snap['SnapshotId'])
                self.savings += monthly_cost
        
        logger.info(f"Snapshot cleanup savings: ${self.savings:.2f}/month")
    
    def cleanup_unattached_volumes(self):
        """Delete EBS volumes not attached to any instance"""
        logger.info("Cleaning up unattached EBS volumes...")
        
        volumes = self.ec2.describe_volumes(
            Filters=[{'Name': 'status', 'Values': ['available']}]
        )['Volumes']
        
        for vol in volumes:
            # Skip if created recently (might be in-use soon)
            age = datetime.now(vol['CreateTime'].tzinfo) - vol['CreateTime']
            if age.days < 7:
                continue
            
            # Estimate: gp3 = $0.08/GB/month
            size_gb = vol['Size']
            monthly_cost = size_gb * 0.08
            
            logger.info(f"Deleting volume {vol['VolumeId']} "
                       f"({size_gb}GB, ~${monthly_cost:.2f}/mo)")
            self.ec2.delete_volume(VolumeId=vol['VolumeId'])
            self.savings += monthly_cost
    
    def cleanup_old_ecr_images(self, keep_count: int = 10):
        """Delete old container images, keeping only recent ones"""
        logger.info("Cleaning up old ECR images...")
        
        repos = self.ecr.describe_repositories()['repositories']
        
        for repo in repos:
            images = self.ecr.describe_images(
                repositoryName=repo['repositoryName'],
                filter={'tagStatus': 'TAGGED'}
            )['imageDetails']
            
            # Sort by push date, keep newest
            images.sort(key=lambda x: x['imagePushedAt'], reverse=True)
            
            for img in images[keep_count:]:
                size_gb = img['imageSizeInBytes'] / (1024**3)
                monthly_cost = size_gb * 0.10  # $0.10/GB/month
                
                logger.info(f"Deleting image {repo['repositoryName']}:"
                           f"{img.get('imageTags', ['untagged'])[0]}")
                self.ecr.batch_delete_image(
                    repositoryName=repo['repositoryName'],
                    imageIds=[{'imageDigest': img['imageDigest']}]
                )
                self.savings += monthly_cost
    
    def stop_dev_instances_after_hours(self):
        """Stop development instances outside business hours"""
        logger.info("Checking dev instances for shutdown...")
        
        # Only run outside business hours
        hour = datetime.now().hour
        if 9 <= hour <= 18:  # 9 AM - 6 PM
            logger.info("Within business hours, skipping")
            return
        
        instances = self.ec2.describe_instances(
            Filters=[
                {'Name': 'tag:Environment', 'Values': ['development', 'dev']},
                {'Name': 'instance-state-name', 'Values': ['running']}
            ]
        )
        
        for reservation in instances['Reservations']:
            for instance in reservation['Instances']:
                instance_id = instance['InstanceId']
                instance_type = instance['InstanceType']
                
                # Rough hourly cost estimate
                hourly_costs = {
                    't3.micro': 0.0104, 't3.small': 0.0208,
                    't3.medium': 0.0416, 't3.large': 0.0832,
                    'm5.large': 0.096, 'm5.xlarge': 0.192
                }
                hourly = hourly_costs.get(instance_type, 0.10)
                # 14 hours shutdown × 22 workdays = 308 hours saved
                monthly_savings = hourly * 308
                
                logger.info(f"Stopping {instance_id} ({instance_type}), "
                           f"saves ~${monthly_savings:.2f}/mo")
                self.ec2.stop_instances(InstanceIds=[instance_id])
                self.savings += monthly_savings
    
    def run_all(self):
        """Execute all cleanup tasks"""
        self.cleanup_old_snapshots()
        self.cleanup_unattached_volumes()
        self.cleanup_old_ecr_images()
        self.stop_dev_instances_after_hours()
        
        logger.info(f"\n💰 TOTAL ESTIMATED MONTHLY SAVINGS: ${self.savings:.2f}")
        return self.savings


if __name__ == '__main__':
    cleaner = ResourceCleaner()
    cleaner.run_all()

# PURPOSE: Provision complete AWS infrastructure for Django application
# WHEN TO USE: Creating/managing cloud resources (VPC, EKS, RDS, etc.)
# 
# WHY TERRAFORM?
# - Declarative: You describe WHAT you want, not HOW to create it
# - Idempotent: Running twice produces same result (safe to re-run)
# - State tracking: Knows what exists, only changes what's different
# - Plan before apply: Preview changes before making them
# 
# INFRASTRUCTURE vs CONFIGURATION:
# - Terraform CREATES resources (the building)
# - Ansible CONFIGURES resources (the furniture inside)
# ============================================================================

# ----------------------------------------------------------------------------
# TERRAFORM SETTINGS BLOCK
# ----------------------------------------------------------------------------
# The 'terraform' block configures Terraform itself, not your infrastructure
terraform {
  # Version constraint - ensures team uses compatible Terraform version
  required_version = ">= 1.5"
  
  # BACKEND CONFIGURATION - Where Terraform stores its state file
  # WHY S3 BACKEND?
  # - Team collaboration: Everyone reads/writes same state
  # - State locking: DynamoDB prevents concurrent modifications
  # - Encryption: State often contains secrets (DB passwords, etc.)
  backend "s3" {
    bucket         = "anihpj-terraform-state"      # S3 bucket name (must exist)
    key            = "infrastructure/terraform.tfstate"  # Path within bucket
    region         = "us-east-1"                   # Bucket region
    encrypt        = true                          # Server-side encryption
    dynamodb_table = "terraform-locks"             # Table for state locking
  }
}

# ----------------------------------------------------------------------------
# AWS PROVIDER CONFIGURATION
# ----------------------------------------------------------------------------
# Providers are plugins that let Terraform interact with cloud APIs
provider "aws" {
  region = var.region  # Variable allows different regions per environment
  
  # DEFAULT TAGS - Applied to ALL resources created by this config
  # WHY? Cost allocation, automation scripts, compliance audits
  default_tags {
    tags = {
      Project     = "anihpj"           # Which project owns this resource
      Environment = var.environment     # dev/staging/production
      ManagedBy   = "terraform"        # Don't modify manually!

# ----------------------------------------------------------------------------
# VPC MODULE - Network Foundation
# ----------------------------------------------------------------------------
# Virtual Private Cloud is your isolated network in AWS
# Think of it as your own data center in the cloud
#
# WHY MODULES?
# - Encapsulation: Complex logic hidden behind simple interface
# - Reusability: Same module for dev/staging/prod with different vars
# - Testing: Module can be tested independently
module "vpc" {
  source = "./modules/vpc"  # Local module path (could also be git URL)
  
  # NAMING: Include environment so you can identify resources easily
  name               = "anihpj-${var.environment}"
  cidr               = var.vpc_cidr                 # e.g., "10.0.0.0/16" = 65,536 IPs
  availability_zones = var.availability_zones       # Multi-AZ for high availability
  
  # SUBNET DESIGN - Layered network architecture for security
  # Public subnets: Load balancers, NAT gateways (internet-facing)
  # Private subnets: Application containers (no public IPs)
  # Database subnets: Only accessible from private subnets
  private_subnets  = var.private_subnet_cidrs     # e.g., ["10.0.1.0/24", "10.0.2.0/24"]
  public_subnets   = var.public_subnet_cidrs      # Where ALB lives
  database_subnets = var.database_subnet_cidrs    # Where RDS lives
  
  # NAT GATEWAY - Lets private subnets reach internet (for updates, etc.)
  # COST TIP: NAT gateways cost ~$32/month each!
  enable_nat_gateway = true
  single_nat_gateway = var.environment != "production"  # Multiple NATs only in prod for HA
}

# ----------------------------------------------------------------------------
# EKS MODULE - Kubernetes Platform
# ----------------------------------------------------------------------------
# Elastic Kubernetes Service - WHERE your Django containers run
#
# WHY EKS (vs self-managed K8s)?
# - AWS manages control plane (masters) - no patching required
# - Integrated with AWS services (ALB, IAM, Secrets Manager)
# - Automatic security updates available
module "eks" {
  source = "./modules/eks"
  
  cluster_name    = "anihpj-${var.environment}"
  cluster_version = "1.28"  # Kubernetes version - upgrade regularly for security!
  
  # NETWORK - Place nodes in private subnets (no public IPs = more secure)
  vpc_id     = module.vpc.vpc_id             # Reference to VPC created above
  subnet_ids = module.vpc.private_subnet_ids  # Nodes get private IPs only
  
  # NODE GROUPS - EC2 instances that run your Kubernetes pods
  # AWS Managed Node Groups handle AMI updates and scaling for you
  node_groups = {
    main = {
      instance_types = var.eks_instance_types  # e.g., ["t3.medium", "t3.large"]
      desired_size   = var.eks_desired_nodes   # Normal running count
      min_size       = var.eks_min_nodes       # Never scale below this
      max_size       = var.eks_max_nodes       # Cost protection - never scale above
    }
  }
  
  # IRSA - IAM Roles for Service Accounts (CRITICAL for security!)
  # WHY: Each pod gets only the AWS permissions it needs
  # NO credentials in code - pods assume roles automatically
  enable_irsa = true
}

# ============================================================================
# DYNAMIC INVENTORY - Auto-discover AWS EC2 instances
# ============================================================================
# PURPOSE: Automatically find servers instead of maintaining static host lists
# WHY DYNAMIC? When using auto-scaling, instances come and go - static IPs are useless
#
# HOW IT WORKS:
# 1. Ansible calls AWS EC2 API
# 2. Filters instances by your tags
# 3. Groups them by Role/Environment tags
# 4. You target groups like "role_webserver" in playbooks
# ============================================================================

# Use the AWS EC2 inventory plugin (not static hosts file)
plugin: amazon.aws.aws_ec2

regions:
  - us-east-1    # Only look in this region (saves API calls and time)

# FILTERS - Which instances to include
# Only instances with these tags AND running state
filters:
  tag:Environment: production    # Only prod instances
  tag:Project: anihpj            # Only our project's instances
  instance-state-name: running   # Skip stopped/terminated instances

# KEYED GROUPS - Automatically create Ansible groups from tags
# This is the MAGIC of dynamic inventory!
keyed_groups:
  # Creates groups like: role_webserver, role_database, role_worker
  - key: tags.Role
    prefix: role
  # Creates groups like: env_production, env_staging
  - key: tags.Environment
    prefix: env

# COMPOSE - Build variables for each host
# Use private IP since Ansible runs from within VPC (via bastion or VPN)
compose:
  ansible_host: private_ip_address   # Connect via private IP, not public

# CONFIGURATION AUTOMATION - Ansible Site Playbook
# ============================================================================
# PURPOSE: Ensure consistent system state across all servers
# WHEN TO USE: After infrastructure exists, to configure and maintain servers
#
# KEY CONCEPT: Ansible is IDEMPOTENT
# - Run it 100 times, same result
# - Only makes changes if needed
# - Safe to run repeatedly (e.g., every 30 minutes via cron)
#
# CONFIGURATION vs INFRASTRUCTURE:
# - Terraform CREATES the servers (EC2 instances)
# - Ansible CONFIGURES the servers (packages, files, services)
# ============================================================================

---
# PLAY 1: Common configuration for ALL servers
# Notice: "hosts: all" targets every server in inventory
- name: Apply common configuration to all servers
  hosts: all               # Target every discovered server
  become: yes              # Run as root (sudo) - most config needs this
  roles:
    - common               # Base packages, users, timezone, NTP
    - security             # Firewall, SSH hardening, fail2ban

# PLAY 2: Web server specific configuration
# Notice: "hosts: role_webserver" only targets servers tagged Role=webserver
- name: Configure web servers
  hosts: role_webserver    # Only servers with tag Role=webserver
  become: yes
  roles:
    - nginx                # Reverse proxy, SSL termination
    - django               # Gunicorn, app deployment, static files
  vars:
    nginx_worker_processes: auto    # CPU cores = workers (auto-detect)
    django_workers: 4               # Gunicorn workers (2-4x CPU cores)

# ============================================================================
# COMMON ROLE - Base Configuration for ALL Servers
# ============================================================================
# PURPOSE: Establish consistent baseline across entire infrastructure
# APPLIES TO: Every server (web, worker, database, etc.)
#
# WHY A COMMON ROLE?
# - Single source of truth for base config
# - Security baseline everywhere
# - Same debugging tools on all servers
# - Consistent user/group setup
# ============================================================================

---
# STEP 1: Update package manager cache
# WHY: apt needs fresh package lists before installing
# cache_valid_time: Don't re-download if updated within 1 hour (3600 seconds)
- name: Update apt cache
  apt:
    update_cache: yes
    cache_valid_time: 3600   # Saves time on repeated runs

# STEP 2: Install essential packages
# WHY: Standard tools every engineer needs for debugging/operations
# state: present means "install if missing, leave alone if exists"
- name: Install essential packages
  apt:
    name:
      - vim           # Editor for quick config fixes
      - htop          # Better process viewer than top
      - curl          # HTTP client for API testing
      - wget          # File downloads
      - git           # Version control
      - unzip         # Extract archives
      - jq            # JSON parser (critical for AWS CLI output)
      - python3-pip   # Python package manager
      - net-tools     # ifconfig, netstat (debugging)
    state: present

# STEP 3: Standardize timezone
# WHY: Consistent timestamps in logs across all servers
# Makes log correlation during incidents much easier
- name: Set timezone
  timezone:
    name: "{{ timezone | default('UTC') }}"   # UTC is safest for distributed systems

# STEP 4: Time synchronization (critical for security)
# WHY: Certificate validation, API auth, and log correlation need accurate time
# chrony is modern replacement for ntpd - handles cloud VM clock drift better
- name: Configure NTP
  apt:
    name: chrony
    state: present
  notify: restart chrony   # Trigger handler if package was just installed

# STEP 5: Application service account
# WHY: Never run apps as root! Principle of least privilege
# groups: www-data allows access to web server sockets/files
- name: Create application user
  user:
    name: anihpj           # Dedicated user for our app
    shell: /bin/bash       # Can SSH in for debugging
    groups: www-data       # Access nginx sockets and static files
    append: yes            # Add to group, don't replace existing groups
    create_home: yes       # Needed for SSH, pip cache, etc.

# STEP 6: SSH key deployment
# WHY: Password auth is disabled; only key-based access allowed
# loop: Add all keys from ssh_authorized_keys variable
- name: Set up SSH authorized keys
  authorized_key:
    user: anihpj
    key: "{{ item }}"
  loop: "{{ ssh_authorized_keys }}"   # Defined in group_vars/all.yml

# STEP 7: Kernel tuning for production workloads
# WHY: Default Linux settings are for desktops, not servers
# These sysctl settings optimize for high network throughput
- name: Configure sysctl for performance
  sysctl:
    name: "{{ item.key }}"
    value: "{{ item.value }}"
    state: present
    reload: yes
  loop:
    - { key: 'net.core.somaxconn', value: '65535' }
    - { key: 'net.ipv4.tcp_max_syn_backlog', value: '65535' }
    - { key: 'vm.swappiness', value: '10' }

- name: Configure log rotation
  template:
    src: logrotate.conf.j2
    dest: /etc/logrotate.d/anihpj
    mode: '0644'

aws s3 ls "s3://$BACKUP_BUCKET/daily/" | while read -r line; do

        aws s3 rm "s3://$BACKUP_BUCKET/daily/$file"

done

curl -X POST "$SLACK_WEBHOOK" \
    -H 'Content-type: application/json' \
    -d '{"text": "✅ Database backup completed: '"$BACKUP_FILE"'"}'

log "Backup complete!"

#!/usr/bin/env python3
"""
health-report.py - Daily infrastructure health report
Posts summary to Slack every morning
"""
import boto3
import requests
from kubernetes import client, config
from datetime import datetime, timedelta
import os

SLACK_WEBHOOK = os.environ['SLACK_WEBHOOK']

class HealthReporter:
    def __init__(self):
        config.load_incluster_config()
        self.k8s = client.CoreV1Api()
        self.cloudwatch = boto3.client('cloudwatch')
    
    def get_pod_health(self) -> dict:
        """Check Kubernetes pod status"""
        pods = self.k8s.list_namespaced_pod('production')
        
        total = len(pods.items)
        running = sum(1 for p in pods.items if p.status.phase == 'Running')
        restarts = sum(
            sum(c.restart_count for c in p.status.container_statuses or [])
            for p in pods.items
        )
        
        return {
            'total': total,
            'running': running,
            'unhealthy': total - running,
            'restarts_24h': restarts
        }
    
    def get_error_rate(self) -> float:
        """Get 5xx error rate from CloudWatch"""
        response = self.cloudwatch.get_metric_statistics(
            Namespace='AWS/ApplicationELB',
            MetricName='HTTPCode_Target_5XX_Count',
            Dimensions=[{'Name': 'LoadBalancer', 'Value': 'app/anihpj-alb/xxx'}],
            StartTime=datetime.utcnow() - timedelta(hours=24),
            EndTime=datetime.utcnow(),
            Period=86400,
            Statistics=['Sum']
        )
        return response['Datapoints'][0]['Sum'] if response['Datapoints'] else 0
    
    def get_database_metrics(self) -> dict:
        """Get RDS metrics"""
        cpu = self.cloudwatch.get_metric_statistics(
            Namespace='AWS/RDS',
            MetricName='CPUUtilization',
            Dimensions=[{'Name': 'DBInstanceIdentifier', 'Value': 'anihpj-production'}],
            StartTime=datetime.utcnow() - timedelta(hours=24),
            EndTime=datetime.utcnow(),
            Period=86400,
            Statistics=['Average', 'Maximum']
        )
        return {
            'cpu_avg': cpu['Datapoints'][0]['Average'] if cpu['Datapoints'] else 0,
            'cpu_max': cpu['Datapoints'][0]['Maximum'] if cpu['Datapoints'] else 0
        }
    
    def generate_report(self) -> str:
        """Generate Slack-formatted report"""
        pods = self.get_pod_health()
        errors = self.get_error_rate()
        db = self.get_database_metrics()
        
        status = "✅" if pods['unhealthy'] == 0 and errors < 100 else "⚠️"
        
        report = f"""
*{status} Daily Health Report - {datetime.now().strftime('%Y-%m-%d')}*

*Kubernetes:*
• Pods: {pods['running']}/{pods['total']} running
• Unhealthy: {pods['unhealthy']}
• Restarts (24h): {pods['restarts_24h']}

*Application:*
• 5xx Errors (24h): {int(errors)}
• Error Rate: {'🔴 High' if errors > 100 else '🟢 Normal'}

*Database:*
• CPU Avg: {db['cpu_avg']:.1f}%
• CPU Max: {db['cpu_max']:.1f}%
• Status: {'🔴 High Load' if db['cpu_max'] > 80 else '🟢 Normal'}
        """
        return report
    
    def send_report(self):
        """Post report to Slack"""
        report = self.generate_report()
        requests.post(SLACK_WEBHOOK, json={'text': report})


if __name__ == '__main__':

"""
auto-healer - Lambda function triggered by CloudWatch alarms
Automatically remediates common issues
"""
import boto3
import json
import os
from kubernetes import client, config

def lambda_handler(event, context):
    """
    Triggered by SNS when CloudWatch alarm fires
    Automatically tries to fix common issues
    """
    # Parse alarm details
    message = json.loads(event['Records'][0]['Sns']['Message'])
    alarm_name = message['AlarmName']
    
    print(f"Processing alarm: {alarm_name}")
    
    # Route to appropriate handler
    handlers = {
        'HighCPU-Web': handle_high_cpu,
        'HighMemory-Web': handle_high_memory,
        'UnhealthyHosts': handle_unhealthy_hosts,
        'HighDBConnections': handle_db_connections,
    }
    
    if alarm_name in handlers:
        result = handlers[alarm_name](message)
        notify_slack(f"🔧 Auto-healer executed for {alarm_name}: {result}")
        return result
    else:
        notify_slack(f"⚠️ Unknown alarm {alarm_name} - manual intervention required")
        return {'status': 'unknown_alarm'}


def handle_high_cpu(message):
    """Scale up pods when CPU is high"""
    config.load_incluster_config()
    apps_v1 = client.AppsV1Api()
    
    # Get current replicas
    deployment = apps_v1.read_namespaced_deployment('anihpj', 'production')
    current = deployment.spec.replicas
    
    # Scale up by 50%
    new_count = min(current + max(2, current // 2), 20)  # Max 20 pods
    
    deployment.spec.replicas = new_count
    apps_v1.patch_namespaced_deployment('anihpj', 'production', deployment)
    
    return {
        'action': 'scale_up',
        'previous': current,
        'new': new_count
    }


def handle_high_memory(message):
    """Restart pods with highest memory usage"""
    config.load_incluster_config()
    v1 = client.CoreV1Api()
    
    # Get pods sorted by memory
    pods = v1.list_namespaced_pod('production', label_selector='app=anihpj')
    
    # Delete oldest pod (likely has memory leaks)
    oldest = min(pods.items, key=lambda p: p.metadata.creation_timestamp)
    v1.delete_namespaced_pod(oldest.metadata.name, 'production')
    
    return {
        'action': 'restart_pod',
        'pod': oldest.metadata.name
    }


def handle_unhealthy_hosts(message):
    """Replace unhealthy pods"""
    config.load_incluster_config()
    v1 = client.CoreV1Api()
    
    pods = v1.list_namespaced_pod('production', label_selector='app=anihpj')
    
    deleted = []
    for pod in pods.items:
        if pod.status.phase != 'Running':
            v1.delete_namespaced_pod(pod.metadata.name, 'production')
            deleted.append(pod.metadata.name)
    
    return {
        'action': 'replace_unhealthy',
        'deleted': deleted
    }


def handle_db_connections(message):
    """Scale down if DB connections exhausted"""
    config.load_incluster_config()
    apps_v1 = client.AppsV1Api()
    
    deployment = apps_v1.read_namespaced_deployment('anihpj', 'production')
    current = deployment.spec.replicas
    
    # Scale down to reduce connections
    new_count = max(3, current - 2)  # Min 3 pods
    
    deployment.spec.replicas = new_count
    apps_v1.patch_namespaced_deployment('anihpj', 'production', deployment)
    
    return {
        'action': 'scale_down',
        'reason': 'db_connection_limit',
        'previous': current,
        'new': new_count
    }

WHEN PYTHON IS THE RIGHT CHOICE

PURPOSE: Examples where Python beats Bash/other languages

RULE OF THUMB:

- If it involves JSON/YAML → Python (built-in support)
- If it's complex logic → Python (readable, testable)

- Error handling is painful in Bash

import json
from datetime import datetime, timedelta

# =========================================================================

def get_github_prs():
    """
    Fetch and process GitHub PRs to find stale ones.
    
    This combines:
    - HTTP request with auth headers
    - JSON parsing
    - List comprehension with filtering
    - Date arithmetic
    
    Bash would be ~100 lines of curl + jq + awk gymnastics!

                pr['created_at'].replace('Z', '+00:00')

                MetricName='CPUUtilization',
                Dimensions=[{'Name': 'InstanceId', 'Value': instance['InstanceId']}],
                StartTime=datetime.utcnow() - timedelta(days=7),
                EndTime=datetime.utcnow(),
                Period=86400,  # Daily average
                Statistics=['Average']
            )
            
            avg_cpu = sum(d['Average'] for d in metrics['Datapoints']) / len(metrics['Datapoints']) if metrics['Datapoints'] else 0
            
            if avg_cpu < 10:  # Less than 10% CPU
                underutilized.append({
                    'instance_id': instance['InstanceId'],
                    'type': instance['InstanceType'],
                    'avg_cpu': round(avg_cpu, 2)
                })
    
    return underutilized

def analyze_logs(log_file: str) -> dict:
    """Parse and analyze application logs"""
    import re
    from collections import defaultdict
    
    errors = defaultdict(list)
    response_times = []
    
    pattern = re.compile(
        r'(?P\d{4}-\d{2}-\d{2} \d{2}:\d{2}:\d{2}) '
        r'(?P\w+) '
        r'(?P.*?) '
        r'response_time=(?P\d+)ms'
    )
    
    with open(log_file) as f:
        for line in f:
            match = pattern.search(line)
            if match:
                data = match.groupdict()
                response_times.append(int(data['time']))
                if data['level'] == 'ERROR':
                    errors[data['message']].append(data['timestamp'])
    
    return {
        'total_requests': len(response_times),
        'avg_response_time': sum(response_times) / len(response_times) if response_times else 0,
        'p95_response_time': sorted(response_times)[int(len(response_times) * 0.95)] if response_times else 0,
        'error_summary': {msg: len(times) for msg, times in errors.items()}
    }

#!/usr/bin/env python3
"""
Secrets Manager Integration
Fetches secrets from AWS Secrets Manager or HashiCorp Vault
"""
import os
import json
import boto3
from functools import lru_cache

class SecretsManager:
    """Unified secrets access for automation scripts"""
    
    def __init__(self, provider: str = "aws"):
        self.provider = provider
        if provider == "aws":
            self._client = boto3.client('secretsmanager')
    
    @lru_cache(maxsize=100)
    def get_secret(self, secret_name: str) -> dict:
        """Fetch secret from secrets manager"""
        if self.provider == "aws":
            response = self._client.get_secret_value(SecretId=secret_name)
            return json.loads(response['SecretString'])
        raise ValueError(f"Unknown provider: {self.provider}")
    
    def get_database_url(self, env: str = "dev") -> str:
        """Get database connection string"""
        secret = self.get_secret(f"anihpj/{env}/database")
        return (
            f"postgres://{secret['username']}:{secret['password']}"
            f"@{secret['host']}:{secret['port']}/{secret['database']}"
        )
    
    def get_api_key(self, service: str, env: str = "dev") -> str:
        """Get API key for external service"""
        secret = self.get_secret(f"anihpj/{env}/api-keys")
        return secret.get(service)

# Usage in scripts
def main():
    secrets = SecretsManager()
    
    # Get database URL
    db_url = secrets.get_database_url(env="staging")
    print(f"Database configured: {db_url[:50]}...")
    
    # Get API keys
    stripe_key = secrets.get_api_key("stripe", env="staging")
    print(f"Stripe key: {stripe_key[:10]}...")

if __name__ == '__main__':
    main()

# The 3-Strike Rule for Automation

## The Rule:
1. **First time**: Do it manually, document the steps
2. **Second time**: Do it manually, refine the documentation
3. **Third time**: Automate it

## Why This Works:
- Prevents premature optimization
- Ensures you understand the problem
- Documents edge cases before coding
- Validates the task is actually recurring

| Factor              | Automate Now | Wait       | Never       |
|---------------------|--------------|------------|-------------|
| Frequency           | Daily+       | Weekly     | Once        |
| Time per execution  | >10 min      | 5-10 min   | <5 min      |
| Error potential     | High         | Medium     | Low         |
| Team benefit        | Everyone     | Some       | Only you    |
| Complexity          | Moderate     | High       | Trivial     |

- Runs daily? +3, weekly? +2, monthly? +1
- Takes >30 min? +3, 10-30 min? +2, <10 min? +1

#!/bin/bash
# ============================================================================
# IDEMPOTENT SCRIPT - Safe to run multiple times!
# ============================================================================
# IDEMPOTENCY: Running the script 1 time or 100 times produces SAME result
#
# WHY IDEMPOTENCY MATTERS:
# - CI/CD may retry failed jobs
# - Ansible/Terraform run repeatedly
# - Users accidentally double-click scripts
# - Cron jobs that overlap
#
# THE PATTERN: Check state → Act only if needed → Verify
# ============================================================================

# ---------------------------------------------------------------------------
# FIX 1: Check before adding (prevents duplicates)
# ---------------------------------------------------------------------------
# BAD:  echo "10.0.0.5 db-server" >> /etc/hosts
#       (adds duplicate line every run!)
# GOOD: Check if line exists first
grep -q "10.0.0.5 db-server" /etc/hosts || \
    echo "10.0.0.5 db-server" >> /etc/hosts
# -q: quiet (no output)
# ||: Execute right side only if left side fails (line not found)

# ---------------------------------------------------------------------------
# FIX 2: Use -p flag for mkdir (creates parent dirs, no error if exists)
# ---------------------------------------------------------------------------
# BAD:  mkdir /opt/app              (fails on second run!)
# GOOD: mkdir -p /opt/app           (creates if missing, silent if exists)
mkdir -p /opt/app

# ---------------------------------------------------------------------------
# FIX 3: Set absolute value, don't increment
# ---------------------------------------------------------------------------
# BAD:  COUNT=$(cat /var/counter)
#       echo $((COUNT + 1)) > /var/counter    (increments every run!)
# GOOD: Set the target state directly
echo "5" > /var/counter    # Always results in "5", regardless of runs

# ---------------------------------------------------------------------------
# FIX 4: Check if user exists before creating
# ---------------------------------------------------------------------------
# BAD:  useradd deploy              (fails if user exists!)
# GOOD: Check first with `id`
id deploy &>/dev/null || useradd deploy
# id: Check if user exists (exit code 0 = exists, 1 = doesn't exist)
# &>/dev/null: Suppress all output (we only care about exit code)
# ||: Only run useradd if id failed (user doesn't exist)

# ---------------------------------------------------------------------------
# FIX 5: Conditional migration (check before running)
# ---------------------------------------------------------------------------
# BAD:  python manage.py migrate    (usually safe, but some aren't!)
# GOOD: Check if specific migration has run
if ! python manage.py showmigrations | grep -q "\\[X\\] 0015_new"; then
    python manage.py migrate myapp 0015_new
fi
# [X] = migration applied, [ ] = not applied

# ============================================================================
# SUMMARY: The idempotency pattern
# ============================================================================
# 1. Check current state
# 2. Only act if state doesn't match desired state
# 3. Verify action succeeded
# Result: Same final state whether run 1x or 100x

Pattern: Check state → Act only if needed → Verify

import boto3
import subprocess

def ensure_s3_bucket(bucket_name: str, region: str = "us-east-1"):
    """Create S3 bucket if it doesn't exist (idempotent)"""
    s3 = boto3.client('s3')
    
    # Check current state
    try:
        s3.head_bucket(Bucket=bucket_name)
        print(f"✓ Bucket {bucket_name} already exists")
        return True
    except s3.exceptions.ClientError as e:
        if e.response['Error']['Code'] == '404':
            # Bucket doesn't exist, create it
            print(f"→ Creating bucket {bucket_name}...")
            s3.create_bucket(
                Bucket=bucket_name,
                CreateBucketConfiguration={'LocationConstraint': region}
            )
            print(f"✓ Bucket {bucket_name} created")
            return True
        raise

def ensure_directory(path: str, owner: str = None, mode: str = "755"):
    """Create directory with correct permissions (idempotent)"""
    import os
    import pwd
    
    # Check if exists with correct permissions
    if os.path.isdir(path):
        # Verify permissions
        current_mode = oct(os.stat(path).st_mode)[-3:]
        if current_mode == mode:
            print(f"✓ Directory {path} exists with correct permissions")
            return True
    
    # Create or fix
    os.makedirs(path, mode=int(mode, 8), exist_ok=True)
    if owner:
        uid = pwd.getpwnam(owner).pw_uid
        os.chown(path, uid, -1)
    
    print(f"✓ Directory {path} configured")
    return True

def ensure_systemd_service(service_name: str, enabled: bool = True, running: bool = True):
    """Ensure systemd service state (idempotent)"""
    
    # Check current state
    is_enabled = subprocess.run(
        ["systemctl", "is-enabled", service_name],
        capture_output=True
    ).returncode == 0
    
    is_running = subprocess.run(
        ["systemctl", "is-active", service_name],
        capture_output=True
    ).returncode == 0
    
    # Only change if needed
    if enabled and not is_enabled:
        subprocess.run(["systemctl", "enable", service_name])
        print(f"→ Enabled {service_name}")
    elif not enabled and is_enabled:
        subprocess.run(["systemctl", "disable", service_name])
        print(f"→ Disabled {service_name}")
    
    if running and not is_running:
        subprocess.run(["systemctl", "start", service_name])
        print(f"→ Started {service_name}")
    elif not running and is_running:
        subprocess.run(["systemctl", "stop", service_name])
        print(f"→ Stopped {service_name}")
    
    print(f"✓ Service {service_name} in desired state")

#!/usr/bin/env python3
"""
Fail-Fast Deployment Script
Stops immediately on any error with clear messaging
"""
import sys
import subprocess
from datetime import datetime

class DeploymentError(Exception):
    """Custom exception for deployment failures"""
    pass

def log(message: str, level: str = "INFO"):
    """Structured logging"""
    timestamp = datetime.now().isoformat()
    print(f"[{timestamp}] [{level}] {message}")

def run_command(cmd: str, description: str) -> str:
    """Run command with fail-fast behavior"""
    log(f"Executing: {description}")
    log(f"Command: {cmd}", "DEBUG")
    
    result = subprocess.run(
        cmd,
        shell=True,
        capture_output=True,
        text=True
    )
    
    if result.returncode != 0:
        log(f"FAILED: {description}", "ERROR")
        log(f"Exit code: {result.returncode}", "ERROR")
        log(f"Stderr: {result.stderr}", "ERROR")
        raise DeploymentError(f"{description} failed")
    
    log(f"SUCCESS: {description}")
    return result.stdout

def deploy(environment: str, version: str):
    """Deploy with fail-fast on any step"""
    
    log(f"Starting deployment of {version} to {environment}")
    
    steps = [
        ("Running pre-flight checks", f"./scripts/preflight.sh {environment}"),
        ("Building Docker image", f"docker build -t anihpj:{version} ."),
        ("Running tests", "docker run anihpj:{version} pytest"),
        ("Pushing to registry", f"docker push registry/anihpj:{version}"),
        ("Updating Kubernetes", f"kubectl set image deploy/anihpj app=anihpj:{version}"),
        ("Waiting for rollout", "kubectl rollout status deploy/anihpj --timeout=300s"),
        ("Running smoke tests", f"./scripts/smoke-test.sh {environment}"),
    ]
    
    completed = []
    try:
        for description, command in steps:
            run_command(command, description)
            completed.append(description)
            
        log(f"✅ Deployment complete: {version} → {environment}", "SUCCESS")
        
    except DeploymentError as e:
        log(f"❌ Deployment failed at: {e}", "ERROR")
        log(f"Completed steps: {completed}", "ERROR")
        log(f"Rolling back...", "WARN")
        
        # Rollback
        run_command(
            "kubectl rollout undo deploy/anihpj",
            "Rollback deployment"
        )
        
        sys.exit(1)

if __name__ == '__main__':
    deploy(
        environment=sys.argv[1],
        version=sys.argv[2]
    )

"""
============================================================================
AUTOMATION ROI CALCULATOR
============================================================================
PURPOSE: Make data-driven decisions about what to automate
USE THIS WHEN: Deciding whether to invest time in automation

THE CORE QUESTION:
"Will the time saved exceed the time invested in automation?"

KEY INSIGHT: Most people UNDERESTIMATE:
- Development time (testing, edge cases, documentation)
- Maintenance time (updates, bug fixes, training)

And OVERESTIMATE:
- How often the task actually runs
- How long the automation will be used

This calculator forces you to be honest about the numbers!
============================================================================
"""
from dataclasses import dataclass
from enum import Enum

# ---------------------------------------------------------------------------
# FREQUENCY ENUM - How often does this task run?
# ---------------------------------------------------------------------------
# The values are executions per year (365 days, 52 weeks, etc.)
# This is CRITICAL - daily tasks are MUCH more valuable to automate than yearly ones
class Frequency(Enum):
    HOURLY = 8760    # 24 hours × 365 days = 8,760 per year
    DAILY = 365      # Once per day × 365 days
    WEEKLY = 52      # Once per week × 52 weeks
    MONTHLY = 12     # Once per month × 12 months
    QUARTERLY = 4    # Once per quarter × 4 quarters
    YEARLY = 1       # Once per year

# ---------------------------------------------------------------------------
# ROI CALCULATOR CLASS
# ---------------------------------------------------------------------------
# Uses Python @dataclass for clean data structure with automatic __init__
# Uses @property for computed values that depend on the inputs
@dataclass
class AutomationROI:
    # INPUTS (you provide these)
    task_name: str                    # What are we automating?
    manual_minutes: float             # How long does it take manually?
    frequency: Frequency              # How often do we do this?
    automation_hours: float           # How long to build the automation?
    maintenance_hours_yearly: float = 0   # Ongoing maintenance (default: 0)
    
    # ---------------------------------------------------------------------------
    # CALCULATED PROPERTIES
    # ---------------------------------------------------------------------------
    
    @property
    def annual_manual_hours(self) -> float:
        """
        Total hours spent on this task per year if done manually.
        
        Formula: (minutes per task / 60) × tasks per year
        Example: 30 min weekly = (30/60) × 52 = 26 hours/year
        """
        return (self.manual_minutes / 60) * self.frequency.value
    
    @property
    def first_year_roi(self) -> float:
        """
        Net hours saved (or lost) in the first year.
        
        Negative = You're still paying off the automation investment
        Positive = You're already ahead!
        
        Reality: Most automation is NEGATIVE in year one!
        That's okay - it's an investment.
        """
        investment = self.automation_hours + self.maintenance_hours_yearly
        return self.annual_manual_hours - investment
    
    @property
    def break_even_months(self) -> float:
        """
        How many months until the automation pays for itself?
        
        Formula: Development hours ÷ monthly time saved
        
        RULE OF THUMB:
        - Break-even < 6 months = Strong YES to automate
        - Break-even 6-12 months = Consider carefully
        - Break-even > 12 months = Probably skip (unless strategic)
        """
        if self.annual_manual_hours == 0:
            return float('inf')   # Never breaks even if task takes 0 time!
        monthly_savings = self.annual_manual_hours / 12
        return self.automation_hours / monthly_savings
    
    @property
    def five_year_roi(self) -> float:
        """Total hours saved over 5 years"""
        investment = self.automation_hours + (self.maintenance_hours_yearly * 5)
        savings = self.annual_manual_hours * 5
        return savings - investment
    
    def report(self) -> str:
        """Generate human-readable report"""
        return f"""
╔══════════════════════════════════════════════════════════╗
║  AUTOMATION ROI ANALYSIS: {self.task_name:<30}║
╠══════════════════════════════════════════════════════════╣
║  Manual effort:                                          ║
║    • {self.manual_minutes:.0f} minutes per execution                        
║    • {self.frequency.name.lower()} frequency                               
║    • {self.annual_manual_hours:.1f} hours/year spent on this task          
║                                                          ║
║  Automation investment:                                  ║
║    • {self.automation_hours:.0f} hours to build                            
║    • {self.maintenance_hours_yearly:.0f} hours/year maintenance             
║                                                          ║
║  ROI Analysis:                                           ║
║    • Break-even: {self.break_even_months:.1f} months                       
║    • First year savings: {self.first_year_roi:+.1f} hours                 
║    • 5-year savings: {self.five_year_roi:+.1f} hours                      
║                                                          ║
║  Recommendation: {"✅ AUTOMATE" if self.break_even_months <= 6 else "⚠️ CONSIDER" if self.break_even_months <= 12 else "❌ SKIP"}

@dataclass

    """Calculate true cost including hidden factors"""
    
    # Basic inputs
    task_name: str
    direct_minutes: float
    frequency_per_year: int
    
    # Team factors
    hourly_rate: float = 75.0  # Engineer cost
    people_who_can_do_it: int = 1
    team_size: int = 5
    
    # Hidden costs
    context_switch_minutes: float = 20.0
    error_rate_percent: float = 5.0
    error_recovery_hours: float = 2.0
    outage_cost_per_hour: float = 1000.0  # Revenue impact
    
    # Automation costs
    automation_dev_hours: float = 0.0
    automation_maintenance_hours_yearly: float = 0.0
    
    @property
    def annual_direct_hours(self) -> float:
        """Direct execution time per year"""
        return (self.direct_minutes / 60) * self.frequency_per_year
    
    @property
    def annual_context_switch_hours(self) -> float:
        """Context switching cost per year"""
        return (self.context_switch_minutes / 60) * self.frequency_per_year
    
    @property
    def annual_error_cost(self) -> float:
        """Expected cost of errors per year"""
        errors_per_year = self.frequency_per_year * (self.error_rate_percent / 100)
        return errors_per_year * (
            self.error_recovery_hours * self.hourly_rate +
            self.outage_cost_per_hour * self.error_recovery_hours
        )
    
    @property
    def knowledge_risk_cost(self) -> float:
        """Annual cost of knowledge concentration"""
        if self.people_who_can_do_it >= self.team_size / 2:
            return 0
        
        # Risk: person leaves, 40 hours to transfer knowledge
        turnover_rate = 0.15  # 15% annual turnover
        knowledge_transfer_hours = 40
        
        probability_of_loss = turnover_rate * (
            self.people_who_can_do_it / self.team_size
        )
        return probability_of_loss * knowledge_transfer_hours * self.hourly_rate
    
    @property
    def total_annual_cost_manual(self) -> float:
        """Total annual cost if done manually"""
        return (
            (self.annual_direct_hours + self.annual_context_switch_hours) * self.hourly_rate +
            self.annual_error_cost +
            self.knowledge_risk_cost
        )

    def total_annual_cost_automated(self) -> float:

            automated_hours * self.hourly_rate +
            self.automation_maintenance_hours_yearly * self.hourly_rate +

        """How much automation saves per year"""
        return self.total_annual_cost_manual - self.total_annual_cost_automated

    def roi_percent(self) -> float:
        """Return on investment percentage"""
        if self.automation_dev_hours == 0:

        """Record an automation execution"""
        self.table.put_item(Item={
            'automation_name': execution.automation_name,
            'timestamp': execution.timestamp.isoformat(),
            'duration_seconds': int(execution.duration_seconds),
            'manual_equivalent_seconds': int(execution.manual_equivalent_seconds),
            'time_saved_seconds': int(execution.time_saved_seconds),
            'success': execution.success
        })
    
    def get_monthly_report(self, year: int, month: int) -> dict:
        """Generate monthly value report"""
        start = datetime(year, month, 1)
        if month == 12:
            end = datetime(year + 1, 1, 1)
        else:
            end = datetime(year, month + 1, 1)
        
        # Query executions
        response = self.table.scan(
            FilterExpression='#ts BETWEEN :start AND :end',
            ExpressionAttributeNames={'#ts': 'timestamp'},
            ExpressionAttributeValues={
                ':start': start.isoformat(),
                ':end': end.isoformat()
            }
        )
        
        items = response['Items']
        
        # Aggregate by automation
        automations = {}
        for item in items:
            name = item['automation_name']
            if name not in automations:
                automations[name] = {
                    'name': name,
                    'executions': 0,
                    'total_time_saved_hours': 0,
                    'success_rate': 0,
                    'successes': 0
                }
            
            automations[name]['executions'] += 1
            automations[name]['total_time_saved_hours'] += item['time_saved_seconds'] / 3600
            if item['success']:
                automations[name]['successes'] += 1
        
        # Calculate success rates
        for auto in automations.values():
            auto['success_rate'] = (auto['successes'] / auto['executions']) * 100
            auto['value_delivered'] = auto['total_time_saved_hours'] * self.hourly_rate
        
        total_hours = sum(a['total_time_saved_hours'] for a in automations.values())
        
        return {
            'period': f"{year}-{month:02d}",
            'total_executions': len(items),
            'total_hours_saved': round(total_hours, 1),
            'total_value_delivered': round(total_hours * self.hourly_rate, 2),
            'automations': list(automations.values())
        }

# When NOT to Automate

## Red Flags - Skip Automation If:

### 1. One-Time Tasks
- Data migration you'll never do again
- Initial setup of a new system
- One-off investigation
→ *Just do it manually*

### 2. Constantly Changing Requirements
- Process changes weekly
- No stable definition yet
- Still experimenting
→ *Wait until process stabilizes*

### 3. Simple + Infrequent
- Task takes 2 minutes
- Done once a month
- Very simple steps
→ *Not worth the investment*

### 4. Complex Edge Cases
- 20 different scenarios
- Human judgment required
- Exceptions are common
→ *Semi-automate: automate the common path*

### 5. Political/Approval Processes
- Requires human sign-off
- Compliance needs human review
- Multiple stakeholder approval
→ *Automate preparation, not decision*

## Real Examples of Bad Automation

### ❌ Automated Everything
**What happened**: Team automated a complex approval workflow
**Time to build**: 3 weeks
**Time saved**: 5 minutes per approval
**Approvals per year**: 10
**ROI**: -99% (wasted 3 weeks to save 50 minutes)

### ❌ Automated Too Early  
**What happened**: Built CI/CD before requirements were clear
**Time to build**: 2 weeks
**Rebuilds needed**: 4 times as requirements changed
**Total time wasted**: 6 weeks

### ❌ Automated the Wrong Thing
**What happened**: Automated report generation
**What they should have automated**: Report distribution
**Result**: Reports generated automatically, then manually emailed

## Decision Checklist

Before automating, ask:

☐ Will this task exist in 6 months?
☐ Is the process stable?
☐ Does it run at least weekly?
☐ Does it take more than 5 minutes?
☐ Is it error-prone when done manually?
☐ Would multiple people benefit?
☐ Is the ROI positive within 6 months?

If fewer than 4 checkmarks: **DON'T AUTOMATE**
