🚀 Quick Implementation Steps

  1. Install: pip install pytest pytest-django
  2. Configure: Create pytest.ini or add [tool.pytest.ini_options] to pyproject.toml

✅ Best Practices

✅ DO

  • Use @pytest.mark.django_db for database tests
  • Create fixtures in conftest.py for reuse
  • Use pytest.raises() for exception testing
  • Keep tests isolated and independent
  • Use descriptive test names
  • Register custom markers in pytest.ini
  • Use parametrize for data-driven tests
  • Clean up resources with fixture teardown

❌ DON'T

  • Share state between tests
  • Use print() for debugging (use -s flag)
  • Forget to mark async tests with @pytest.mark.asyncio
  • Hardcode absolute paths
  • Test implementation details
  • Skip tests without good reason
  • Use time.sleep() in tests
  • Depend on test execution order

📊 Constants

Pytest constants provide version information and special parameters that can be used throughout your test suite. These constants help you write version-aware tests and control test naming behavior.

pytest.__version__

Full Description

The pytest.__version__ constant returns the current installed version of pytest as a string. This is useful for checking pytest version compatibility in your test suite or when debugging test failures that might be version-specific. The version string follows semantic versioning (MAJOR.MINOR.PATCH) format.

Syntax

import pytest
version = pytest.__version__

Parameters Description

This constant does not accept any parameters. It is a read-only string attribute that contains the version information.

Attributes

Type: str

Format: MAJOR.MINOR.PATCH (e.g., "7.0.0")

Access: Read-only attribute

Module: pytest

Methods

This constant has no methods. It is a simple string constant. However, you can use standard Python string methods on it such as:

  • .split('.') - Split version into components
  • .startswith() - Check version prefix
  • comparison operators - Compare versions (use packaging.version for proper comparison)

Return Value

Returns: A string representing the current pytest version in the format "MAJOR.MINOR.PATCH"

Example return values: "7.0.0", "7.4.3", "8.0.0rc1"

pytest.HIDDEN_PARAM

Full Description

The pytest.HIDDEN_PARAM constant (added in pytest 8.4) is a special marker that can be passed to the ids parameter of pytest.mark.parametrize() or to the id parameter of pytest.param() to hide a parameter set from the test name. This is useful when you want to run parametrized tests but don't want certain parameter values to appear in the test output. It can only be used at most once per test since test names need to be unique.

Syntax

import pytest

# Use in parametrize
@pytest.mark.parametrize("param", [value1, value2], ids=[pytest.HIDDEN_PARAM, "visible"])
def test_function(param):
    pass

# Use in param
@pytest.mark.parametrize("value", [
    pytest.param(data, id=pytest.HIDDEN_PARAM),
    pytest.param(data2, id="named")
])

Parameters Description

This constant does not accept parameters. It is used as a value assignment to hide parameter identifiers in test names.

  • Usage context: Only in ids parameter of parametrize or id parameter of pytest.param()
  • Limitation: Can only be used once per test (test names must be unique)
  • Purpose: Hide sensitive or verbose parameter values from test output

Attributes

Type: Special marker constant

Added: pytest 8.4

Module: pytest

Usage: Test parametrization and naming control

Methods

This constant has no methods. It serves as a special marker value that pytest recognizes during test collection and naming.

Return Value

Returns: N/A - This is a marker constant used for test naming control, not a callable function.

pytest.version_tuple

Full Description

The pytest.version_tuple constant (added in pytest 7.0) returns the current pytest version as a tuple of integers and strings, making version comparisons more convenient than string parsing. The tuple format is (major, minor, patch) for stable releases, and (major, minor, prerelease_string) for pre-releases. This allows for easy programmatic version checking using standard Python tuple comparison operators.

Syntax

import pytest
version = pytest.version_tuple

# Example returns:
# (7, 4, 3) for stable version 7.4.3
# (7, 0, '0rc1') for pre-release 7.0.0rc1

Parameters Description

This constant does not accept any parameters. It is a read-only tuple attribute containing version components.

  • Stable versions: Returns tuple of three integers (major, minor, patch)
  • Pre-releases: Last component is a string (e.g., '0rc1', '1a1', '2b3')
  • Comparison: Can be compared directly using Python operators (<, <=, ==, >=, >)

Attributes

Type: tuple

Format: (major: int, minor: int, patch: int | str)

Added: pytest 7.0

Access: Read-only attribute

Module: pytest

Methods

This constant has no methods. It is a tuple, so standard tuple operations apply:

  • Indexing: version_tuple[0] for major version
  • Unpacking: major, minor, patch = version_tuple
  • Slicing: version_tuple[:2] for major.minor
  • Comparison: Direct comparison with other tuples

Return Value

Returns: A tuple representing the version

Stable version example: (7, 4, 3) for pytest 7.4.3

Pre-release example: (7, 0, '0rc1') for pytest 7.0.0rc1

pytest.approx

Full Description

The pytest.approx function provides a convenient way to compare floating-point numbers with a tolerance, avoiding issues with floating-point precision. It's especially useful when testing calculations involving money, percentages, or any mathematical operations that may produce small rounding errors. In Django applications, it's commonly used to validate salary calculations, price computations, statistical aggregations, and financial data where exact floating-point comparison would be inappropriate.

Syntax

pytest.approx(expected, rel=None, abs=None, nan_ok=False)

Parameters Description

  • expected (float | Sequence[float] | Mapping[str, float]): The expected value or collection of values to compare against.
  • rel (float | None): Relative tolerance (as a fraction). Default is 1e-6 (0.0001%).
  • abs (float | None): Absolute tolerance. Default is 1e-12.
  • nan_ok (bool): If True, NaN values are considered equal. Default is False.

Return Value

Returns: An ApproxScalar, ApproxSequencelike, or ApproxMapping object that can be compared with actual values using standard comparison operators.

        dlon = math.radians(lon2 - lon1)

             math.cos(math.radians(lat1)) * math.cos(math.radians(lat2)) * 
             math.sin(dlon/2) ** 2)
        c = 2 * math.asin(math.sqrt(a))
        
        return R * c

pytest.fail

Full Description

The pytest.fail function explicitly fails a test with a given message. It's useful when you need to fail a test based on complex conditions that can't be easily expressed with simple assertions, or when you want to provide detailed failure messages. In Django applications, it's commonly used for conditional test failures, validation of business logic constraints, and providing detailed error context when tests encounter unexpected states.

Syntax

pytest.fail(msg='', pytrace=True)

Parameters Description

  • msg (str): The failure message to display. Default is empty string.
  • pytrace (bool): Whether to show Python traceback. Default is True.

Return Value

Returns: NoReturn - This function raises a Failed exception and never returns.

session: Session - Pytest session

path: Path - Filesystem path

nodeid: str - ::-separated collection address

keywords: Keywords/markers dictionary

Key Methods

iter_parents(): Iterate up to root

listchain(): List all parents down to self

add_marker(marker): Add marker dynamically

iter_markers(name): Iterate over markers

get_closest_marker(name): Get first matching marker

getparent(cls): Get closest parent of type

Example: Working with Node in fixtures
import pytest
from jobpost.models import JobPost

def node_info(request):

@pytest.mark.django_db

Collector

Full Description

The Collector class is the base class for all collector nodes (directories, modules, classes). Collectors create children through their collect() method.

Key Methods

collect(): Abstract method - return children (items and collectors)

repr_failure(excinfo): Return failure representation

Item

Full Description

The Item class is the base class for all test items (leaf nodes in the collection tree). Each Item represents a single test invocation.

Key Properties and Methods

user_properties: list[tuple] - User-defined properties

runtest(): Abstract - run the test

add_report_section(when, key, content): Add report section

reportinfo(): Get (path, lineno, name) for reports

location: tuple - (relfspath, lineno, testname)

Session

Full Description

The Session class is the root collector of the collection tree. It collects initial paths and manages the overall test session.

Key Properties and Methods

startpath: Path - Path pytest was invoked from

items: List of collected test items

testscollected: Number of collected tests

isinitpath(path): Check if path was given on command line

perform_collect(args, genitems): Perform collection phase

Example: Using Session in hooks
import pytest

def pytest_collection_finish(session):
    """Called after collection is complete"""
    
    print(f"\\nCollection Summary:")
    print(f"  Start path: {session.startpath}")
    print(f"  Tests collected: {session.testscollected}")
    print(f"  Items: {len(session.items)}")
    
    # Group tests by module
    modules = {}
    for item in session.items:
        module = item.module.__name__
        modules.setdefault(module, []).append(item.name)
    
    print("\\nTests by module:")
    for module, tests in modules.items():
        print(f"  {module}: {len(tests)} tests")

# Output:
# Collection Summary:
#   Start path: /home/user/myproject
#   Tests collected: 25
#   Items: 25
#
# Tests by module:
#   test_jobpost: 10 tests
#   test_skills: 8 tests
#   test_location: 7 tests

🌐 Global Variables

Global variables can be defined in test modules or conftest.py files to customize test collection and apply markers globally. These variables are processed by pytest during collection.

collect_ignore

Full Description

The collect_ignore variable is a list of paths (strings or Path objects) that should be excluded from test collection. Define it in conftest.py to skip specific files or directories.

Example: Ignoring files in conftest.py
# conftest.py
import sys

# Ignore specific files
collect_ignore = [
    "setup.py",
    "conftest_backup.py",
    "tests/legacy/",
]

# Conditionally ignore based on Python version
if sys.version_info < (3, 10):
    collect_ignore.append("tests/test_py310_features.py")

# Ignore slow tests in CI
import os
if os.environ.get('CI'):
    collect_ignore.append("tests/test_slow_integration.py")

# Output when running pytest:
# collected 45 items (2 skipped due to collect_ignore)

collect_ignore_glob

Full Description

The collect_ignore_glob variable uses glob patterns to exclude files from collection. Useful for excluding files by pattern rather than exact path.

Example: Using glob patterns
# conftest.py

# Ignore files matching patterns
collect_ignore_glob = [
    "*_ignore.py",      # Any file ending with _ignore.py
    "test_*_slow.py",   # Slow test files
    "**/deprecated/*.py", # All files in deprecated folders
    "*.bak",            # Backup files
]

# Output:
# Files ignored: test_api_ignore.py, test_db_slow.py, deprecated/test_old.py

pytest_plugins

Full Description

The pytest_plugins variable registers plugins to be loaded. Can be a string or sequence of strings specifying plugin module names.

Example: Registering plugins
# conftest.py

# Single plugin
pytest_plugins = "myapp.testing.plugin"

# Multiple plugins
pytest_plugins = [
    "pytest_django",
    "pytest_cov",
    "myapp.testing.fixtures",
    "myapp.testing.marks",
]

# In test module - load specific fixtures
pytest_plugins = ["tests.fixtures.jobpost_fixtures"]

# This loads custom fixtures from the specified module

Environment variables control pytest's behavior without modifying configuration files. They're especially useful in CI/CD pipelines and for temporary configuration changes.

CI / BUILD_NUMBER

Full Description

When CI or BUILD_NUMBER is set to any non-empty value, pytest acknowledges it's running in a CI environment and adjusts behavior (e.g., disables output truncation).

Example: CI-aware testing
# Set in CI pipeline
# CI=true pytest tests/

# conftest.py - CI-specific configuration
import os
import pytest

def pytest_configure(config):
    """Configure based on CI environment"""
    
    if os.environ.get('CI'):
        print("\\n🔄 Running in CI mode")
        # Adjust settings for CI
        config.option.verbose = 2
        config.option.tb = 'short'
    else:
        print("\\n💻 Running locally")

@pytest.fixture
def ci_mode():
    """Fixture that indicates CI mode"""
    return bool(os.environ.get('CI') or os.environ.get('BUILD_NUMBER'))

def test_with_ci_awareness(ci_mode):
    if ci_mode:
        print("Running in CI - using mock services")
    else:
        print("Running locally - using real services")

PYTEST_VERSION

Full Description

The PYTEST_VERSION variable is set by pytest during the test session. It contains the pytest version string and can be used to detect if code is running within pytest.

Example: Detecting pytest execution
import os

def get_db_connection():
    """Get database connection - use test DB if in pytest"""

        # Running in pytest - use test database
        print(f"Pytest {os.environ['PYTEST_VERSION']} detected")
        return get_test_database()
    else:
        # Normal execution - use production database
        return get_production_database()

# In application code
def some_function():
    if 'PYTEST_VERSION' in os.environ:
        # Skip certain operations during testing
        print("Skipping analytics in test mode")
        return
    
    send_analytics_event()

⚙️ Configuration Options

Configuration options can be set in pytest.ini, pyproject.toml, tox.ini, or setup.cfg. These options control test collection, execution, and reporting behavior.

addopts

Full Description

The addopts option adds command-line arguments that are always used when running pytest.

Example: pytest.ini configuration
# pytest.ini
[pytest]
addopts = -v --tb=short --strict-markers -ra

# pyproject.toml
[tool.pytest.ini_options]
addopts = [
    "-v",
    "--tb=short",
    "--strict-markers",
    "-ra",
    "--cov=jobpost",
    "--cov-report=term-missing",
]

# Now "pytest" automatically includes these options
# Equivalent to: pytest -v --tb=short --strict-markers -ra

cache_dir

Full Description

The cache_dir option sets the directory for pytest's cache (used by --lf, --ff, etc.). Default is .pytest_cache.

Example: Custom cache directory
# pytest.ini
[pytest]
cache_dir = .cache/pytest

# pyproject.toml
[tool.pytest.ini_options]
cache_dir = ".cache/pytest"

# Useful for CI - shared cache location
# cache_dir = /tmp/pytest_cache

markers

Full Description

The markers option registers custom markers. With --strict-markers, only registered markers are allowed.

Example: Registering markers
# pytest.ini
[pytest]
markers =
    slow: marks tests as slow (deselect with '-m "not slow"')
    integration: marks tests as integration tests
    unit: marks tests as unit tests
    smoke: marks tests for smoke testing
    database: marks tests that require database

# pyproject.toml
[tool.pytest.ini_options]
markers = [
    "slow: marks tests as slow",
    "integration: integration tests",
    "smoke: smoke tests",
]

# Usage

def test_large_dataset():
    pass

# pytest -m smoke

Full Description

The testpaths option specifies directories to search for tests when no paths are given on the command line.

python_files

Full Description

The python_files option specifies file patterns for test module discovery. Default is test_*.py *_test.py.

💻 Command-Line Flags

Command-line flags control pytest's behavior at runtime. These options modify test selection, output formatting, debugging, and execution flow.

-k (expression)

Full Description

The -k flag filters tests by name using substring expressions. Supports and, or, not operators.

Examples: Test selection with -k
# Run tests containing "jobpost"
pytest -k "jobpost"

# Run tests containing "create" OR "update"
pytest -k "create or update"

# Run tests containing "test" but NOT "slow"
pytest -k "test and not slow"

# Complex expression
pytest -k "(create or update) and not integration"

# Match parametrized test IDs
pytest -k "test_skill[Python]"

-m (markers)

Full Description

The -m flag selects tests by marker expressions. Only tests with matching markers run.

Examples: Marker selection
# Run slow tests
pytest -m slow

# Run tests that are NOT slow
pytest -m "not slow"

# Run database tests that aren't integration tests
pytest -m "database and not integration"

# Run smoke OR unit tests
pytest -m "smoke or unit"

-v (verbose)

Full Description

The -v flag increases output verbosity. Can be used multiple times (-vv, -vvv) for more detail.

Examples: Verbosity levels
# Normal verbosity
pytest tests/

# Verbose - show test names

# Very verbose - show more details
pytest -vv tests/

# Quiet mode - minimal output
pytest -q tests/

Full Description

The -x or --exitfirst flag stops after the first test failure. Useful for debugging.

Example: Exit on first failure
# Stop at first failure
pytest -x tests/

pytest --maxfail=3 tests/

Full Description

⚠️ Exceptions & Warnings

Pytest exceptions and warnings indicate problems during test collection, execution, or configuration. Understanding these helps in debugging test failures and writing robust test suites.

UsageError

Full Description

The UsageError exception is raised when pytest is invoked incorrectly or with invalid options.

Common causes and solutions
# UsageError: unrecognized argument: --invalid
# Solution: Check spelling of command-line options

# UsageError: file not found: tests/nonexistent.py
# Solution: Verify file path exists

# UsageError: 'str' object is not callable
# Solution: Check pytest.ini syntax

# Catching in plugins
import pytest

def pytest_configure(config):
    try:
        config.getoption("--nonexistent")
    except pytest.UsageError:
        print("Option not available")

FixtureLookupError

Full Description

The FixtureLookupError exception is raised when a requested fixture cannot be found.

Common causes and solutions
# E: fixture 'db' not found
# Solution: Install pytest-django or define the fixture

# E: fixture 'my_fixture' not found
# Solution: Check conftest.py location and fixture name spelling

# Example of catching
def test_with_optional_fixture(request):
    try:
        value = request.getfixturevalue('optional_fixture')
    except pytest.FixtureLookupError:
        value = "default_value"
    
    assert value is not None
