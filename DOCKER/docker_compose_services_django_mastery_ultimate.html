Overview

Docs: Compose Services Reference

Services configuration defines containers that make up your application.

Full description

A service definition contains configuration that is applied to each container started for that service. Docker Compose services allow you to define how containers should run, including their resource constraints, networking, volumes, and dependencies. For Django applications like our JobPost management system, services typically include web servers, databases, caches, task queues, and monitoring tools working together as a cohesive application stack.

Syntax

services:
  service_name:
    attribute: value
    another_attribute:
      - list_item
      - another_item

Parameters description

  • services: Top-level key containing all service definitions
  • service_name: Unique name for the service within the compose file
  • attributes: Configuration options that define how the service runs
  • values: Can be strings, numbers, lists, or objects depending on the attribute
  • inheritance: Services can extend other services or use external configurations

Return value

Defines containerized services that Docker Compose manages as a single application stack. Returns a complete application environment with networking, storage, and service dependencies.

Example 1: Basic service definition

services:
  jobpost-web:
    image: python:3.11-slim
    command: python manage.py runserver 0.0.0.0:8000
    ports:
      - "8000:8000"
    environment:
      - DEBUG=True
# Basic Django web service # Uses Python 3.11 slim image # Runs development server on port 8000 # Sets debug mode for development

Example 2: Service with database dependency

services:
  jobpost-web:
    build: .
    depends_on:
      - db
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/jobpost_db
  
  db:
    image: postgres:15
    environment:
      - POSTGRES_DB=jobpost_db
      - POSTGRES_USER=user
      - POSTGRES_PASSWORD=pass
# Web service depends on database # PostgreSQL database for JobPost models # Environment variables configure connections # Compose manages service startup order

annotations

Docs: annotations attribute

Adds metadata to containers.

Full description

Annotations add metadata to containers as key-value pairs. This metadata can be used by orchestration tools, monitoring systems, or custom tooling to make decisions about container management. For Django applications, annotations are useful for tagging services with environment information, team ownership, or deployment metadata that helps with JobPost application management.

Syntax

services:
  service_name:
    annotations:
      key1: "value1"
      key2: "value2"
      com.company.team: "backend"

Parameters description

  • key-value pairs: Arbitrary metadata as string key-value pairs
  • namespacing: Recommended to use reverse DNS notation for organization
  • string values: All values must be strings, numbers should be quoted
  • case sensitive: Keys are case-sensitive
  • tooling integration: Used by external tools for container management decisions

Return value

Applies metadata annotations to the container that can be read by orchestration tools and monitoring systems.

Example 1: Team and project annotations

services:
  jobpost-web:
    image: jobpost:latest
    annotations:
      com.company.team: "platform-engineering"
      com.company.project: "jobpost"
      com.company.environment: "production"
      version: "1.5.2"
# Organizational metadata for JobPost service # Team ownership information # Project and environment identification # Version tracking for deployments

Example 2: Monitoring and alerts annotations

services:
  jobpost-web:
    image: jobpost:latest
    annotations:
      prometheus.io/scrape: "true"
      prometheus.io/port: "8000"
      prometheus.io/path: "/metrics"
      alert.oncall: "platform-team"
      backup.required: "true"
# Prometheus monitoring configuration # Metrics scraping endpoint # Alert routing information # Backup requirements specification

build

Docs: build attribute

Configuration options for building Docker images from source.

Full description

Build configuration tells Compose how to build a Docker image for the service. You can specify a simple path to a build context, or configure advanced build options including build arguments, target stages, and dockerfile paths. For Django JobPost applications, build configuration manages how your application code, dependencies, and static files are built into production-ready container images.

Syntax

# Simple form
services:
  web:
    build: .

# Extended form
services:
  web:
    build:
      context: .
      dockerfile: Dockerfile
      target: production
      args:
        - ARG_NAME=value

Parameters description

  • context: Build context directory path (required)
  • dockerfile: Alternative Dockerfile name (optional)
  • target: Multi-stage build target (optional)
  • args: Build arguments passed to Dockerfile (optional)
  • cache_from: Images to use for build cache (optional)
  • network: Network mode during build (optional)
  • shm_size: Shared memory size during build (optional)

Return value

Builds a Docker image from source code and configuration. Returns a container image that can be run as a service in the compose stack.

Example 1: Simple build from current directory

services:
  jobpost-web:
    build: .
    ports:
      - "8000:8000"
    environment:
      - DJANGO_SETTINGS_MODULE=myproject.settings.production
# Builds JobPost Django app from current directory # Uses default Dockerfile in context root # Production-ready Django configuration

Example 2: Multi-stage build with arguments

services:
  jobpost-web:
    build:
      context: .
      dockerfile: Dockerfile.prod
      target: production
      args:
        - DJANGO_VERSION=5.2
        - BUILD_ENV=production
        - PYTHON_VERSION=3.11
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/jobpost_db
# Custom Dockerfile for production builds # Multi-stage build targeting production stage # Build arguments for Django and Python versions # Configurable build environment

Example 3: Build with cache optimization

services:
  jobpost-web:
    build:
      context: .
      cache_from:
        - jobpost:latest
        - python:3.11-slim
      network: host
      shm_size: 2gb
      args:
        REQUIREMENTS_FILE: requirements/production.txt
# Cache from existing JobPost images for faster builds # Host networking during build for package downloads # Increased shared memory for build performance # Custom requirements file for production dependencies

attach

Docs: attach attribute

Controls whether Compose attaches to the service on startup.

Full description

The attach attribute controls whether Docker Compose attaches to the service's containers on startup, allowing you to see the container's output in the terminal. When set to false, the service runs in the background without showing output. This is particularly useful for Django applications where you might want to see web server logs but not database or cache service logs during development.

Syntax

services:
  service_name:
    attach: true  # or false

Parameters description

  • true: Default behavior, attach to container and show output
  • false: Run service in background without showing output
  • applies to: Controls terminal attachment during docker-compose up
  • logs access: Use docker-compose logs to view output when attach is false

Return value

Controls output visibility during compose startup. Returns attached or detached container execution mode.

Example 1: Attach to Django web service only

services:
  jobpost-web:
    build: .
    ports: ["8000:8000"]
    attach: true  # Show Django development server logs

  db:
    image: postgres:15-alpine
    attach: false  # Hide PostgreSQL logs during startup
    environment:
      - POSTGRES_DB=jobpost_db

  redis:
    image: redis:7-alpine
    attach: false  # Hide Redis logs during startup
# Only JobPost Django web service shows logs # Database and Redis run silently in background # Cleaner development console output # Use docker-compose logs db to view database logs when needed

Example 2: Background services for production

services:
  jobpost-web:
    build: .
    attach: false  # Production web service runs silently

  worker:
    build: .
    command: celery -A myproject worker
    attach: true   # Show Celery worker logs for monitoring

  db:
    image: postgres:15-alpine
    attach: false  # Database runs in background
# Production setup with selective log visibility # Only Celery worker shows output for task monitoring # Web service and database run silently # Focused monitoring on background task processing

blkio_config

Docs: blkio_config attribute

Configures block I/O constraints for the service.

Full description

Block I/O configuration allows you to set constraints on disk I/O operations for containers. This includes setting device read/write rates, weights, and device mappings. For Django applications with heavy database operations or file processing, blkio_config helps ensure fair resource allocation and prevents I/O-intensive services from starving other containers in the JobPost application stack.

Syntax

services:
  service_name:
    blkio_config:
      weight: 300
      weight_device:
        - path: /dev/sda
          weight: 400
      device_read_bps:
        - path: /dev/sdb
          rate: '12mb'
      device_write_bps:
        - path: /dev/sdb
          rate: '1024k'

Parameters description

  • weight: Block I/O weight (10-1000), higher values get more I/O time
  • weight_device: Per-device weight settings
  • device_read_bps: Read rate limit per device in bytes per second
  • device_write_bps: Write rate limit per device in bytes per second
  • device_read_iops: Read operations per second limit per device
  • device_write_iops: Write operations per second limit per device

Return value

Applies block I/O constraints to the container. Returns controlled disk I/O access that prevents resource contention.

Example 1: Database I/O prioritization

services:
  jobpost-db:
    image: postgres:15-alpine
    blkio_config:
      weight: 800  # High I/O priority for database
      device_read_bps:
        - path: /dev/sda
          rate: '100mb'  # Fast read access for JobPost queries
      device_write_bps:
        - path: /dev/sda
          rate: '50mb'   # Moderate write speed
    volumes:
      - postgres_data:/var/lib/postgresql/data

  jobpost-web:
    build: .
    blkio_config:
      weight: 300  # Lower I/O priority for web service
# PostgreSQL gets high I/O priority for JobPost data # Database optimized for fast reads (100MB/s) # Moderate write speed for transaction safety # Web service uses lower I/O priority

Example 2: File processing service limits

services:
  jobpost-file-processor:
    build: .
    command: python manage.py process_uploads
    blkio_config:
      weight: 200  # Low priority for batch processing
      device_write_bps:
        - path: /dev/sdb
          rate: '20mb'  # Limit file processing writes
      device_write_iops:
        - path: /dev/sdb
          rate: 1000    # Limit write operations per second
    volumes:
      - uploads:/app/uploads
# File processing service with I/O limits # Prevents overwhelming storage with bulk operations # Controlled write speed for JobPost file uploads # IOPS limiting for sustained performance

cpu_count

Docs: cpu_count attribute

Sets the number of usable CPUs for the container.

Full description

CPU count limits the number of CPUs available to a container, simulating a system with fewer processors. This is useful for testing application behavior under different CPU constraints or ensuring consistent performance across different deployment environments. For Django applications, cpu_count helps control resource usage for different services in the JobPost stack.

Syntax

services:
  service_name:
    cpu_count: 2

Parameters description

  • integer value: Number of CPUs to make available to the container
  • simulation: Simulates a system with the specified number of CPUs
  • scheduling: Affects CPU scheduler behavior within the container
  • testing: Useful for performance testing under CPU constraints
  • Windows/Mac: Primarily effective on Windows and macOS Docker Desktop

Return value

Limits the container to see only the specified number of CPUs. Returns controlled CPU visibility for application testing and resource management.

Example 1: Multi-tier CPU allocation

services:
  jobpost-web:
    build: .
    cpu_count: 4  # Web service gets 4 CPUs
    ports: ["8000:8000"]
    environment:
      - DJANGO_SETTINGS_MODULE=myproject.settings.production

  jobpost-worker:
    build: .
    command: celery -A myproject worker
    cpu_count: 2  # Background workers get 2 CPUs
    
  jobpost-db:
    image: postgres:15-alpine
    cpu_count: 2  # Database gets 2 CPUs
    environment:
      - POSTGRES_DB=jobpost_db
# Tiered CPU allocation for JobPost services # Web service gets most CPUs for request handling # Workers and database get moderate CPU allocation # Controlled resource distribution across stack

Example 2: Development environment limiting

services:
  jobpost-web:
    build: .
    cpu_count: 1  # Single CPU for development testing
    environment:
      - DEBUG=True
      - DATABASE_URL=postgresql://user:pass@db:5432/jobpost_db
    
  db:
    image: postgres:15-alpine
    cpu_count: 1  # Single CPU for development database
# Development environment with CPU constraints # Tests JobPost application under limited resources # Simulates deployment on smaller instances # Helps identify performance bottlenecks early

command

Docs: command attribute

Overrides the default command executed by the container.

Full description

The command attribute overrides the default command specified in the Dockerfile's CMD instruction. This allows you to run different commands for the same image in different services or environments. For Django applications, command is essential for running different aspects of the JobPost application like web servers, background workers, management commands, or development tools from the same base image.

Syntax

# String form
services:
  service_name:
    command: python manage.py runserver 0.0.0.0:8000

# List form (recommended)
services:
  service_name:
    command: ["python", "manage.py", "runserver", "0.0.0.0:8000"]

Parameters description

  • string form: Command as a single string, parsed by shell
  • list form: Command as array of strings, executed directly
  • overrides CMD: Replaces the default CMD from Dockerfile
  • working directory: Runs from container's working directory
  • environment access: Has access to all environment variables
  • signal handling: List form provides better signal handling

Return value

Executes the specified command when the container starts. Returns the process that runs as the main container process.

Example 1: Different Django services from same image

services:
  jobpost-web:
    build: .
    command: ["python", "manage.py", "runserver", "0.0.0.0:8000"]
    ports: ["8000:8000"]

  jobpost-worker:
    build: .
    command: ["celery", "-A", "myproject", "worker", "--loglevel=info"]
    depends_on: [redis]

  jobpost-beat:
    build: .
    command: ["celery", "-A", "myproject", "beat", "--loglevel=info"]
    depends_on: [redis]

  jobpost-migrate:
    build: .
    command: ["python", "manage.py", "migrate"]
    depends_on: [db]
# Same JobPost image runs different services # Web server for handling HTTP requests # Celery worker for background JobPost processing # Beat scheduler for periodic tasks # Migration service for database setup

Example 2: Development vs production commands

services:
  jobpost-web-dev:
    build: .
    command: ["python", "manage.py", "runserver", "0.0.0.0:8000"]
    environment:
      - DEBUG=True
    profiles: ["dev"]

  jobpost-web-prod:
    build: .
    command: ["gunicorn", "--bind", "0.0.0.0:8000", "--workers", "4", "myproject.wsgi:application"]
    environment:
      - DJANGO_SETTINGS_MODULE=myproject.settings.production
    profiles: ["prod"]
# Different commands for development and production # Development uses Django runserver with debug # Production uses Gunicorn with multiple workers # Profiles control which service runs in each environment

Example 3: Management and utility commands

services:
  jobpost-collectstatic:
    build: .
    command: ["python", "manage.py", "collectstatic", "--noinput"]
    volumes:
      - static_volume:/app/staticfiles
    profiles: ["deploy"]

  jobpost-shell:
    build: .
    command: ["python", "manage.py", "shell_plus"]

Docs: depends_on attribute

Full description

The depends_on attribute controls the order in which services are started and stopped. Compose starts services in dependency order, ensuring that dependent services start before services that depend on them. For Django applications, this is crucial for ensuring databases, caches, and message brokers are ready before the JobPost web application and workers attempt to connect to them.

Syntax

# Simple form - startup order only
services:
  web:
    depends_on:
      - db
      - redis

services:
  web:
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_started

Parameters description

  • simple list: Service names that must start before this service
  • extended form: Allows specifying conditions for dependency readiness
  • service_started: Waits for service container to start (default)
  • service_healthy: Waits for service to report healthy status
  • service_completed_successfully: Waits for service to complete and exit successfully
  • startup order: Controls only startup order, not readiness

Return value

Ensures proper service startup order and dependency management. Returns coordinated service orchestration that prevents connection failures.

10. restart

Docs: Restart Reference • Restart Policies

Full description

The restart attribute defines automatic restart behavior for containers when they exit or fail. It ensures service availability and resilience by automatically recovering from failures, system reboots, and unexpected shutdowns, critical for production Django applications requiring high availability.

Syntax

services:
  servicename:
    restart: no|always|on-failure|unless-stopped
    # OR with failure count limit (Docker Swarm only)

Parameters description

  • on-failure: Restart only when container exits with non-zero status
  • unless-stopped: Restart always except when explicitly stopped by user
  • on-failure:N: Restart on failure with maximum retry count (Swarm mode only)

Return value

Configures Docker daemon to automatically restart containers based on specified conditions, ensuring service continuity and reducing manual intervention for routine failures or system maintenance operations.

Full description

The build attribute configures how Docker Compose builds container images from source code using a Dockerfile. 13. healthcheck

Docs: Healthcheck Reference • Dockerfile Healthcheck

Full description

The healthcheck attribute configures container health monitoring by defining commands that check service status and availability. It enables automatic detection of unhealthy containers, facilitating orchestration decisions and ensuring Django application reliability through periodic health verification and recovery mechanisms.

Syntax

services:
  servicename:
    healthcheck:
      test: ["CMD", "command", "arg1", "arg2"]
      # OR
      test: "command arg1 arg2"
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 60s
      start_interval: 5s
    # OR disable healthcheck
    healthcheck:
      disable: true

Parameters description

  • test: Command to run for health check (array or string format)
  • interval: Time between health checks (default: 30s)
  • timeout: Maximum time allowed for health check command (default: 30s)
  • retries: Number of consecutive failures before marking unhealthy (default: 3)
  • start_period: Grace period before health checks start (default: 0s)
  • start_interval: Time between health checks during start period
  • disable: Set to true to disable inherited healthcheck

Return value

Provides container health status monitoring with automatic unhealthy container detection, enabling orchestration systems to restart failed services and maintain application availability through continuous health verification.

Full description

21. secrets

Docs: Secrets Reference • Docker Secrets

Full description

The secrets attribute grants service access to sensitive data managed by Docker secrets system. It provides secure handling of passwords, API keys, certificates, and other confidential information for Django applications without exposing secrets in environment variables or configuration files.

Syntax

services:
  servicename:
    secrets:
      - secret_name
      - source: secret_name
        target: /path/in/container
        uid: "1000"
        gid: "1000"
        mode: 0400

Parameters description

  • Short syntax: List of secret names to mount at default locations
  • Long syntax: Detailed secret mounting configuration
    • source: Name of the secret as defined in top-level secrets section
    • target: Path where secret is mounted in container (default: /run/secrets/<secret_name>)
    • uid/gid: User and group ownership of secret file
    • mode: File permissions for secret (octal format)

Return value

Mounts secrets as read-only files inside containers, enabling secure access to sensitive data for Django applications with proper file permissions and ownership controls for enhanced security.

23. deploy

Docs: Deploy Reference • Swarm Services

Full description

The deploy attribute configures deployment and runtime constraints for services in Docker Swarm mode. It defines resource limits, replica counts, placement constraints, and update strategies for Django applications, enabling production-grade orchestration with high availability and scalability.

Syntax

services:
  servicename:
    deploy:
      replicas: 3
      mode: replicated|global
      resources:
        limits: {cpus: '1.0', memory: 1G}
        reservations: {cpus: '0.5', memory: 512M}
      placement:
        constraints: [node.role == worker]
        preferences: [spread: node.labels.zone]
      update_config:
        parallelism: 2
        delay: 10s
      restart_policy:
        condition: on-failure

Parameters description

  • replicas: Number of service replicas to run
  • mode: replicated (specific count) or global (one per node)
  • resources: CPU and memory limits/reservations
  • placement: Node selection constraints and preferences
  • update_config: Rolling update configuration
  • restart_policy: Container restart behavior in Swarm
  • labels: Service-level labels for discovery

Return value

Configures Docker Swarm deployment characteristics for production Django applications with high availability, resource management, and automated scaling capabilities across multiple nodes.

26. extends

Docs: Extends Reference • Multiple Compose Files

Full description

The extends attribute enables service inheritance from other services, promoting configuration reuse and reducing duplication in Django application deployments. It supports service template patterns, environment-specific customization, and modular architecture through composition.

Syntax

services:
  servicename:
    extends:
      file: common.yml
      service: base-service
    # Additional or override configuration

Parameters description

  • file: Path to Compose file containing the base service (optional if same file)
  • service: Name of the service to extend from
  • Inheritance rules: All base service configuration is inherited
  • Override capability: Extending service can override any inherited settings
  • Exclusions: Some attributes like depends_on, links are not inherited

Return value

Creates a new service that inherits all configuration from a base service, with the ability to override or extend specific settings for Django application customization and environment-specific deployment.

27. extra_hosts

Docs: Extra Hosts Reference • Managing /etc/hosts

Full description

The extra_hosts attribute adds custom hostname-to-IP mappings to the container's /etc/hosts file, enabling Django applications to resolve custom hostnames for external services, development environments, and legacy system integration without DNS configuration.

Syntax

services:
  servicename:
    extra_hosts:
      - "hostname:IP"
      - "example.com:192.168.1.100"

Parameters description

  • hostname:IP format: Custom hostname mapped to specific IP address
  • Multiple entries: List of hostname-IP mappings for various services
  • Local resolution: Entries added to container's /etc/hosts file
  • Override DNS: Takes precedence over DNS resolution for specified hostnames
  • Development use: Common for local development and testing environments

Return value

Configures custom hostname resolution within containers, enabling Django applications to connect to external services, development tools, and legacy systems using predefined hostname-to-IP mappings.

Examples

28. dns

Docs: DNS Reference • Network Settings

Full description

The dns attribute configures custom DNS servers for container name resolution, enabling Django applications to use specific DNS providers, corporate DNS infrastructure, or specialized DNS services for enhanced security, performance, and network policy compliance.

Syntax

services:
  servicename:
    dns:
      - 8.8.8.8
      - 8.8.4.4
    # OR single DNS server
    dns: 1.1.1.1

Parameters description

  • IP addresses: List of DNS server IP addresses for name resolution
  • Order significance: DNS servers queried in specified order
  • Override default: Replaces system default DNS configuration
  • Corporate DNS: Enables use of internal/corporate DNS servers
  • Security DNS: Supports DNS filtering and security services

Return value

Configures container DNS resolution to use specified DNS servers, enabling Django applications to resolve hostnames through custom DNS infrastructure for security, performance, and network policy requirements.
