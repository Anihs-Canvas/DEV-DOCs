1) Overview

Docs: Dockerfile Overview

Docker can build images automatically by reading instructions from a Dockerfile.

Full description

A Dockerfile is a text document that contains all the commands a user could call on the command line to assemble an image. Using docker build users can create an automated build that executes several command-line instructions in succession. The docker build command builds an image from a Dockerfile and a context. For Django applications like our JobPost management system, Dockerfiles define the complete runtime environment including Python version, dependencies, application code, and configuration.

Syntax

# Comment
INSTRUCTION arguments

Parameters description

  • Build context: Set of files located in the specified PATH or URL
  • Instructions: Define the steps to build the image
  • Layers: Each instruction creates a new layer in the image
  • Caching: Docker caches layers to speed up subsequent builds
  • Security: Dockerfile should follow security best practices

Return value

Docker build process creates a container image that can be run to execute your Django JobPost application.

2) Format

Docs: Format

The format of a Dockerfile follows specific rules for instructions, comments, and line continuations.

Full description

The format of a Dockerfile is: instruction arguments. Instructions are not case-sensitive, but convention is to use UPPERCASE to distinguish them from arguments. Docker runs instructions in a Dockerfile in order. A Dockerfile must begin with a FROM instruction, which specifies the base image from which you are building. For Django applications, proper formatting ensures reliable builds and maintainable Dockerfiles.

Syntax

# Comment
INSTRUCTION arguments
INSTRUCTION \
  argument1 \
  argument2

Parameters description

  • Comments: Lines beginning with # are treated as comments
  • Line continuation: Backslash (\) at end of line continues instruction
  • Whitespace: Leading and trailing whitespace is ignored
  • Case insensitive: Instructions are case insensitive but UPPERCASE is convention
  • Order matters: Instructions are executed in order from top to bottom

Return value

Properly formatted Dockerfile enables successful image builds with predictable layer creation.

5 short examples with Django formatting

# Version: 1.0.0

Parser Directives

Docs: Parser Directives

Optional directives that affect how subsequent lines in a Dockerfile are handled. These must appear at the top of the Dockerfile.

Full description

Parser directives are optional, and affect the way in which subsequent lines in a Dockerfile are handled. Parser directives don't add layers to the build, and don't show up as build steps. Parser directives are written as a special type of comment in the form # directive=value. A single directive may only be used once.

Syntax

# directive=value
# syntax=docker/dockerfile:1


# escape=\ or 
# escape=`


# check=skip=checks|all
# check=error=boolean
# check=skip=JSONArgsRecommended,StageNameCasing

Parameters description

  • syntax: Declares the Dockerfile syntax version to use for the build
  • escape: Sets the character used to escape characters in a Dockerfile
  • check: Configures how build checks are evaluated
  • Position: Must be at the top of Dockerfile before any instructions
  • Case sensitivity: Directive keys are case-insensitive, values are case-sensitive

Return value

Controls Dockerfile parsing behavior without affecting the resulting image layers.

syntax

Use the syntax parser directive to declare the Dockerfile syntax version to use for the build. If unspecified, BuildKit uses a bundled version of the Dockerfile frontend.

check

The check directive is used to configure how build checks are evaluated. By default, all checks are run, and failures are treated as warnings. You can disable specific checks or treat warnings as errors.

You can disable specific checks using #check=skip=. To specify multiple checks to skip, separate them with a comma:

# check=skip=JSONArgsRecommended,StageNameCasing

To disable all checks, use #check=skip=all.

By default, builds with failing build checks exit with a zero status code despite warnings. To make the build fail on warnings, set #check=error=true.

# check=error=true

Environment Replacement

Docs: Environment Replacement

Environment variables can be used in certain instructions as variables to be interpreted by the Dockerfile. Supports bash-style variable substitution.

Full description

Environment variables (declared with the ENV statement) can also be used in certain instructions as variables to be interpreted by the Dockerfile. Escapes are also handled for including variable-like syntax into a statement literally. Environment variables are notated in the Dockerfile either with $variable_name or ${variable_name}.

Syntax

$variable_name
${variable_name}
${variable:-word}     # Default value if variable not set
${variable:+word}     # Use word if variable is set
${variable#pattern}   # Remove shortest match from start
${variable##pattern}  # Remove longest match from start
${variable%pattern}   # Remove shortest match from end
${variable%%pattern}  # Remove longest match from end

Parameters description

  • $variable_name: Simple variable substitution
  • ${variable_name}: Brace syntax for complex cases
  • :-word: Default value if variable is unset
  • :+word: Alternative value if variable is set
  • #pattern: Remove prefix pattern (shortest match)
  • ##pattern: Remove prefix pattern (longest match)
  • %pattern: Remove suffix pattern (shortest match)
  • %%pattern: Remove suffix pattern (longest match)

Return value

Substituted text values in Dockerfile instructions at build time.

Shell and Exec Form

Docs: Shell and Exec Form

RUN, CMD, and ENTRYPOINT instructions have two possible forms: shell form and exec form. Understanding the differences is crucial for Django applications.

📋 Shell and Exec Form Overview

Exec Form - JSON array format for direct execution
Shell Form - String format with shell processing
Different Shell - Custom shell configuration techniques

Full description

The RUN, CMD, and ENTRYPOINT instructions all have two possible forms: INSTRUCTION ["executable","param1","param2"] (exec form) and INSTRUCTION command param1 param2 (shell form). The exec form makes it possible to avoid shell string munging, and to invoke commands using a specific command shell, or any other executable.

Syntax

# Exec form
INSTRUCTION ["executable", "param1", "param2"]

# Shell form  
INSTRUCTION command param1 param2

Parameters description

  • Exec form: Parsed as JSON array, must use double quotes
  • Shell form: String format, automatically uses command shell
  • Variable substitution: Shell form supports variables, exec form doesn't
  • Signal handling: Exec form receives signals directly
  • PID 1: Exec form runs as PID 1, shell form runs under shell

Return value

Different process execution behavior affecting signal handling and variable substitution.

Exec Form

The exec form is parsed as a JSON array, which means that you must use double-quotes (") around words, not single-quotes ('). The exec form doesn't automatically invoke a command shell.

Full description

4) RUN

Docs: RUN instruction

Execute any commands in a new layer on top of the current image and commit the results.

📋 RUN Instruction Overview

Cache Invalidation - Optimizing layer caching strategies
RUN --device - Device access during build
RUN --mount - Advanced mount options (bind, cache, secret, ssh)
RUN --network - Network configuration options
RUN --security - Security context management

Full description

The RUN instruction will execute any commands in a new layer on top of the current image and commit the results. The resulting committed image will be used for the next step in the Dockerfile. Layering RUN instructions and generating commits conforms to the core concepts of Docker where commits are cheap and containers can be created from any point in an image's history. For Django JobPost applications, RUN is used for installing dependencies, setting up the environment, and preparing the application.

Syntax

# Shell form
RUN <command>

# Exec form
RUN ["executable", "param1", "param2"]

# Advanced options
RUN --mount=<mount-options> <command>
RUN --network=<network-mode> <command>
RUN --security=<security-mode> <command>

Parameters description

  • Shell form: Commands run in a shell (/bin/sh -c on Linux). Supports shell features like pipes, variables
  • Exec form: Commands run directly without shell processing. More reliable for signal handling
  • command: The shell command to execute
  • executable: Direct executable path in exec form
  • param1, param2: Arguments passed to the executable
  • --mount: Mount filesystems during command execution (cache, bind, secret, ssh)
  • --network: Set network mode during command execution (default, none, host)
  • --security: Set security mode (insecure, sandbox)

Return value

Creates a new image layer with the results of the command execution. Returns exit code 0 for success, non-zero for failure (stops build).

The cache for RUN instructions isn't invalidated automatically during the next build. The cache for a RUN instruction can be invalidated by using the --no-cache flag, for example docker build --no-cache. The cache for RUN instructions can be invalidated by ADD and COPY instructions.

# System packages cached until Dockerfile changes # Python dependencies cached until requirements.txt changes # Cache invalidation cascades to subsequent layers # JobPost app build optimization through layer ordering

9) ADD

Docs: ADD instruction

Copies new files, directories or remote file URLs and adds them to the filesystem of the image.

📋 ADD Instruction Overview

Source - Source file and directory handling
Destination - Target path configuration
--keep-git-dir - Preserve git metadata
--checksum - File integrity verification
--chown --chmod - Ownership and permissions
--link - Efficient linking strategy
--exclude - Pattern-based exclusions

Full description

The ADD instruction copies new files, directories or remote file URLs from <src> and adds them to the filesystem of the image at the path <dest>. Multiple <src> resources may be specified but if they are files or directories, their paths are interpreted as relative to the source of the context of the build. Each <src> may contain wildcards and matching will be done using Go's filepath.Match rules.

ADD or COPY

ADD and COPY are functionally similar. COPY supports basic copying of files into the container, from the build context or from a stage in a multi-stage build. ADD supports features for fetching files from remote HTTPS and Git URLs, and extracting tar files automatically when adding files from the build context.

Syntax

ADD [--chown=<user>:<group>] [--chmod=<perms>] [--checksum=<checksum>] <src>... <dest>
ADD [--chown=<user>:<group>] [--chmod=<perms>] ["<src>",... "<dest>"]

# With BuildKit features
ADD --link [--chown=<user>:<group>] [--chmod=<perms>] <src>... <dest>
ADD --exclude=<path> <src>... <dest>

Parameters description

  • src: Source files, directories, or URLs to copy. Supports wildcards and multiple sources
  • dest: Destination path in the container filesystem
  • --chown: Set file ownership (user:group). Only supported on Linux containers
  • --chmod: Set file permissions. Only supported on Linux containers
  • --checksum: Verify downloaded files against provided checksum
  • --link: Enable improved caching and faster builds with BuildKit
  • --exclude: Exclude files and directories matching pattern
  • Wildcards: Use Go filepath.Match rules for pattern matching
  • Auto-extraction: Automatically extracts compressed archives (tar, gzip, etc.)

Return value

Copies files and directories to the container filesystem. Returns a new image layer containing the copied content. Automatically extracts compressed archives and can fetch remote URLs.

Source

The <src> path must be inside the context of the build; you cannot ADD ../something /something, because the first step of a docker build is to send the context directory (and subdirectories) to the docker daemon.

Example: Django project files

ADD myproject/ /app/
ADD requirements.txt /app/
ADD static/*.css /app/static/css/
# Copies Django project directory # Copies requirements file # Copies CSS files matching pattern # All sources relative to build context

You can specify multiple source files or directories with ADD. The last argument must always be the destination. For example, to add two files, file1.txt and file2.txt, from the build context to /usr/src/things/ in the build container:
ADD file1.txt file2.txt /usr/src/things/

If you specify multiple source files, either directly or using a wildcard, then the destination must be a directory (must end with a slash /).

To add files from a remote location, you can specify a URL or the address of a Git repository as the source. For example:
ADD https://example.com/archive.zip /usr/src/things/
ADD git@github.com:user/repo.git /usr/src/things/

Destination

The <dest> is an absolute path, or a path relative to WORKDIR, into which the source will be copied inside the destination container. If <dest> doesn't exist, it is created along with all missing directories in its path.

Example: Django destination paths

WORKDIR /app
ADD requirements.txt .          # Copies to /app/requirements.txt
ADD static/ /app/static/       # Copies to /app/static/
ADD config/ ./config/          # Copies to /app/config/
# Destination paths relative to WORKDIR # Directories created automatically # JobPost static and config files organized

If the destination path begins with a forward slash, it's interpreted as an absolute path, and the source files are copied into the specified destination relative to the root of the current build stage.

# create /abs/test.txt
ADD test.txt /abs/
Trailing slashes are significant. For example, 
  ADD test.txt /abs creates a file at /abs, 
  whereas ADD test.txt /abs/ creates /abs/test.txt.

If the destination path doesn't begin with a leading slash, it's interpreted as relative to the working directory of the build container.

WORKDIR /usr/src/app
# create /usr/src/app/rel/test.txt
ADD test.txt rel/

If destination doesn't exist, it's created, along with all missing directories in its path.

If the source is a file, and the destination doesn't end with a trailing slash, the source file will be written to the destination path as a file.

When adding a git repository, the --keep-git-dir=true flag can be used to keep the .git directory.

Full description

The COPY instruction copies new files or directories from <src> and adds them to the filesystem of the container at the path <dest>. Multiple <src> resources may be specified but the paths of files and directories will be interpreted as relative to the source of the context of the build. COPY is preferred over ADD for simple file copying as it's more transparent and predictable.

📋 ENTRYPOINT Instruction Overview

Full description

An ENTRYPOINT allows you to configure a container that will run as an executable. Command line arguments to docker run <image> will be appended after all elements in an exec form ENTRYPOINT, and will override all elements specified using CMD. This allows arguments to be passed to the entry point.

Syntax

# Exec form (preferred)
ENTRYPOINT ["executable", "param1", "param2"]

# Shell form
ENTRYPOINT command param1 param2

Parameters description

  • Exec form: Recommended. Runs executable directly without shell processing
  • Shell form: Commands are run in a shell, preventing signal forwarding
  • executable: The command or program to execute as container entry point
  • param1, param2: Default arguments passed to the executable
  • CMD interaction: CMD provides default arguments that can be overridden
  • Runtime arguments: docker run arguments are appended to ENTRYPOINT
  • Signal handling: Exec form ensures proper signal forwarding to the process
  • Override: Use --entrypoint flag to override ENTRYPOINT at runtime

Return value

Configures the container's main process and entry point. Returns the primary executable that will run when the container starts, with proper signal handling and argument passing.

#!/bin/bash

# Colors for output

GREEN='\033[0;32m'

    until python manage.py dbshell --command="SELECT 1;" >/dev/null 2>&1; do
        echo -e "${YELLOW}⏳ Database is unavailable - sleeping${NC}"
        sleep 2
    done
    echo -e "${GREEN}✅ Database is available${NC}"
}

run_migrations() {
    echo -e "${YELLOW}🔄 Running database migrations...${NC}"
    python manage.py migrate --noinput

}

collect_static() {
    echo -e "${YELLOW}📦 Collecting static files...${NC}"
    python manage.py collectstatic --noinput --clear
    echo -e "${GREEN}✅ Static files collected${NC}"
}

# Function to create superuser
create_superuser() {

seed_demo_data() {

try:

"

health_check() {
    echo -e "${YELLOW}🏥 Running application health checks...${NC}"

        echo -e "${RED}❌ Django configuration check failed${NC}"
        exit 1
    }
    
    # Check database connectivity

        echo -e "${RED}❌ Database connectivity check failed${NC}"

}

# Main execution logic
main() {
    echo -e "${GREEN}=== JobPost Django Application Startup ===${NC}"
    echo -e "Environment: ${DJANGO_SETTINGS_MODULE:-development}"
    echo -e "Debug Mode: ${DEBUG:-False}"
    echo -e "Database: ${DATABASE_URL:-sqlite}"

    wait_for_db
    
    # Database setup
    run_migrations
    
    # Static files (only in production)

        collect_static
    fi
    
    # Create superuser if specified
    create_superuser
    
    # Seed demo data if requested
    seed_demo_data
    
    # Run health checks
    health_check
    
    echo -e "${GREEN}🎉 JobPost application startup completed successfully${NC}"
    echo -e "${GREEN}🚀 Starting application server...${NC}"
    
    # Execute the main command
    exec "$@"
}

Docs: ARG instruction

Defines a variable that users can pass at build-time to the builder with the docker build command.

📋 ARG Instruction Overview

Using Variables - Variable substitution techniques

Full description

The ARG instruction defines a variable that users can pass at build-time to the builder with the docker build command using the --build-arg <varname>=<value> flag. If a user specifies a build argument that was not defined in the Dockerfile, the build outputs a warning. A Dockerfile may include one or more ARG instructions.

Syntax

ARG <name>[=<default value>]

# Examples
ARG VERSION
ARG PYTHON_VERSION=3.11
ARG BUILDPLATFORM
ARG TARGETPLATFORM

Parameters description

  • name: The variable name that can be referenced in the Dockerfile
  • default value: Optional default value used if --build-arg is not provided
  • Build-time only: ARG variables are only available during build, not in running containers
  • Scope: ARG before FROM applies to FROM line only, ARG after FROM applies to current stage
  • Predefined ARGs: Docker provides predefined ARGs like BUILDPLATFORM, TARGETPLATFORM
  • Environment substitution: Can be used in ENV, ADD, COPY and other instructions
  • Security: ARG values may be visible in docker history, avoid secrets
  • Multi-stage: ARG scope is limited to the build stage where it's defined

Return value

Defines build-time variables that can be used in subsequent instructions. Returns variable values that are substituted during the build process but not persisted in the final image.

Default values

An ARG instruction can optionally include a default value. If an ARG instruction has a default value and if there is no value passed at build-time, the builder uses the default.

Example: Django version with default

ARG DJANGO_VERSION=5.2
ARG PYTHON_VERSION=3.11

RUN pip install Django==${DJANGO_VERSION}
# Default Django 5.2 and Python 3.11 # Can be overridden at build time: # docker build --build-arg DJANGO_VERSION=5.2 . # JobPost app with configurable versions

Scope

An ARG variable definition comes into effect from the line on which it is defined in the Dockerfile not from the argument's use on the command line or elsewhere. Variables defined in one stage are not available in subsequent stages unless re-declared.

Example: ARG scope across stages

ARG GLOBAL_VERSION=1.0.0

FROM python:3.11 AS builder
ARG GLOBAL_VERSION
ARG BUILD_DEPS=build-essential
RUN echo "Builder stage - Version: ${GLOBAL_VERSION}, Deps: ${BUILD_DEPS}"

FROM python:3.11-slim AS runtime
ARG GLOBAL_VERSION
# BUILD_DEPS not available here unless redeclared
RUN echo "Runtime stage - Version: ${GLOBAL_VERSION}"
# GLOBAL_VERSION available in both stages # BUILD_DEPS only available in builder stage # Must redeclare ARG in each stage that uses it # JobPost app multi-stage build with shared config

Example: ARG vs ENV precedence

Full description

ONBUILD adds a trigger instruction to the image that will be executed at a later time, when the image is used as the base for another build. 21) HEALTHCHECK

Docs: HEALTHCHECK instruction

Tells Docker how to test a container to check that it is still working.

Full description

The HEALTHCHECK instruction tells Docker how to test a container to check that it is still working. This can detect cases such as a web server that is stuck in an infinite loop and unable to handle new connections, even though the server process is still running. For Django JobPost applications, health checks ensure the application can serve requests, database connections work, and critical services like user authentication and job posting functionality remain available.

The health check will first run interval seconds after the container is started, and then again interval seconds after each previous check completes.
If a single run of the check takes longer than timeout seconds then the check is considered to have failed. The process performing the check is abruptly stopped with a SIGKILL.

It takes retries consecutive failures of the health check for the container to be considered unhealthy.

start period provides initialization time for containers that need time to bootstrap. Probe failure during that period will not be counted towards the maximum number of retries. However, if a health check succeeds during the start period, the container is considered started and all consecutive failures will be counted towards the maximum number of retries.

start interval is the time between health checks during the start period. This option requires Docker Engine version 25.0 or later.

There can only be one HEALTHCHECK instruction in a Dockerfile. If you list more than one then only the last HEALTHCHECK will take effect.

The command after the CMD keyword can be either a shell command (e.g. HEALTHCHECK CMD /bin/check-running) or an exec array (as with other Dockerfile commands; see e.g. ENTRYPOINT for details).

Syntax

HEALTHCHECK [OPTIONS] CMD command
HEALTHCHECK NONE  # Disable health check

Parameters description

  • --start-period=DURATION: Grace period before checks start (default: 0s)
  • --start-interval=DURATION: Time between health checks during start period (default: 5s)
  • --interval=DURATION: Time between health checks (default: 30s)
  • --timeout=DURATION: Maximum time for single check (default: 30s)
  • --retries=N: Consecutive failures needed to mark unhealthy (default: 3)
  • CMD command: Command to run for health check

Return value

Returns exit status: 0 (healthy), 1 (unhealthy), or 2 (reserved). Container marked as unhealthy after consecutive failures.

# .github/workflows/django-docker.yml
name: Django Docker CI/CD

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}/jobboard

jobs:
  test:
    runs-on: ubuntu-latest
    
    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_PASSWORD: postgres
          POSTGRES_DB: jobboard_test
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

    steps:
    - name: Checkout code
      uses: actions/checkout@v4

    - name: Set up Docker Buildx
      uses: docker/setup-buildx-action@v3

    - name: Build test image
      uses: docker/build-push-action@v5
      with:
        context: .
        target: testing
        push: false
        tags: jobboard:test
        cache-from: type=gha
        cache-to: type=gha,mode=max

    - name: Run tests in container
      run: |
        docker run --rm \
          --network ${{ job.services.postgres.network }} \
          -e DATABASE_URL=postgresql://postgres:postgres@postgres:5432/jobboard_test \
          jobboard:test \
          python -m pytest tests/ --cov=jobboard --cov-report=xml

  security-scan:
    runs-on: ubuntu-latest
    needs: test
    
    steps:
    - name: Checkout code
      uses: actions/checkout@v4

    - name: Run Trivy vulnerability scanner
      uses: aquasecurity/trivy-action@master
      with:
        image-ref: 'jobboard:test'
        format: 'sarif'
        output: 'trivy-results.sarif'

    - name: Upload Trivy scan results
      uses: github/codeql-action/upload-sarif@v2
      with:
        sarif_file: 'trivy-results.sarif'

  build-and-push:
    runs-on: ubuntu-latest
    needs: [test, security-scan]
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    
    steps:
    - name: Checkout code
      uses: actions/checkout@v4

    - name: Set up Docker Buildx
      uses: docker/setup-buildx-action@v3

    - name: Log in to Container Registry
      uses: docker/login-action@v3
      with:
        registry: ${{ env.REGISTRY }}
        username: ${{ github.actor }}
        password: ${{ secrets.GITHUB_TOKEN }}

    - name: Extract metadata
      id: meta
      uses: docker/metadata-action@v5
      with:
        images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
        tags: |
          type=ref,event=branch
          type=ref,event=pr

#!/usr/bin/env python3
import requests
import time
import json
import yaml
import os
from datetime import datetime

class HealthChecker:
    def __init__(self, config_file='health_config.yaml'):
        with open(config_file, 'r') as f:
            self.config = yaml.safe_load(f)
    
    def check_service(self, service):
        try:
            response = requests.get(
                service['url'], 
                timeout=service.get('timeout', 10)
            )
            
            if response.status_code == 200:
                return {
                    'status': 'healthy',
                    'response_time': response.elapsed.total_seconds(),
                    'timestamp': datetime.now().isoformat()
                }
            else:
                return {
                    'status': 'unhealthy',
                    'status_code': response.status_code,
                    'timestamp': datetime.now().isoformat()
                }
        except Exception as e:
            return {
                'status': 'unhealthy',
                'error': str(e),
                'timestamp': datetime.now().isoformat()
            }
    
    def run_checks(self):
        results = {}
        for service_name, service_config in self.config['services'].items():
            results[service_name] = self.check_service(service_config)
        
        return results
    
    def continuous_monitoring(self):
        while True:
            results = self.run_checks()
            
            # Log results
            print(json.dumps(results, indent=2))
            
            # Send alerts if configured
            self.send_alerts(results)
            
            time.sleep(self.config.get('interval', 60))
    
    def send_alerts(self, results):
        for service_name, result in results.items():
            if result['status'] == 'unhealthy':
                print(f"ALERT: {service_name} is unhealthy: {result}")

if __name__ == '__main__':
    checker = HealthChecker()
    checker.continuous_monitoring()
EOF

#!/usr/bin/env python3
import json
import redis
import time
from datetime import datetime
from elasticsearch import Elasticsearch

class LogAggregator:
    def __init__(self):
        self.redis_client = redis.Redis(host='redis', port=6379, db=0)
        self.es = Elasticsearch(['elasticsearch:9200'])
    
    def process_logs(self):
        while True:
            try:
                # Get logs from Redis queue
                log_data = self.redis_client.blpop(['django_logs'], timeout=5)
                
                if log_data:
                    log_entry = json.loads(log_data[1])
                    
                    # Enhance log entry
                    log_entry['@timestamp'] = datetime.now().isoformat()
                    log_entry['environment'] = 'production'
                    
                    # Index to Elasticsearch
                    self.es.index(
                        index=f"django-logs-{datetime.now().strftime('%Y-%m')}",
                        body=log_entry
                    )
                    
                    print(f"Processed log: {log_entry.get('message', 'No message')}")
                
            except Exception as e:
                print(f"Error processing logs: {e}")
                time.sleep(1)

if __name__ == '__main__':
    aggregator = LogAggregator()
    aggregator.process_logs()
EOF

RUN chmod +x log_aggregator.py

CMD ["python", "log_aggregator.py"]
