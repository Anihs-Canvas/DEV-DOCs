Overview

Docs: Compose Services Reference

Services configuration defines containers that make up your application.

Full description

A service definition contains configuration that is applied to each container started for that service. Docker Compose services allow you to define how containers should run, including their resource constraints, networking, volumes, and dependencies. For Django applications like our JobPost management system, services typically include web servers, databases, caches, task queues, and monitoring tools working together as a cohesive application stack.

Syntax

services:
  service_name:
    attribute: value
    another_attribute:
      - list_item
      - another_item

Parameters description

  • services: Top-level key containing all service definitions
  • service_name: Unique name for the service within the compose file
  • attributes: Configuration options that define how the service runs
  • values: Can be strings, numbers, lists, or objects depending on the attribute
  • inheritance: Services can extend other services or use external configurations

Return value

Defines containerized services that Docker Compose manages as a single application stack. Returns a complete application environment with networking, storage, and service dependencies.

Example 1: Basic service definition

services:
  jobpost-web:
    image: python:3.11-slim
    command: python manage.py runserver 0.0.0.0:8000
    ports:
      - "8000:8000"
    environment:
      - DEBUG=True
# Basic Django web service # Uses Python 3.11 slim image # Runs development server on port 8000 # Sets debug mode for development

Example 2: Service with database dependency

services:
  jobpost-web:
    build: .
    depends_on:
      - db
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/jobpost_db
  
  db:
    image: postgres:15
    environment:
      - POSTGRES_DB=jobpost_db
      - POSTGRES_USER=user
      - POSTGRES_PASSWORD=pass
# Web service depends on database # PostgreSQL database for JobPost models # Environment variables configure connections # Compose manages service startup order

Docs: annotations attribute

Adds metadata to containers.

Full description

Annotations add metadata to containers as key-value pairs. This metadata can be used by orchestration tools, monitoring systems, or custom tooling to make decisions about container management. For Django applications, annotations are useful for tagging services with environment information, team ownership, or deployment metadata that helps with JobPost application management.

Syntax

services:
  service_name:
    annotations:
      key1: "value1"
      key2: "value2"
      com.company.team: "backend"

Parameters description

  • key-value pairs: Arbitrary metadata as string key-value pairs
  • namespacing: Recommended to use reverse DNS notation for organization
  • string values: All values must be strings, numbers should be quoted
  • case sensitive: Keys are case-sensitive
  • tooling integration: Used by external tools for container management decisions

Return value

Applies metadata annotations to the container that can be read by orchestration tools and monitoring systems.

Example 1: Team and project annotations

services:
  jobpost-web:
    image: jobpost:latest
    annotations:
      com.company.team: "platform-engineering"
      com.company.project: "jobpost"
      com.company.environment: "production"
      version: "1.5.2"
# Organizational metadata for JobPost service # Team ownership information # Project and environment identification # Version tracking for deployments

Example 2: Monitoring and alerts annotations

services:
  jobpost-web:
    image: jobpost:latest
    annotations:
      prometheus.io/scrape: "true"

12. build

Docs: Build Reference • Build Configuration

Full description

The build attribute configures how Docker Compose builds container images from source code using a Dockerfile. It supports build context, Dockerfile location, build arguments, target stages, and cache options, enabling customized image creation for Django applications with specific build requirements and optimization strategies.

Syntax

services:
  servicename:
    build: ./path  # Simple context path
    # OR advanced configuration
    build:
      context: ./path
      dockerfile: Dockerfile.prod
      args:
        - ARG_NAME=value
      target: production
      cache_from:
        - myregistry/myapp:latest
      cache_to:
        - myregistry/myapp:cache
      labels:
        - "com.example.description=My app"
      network: host
      shm_size: 2gb

Parameters description

  • context: Build context directory path (required)
  • dockerfile: Path to Dockerfile relative to context (default: Dockerfile)
  • args: Build-time variables passed to Dockerfile ARG instructions
  • target: Target stage in multi-stage Dockerfile build
  • cache_from: Images to use for build cache (BuildKit)
  • cache_to: Cache export destination (BuildKit)
  • labels: Metadata labels to add to built image
  • network: Network mode for RUN instructions during build
  • shm_size: Shared memory size for build containers

Return value

Builds and tags a Docker image from source code, making it available for service container creation with customized build configuration, optimization settings, and metadata appropriate for Django application deployment requirements.

Examples

13. healthcheck

Docs: Healthcheck Reference • Dockerfile Healthcheck

Full description

The healthcheck attribute configures container health monitoring by defining commands that check service status and availability. It enables automatic detection of unhealthy containers, facilitating orchestration decisions and ensuring Django application reliability through periodic health verification and recovery mechanisms.

Syntax

services:
  servicename:
    healthcheck:
      test: ["CMD", "command", "arg1", "arg2"]
      # OR
      test: "command arg1 arg2"
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 60s
      start_interval: 5s
    # OR disable healthcheck
    healthcheck:
      disable: true

Parameters description

  • test: Command to run for health check (array or string format)
  • interval: Time between health checks (default: 30s)
  • timeout: Maximum time allowed for health check command (default: 30s)
  • retries: Number of consecutive failures before marking unhealthy (default: 3)
  • start_period: Grace period before health checks start (default: 0s)
  • start_interval: Time between health checks during start period
  • disable: Set to true to disable inherited healthcheck

Return value

Provides container health status monitoring with automatic unhealthy container detection, enabling orchestration systems to restart failed services and maintain application availability through continuous health verification.

Examples

23. deploy

Docs: Deploy Reference • Swarm Services

Full description

The deploy attribute configures deployment and runtime constraints for services in Docker Swarm mode. It defines resource limits, replica counts, placement constraints, and update strategies for Django applications, enabling production-grade orchestration with high availability and scalability.

Syntax

services:
  servicename:
    deploy:
      replicas: 3
      mode: replicated|global
      resources:
        limits: {cpus: '1.0', memory: 1G}
        reservations: {cpus: '0.5', memory: 512M}
      placement:
        constraints: [node.role == worker]
        preferences: [spread: node.labels.zone]
      update_config:
        parallelism: 2
        delay: 10s
      restart_policy:
        condition: on-failure

Parameters description

  • replicas: Number of service replicas to run
  • mode: replicated (specific count) or global (one per node)
  • resources: CPU and memory limits/reservations
  • placement: Node selection constraints and preferences
  • update_config: Rolling update configuration
  • restart_policy: Container restart behavior in Swarm
  • labels: Service-level labels for discovery

Return value

Configures Docker Swarm deployment characteristics for production Django applications with high availability, resource management, and automated scaling capabilities across multiple nodes.

25. profiles

Docs: Profiles Reference • Compose Profiles

Full description

The profiles attribute enables selective service activation based on named profiles, allowing conditional deployment of Django application components. It provides environment-specific service management, feature toggling, and modular architecture deployment without maintaining separate Compose files.

Syntax

services:
  servicename:
    profiles:
      - profile1
      - profile2
    # Service only starts when profile is active

Parameters description

  • Profile names: List of profile identifiers that activate this service
  • Activation: Services start only when their profile is explicitly enabled
  • Command usage: docker compose --profile profile1 up
  • Multiple profiles: Service starts if any assigned profile is active
  • No profile: Services without profiles always start by default

Return value

Enables conditional service deployment based on active profiles, providing flexible environment management and feature toggling for Django applications across development, testing, and production scenarios.

Examples

26. extends

Docs: Extends Reference • Multiple Compose Files

Full description

The extends attribute enables service inheritance from other services, promoting configuration reuse and reducing duplication in Django application deployments. It supports service template patterns, environment-specific customization, and modular architecture through composition.

Syntax

services:
  servicename:
    extends:
      file: common.yml
      service: base-service
    # Additional or override configuration

Parameters description

  • file: Path to Compose file containing the base service (optional if same file)
  • service: Name of the service to extend from
  • Inheritance rules: All base service configuration is inherited
  • Override capability: Extending service can override any inherited settings
  • Exclusions: Some attributes like depends_on, links are not inherited

Return value

Creates a new service that inherits all configuration from a base service, with the ability to override or extend specific settings for Django application customization and environment-specific deployment.

Examples
