docker build

Official Docs: docker build | Docker CLI: docker build

Build Docker images from a Dockerfile and a "context" - a set of files at a specified location.

Full Description

The docker build command constructs Docker images by reading instructions from a Dockerfile. For your anihpj Django project, this command processes your Dockerfile to create container images that include your Django application code, Python dependencies from requirements.txt, static files, database migrations, and all necessary runtime configurations. The build process creates layered images where each Dockerfile instruction creates a new layer, enabling efficient caching and faster subsequent builds.

When building your anihpj Django application, Docker build reads the build context (typically your project directory), copies relevant files into the container, installs Python packages, configures Django settings, runs collectstatic for production deployments, and creates the final image. The resulting image can be used to run containers that serve your jobpost app APIs, handle user authentication, manage author profiles, process location-based queries, and execute JobPost operations.

Syntax

# Basic build from current directory
docker build .

# Build with custom tag
docker build -t {image-name}:{tag} .

# Build with Dockerfile in different location
docker build -f {dockerfile-path} {context-path}

# Build with build arguments
docker build --build-arg {ARG_NAME}={value} -t {image-name} .

# Build with target stage (multi-stage builds)
docker build --target {stage-name} -t {image-name} .

# Build with custom platform
docker build --platform {platform} -t {image-name} .

# Build with progress output
docker build --progress=plain -t {image-name} .

# Build without cache
docker build --no-cache -t {image-name} .

# Build with cache from another image
docker build --cache-from {image-name} -t {new-image} .

Parameters Description

  • context-path: The build context path (usually current directory .). Contains Dockerfile, source code, and files needed for the build. For Django projects, this includes manage.py, apps, requirements.txt, static files, and templates.
  • -t, --tag: Name and optionally tag for the image in format name:tag. Use descriptive tags like anihpj/jobpost-app:v1.0.0 or anihpj/django-dev:latest for different environments and versions.
  • -f, --file: Path to the Dockerfile (default is {context}/Dockerfile). Useful when you have multiple Dockerfiles like Dockerfile.dev, Dockerfile.prod, or Dockerfile.test for different Django deployment scenarios.
  • --build-arg: Set build-time variables that can be used in Dockerfile with ARG instruction. Common Django args include DJANGO_SETTINGS_MODULE, DEBUG, SECRET_KEY, DATABASE_URL for environment-specific builds.
  • --target: Set target build stage for multi-stage builds. Useful for Django apps with separate development, testing, and production stages in a single Dockerfile.
  • --platform: Set platform if server is multi-platform capable. Specify platforms like linux/amd64, linux/arm64 for Django deployments across different architectures.
  • --progress: Set type of progress output (auto, plain, tty). Use 'plain' for CI/CD pipelines building Django images to get detailed build logs.
  • --no-cache: Do not use cache when building the image. Forces complete rebuild of Django application, useful when you want to ensure latest dependencies or clean build environment.
  • --cache-from: Images to consider as cache sources. Speeds up Django image builds by reusing layers from previous builds or base images.
  • --pull: Always attempt to pull newer version of base image. Ensures your Django image uses latest security updates from base Python images.
  • --quiet, -q: Suppress build output and print image ID on success. Useful for automated Django deployment scripts.
  • --rm: Remove intermediate containers after successful build (default true). Keeps Docker environment clean during Django image builds.
  • --force-rm: Always remove intermediate containers even if build fails. Prevents accumulation of failed Django build containers.

Attributes

  • Image ID: Unique identifier for the built image (SHA256 hash). Used to reference the specific Django image version.
  • Build Context Size: Size of files sent to Docker daemon. Monitor this for Django projects to optimize build performance.
  • Total Build Time: Time taken for complete build process. Track this metric for Django application build optimization.
  • Layer Count: Number of layers in the final image. Each Dockerfile instruction creates a layer affecting Django image size and startup time.
  • Cache Status: Indicates which layers were cached vs rebuilt. Helps optimize Django build times through effective layer caching.

Methods

Build with Build Arguments

Syntax: docker build --build-arg {ARG}={VALUE} -t {image} {context}

Parameters: ARG (string) - Build argument name defined in Dockerfile with ARG instruction, VALUE (string) - Value to pass for the argument

Pass build-time variables to customize Django application builds for different environments, database configurations, or feature flags.

Multi-stage Build Target

Syntax: docker build --target {stage} -t {image} {context}

Parameters: stage (string) - Name of the build stage from Dockerfile FROM instruction with AS alias

Build specific stages of multi-stage Dockerfiles, useful for creating separate Django development, testing, and production images from single Dockerfile.

Cross-platform Build

Syntax: docker build --platform {platform} -t {image} {context}

Parameters: platform (string) - Target platform in format os/architecture like linux/amd64, linux/arm64

Build Django images for specific platforms, essential for deploying to different cloud providers or ARM-based servers.

Return Value

Returns the image ID of successfully built Docker image. On failure, returns non-zero exit code with error details. Build output includes layer information, cache status, and final image summary.

  • Success (0): Image built successfully, returns image ID
  • Error (1): Build failed due to Dockerfile syntax errors, missing files, or build context issues
  • Error (125): Docker daemon error or invalid command usage

Official Docs: docker builder | Docker CLI: docker builder

Manage the builder environment for Docker builds, including cache management and builder instance configuration.

Full Description

The docker builder command manages Docker build environments, providing advanced build cache management and builder instance configuration. For your anihpj Django project, this command helps optimize build performance by managing build cache, configuring build environments, and cleaning up unnecessary build artifacts. The builder system enables faster iterative development by reusing cached layers from previous Django application builds.

Builder management is crucial for Django development workflows where you frequently rebuild images during development cycles. Proper cache management ensures that your jobpost app builds leverage cached layers for unchanged dependencies, static files, and Django setup commands, significantly reducing build times for Skills, User, Author, Location, and JobPost service containers.

Syntax

# Show builder disk usage
docker builder prune

# Remove build cache (interactive)
docker builder prune

# Remove all unused build cache without prompt
docker builder prune -f

# Remove build cache with specific filters
docker builder prune --filter until=24h

# Show builder storage usage
docker system df

# Remove build cache older than specified time
docker builder prune --filter until=72h --force

Parameters Description

  • prune: Remove unused build cache to free up disk space. Essential for Django development environments where frequent builds can accumulate significant cache data from dependency installations, static file collection, and application layer builds.
  • -f, --force: Do not prompt for confirmation before removing cache. Useful in automated Django CI/CD pipelines where interactive prompts would halt the build process.
  • --filter: Provide filter values to selectively remove cache entries. Common filters include 'until' for time-based cleanup, 'label' for labeled cache entries, and 'type' for specific cache types in Django build workflows.
  • --all, -a: Remove all unused cache, not just dangling cache. More aggressive cleanup that removes all build cache including partially used cache from Django application layers.
  • --keep-storage: Amount of disk space to keep for cache. Specify in bytes, KB, MB, or GB to maintain optimal cache size for Django project builds without exhausting disk space.

Attributes

  • Cache Size: Total disk space consumed by build cache. Monitor this to optimize Django build performance while managing storage resources.
  • Reclaimable Space: Amount of space that can be freed by pruning unused cache. Important metric for maintaining efficient Django development environments.
  • Cache Entries: Number of cached build layers stored. Each Django build instruction creates cache entries that can be reused in subsequent builds.
  • Last Used: Timestamp when cache entries were last accessed. Helps identify stale cache from Django builds that can be safely removed.

Methods

Prune with Time Filter

Syntax: docker builder prune --filter until={time}

Parameters: time (duration) - Time duration like 24h, 7d, 1w for removing cache older than specified time

Remove Django build cache older than specified duration to maintain recent cache while freeing space from outdated dependency and application builds.

Forced Cache Cleanup

Syntax: docker builder prune --all --force

Parameters: --all (boolean) - Remove all cache including active cache, --force (boolean) - Skip confirmation prompt

Aggressively clean all build cache without prompts, useful for Django development environment reset or CI/CD pipeline cleanup stages.

Return Value

Returns information about cache cleanup operations including total space reclaimed, number of cache entries removed, and remaining cache statistics. Success operations return zero exit code, while errors return non-zero codes with descriptive messages.

  • Success (0): Cache cleanup completed successfully
  • Error (1): General error in cache operations
  • Error (125): Docker daemon error or invalid command usage

docker buildx

Official Docs: docker buildx | Docker CLI: docker buildx

Extended build capabilities with support for multi-platform builds, advanced caching, and BuildKit features for modern container workflows.

Full Description

The docker buildx command provides extended build capabilities beyond the standard docker build functionality. For your anihpj Django project, buildx enables multi-platform container builds, advanced caching strategies, concurrent builds, and modern BuildKit features. This is essential when deploying Django applications across different architectures (AMD64, ARM64) or when building for multiple environments simultaneously.

Buildx is particularly valuable for Django development teams that need to build container images for different deployment targets - from local development on Apple Silicon Macs to production deployment on AMD64 cloud infrastructure. The command manages multiple builder instances, supports BuildKit's advanced features like cache mounts and secrets, and enables efficient CI/CD pipelines for your jobpost application across various platforms.

Syntax

# List available builders
docker buildx ls

# Create new builder instance
docker buildx create --name {builder-name} --driver {driver}

# Use specific builder
docker buildx use {builder-name}

# Build for multiple platforms
docker buildx build --platform linux/amd64,linux/arm64 -t {image-name} .

# Build with advanced features
docker buildx build --cache-from type=registry,ref={cache-image} --cache-to type=registry,ref={cache-image} .

# Inspect builder capabilities
docker buildx inspect {builder-name}

# Remove builder instance
docker buildx rm {builder-name}

# Bootstrap builder (ensure it's ready)
docker buildx inspect --bootstrap {builder-name}

Parameters Description

  • --platform: Target platforms for multi-architecture builds in format os/architecture. Specify multiple platforms comma-separated like linux/amd64,linux/arm64 for Django applications that need to run on different infrastructures.
  • --builder: Name of builder instance to use for the build. Different builders can have different configurations, drivers, and capabilities optimized for specific Django deployment scenarios.
  • --driver: Driver for the builder instance (docker, docker-container, kubernetes). Each driver provides different capabilities for Django build workflows and deployment environments.
  • --cache-from: External cache sources to import during build. Supports registry, local, gha (GitHub Actions), s3, and other cache backends to accelerate Django application builds.
  • --cache-to: External cache destinations to export after build. Enables cache sharing between Django development team members and CI/CD systems.
  • --push: Push the built image to registry immediately after successful build. Streamlines Django application deployment by combining build and publish steps.
  • --load: Load build result to local Docker images after build. Useful for testing Django applications locally after multi-platform builds.
  • --output: Output destination and format for build results. Can export to local filesystem, registry, or other formats beyond standard Docker images.

Attributes

  • Builder Name: Unique identifier for builder instances. Each builder can have different configurations for various Django deployment scenarios.
  • Driver Type: Builder driver that determines available features and performance characteristics for Django builds.
  • Platform Support: List of architectures and operating systems supported by the builder for Django multi-platform deployments.
  • Node Endpoints: Network endpoints where builder nodes are available, important for distributed Django build workflows.
  • Builder Status: Current state of builder instance (running, stopped, error) affecting Django build availability.

Methods

Create Multi-Platform Builder

Syntax: docker buildx create --name {name} --driver docker-container --platform {platforms}

Parameters: name (string) - Builder instance name, driver (string) - Builder driver type, platform (string) - Supported platforms

Create builder instances optimized for Django multi-platform builds supporting different deployment architectures.

Build with Registry Cache

Syntax: docker buildx build --cache-from type=registry,ref={ref} --cache-to type=registry,ref={ref}

Parameters: ref (string) - Registry reference for cache storage and retrieval

Enable distributed cache sharing for Django builds across development team and CI/CD systems using registry-based cache.

Concurrent Platform Build

Syntax: docker buildx build --platform {platform1},{platform2} --push -t {image}

Parameters: platform (string) - Comma-separated list of target platforms

Build Django applications for multiple architectures simultaneously and push to registry for cross-platform deployment.

Return Value

Returns build results including image digests, manifest information, and platform-specific details. For multi-platform builds, returns manifest list with references to platform-specific images. Build failures return detailed error information including platform-specific failure details.

  • Success (0): Build completed successfully for all requested platforms
  • Error (1): Build failed on one or more platforms with detailed error messages
  • Error (125): BuildKit daemon error or invalid buildx configuration

Official Docs: docker buildx bake | Docker CLI: docker buildx bake

Build multiple images concurrently from a single configuration file, enabling complex multi-service Django application deployments with coordinated builds.

Full Description

The docker buildx bake command enables building multiple related container images concurrently from a single configuration file (HCL, JSON, or docker-compose.yml). For your anihpj Django project, bake is essential when you need to build multiple services simultaneously - such as the web application, worker processes, database initialization containers, and auxiliary services like Redis or Nginx.

Bake configurations allow you to define complex build relationships, shared variables, inheritance between targets, and conditional builds based on environment variables. This is particularly powerful for Django microservices architectures where you need to build the main Django app, Celery workers, database migration containers, static file servers, and monitoring services all with consistent tagging, platform support, and build arguments.

Syntax

# Bake all targets from default bake file
docker buildx bake

# Bake specific targets
docker buildx bake {target1} {target2}

# Bake with custom configuration file  
docker buildx bake -f {bake-file.hcl}

# Bake with variable overrides
docker buildx bake --set {target.args.VAR=value}

# Bake with JSON override
docker buildx bake --set {target.platforms=linux/amd64,linux/arm64}

# Print configuration without building
docker buildx bake --print

# Bake from docker-compose file
docker buildx bake -f docker-compose.yml --push

# Bake with progress output
docker buildx bake --progress {auto|plain|tty}

Parameters Description

  • -f, --file: Bake configuration file path (docker-bake.hcl, docker-compose.yml, or JSON). Defines all build targets, dependencies, and shared configurations for Django services.
  • --print: Print the resolved build configuration without executing builds. Useful for validating Django service build definitions and debugging complex configurations.
  • --progress: Set type of progress output during concurrent builds (auto, plain, tty). Important for monitoring multiple Django service builds simultaneously.
  • --push: Push all built images to registry after successful builds. Enables automated deployment workflows for entire Django application stack.
  • --load: Load built images to local Docker daemon. Useful for testing complete Django application stack locally after multi-service builds.
  • --set: Override build configuration values dynamically. Allows runtime customization of Django build arguments, platforms, or outputs without modifying bake files.
  • --no-cache: Build all targets without using cache. Ensures clean builds for Django services when dependency or configuration changes require complete rebuilds.
  • --pull: Always attempt to pull newer versions of base images for all targets. Keeps Django service builds up-to-date with latest security patches.

Attributes

  • Target Names: Identifiers for individual build targets representing different Django services or deployment configurations.
  • Group Definitions: Collections of related targets that can be built together, useful for Django application tiers (frontend, backend, workers).
  • Variable Inheritance: Shared build arguments and configurations across multiple Django service targets.
  • Platform Matrix: Platform specifications that apply across multiple targets for consistent Django cross-platform deployments.
  • Output Destinations: Registry locations, local paths, or other output formats for built Django service images.

Methods

Multi-Service Django Build

Syntax: docker buildx bake -f django-services.hcl --push

Parameters: file (string) - HCL configuration file defining Django service targets

Build all Django application services (web, workers, utilities) concurrently with shared configuration and coordinated deployment.

Environment-Specific Configuration

Syntax: docker buildx bake --set webapp.args.DJANGO_ENV=production --set webapp.platforms=linux/amd64

Parameters: target configuration overrides for Django environment-specific builds

Override Django build configurations for different deployment environments (development, staging, production) without modifying bake files.

Selective Target Building

Syntax: docker buildx bake django-web celery-worker

Parameters: target names to build selectively from Django application stack

Build specific Django services while maintaining shared configurations and dependencies between targets.

Return Value

Returns build results for all targets including individual success/failure status, image digests, and timing information. Failed targets provide detailed error messages while successful targets return registry information and image metadata.

  • Success (0): All specified targets built successfully
  • Error (1): One or more targets failed with individual error details
  • Error (2): Configuration file parsing error or invalid target definitions

docker buildx create

Official Docs: docker buildx create | Docker CLI: docker buildx create

Create new builder instances with custom configurations for specialized Django build requirements, multi-platform support, and advanced BuildKit features.

Full Description

The docker buildx create command creates new builder instances that can be customized with specific drivers, platforms, and configurations tailored to your anihpj Django project's build requirements. Unlike the default docker builder, custom builders enable advanced BuildKit features, multi-platform builds, remote builder connections, and specialized build environments.

For Django development teams, custom builders are essential when you need to build for multiple architectures (ARM64 for Apple Silicon development, AMD64 for cloud deployment), use advanced caching strategies, connect to remote build clusters, or isolate builds between different projects or environments. Each builder instance can have its own configuration, supported platforms, and performance optimizations specific to Django application deployment scenarios.

Syntax

# Create basic builder instance
docker buildx create --name {builder-name}

# Create with specific driver
docker buildx create --name {builder-name} --driver {driver-type}

# Create with platform support
docker buildx create --name {builder-name} --platform {platforms}

# Create with remote node
docker buildx create --name {builder-name} {node-endpoint}

# Create with configuration
docker buildx create --name {builder-name} --config {config-file}

# Create and use immediately  
docker buildx create --name {builder-name} --use

# Create with driver options
docker buildx create --name {builder-name} --driver docker-container --driver-opt {option=value}

# Create with BuildKit image
docker buildx create --name {builder-name} --driver docker-container --driver-opt image={buildkit-image}

Parameters Description

  • --name: Builder instance name for identification and management. Must be unique across all builders for Django project organization and team coordination.
  • --driver: Builder driver type (docker, docker-container, kubernetes, remote). Each driver provides different capabilities for Django build scenarios and deployment environments.
  • --platform: Comma-separated list of supported platforms (linux/amd64, linux/arm64, etc.). Essential for Django cross-platform deployments and development team compatibility.
  • --use: Automatically switch to use the newly created builder. Streamlines Django development workflow by making the new builder immediately active.
  • --driver-opt: Driver-specific options for performance tuning and feature configuration. Optimizes Django build performance and enables specialized BuildKit features.
  • --config: Configuration file path for advanced builder settings. Enables team-wide Django build standardization and complex build environment setup.
  • --append: Add node to existing builder instead of creating new one. Useful for expanding Django build capacity or adding geographic build nodes.
  • --node: Node name for multi-node builders. Supports distributed Django builds across multiple machines or cloud regions.

Attributes

  • Builder Name: Unique identifier for the builder instance, used for management and team coordination in Django projects.
  • Driver Type: Determines builder capabilities, performance characteristics, and available features for Django builds.
  • Supported Platforms: List of architectures the builder can target for Django multi-platform deployments.
  • Node Configuration: Network endpoints, credentials, and connection details for distributed Django build scenarios.
  • Build Features: Available BuildKit features like cache mount, secret mount, SSH forwarding for advanced Django builds.

Methods

Create Multi-Platform Builder

Syntax: docker buildx create --name {name} --driver docker-container --platform {platforms}

Parameters: name (string) - Builder identifier, platforms (string) - Target architectures

Create builder instances capable of building Django applications for multiple architectures simultaneously.

Create Remote Builder Connection

Syntax: docker buildx create --name {name} --driver remote tcp://{host:port}

Parameters: name (string) - Builder identifier, host (string) - Remote BuildKit daemon endpoint

Connect to remote BuildKit instances for distributed Django builds and cloud-based build acceleration.

Create Kubernetes Builder

Syntax: docker buildx create --name {name} --driver kubernetes --driver-opt namespace={ns}

Parameters: name (string) - Builder identifier, namespace (string) - Kubernetes namespace

Create builders that run Django builds as Kubernetes jobs for scalable container orchestration.

Return Value

Returns builder instance identifier and configuration details upon successful creation. Provides builder name, driver type, and initial status information. Failed creations return specific error messages about configuration issues or connectivity problems.

  • Success (0): Builder created successfully with valid configuration
  • Error (1): Creation failed due to invalid parameters or connectivity issues
  • Error (2): Driver initialization error or platform incompatibility

docker buildx imagetools

Official Docs: docker buildx imagetools | Docker CLI: docker buildx imagetools

Inspect and manipulate multi-platform container images and manifests for advanced Django deployment scenarios, registry management, and cross-platform compatibility verification.

Full Description

The docker buildx imagetools command provides advanced image inspection and manipulation capabilities for multi-platform container images. For your anihpj Django project, imagetools is essential when managing complex deployment scenarios involving multiple architectures, validating cross-platform builds, creating custom multi-platform manifests, and troubleshooting registry-related issues.

Imagetools enables Django development teams to inspect manifest lists, verify platform-specific image layers, create custom multi-platform images from existing single-platform builds, and manage registry metadata. This is particularly valuable for Django applications deployed across different cloud providers, development environments with mixed architectures (Intel and ARM), and CI/CD pipelines requiring detailed image validation and manipulation.

Syntax

# Inspect multi-platform image manifest
docker buildx imagetools inspect {image-ref}

# Create multi-platform manifest from existing images
docker buildx imagetools create -t {target-ref} {source-ref1} {source-ref2}

# Inspect specific platform image
docker buildx imagetools inspect --format json {image-ref}

# Create manifest with specific tags
docker buildx imagetools create -t {target}:latest -t {target}:v1.0 {source}

# Inspect and extract platform details
docker buildx imagetools inspect --raw {image-ref}

# Create from multi-platform sources
docker buildx imagetools create -t {target} {source}@{digest1} {source}@{digest2}

# Inspect with custom output format
docker buildx imagetools inspect --format '{{ .Manifest.MediaType }}' {image-ref}

Parameters Description

  • inspect: Display detailed information about image manifests, platforms, and metadata. Essential for Django deployment validation and troubleshooting multi-platform builds.
  • create: Create new multi-platform manifests by combining existing images. Useful for Django applications when building platform-specific images separately and combining them.
  • -t, --tag: Tag names for the created manifest. Supports multiple tags for Django versioning strategies and deployment environment labeling.
  • --format: Output format for inspect command (json, raw, or template). Enables automated Django deployment pipeline integration and custom analysis tools.
  • --raw: Display raw manifest JSON without formatting. Provides complete manifest details for Django deployment automation and registry debugging.
  • --dry-run: Show what would be created without actually creating the manifest. Useful for Django deployment validation and testing manifest creation logic.
  • --append: Add platform images to existing multi-platform manifest instead of replacing. Enables incremental Django multi-platform deployment strategies.
  • --platform: Filter inspect output to specific platforms. Helps Django teams focus on relevant architecture-specific information during deployment verification.

Attributes

  • Manifest Type: Identifies whether image is single-platform or multi-platform manifest list for Django deployment planning.
  • Platform Details: Architecture, OS, and variant information for each platform in Django cross-platform deployments.
  • Layer Information: Digests, sizes, and metadata for image layers enabling Django deployment optimization and troubleshooting.
  • Registry Metadata: Creation time, labels, annotations, and other registry-specific information for Django image management.
  • Digest References: Immutable content-addressable identifiers for precise Django deployment targeting and version control.

Methods

Inspect Multi-Platform Manifest

Syntax: docker buildx imagetools inspect {image-ref}

Parameters: image-ref (string) - Registry reference to multi-platform image

Analyze Django application images for platform compatibility, layer structure, and deployment readiness across architectures.

Create Combined Manifest

Syntax: docker buildx imagetools create -t {target} {source1} {source2}

Parameters: target (string) - New manifest reference, sources (strings) - Platform-specific image references

Combine separately built Django platform images into unified multi-platform deployment manifest for registry distribution.

Extract Platform-Specific Image

Syntax: docker buildx imagetools inspect --platform {platform} {image-ref}

Parameters: platform (string) - Target platform specification, image-ref (string) - Multi-platform image reference

Extract specific platform details from Django multi-platform images for targeted deployment and debugging scenarios.

Return Value

Returns detailed manifest information, platform specifications, and image metadata. For inspect operations, provides manifest structure, layer details, and platform availability. Create operations return new manifest digest and reference information.

  • Success (0): Operation completed successfully with valid manifest data
  • Error (1): Image not found or insufficient permissions for registry access
  • Error (2): Invalid manifest format or corrupted image data
