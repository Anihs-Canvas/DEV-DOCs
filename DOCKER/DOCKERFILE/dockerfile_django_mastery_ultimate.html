1) Overview

Docs: Dockerfile Overview

Docker can build images automatically by reading instructions from a Dockerfile.

Full description

A Dockerfile is a text document that contains all the commands a user could call on the command line to assemble an image. Using docker build users can create an automated build that executes several command-line instructions in succession. The docker build command builds an image from a Dockerfile and a context. For Django applications like our JobPost management system, Dockerfiles define the complete runtime environment including Python version, dependencies, application code, and configuration.

Syntax

# Comment
INSTRUCTION arguments

Parameters description

  • Build context: Set of files located in the specified PATH or URL
  • Instructions: Define the steps to build the image
  • Layers: Each instruction creates a new layer in the image
  • Caching: Docker caches layers to speed up subsequent builds
  • Security: Dockerfile should follow security best practices

Return value

Docker build process creates a container image that can be run to execute your Django JobPost application.

5 short examples with Django context

Example 1: Basic Django Dockerfile

FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
# Basic Django JobPost application Dockerfile # Creates 6 layers total # Ready for development deployment

Example 2: Production Django build

FROM python:3.11-slim
ENV PYTHONUNBUFFERED=1
RUN apt-get update && apt-get install -y postgresql-client
COPY requirements.txt .
RUN pip install -r requirements.txt gunicorn
COPY . /app
WORKDIR /app
RUN python manage.py collectstatic --noinput
EXPOSE 8000
CMD ["gunicorn", "myproject.wsgi:application"]
# Production-ready Django JobPost application # Includes PostgreSQL client and Gunicorn # Static files collected during build

Example 3: Multi-stage Django build

FROM python:3.11 AS builder
COPY requirements.txt .
RUN pip install -r requirements.txt

FROM python:3.11-slim
COPY --from=builder /usr/local/lib/python3.11/site-packages /usr/local/lib/python3.11/site-packages
COPY . /app
WORKDIR /app
CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
# Multi-stage build for optimized Django image # Builder stage installs dependencies # Runtime stage copies only needed packages

Example 4: Django with build arguments

ARG PYTHON_VERSION=3.11
ARG DJANGO_ENV=production
FROM python:${PYTHON_VERSION}-slim
ENV DJANGO_SETTINGS_MODULE=myproject.settings.${DJANGO_ENV}
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . /app
WORKDIR /app
# Configurable Django build with arguments # Python version and environment configurable # JobPost app adaptable to different environments

Example 5: Django with health checks

FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
HEALTHCHECK --interval=30s --timeout=10s --start-period=40s \
  CMD python manage.py check || exit 1
EXPOSE 8000
CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
# Django JobPost app with health monitoring # Health check uses Django's built-in check command # Container health status available to orchestrators

2) Format

Docs: Format

The format of a Dockerfile follows specific rules for instructions, comments, and line continuations.

Full description

The format of a Dockerfile is: instruction arguments. Instructions are not case-sensitive, but convention is to use UPPERCASE to distinguish them from arguments. Docker runs instructions in a Dockerfile in order. A Dockerfile must begin with a FROM instruction, which specifies the base image from which you are building. For Django applications, proper formatting ensures reliable builds and maintainable Dockerfiles.

Syntax

# Comment
INSTRUCTION arguments
INSTRUCTION \
  argument1 \
  argument2

Parameters description

  • Comments: Lines beginning with # are treated as comments
  • Line continuation: Backslash (\) at end of line continues instruction
  • Whitespace: Leading and trailing whitespace is ignored
  • Case insensitive: Instructions are case insensitive but UPPERCASE is convention
  • Order matters: Instructions are executed in order from top to bottom

Return value

Properly formatted Dockerfile enables successful image builds with predictable layer creation.

5 short examples with Django formatting

Example 1: Comments and documentation

# Django JobPost Application Dockerfile
# Maintainer: dev@company.com
# Version: 1.0.0

FROM python:3.11-slim
# Install system dependencies for PostgreSQL
RUN apt-get update && apt-get install -y postgresql-client
# Well-documented Dockerfile structure # Comments explain purpose and dependencies # Clear documentation for JobPost application

Example 2: Line continuation for readability

RUN apt-get update && apt-get install -y \
    postgresql-client \
    redis-tools \
    curl \
    && rm -rf /var/lib/apt/lists/*
# Multi-line instruction with backslash continuation # Better readability for complex RUN commands # Installs dependencies needed for JobPost app

Example 3: Environment variables formatting

ENV PYTHONUNBUFFERED=1 \
    PYTHONDONTWRITEBYTECODE=1 \
    DJANGO_SETTINGS_MODULE=myproject.settings.production \
    DATABASE_URL=postgresql://user:pass@db:5432/jobpost_db
# Multiple environment variables with continuation # Clean formatting for Django configuration # JobPost database and Python settings configured

Example 4: Case sensitivity demonstration

# Both forms work but UPPERCASE is preferred
from python:3.11-slim
WORKDIR /app
copy requirements.txt .
RUN pip install -r requirements.txt
# Mixed case instructions work but not recommended # UPPERCASE convention improves readability # Better to stick with consistent formatting

Example 5: Complex Django setup formatting

# Django JobPost production setup
FROM python:3.11-slim AS base

# System dependencies
RUN apt-get update && apt-get install -y \
    # PostgreSQL client for database operations
    postgresql-client \
    # Redis CLI for cache operations  
    redis-tools \
    # Cleanup
    && rm -rf /var/lib/apt/lists/*

# Python environment
ENV PYTHONUNBUFFERED=1 \
    PYTHONDONTWRITEBYTECODE=1

# Application setup
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
# Professional Dockerfile formatting # Clear sections with comments # Logical grouping of related instructions # JobPost production environment ready

3) FROM

Docs: FROM instruction

Initializes a new build stage and sets the Base Image for subsequent instructions.

Full description

The FROM instruction initializes a new build stage and sets the Base Image for subsequent instructions. As such, a valid Dockerfile must start with a FROM instruction. The image can be any valid image – it is especially easy to start by pulling an image from the Public Repositories. For Django JobPost applications, choosing the right base image affects security, size, and available tools.

Syntax

FROM [--platform=<platform>] <image> [AS <name>]
FROM [--platform=<platform>] <image>[:<tag>] [AS <name>]
FROM [--platform=<platform>] <image>[@<digest>] [AS <name>]

Parameters description

  • image: Base image name from registry or local
  • tag: Image version tag (defaults to 'latest' if omitted)
  • digest: SHA256 digest for exact image identification
  • --platform: Target platform (linux/amd64, linux/arm64, etc.)
  • AS name: Assigns name to build stage for multi-stage builds

Return value

Establishes the base filesystem and environment for subsequent Dockerfile instructions.

Understand how ARG and FROM interact

ARG is the only instruction that may precede FROM in the Dockerfile. ARG declared before FROM is outside of a build stage, so it can't be used in any instruction after a FROM. To use the default value of an ARG declared before the first FROM use an ARG instruction without a value inside of a build stage.

Example 1: ARG before FROM

ARG PYTHON_VERSION=3.11
FROM python:${PYTHON_VERSION}-slim
ARG PYTHON_VERSION
RUN echo "Using Python version: ${PYTHON_VERSION}"
# ARG before FROM sets build-time variable # Can be used in FROM instruction # Must redeclare ARG after FROM to use in subsequent instructions # Django app built with specified Python version

Example 2: Multiple ARGs with FROM

ARG PYTHON_VERSION=3.11
ARG VARIANT=slim
FROM python:${PYTHON_VERSION}-${VARIANT}
ARG PYTHON_VERSION
ARG VARIANT
LABEL python.version=${PYTHON_VERSION} \
      image.variant=${VARIANT}
# Multiple ARGs configure base image selection # Variables available in FROM instruction # Redeclared after FROM for use in LABEL # JobPost app metadata includes build parameters

Example 3: Conditional FROM with ARG

ARG BUILD_ENV=production
FROM python:3.11-slim AS development
FROM python:3.11-alpine AS production
FROM ${BUILD_ENV} AS final
ARG BUILD_ENV
RUN echo "Building for environment: ${BUILD_ENV}"
# ARG determines which base image to use # Multi-stage build with conditional selection # Final stage determined by BUILD_ENV argument # JobPost app optimized for target environment

Example 4: Default ARG values

ARG REGISTRY=docker.io
ARG IMAGE=python
ARG TAG=3.11-slim
FROM ${REGISTRY}/${IMAGE}:${TAG}
# ARGs are now out of scope, need to redeclare
ARG REGISTRY
ARG IMAGE  
ARG TAG
RUN echo "Base image: ${REGISTRY}/${IMAGE}:${TAG}"
# Configurable registry and image selection # Default values provided for all ARG variables # Full image path constructed from components # JobPost app deployable from different registries

Example 5: ARG scope demonstration

ARG GLOBAL_ARG=global_value
FROM python:3.11-slim AS stage1
ARG GLOBAL_ARG
RUN echo "Stage 1 - Global ARG: ${GLOBAL_ARG}"

FROM python:3.11-slim AS stage2
ARG GLOBAL_ARG=stage2_default
RUN echo "Stage 2 - Global ARG: ${GLOBAL_ARG}"
# Global ARG available to all stages # Each stage must redeclare ARG to use it # Different default values per stage possible # JobPost app multi-stage build with shared configuration

4) RUN

Docs: RUN instruction

Execute any commands in a new layer on top of the current image and commit the results.

Full description

The RUN instruction will execute any commands in a new layer on top of the current image and commit the results. The resulting committed image will be used for the next step in the Dockerfile. Layering RUN instructions and generating commits conforms to the core concepts of Docker where commits are cheap and containers can be created from any point in an image's history. For Django JobPost applications, RUN is used for installing dependencies, setting up the environment, and preparing the application.

Syntax

# Shell form
RUN <command>

# Exec form
RUN ["executable", "param1", "param2"]

# Advanced options
RUN --mount=<mount-options> <command>
RUN --network=<network-mode> <command>
RUN --security=<security-mode> <command>

Parameters description

  • Shell form: Commands run in a shell (/bin/sh -c on Linux). Supports shell features like pipes, variables
  • Exec form: Commands run directly without shell processing. More reliable for signal handling
  • command: The shell command to execute
  • executable: Direct executable path in exec form
  • param1, param2: Arguments passed to the executable
  • --mount: Mount filesystems during command execution (cache, bind, secret, ssh)
  • --network: Set network mode during command execution (default, none, host)
  • --security: Set security mode (insecure, sandbox)

Return value

Creates a new image layer with the results of the command execution. Returns exit code 0 for success, non-zero for failure (stops build).

Cache invalidation for RUN instructions

The cache for RUN instructions isn't invalidated automatically during the next build. The cache for a RUN instruction can be invalidated by using the --no-cache flag, for example docker build --no-cache. The cache for RUN instructions can be invalidated by ADD and COPY instructions.

Example 1: Cache-friendly Django setup

# Dependencies change less frequently than source code
COPY requirements.txt .
RUN pip install -r requirements.txt
# Source code changes more frequently
COPY . /app
# requirements.txt copied first for better caching # pip install layer cached unless requirements change # Source code changes don't invalidate dependency cache # Faster builds for JobPost Django development

Example 2: Cache invalidation example

RUN apt-get update && apt-get install -y postgresql-client
# This layer will be cached
COPY requirements.txt .
RUN pip install -r requirements.txt
# If requirements.txt changes, this and subsequent layers rebuild
# System packages cached until Dockerfile changes # Python dependencies cached until requirements.txt changes # Cache invalidation cascades to subsequent layers # JobPost app build optimization through layer ordering

RUN --device

Similar to the --device flag for docker run, --device allows you to provide devices to the RUN instruction. This is useful for builds that need access to specific hardware devices during the build process.

Example: GPU access during build

RUN --device=/dev/nvidia0 python setup.py install_gpu_support
# Provides GPU device access during build # Useful for compiling CUDA-enabled packages # JobPost app with machine learning features

RUN --mount

RUN --mount allows you to create mounts that process running as part of the build can access. This can be used to bind files from other parts of the build without permanently adding them to the build context or final image, access build secrets without leaking them into the image, or create cache mounts to speed up builds.

RUN --mount=type=bind

Bind mount from context

RUN --mount=type=bind,source=scripts,target=/scripts \
    /scripts/install_django_deps.sh
# Mounts scripts directory during build # Scripts not included in final image # Clean builds without temporary files

RUN --mount=type=cache

Pip cache mount

RUN --mount=type=cache,target=/root/.cache/pip \
    pip install django==5.2 psycopg2-binary
# Pip cache persisted across builds # Faster Django dependency installation # JobPost requirements installed efficiently

Apt cache mount

RUN --mount=type=cache,target=/var/cache/apt \
    --mount=type=cache,target=/var/lib/apt/lists \
    apt-get update && apt-get install -y postgresql-client
# APT package cache preserved # System packages for JobPost app cached # Reduced download time on rebuilds

RUN --mount=type=tmpfs

Temporary filesystem mount

RUN --mount=type=tmpfs,target=/tmp \
    python manage.py test --keepdb
# Temporary filesystem for test data # Fast in-memory storage during build # JobPost Django tests run efficiently

RUN --mount=type=secret

Secret mount for credentials

RUN --mount=type=secret,id=pip_config,target=/etc/pip.conf \
    pip install --trusted-host private.pypi.org django-extensions
# Private PyPI credentials mounted securely # Secrets not stored in image layers # Private Django packages for JobPost

RUN --mount=type=ssh

SSH mount for git operations

RUN --mount=type=ssh \
    git clone git@github.com:company/jobpost-private.git /tmp/private
# SSH agent forwarded to build # Private git repositories accessible # JobPost private modules cloned securely

RUN --network

RUN --network allows control over which networking environment the command is run in.

RUN --network=default

Default network access

RUN --network=default \
    pip install django==5.2
# Standard network access enabled # Can download from PyPI and other sources # Default behavior for JobPost dependencies

RUN --network=none

No network access

RUN --network=none \
    python manage.py collectstatic --noinput
# Network access disabled during execution # Prevents data exfiltration during build # JobPost static files collected securely

RUN --network=host

Host network access

RUN --network=host \
    curl -o requirements.txt http://localhost:8080/requirements.txt
# Direct access to host network # Can reach services on host machine # JobPost requirements from local development server

RUN --security

RUN --security=insecure is used to allow the build to run insecure operations, such as privileged operations, during the build.

Insecure operations

RUN --security=insecure \
    mount -t tmpfs tmpfs /mnt && \
    python manage.py migrate --run-syncdb
# Allows privileged operations during build # Use sparingly and only when necessary # JobPost database initialization with special requirements

5) CMD

Docs: CMD instruction

Provides defaults for an executing container.

Full description

CMD specifies the default command to run when the container starts. Unlike RUN which executes during build time, CMD runs when the container is started. There can only be one CMD instruction in a Dockerfile - if multiple are listed, only the last one takes effect. For Django projects, CMD typically starts the web server, runs migrations, or starts a specific Django management command. CMD can be overridden when running the container.

Syntax

# Exec form (preferred)
CMD ["executable","param1","param2"]

# Shell form
CMD command param1 param2

# Default parameters for ENTRYPOINT
CMD ["param1","param2"]

Parameters description

  • Exec form: Recommended. Runs executable directly without shell processing
  • Shell form: Commands are run in a shell, allowing shell features like variables and pipes
  • Default parameters: Used with ENTRYPOINT to provide default arguments
  • executable: The command or program to execute
  • param1, param2: Arguments passed to the executable

Return value

Sets the default command for container execution. Returns the process that will run when the container starts, unless overridden at runtime.

Example 1: Django development server

CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
# Starts Django development server # Binds to all interfaces on port 8000 # Ideal for JobPost application development # Can be overridden at runtime

Example 2: Production with Gunicorn

CMD ["gunicorn", "--bind", "0.0.0.0:8000", "--workers", "4", "myproject.wsgi:application"]
# Production-ready WSGI server # 4 worker processes for handling requests # Serves JobPost Django application # Suitable for production deployment

Example 3: Shell form with variables

ENV DJANGO_PORT=8000
CMD python manage.py runserver 0.0.0.0:$DJANGO_PORT
# Uses environment variable for port # Shell form allows variable expansion # Flexible port configuration # JobPost app adapts to environment

Example 4: Default parameters with ENTRYPOINT

ENTRYPOINT ["python", "manage.py"]
CMD ["runserver", "0.0.0.0:8000"]
# ENTRYPOINT + CMD combination # Default runs development server # Can override CMD: docker run app collectstatic # Maintains python manage.py as base command

Example 5: Multi-step startup script

CMD ["sh", "-c", "python manage.py migrate && python manage.py collectstatic --noinput && python manage.py runserver 0.0.0.0:8000"]
# Runs migrations before starting server # Collects static files automatically # Comprehensive startup for JobPost app # Ensures database and static files are ready

6) LABEL

Docs: LABEL instruction

Adds metadata to an image.

Full description

LABEL adds metadata to an image. A LABEL is a key-value pair. To include spaces within a LABEL value, use quotes and backslashes as you would in command-line parsing. Labels are additive including LABELs in FROM images. If Docker encounters a label/key that already exists, the new value overrides any previous labels with identical keys. For Django projects, LABELs provide essential metadata about the application version, maintainer, description, and operational information for the JobPost system.

Syntax

LABEL <key>=<value> <key>=<value> <key>=<value> ...

# Multi-line format
LABEL version="1.0" \
      description="Django JobPost Application" \
      maintainer="Platform Team"

Parameters description

  • key: The metadata key name (case-sensitive)
  • value: The metadata value (supports spaces when quoted)
  • Multiple pairs: Single LABEL instruction can set multiple key-value pairs
  • Line continuation: Use backslash for multi-line format
  • Quotes: Required for values containing spaces or special characters

Return value

Adds metadata to the image that can be inspected with docker inspect. Returns metadata that is preserved in the image and accessible to runtime tools and orchestration systems.

Example 1: Basic application metadata

LABEL version="1.0"
LABEL description="Django JobPost Management System"
LABEL maintainer="platform-team@company.com"

Example 2: Multi-line consolidated format

LABEL version="2.1.0" \

Example 3: OCI annotations for JobPost

Example 5: Django-specific metadata

7) EXPOSE

Docs: EXPOSE instruction

Informs Docker that the container listens on the specified network ports at runtime.

Full description

EXPOSE informs Docker that the container listens on the specified network ports at runtime. The EXPOSE instruction does not actually publish the port. It functions as a type of documentation between the person who builds the image and the person who runs the container, about which ports are intended to be published. For Django applications, EXPOSE typically declares the web server port (8000, 80, 443) and any additional service ports like database connections or Redis caching that the JobPost application uses.

Syntax

EXPOSE <port> [<port>/<protocol>...]

# Multiple ports

9) ADD

Docs: ADD instruction

Copies new files, directories or remote file URLs and adds them to the filesystem of the image.

Full description

The ADD instruction copies new files, directories or remote file URLs from <src> and adds them to the filesystem of the image at the path <dest>. Multiple <src> resources may be specified but if they are files or directories, their paths are interpreted as relative to the source of the context of the build. Each <src> may contain wildcards and matching will be done using Go's filepath.Match rules.

Syntax

ADD [--chown=<user>:<group>] [--chmod=<perms>] [--checksum=<checksum>] <src>... <dest>
ADD [--chown=<user>:<group>] [--chmod=<perms>] ["<src>",... "<dest>"]

# With BuildKit features
ADD --link [--chown=<user>:<group>] [--chmod=<perms>] <src>... <dest>
ADD --exclude=<path> <src>... <dest>

Parameters description

  • src: Source files, directories, or URLs to copy. Supports wildcards and multiple sources
  • dest: Destination path in the container filesystem
  • --chown: Set file ownership (user:group). Only supported on Linux containers
  • --chmod: Set file permissions. Only supported on Linux containers
  • --checksum: Verify downloaded files against provided checksum
  • --link: Enable improved caching and faster builds with BuildKit
  • --exclude: Exclude files and directories matching pattern
  • Wildcards: Use Go filepath.Match rules for pattern matching
  • Auto-extraction: Automatically extracts compressed archives (tar, gzip, etc.)

Return value

Copies files and directories to the container filesystem. Returns a new image layer containing the copied content. Automatically extracts compressed archives and can fetch remote URLs.

Source

The <src> path must be inside the context of the build; you cannot ADD ../something /something, because the first step of a docker build is to send the context directory (and subdirectories) to the docker daemon.

Example: Django project files

# Copies Django project directory # Copies requirements file # Copies CSS files matching pattern # All sources relative to build context

Destination

The <dest> is an absolute path, or a path relative to WORKDIR, into which the source will be copied inside the destination container. If <dest> doesn't exist, it is created along with all missing directories in its path.

Example: Django destination paths

Example: Optimized caching

10) COPY

Docs: COPY instruction

Copies new files or directories and adds them to the filesystem of the container.

Full description

The COPY instruction copies new files or directories from <src> and adds them to the filesystem of the container at the path <dest>. Multiple <src> resources may be specified but the paths of files and directories will be interpreted as relative to the source of the context of the build. COPY is preferred over ADD for simple file copying as it's more transparent and predictable.

Syntax

COPY [--chown=<user>:<group>] [--chmod=<perms>] <src>... <dest>
COPY [--chown=<user>:<group>] [--chmod=<perms>] ["<src>",... "<dest>"]

# Multi-stage builds
COPY --from=<stage> [--chown=<user>:<group>] [--chmod=<perms>] <src>... <dest>

# BuildKit features
COPY --link [--chown=<user>:<group>] [--chmod=<perms>] <src>... <dest>
COPY --parents [--chown=<user>:<group>] [--chmod=<perms>] <src>... <dest>
COPY --exclude=<path> <src>... <dest>

Parameters description

  • src: Source files or directories to copy. Supports wildcards and multiple sources
  • dest: Destination path in the container filesystem
  • --chown: Set file ownership (user:group). Only supported on Linux containers
  • --chmod: Set file permissions. Only supported on Linux containers
  • --from: Copy from another build stage or image in multi-stage builds
  • --link: Enable improved caching and faster builds with BuildKit
  • --parents: Preserve parent directory structure when copying
  • --exclude: Exclude files and directories matching pattern
  • Wildcards: Use Go filepath.Match rules for pattern matching
  • No auto-extraction: Unlike ADD, COPY does not extract archives

Return value

Copies files and directories to the container filesystem. Returns a new image layer containing the copied content. Does not extract archives or fetch remote URLs.

Source

The <src> path must be inside the context of the build; you cannot COPY ../something /something. All new files and directories are copied with a UID and GID of 0, unless the optional --chown flag specifies a given username, groupname, or UID/GID combination.

Example: Django source files

COPY myproject/ /app/myproject/
COPY static/css/*.css /app/static/css/
# Django management script copied # Project directory structure preserved # CSS files copied with wildcard pattern

Destination

The <dest> is an absolute path, or a path relative to WORKDIR, into which the source will be copied inside the destination container. If <dest> ends with a trailing slash /, it will be considered a directory and the contents of <src> will be written at <dest>/base(<src>).

Example: Destination behavior

COPY requirements.txt /app/requirements.txt  # File destination
COPY requirements.txt /app/                   # Directory destination
COPY static/ /app/static/                     # Directory to directory
# Explicit file destination # File copied to directory with same name # Directory contents copied to destination

COPY --from

The COPY instruction can optionally accept a flag --from=<name> that can be used to set the source location to a previous build stage (created with FROM .. AS <name>) that will be used instead of a build context sent by the user.

# Nginx configuration from official image # Redis CLI tool from official image # Selective file extraction from other images

The COPY instruction supports setting ownership and permissions using --chown and --chmod flags, similar to ADD.

Example: Django file permissions

# Django management script with execute permissions # Application files with proper ownership # JobPost app security best practices

The --link flag enables advanced caching behavior where the contents can change without invalidating the cache as long as the metadata doesn't change.

Example: Optimized Django builds

# Advanced caching for requirements file # Application directory with link optimization # Faster JobPost development iterations

The --parents flag preserves parent directories when copying files, maintaining the directory structure.

Example: Preserve directory structure

11) ENTRYPOINT

Docs: ENTRYPOINT instruction

Allows you to configure a container that will run as an executable.

Full description

An ENTRYPOINT allows you to configure a container that will run as an executable. Command line arguments to docker run <image> will be appended after all elements in an exec form ENTRYPOINT, and will override all elements specified using CMD. This allows arguments to be passed to the entry point.

Syntax

# Exec form (preferred)
ENTRYPOINT ["executable", "param1", "param2"]

# Shell form
ENTRYPOINT command param1 param2

Parameters description

  • Exec form: Recommended. Runs executable directly without shell processing
  • Shell form: Commands are run in a shell, preventing signal forwarding
  • executable: The command or program to execute as container entry point
  • param1, param2: Default arguments passed to the executable
  • CMD interaction: CMD provides default arguments that can be overridden
  • Runtime arguments: docker run arguments are appended to ENTRYPOINT
  • Signal handling: Exec form ensures proper signal forwarding to the process
  • Override: Use --entrypoint flag to override ENTRYPOINT at runtime

Return value

Configures the container's main process and entry point. Returns the primary executable that will run when the container starts, with proper signal handling and argument passing.

Exec form ENTRYPOINT example

You can use the exec form of ENTRYPOINT to set fairly stable default commands and arguments and then use either form of CMD to set additional defaults that are more likely to be changed.

Example: Django management wrapper

CMD ["runserver", "0.0.0.0:8000"]

# docker run myapp                    # runs: python manage.py runserver 0.0.0.0:8000
# docker run myapp migrate           # runs: python manage.py migrate
# docker run myapp shell             # runs: python manage.py shell
# ENTRYPOINT sets fixed command prefix # CMD provides default arguments # Container becomes Django management tool # JobPost app management commands simplified

Example: Django with initialization

CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]

12) ARG

Docs: ARG instruction

Defines a variable that users can pass at build-time to the builder with the docker build command.

Full description

The ARG instruction defines a variable that users can pass at build-time to the builder with the docker build command using the --build-arg <varname>=<value> flag. If a user specifies a build argument that was not defined in the Dockerfile, the build outputs a warning. A Dockerfile may include one or more ARG instructions.

Syntax

ARG <name>[=<default value>]

# Examples
ARG VERSION
ARG PYTHON_VERSION=3.11
ARG BUILDPLATFORM
ARG TARGETPLATFORM

Parameters description

  • name: The variable name that can be referenced in the Dockerfile
  • default value: Optional default value used if --build-arg is not provided
  • Build-time only: ARG variables are only available during build, not in running containers
  • Scope: ARG before FROM applies to FROM line only, ARG after FROM applies to current stage
  • Predefined ARGs: Docker provides predefined ARGs like BUILDPLATFORM, TARGETPLATFORM
  • Environment substitution: Can be used in ENV, ADD, COPY and other instructions
  • Security: ARG values may be visible in docker history, avoid secrets
  • Multi-stage: ARG scope is limited to the build stage where it's defined

Return value

Defines build-time variables that can be used in subsequent instructions. Returns variable values that are substituted during the build process but not persisted in the final image.

Default values

An ARG instruction can optionally include a default value. If an ARG instruction has a default value and if there is no value passed at build-time, the builder uses the default.

Example: Django version with default

ARG DJANGO_VERSION=5.2
ARG PYTHON_VERSION=3.11
FROM python:${PYTHON_VERSION}-slim
RUN pip install Django==${DJANGO_VERSION}
# Default Django 5.2 and Python 3.11 # Can be overridden at build time: # docker build --build-arg DJANGO_VERSION=5.2 . # JobPost app with configurable versions

Scope

An ARG variable definition comes into effect from the line on which it is defined in the Dockerfile not from the argument's use on the command line or elsewhere. Variables defined in one stage are not available in subsequent stages unless re-declared.

Example: ARG scope across stages

ARG GLOBAL_VERSION=1.0.0

FROM python:3.11 AS builder
ARG GLOBAL_VERSION
ARG BUILD_DEPS=build-essential
RUN echo "Builder stage - Version: ${GLOBAL_VERSION}, Deps: ${BUILD_DEPS}"

FROM python:3.11-slim AS runtime
ARG GLOBAL_VERSION
# BUILD_DEPS not available here unless redeclared
RUN echo "Runtime stage - Version: ${GLOBAL_VERSION}"
# GLOBAL_VERSION available in both stages # BUILD_DEPS only available in builder stage # Must redeclare ARG in each stage that uses it # JobPost app multi-stage build with shared config

Using ARG variables

You can use an ARG or an ENV instruction to specify variables that are available to the RUN instruction. Environment variables defined using the ENV instruction always override an ARG instruction of the same name.

Example: ARG vs ENV precedence

ARG DATABASE_URL=sqlite:///db.sqlite3
ENV DATABASE_URL=postgresql://user:pass@db:5432/jobpost_db
RUN echo "Database URL: ${DATABASE_URL}"  # Uses ENV value
# ENV takes precedence over ARG # Runtime value: postgresql://user:pass@db:5432/jobpost_db # Build-time ARG overridden by runtime ENV

Example: ARG to ENV conversion

ARG BUILD_ENV=production
ARG SECRET_KEY
ENV DJANGO_SETTINGS_MODULE=myproject.settings.${BUILD_ENV}
ENV SECRET_KEY=${SECRET_KEY}
RUN echo "Settings: ${DJANGO_SETTINGS_MODULE}, Secret: ${SECRET_KEY}"
# ARG values converted to ENV for runtime use # Build arguments become environment variables # JobPost app configuration from build to runtime

Predefined ARGs

Docker has a set of predefined ARG variables that you can use without a corresponding ARG instruction in the Dockerfile.

Example: HTTP proxy settings

# These ARGs are automatically available:
# HTTP_PROXY, http_proxy, HTTPS_PROXY, https_proxy, FTP_PROXY, ftp_proxy, NO_PROXY, no_proxy

FROM python:3.11-slim
RUN pip install --proxy=${HTTP_PROXY} Django==5.2
# Proxy settings automatically available # No ARG declaration needed # Build-time proxy configuration for JobPost dependencies

Automatic platform ARGs in the global scope

These are automatically available and can be used anywhere in the Dockerfile without declaring them with ARG.

Example: Platform-specific builds

FROM python:3.11-slim
RUN echo "Building for: ${TARGETPLATFORM}, ${TARGETOS}, ${TARGETARCH}"

16) VOLUME

Docs: VOLUME instruction

Creates a mount point with the specified name and marks it as holding externally mounted volumes.

Full description

VOLUME creates a mount point with the specified name and marks it as holding externally mounted volumes from native host or other containers. The value can be a JSON array, VOLUME ["/var/log/"], or a plain string with multiple arguments, such as VOLUME /var/log or VOLUME /var/log /var/db. For Django applications, VOLUME is essential for persisting data like database files, media uploads, logs, and other stateful content that should survive container restarts and updates.

Syntax

VOLUME ["/path/to/volume"]
VOLUME /path/to/volume
VOLUME /path/one /path/two

# JSON array format (recommended)
VOLUME ["/var/log", "/var/db"]

# Multiple separate volumes
VOLUME /app/media
VOLUME /app/logs

Parameters description

  • path: Absolute path in the container filesystem to create as volume mount point
  • JSON array: Recommended format using double quotes for multiple volumes
  • Multiple arguments: Space-separated paths in single VOLUME instruction
  • Anonymous volumes: Docker creates anonymous volumes if no host path is specified
  • Permission inheritance: Volume inherits permissions set before VOLUME declaration
  • Data persistence: Data in volumes persists beyond container lifecycle
  • Performance: Volumes bypass container filesystem layer for better I/O performance
  • Host-dependent: Volume behavior varies between host operating systems

Return value

Creates mount points in the container filesystem that can be bound to host directories or Docker-managed volumes. Returns persistent storage locations that survive container restarts and removals.

Notes about specifying volumes

Keep the following things in mind about volumes in the Dockerfile.

Important Volume Considerations

  • Change of ownership and permission: Changing the volume from within the Dockerfile after it has been declared will be discarded
  • JSON formatting: The list is parsed as a JSON array, which means you must use double-quotes (") around words not single-quotes (')
  • Host-specific volumes: Volumes are host-dependent and may not be available on all hosts
  • Anonymous volumes: If no host path is specified, Docker creates an anonymous volume

Example: Django data persistence

# Create django user and set ownership BEFORE VOLUME

RUN mkdir -p /app/media /app/logs && chown -R django:django /app

# Declare volumes AFTER setting permissions
VOLUME ["/app/media", "/app/logs"]

# Changes to these paths after VOLUME will be discarded
USER django
# Permissions set before VOLUME declaration # Django media files and logs persisted # User changes after VOLUME are ignored # JobPost uploads and logs survive container restarts

# ❌ Wrong: changing permissions after VOLUME

# ✅ Correct: set permissions before VOLUME

# Permission changes after VOLUME are discarded # Always set ownership before declaring VOLUME # PostgreSQL data directory properly configured # JobPost database data persisted correctly

Example: JSON format requirement

# ✅ Correct JSON format
VOLUME ["/app/media", "/app/logs", "/app/staticfiles"]

# ❌ Wrong: single quotes

# ✅ Alternative single path format

# JSON array format with double quotes # Multiple volumes in one instruction # Single volume can omit brackets # JobPost app volumes properly declared

17) ONBUILD

Docs: ONBUILD instruction

Adds a trigger instruction to the image that will be executed when the image is used as a base for another build.

Full description

ONBUILD adds a trigger instruction to the image that will be executed at a later time, when the image is used as the base for another build. The trigger will be executed in the context of the downstream build, as if it had been inserted immediately after the FROM instruction in the downstream Dockerfile. For Django projects, ONBUILD is useful for creating base images that automatically handle common setup tasks for child images.

Syntax

ONBUILD [INSTRUCTION]

Parameters description

  • INSTRUCTION: Any Dockerfile instruction except FROM, MAINTAINER, or another ONBUILD
  • Delayed execution: Instruction executes when image is used as base, not during current build
  • Build context: ONBUILD instructions use the child image's build context
  • Inheritance: ONBUILD instructions are inherited but not triggered by grandchild images
  • Restrictions: Cannot use FROM, MAINTAINER, or nested ONBUILD instructions
  • Trigger order: ONBUILD instructions execute in the order they were added
  • Build stage: Triggers execute after FROM but before any other child instructions
  • Cache behavior: ONBUILD instructions can affect build cache of child images

Adds trigger metadata to the image. Returns instructions that will execute when this image is used as a base for child builds, enabling reusable base image patterns.

Troubleshooting Tips

  • Layer caching: Order Dockerfile instructions from least to most frequently changing
  • Image size: Use multi-stage builds and .dockerignore to reduce size
  • Security: Always run as non-root user and scan for vulnerabilities
  • Performance: Use BuildKit cache mounts for dependencies
  • Debugging: Use docker exec -it container bash to inspect running containers
  • Logs: View container logs with docker logs container_name
  • Health checks: Implement proper health endpoints in Django
  • Environment: Use .env files for local development
  • Static files: Ensure STATIC_ROOT is properly configured
  • Database: Always run migrations in production deployments
