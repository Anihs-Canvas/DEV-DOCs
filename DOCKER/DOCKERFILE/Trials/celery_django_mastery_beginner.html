    1) Introduction to Celery

    Docs: App (Celery) • Task/@task

    What is Celery? A distributed task queue for running background jobs asynchronously and on schedules.

    Full description

    Celery is a reliable, scalable system for running tasks outside of the web request/response cycle. In Django projects, Celery offloads CPU-bound or I/O-bound work so your views remain fast and responsive. It uses a message broker (like Redis or RabbitMQ) to queue tasks and optional result backends to track results.

    • Tasks: Python callables executed asynchronously or on schedule.
    • Workers: Long-running processes that execute tasks.
    • Broker: Redis/RabbitMQ transports messages from producers to workers.
    • Result backend: Stores task states and return values.

    Use Celery when you need to send emails, crunch data, integrate external APIs, generate reports, or schedule recurring jobs (cleanups, aggregations) without slowing user requests.

    Celery vs other task queues:
    • Celery is mature, feature-rich (ETA, countdown, retries, chords, chains).
    • RQ: simpler, Redis-only, fewer primitives.
    • Dramatiq/Huey: lightweight alternatives with fewer features.
    Rule of thumb: If you need robust scheduling, routing, time limits, and complex workflows in Django, Celery is often the right choice.

    Syntax (full)

    # 1) Create a Celery app (Django will usually do this in project/celery.py)
from celery import Celery

# Common constructor usage (broker + backend)
app = Celery(
    "project",                     # main name
    broker="redis://localhost:6379/0",   # or amqp://...
    backend="redis://localhost:6379/1",
    include=["jobs.tasks"],         # optional eager import of tasks
)

# You can also load settings via: app.config_from_object("django.conf:settings", namespace="CELERY")

# 2) Define a task (shared_task is preferred inside Django apps)
from celery import shared_task

@shared_task(
    name="jobs.add",               # custom import-safe name
    bind=False,                     # if True, first arg is self (request, retry, etc.)
    serializer="json",             # payload serializer
    ignore_result=False,            # True to skip storing result
    acks_late=False,                # ack only after run completes (safer on crash)
    reject_on_worker_lost=False,    # make broker re-deliver if worker dies
    rate_limit=None,                # e.g., "10/m"
    soft_time_limit=None,           # seconds (graceful)
    time_limit=None,                # seconds (hard kill)
    autoretry_for=(),               # tuple of Exceptions for auto retry
    retry_backoff=False,            # bool|int (exponential backoff)
    retry_backoff_max=None,         # cap for backoff seconds
    retry_jitter=True,              # add randomness to backoff
    retry_kwargs=None               # e.g., {"max_retries": 3}
)
def add(x, y):
    return x + y

# 3) Send task (delay is shorthand for apply_async(args=(...)))
res = add.delay(2, 2)

# 4) Full call signature for apply_async
# Task.apply_async(
#   args=None, kwargs=None, task_id=None, countdown=None, eta=None,
#   expires=None, priority=None, queue=None, exchange=None, routing_key=None,
#   serializer=None, headers=None, compression=None,
#   link=None, link_error=None, shadow=None,
#   retry=None, retry_policy=None,
#   time_limit=None, soft_time_limit=None,
#   **broker_transport_options
# )

# 5) Result API (AsyncResult)
value = res.get(timeout=10, propagate=True)   # blocks until ready or raises
res.ready()        # bool
res.state          # "PENDING"|"STARTED"|"RETRY"|"SUCCESS"|"FAILURE"
res.successful()   # bool
res.failed()       # bool
res.revoke(terminate=False, signal=None)  # cancel/revoke
res.forget()       # drop from backend

    Parameters — full reference for syntax above

    • Celery(...):
      • main: App name (string).
      • broker: Broker URL (e.g., redis://, amqp://).
      • backend: Result backend URL (redis://, rpc://, db+scheme).
      • include: List of modules to import at startup.
      • config_from_object: Load config object/module (e.g., Django settings with namespace).
    • @shared_task / @app.task options:
      • name: Override canonical task name (recommended for stable routing).
      • bind: If True, first arg is self to access self.request, self.retry().
      • base: Custom Task subclass to add hooks/behavior.
      • serializer: Message serializer name (json, pickle, yaml; prefer json).
      • ignore_result: Don’t store results (reduces backend load).
      • acks_late: Ack after task completes; re-queue on worker crash.
      • reject_on_worker_lost: Nack message on worker loss for safe redelivery.
      • rate_limit: Rate string like "5/m", "10/s".
      • soft_time_limit / time_limit: Per-task limits in seconds.
      • autoretry_for: Tuple of Exceptions triggering automatic retries.
      • retry_backoff: True|int; if True uses exponential backoff starting at 1s.
      • retry_backoff_max: Max backoff seconds cap.
      • retry_jitter: Randomize backoff to avoid thundering herd.
      • retry_kwargs: Extra retry args (e.g., {"max_retries": 3, "countdown": 5}).
      • default_retry_delay / max_retries: Defaults when calling self.retry().
      • throws: Iterable of exception classes considered expected (don’t mark FAILURE).
      • track_started: Emit STARTED state updates.
      • queue / priority / exchange / routing_key: Default routing hints.
    • Task.apply_async(...):
      • args, kwargs: Positional/keyword args for the task.
      • task_id: Custom id (UUID string); rarely needed.
      • countdown: Seconds to wait before executing.
      • eta: Exact datetime (aware) to execute.
      • expires: Seconds or datetime after which task is discarded.
      • priority: Broker-dependent priority (0 is highest on some brokers).
      • queue, exchange, routing_key: Route selection.
      • serializer: Override serializer for this send.
      • headers: Custom message headers.
      • compression: e.g., "gzip" (if enabled).
      • link / link_error: Success/error callbacks (signatures).
      • shadow: Alternative task name shown in logs/monitoring.
      • retry: Whether to retry broker send failures.
      • retry_policy: Dict: {"max_retries", "interval_start", "interval_step", "interval_max"}.
      • time_limit / soft_time_limit: Per-invocation limits.
    • AsyncResult:
      • get(timeout, propagate): Return value or raise (with traceback) on failure/timeout.
      • ready(), successful(), failed(): State helpers.
      • state / status: Current state string.
      • revoke(terminate=False): Cancel task (optionally terminate if running).
      • forget(): Remove from backend storage.

    More features you can use now

    • Canvas: Compose tasks with sig.s(), chain, group, chord.
    • Routing: Send to specific queue or set routes in settings.
    • Scheduling: Use countdown/eta or Celery Beat with crontab/timedelta.
    • Limits: rate_limit, soft_time_limit, time_limit to protect dependencies.
    • Retries: autoretry_for + backoff/jitter or manual self.retry().

    Return value

    • Scheduling a task returns AsyncResult (task id + APIs to inspect state, get result, revoke, etc.).
    • Task code can return any JSON-serializable value (or pickle if configured, not recommended).

    2) Setting Up Celery

    Docs: App • Configuration • Worker CLI • Beat CLI

    Full description

    To use Celery with Django, you need:

    1. Celery installed in your environment.
    2. A message broker (Redis or RabbitMQ). Redis is popular for local/dev.
    3. Django integration via a celery.py module and proper settings.
    4. Workers started with the same DJANGO_SETTINGS_MODULE.

    Syntax (full + CLI)

    # Installation
pip install "celery[redis]"  # Redis broker + Celery
# or
pip install celery
pip install redis

# Run Redis locally (Docker)
docker run -d --name redis -p 6379:6379 redis:7-alpine

CELERY_BROKER_URL = "redis://localhost:6379/0"
CELERY_RESULT_BACKEND = "redis://localhost:6379/1"

CELERY_ACCEPT_CONTENT = ["json"]
CELERY_TASK_SERIALIZER = "json"
CELERY_RESULT_SERIALIZER = "json"
CELERY_TIMEZONE = "UTC"
    # project/celery.py
import os
from celery import Celery

os.environ.setdefault("DJANGO_SETTINGS_MODULE", "project.settings")
app = Celery("project")
app.config_from_object("django.conf:settings", namespace="CELERY")
app.autodiscover_tasks()

__all__ = ("celery_app",)

    Parameters — full reference

    • BROKER_URL / RESULT_BACKEND: Transport and storage for messages and results.
    • ACCEPT_CONTENT / SERIALIZER: Prefer JSON; avoid pickle in production.
    • TIMEZONE / ENABLE_UTC: Controls scheduling and timestamps; ensure alignment with Django TIME_ZONE.
    • WORKER_CONCURRENCY: Number of processes/threads; mirrors --concurrency.
    • WORKER_PREFETCH_MULTIPLIER: How many tasks a worker prefetches; tune to reduce starvation.

    • TASK_ACKS_LATE/REJECT_ON_WORKER_LOST: Delivery semantics for crash safety.
    • TASK_TRACK_STARTED: Emits STARTED state (slight overhead).
    • TASK_DEFAULT_QUEUE / TASK_QUEUES / TASK_ROUTES: Queue declarations and static routing.
    • BEAT_SCHEDULE: Dict of periodic schedules (cron/timedelta) + args/kwargs/options.
    • RESULT_EXPIRES: TTL for backend result records.
    • CLI worker options: --pool (prefork|threads|gevent|solo), --autoscale max,min, -Q queues, -n node name.
    • CLI beat options: --scheduler, --pidfile, --logfile.

    3) Writing Your First Task

    Docs: Task/@task • Canvas

    Full description

    Tasks are regular Python functions decorated with @shared_task (recommended for Django apps) or @app.task. You call them with .delay() or .apply_async(). Workers pick them up and execute in the background.

    Syntax (full)

    from celery import shared_task

@shared_task(name="jobs.compute_popular_skills", bind=False)
def compute_popular_skills():
    # ...
    return ["Python", "Django"]

# call
res = compute_popular_skills.delay()
value = res.get()
    # Build signatures explicitly
from celery import chain, group

sig = compute_popular_skills.s()     # mutable signature (prior result prepended in chains)
imm = compute_popular_skills.si()    # immutable signature (no arg injection)

# Call APIs
# .delay(*args, **kwargs) → shorthand for .apply_async(args=args, kwargs=kwargs)
# .apply_async(args=None, kwargs=None, countdown=None, eta=None, queue=None, priority=None,
#              expires=None, serializer=None, headers=None, link=None, link_error=None, retry=None,
#              retry_policy=None, time_limit=None, soft_time_limit=None)

# Synchronous (in-process) execution for debugging
sync_res = compute_popular_skills.apply(args=(), kwargs={})  # executes now
print(sync_res.get())

    Parameters — full reference

    • name: Set a custom task name (useful for routing/versioning).
    • bind: If True, first arg is self, giving access to self.request, self.retry(...), etc.
    • queue (via apply_async): Send to a specific queue.
    • countdown/eta: Schedule in the future.
    • apply(): Run immediately in current process; returns EagerResult/AsyncResult.
    • .s/.si: Build (im)mutable signatures for Canvas workflows.

    Return value

    Any JSON-serializable Python object. Caller gets an AsyncResult that can fetch the return value with .get().

    4) Task Results

    Docs: AsyncResult/GroupResult • Result backend config

    Full description

    Celery tasks can store results to a backend for inspection and retrieval. Common choices are Redis, Database (via django-celery-results), or RPC. Each AsyncResult exposes methods like .status, .ready(), and .get().

    Syntax (full)

    # settings.py
CELERY_RESULT_BACKEND = "redis://localhost:6379/1"

# calling & retrieving
res = some_task.delay(123)
print(res.status)  # PENDING, STARTED, SUCCESS, FAILURE, etc.
value = res.get(timeout=10)  # blocks until ready or timeout
    # AsyncResult / GroupResult / ChainResult quick reference
ar = res                           # AsyncResult
ar.state                           # "PENDING"|"RECEIVED"|"STARTED"|"RETRY"|"SUCCESS"|"FAILURE"
ar.ready()                         # bool
ar.successful(); ar.failed()       # bools
ar.get(timeout=None, propagate=True)  # return or raise
ar.result                          # raw return value or exception instance (if finished)
ar.traceback                       # str or None when failed
ar.forget()                        # remove from backend
ar.revoke(terminate=False)         # cancel

# Collections
# - GroupResult: .get() → list of results; iterate per child.
# - ChainResult: .get() → last result; .results for each link.

    Parameters — full reference

    • RESULT_BACKEND: Redis/RPC/DB; controls persistence and availability of states.
    • State model: PENDING → RECEIVED → STARTED → RETRY|SUCCESS|FAILURE.
    • get(timeout, propagate): If propagate=False, return exception instance instead of raising.
    • result/traceback: Inspect terminal outcome; useful for logs/tests.
    • forget(): Delete stored result to free backend space.
    • ignore_result: Set on task decorator to skip backend writes.

    Return value

    AsyncResult.get() returns the value returned by the task (e.g., list of titles). State APIs return strings/booleans.

    5 short examples (with printed QuerySet outputs)

    Example 1: Check status then get

    5) Error Handling & Retries

    Docs: Task retries • Signals

    Full description

    Celery provides robust retry mechanisms for transient failures. You can use autoretry_for with retry_backoff, or manually call self.retry() when bound (bind=True). Time limits help protect worker resources.

    Syntax (full)

    from celery import shared_task

@shared_task(autoretry_for=(ConnectionError,),
             retry_backoff=True, retry_kwargs={"max_retries": 3})
def fetch_remote_stat():
    # ...
    return 1
    @shared_task(bind=True)
def must_have_skill(self, skill):
    if not exists(skill):
        raise self.retry(countdown=5, max_retries=2)
    return "ok"
    # self.retry signature (common params)
# self.retry(exc=None, countdown=None, eta=None, max_retries=None, **kwargs)
# Notes:
# - Raises celery.exceptions.Retry by default (propagates to caller if eager or get(propagate=True)).
# - Respects task's default max_retries and default_retry_delay when not provided.
# - Works only when bind=True (self available).

    Parameters — full reference

    • autoretry_for: Tuple of exceptions that should trigger automatic retries.
    • retry_backoff: Exponential backoff (True/number) between retries.
    • retry_kwargs: Extra retry kwargs (e.g., max_retries).
    • bind=True: Lets you call self.retry() manually.
    • soft_time_limit/time_limit: Graceful vs hard kill after seconds.
    • acks_late: Only acknowledge after task completes, making it re-queued on crash.
    • self.retry(...): Supports countdown/eta, max_retries, and propagates a Retry exception unless handled.
    • default_retry_delay / max_retries: Per-task defaults used by self.retry.

    Return value

    On success, returns your typical value; on final failure after retries, state becomes FAILURE and .get() raises the exception.

    5 short examples (with printed QuerySet outputs)

    Example 1: Retry if skill is missing

    # jobs/tasks.py
from celery import shared_task
from jobs.models import Skills, JobPost

@shared_task(bind=True, autoretry_for=(ValueError,), retry_backoff=True, retry_kwargs={"max_retries": 1})

                .distinct())

    6) Periodic Tasks

    Docs: Schedules • Beat

    Full description

    Celery Beat is a scheduler that sends tasks at regular intervals or cron-like schedules. Use it for maintenance, reports, cache refreshes, etc. In Django, define schedules in settings or via django-celery-beat (database-stored schedules).

    Syntax (full)

    # settings.py (simple built-in schedule)
from celery.schedules import crontab

CELERY_BEAT_SCHEDULE = {
    "nightly-high-paying-summary": {
        "task": "jobs.tasks.high_paying_titles",
        "schedule": crontab(minute=0, hour=0),  # every midnight UTC
        "args": (85000,),
    },
}
    # run beat
celery -A project beat -l info

# run worker to consume
celery -A project worker -l info
    # Advanced CELERY_BEAT_SCHEDULE entry (with options)
from datetime import timedelta
from celery.schedules import crontab

CELERY_BEAT_SCHEDULE = {
  "skills-refresh": {
    "task": "jobs.tasks.skill_names_sorted",
    "schedule": timedelta(minutes=15),
    "kwargs": {"only_active": True},
    "options": {"queue": "analytics", "expires": 600},
  },
  "ca-6am": {
    "task": "jobs.tasks.titles_in_state",
    "schedule": crontab(minute=0, hour=6, day_of_week="*", day_of_month="*", month_of_year="*"),
    "args": ("CA",),
  },
}

# Timezone
CELERY_TIMEZONE = "UTC"  # or "America/Los_Angeles"; affects crontab/timedelta execution time

    Parameters — full reference

    • crontab(...): Cron-like schedule (minute, hour, day_of_week, day_of_month, month_of_year).
    • schedule: crontab(...) or timedelta; also supports custom schedules.
    • args/kwargs: Passed to the task when scheduled.
    • options: Route/time-control per entry (e.g., queue, priority, expires, time limits).
    • CELERY_TIMEZONE: Ensures crontab runs at expected local time.
    • django-celery-beat: Manage schedules in DB via Admin (install separately).

    Return value

    7) Best Practices (Beginner)

    Docs: Configuration • Canvas

    Full description

    • Structure: Keep tasks in app/tasks.py, keep them small and composable.
    • Idempotency: Safe to re-run. Use checksums, upserts (update_or_create), natural keys to avoid duplicates.
    • Database access: Use select_related/prefetch_related to optimize queries.
    • Retries: Only retry transient errors; cap max_retries.
    • Serialization: Prefer JSON; pass IDs instead of full objects.
    • Observability: Logging, metrics, tracing; consider Flower or Prometheus exporters.

    Syntax (full)

    # Optimized query access in a task
@shared_task
def optimized_titles():
    from jobs.models import JobPost
    qs = (JobPost.objects
          .select_related("author", "location")
          .prefetch_related("skills")
          .order_by("title")
          .values_list("title", flat=True))
    return list(qs)

    Parameters — full reference

    • select_related(...): SQL join to fetch related FK/OneToOne in one query.
    • prefetch_related(...): Separate query + in-Python join for ManyToMany/Reverse FK.
    • update_or_create(defaults=..., **lookup): Idempotent upsert pattern.
    • rate_limit (task option): Throttle noisy tasks, e.g., @shared_task(rate_limit="10/m").
    • acks_late + reject_on_worker_lost: Safer delivery for critical tasks.
    • apply_async(queue=..., priority=...): Simple routing and prioritization.

    More tools to adopt early

    • Canvas: Use chain/group for fan-in/fan-out.
    • Monitoring: Flower for live task inspection.
    • Testing: CELERY_TASK_ALWAYS_EAGER=True and CELERY_TASK_EAGER_PROPAGATES=True.

    Return value

    Tasks should return lightweight structures: IDs, titles, counts, or small dicts — avoid returning huge payloads via the result backend.

    Debug tips

    • Set CELERY_TASK_ALWAYS_EAGER=True to run tasks inline when debugging logic.
    • Use @shared_task(bind=True) and log self.request.id, args.
    • Wrap external calls with retries and tight timeouts.

    Syntax (full control API)

    # Inside Django shell or a management command
from celery import current_app as app

# Inspect workers
i = app.control.inspect()
i.ping()             # health checks
i.active()           # tasks currently executing
i.reserved()         # tasks reserved by workers
i.registered()       # registered task names
i.scheduled()        # ETA/countdown tasks scheduled

# Control operations

app.control.broadcast("pool_restart")           # e.g., restart worker pool
app.control.broadcast("rate_limit", arguments={"task_name": "jobs.tasks.title_by_slug", "rate": "10/m"})

# Revoke with terminate (dangerous if not idempotent)
app.control.revoke("<task_id>", terminate=True)

    Parameters — quick reference

    • inspect(): Returns a probe object; methods return dict keyed by worker hosjobpost.
    • revoke(task_id, terminate, signal): Cancel a task; terminate sends a signal to kill a running task.
    • broadcast(command, arguments): Send low-level control commands to all workers.
    • Idempotency: If you terminate tasks, ensure they’re idempotent or guarded to avoid partial writes.

    API Cheat Sheet

    Docs: Reference Index

    SUMMARY

    End-to-end at a glance — the full flow from local dev to production.

    1. Configure Celery in Django: Create the Celery app, load settings, set broker_url and result_backend.
    2. Declare tasks: Keep them small and idempotent; choose @shared_task vs @app.task; set time limits, rate limits, and retries.
    3. Start infrastructure: Run Redis (or your chosen broker/backend) locally or via Docker.
    4. Run processes: Start worker(s) for execution and optionally beat for periodic schedules.
    5. Produce work: From Django code or shell, enqueue with .delay()/.apply_async() using small JSON-safe args.
    6. Observe & control: Use inspect() for health/queues; revoke tasks; restart pools; optionally run Flower and emit metrics.
    7. Compose workflows: Build chains, groups, and chords; attach success/error callbacks.
    8. Route & prioritize: Define queues/exchanges; map task names to lanes; set priorities to isolate heavy work.
    9. Harden & scale: Tune acks_late, prefetch, concurrency, rate limits, and soft/hard time limits; consider autoscaling.
    10. Serialize & trace: Prefer JSON serializers; add headers for tracing/correlation (e.g., trace_id).
    11. Test & CI: Use task_always_eager/task_eager_propagates in unit tests; run integration tests against a real worker.
    12. Deploy: Configure env vars, health checks, and result expiration; keep secrets out of code.
    13. Troubleshoot: Apply retries/backoff wisely, inspect stuck tasks, handle dead letters, and roll forward/back safely.

    Tip: Each step below includes a compact snippet you can copy into your project.

    Skeletal Structure

    +--------------------+         +---------------------+         +------------------+
| Django (caller)   | --msg--> | Broker / Queues     | --msg--> | Worker / Pool    |
+-------------------+         +---------------------+         +------------------+
         |                             ^      ^                          |
         | AsyncResult.get()/state     |      | Beat (periodic)          v
         v                             |      +-- routes / priorities  Result
  Monitoring / Control                 |                                 Backend
 (inspect, revoke, Flower) <-----------+                             (optional)

Flow:
1) Django code enqueues a task via delay()/apply_async().
2) Message lands in a queue on the broker (optionally routed/prioritized).
3) Worker processes poll the queue; a worker executes the task.
4) Result (if stored) is written to the result backend; client checks AsyncResult.
5) Beat schedules periodic tasks by publishing to the broker on schedule.
6) Operators observe/control via inspect(), revoke(), Flower, and logs/metrics.

Notes:
- Prefer idempotent tasks if using acks_late / retries.
- Keep payloads JSON-serializable; pass IDs/strings, not ORM instances.
- Consider a DLQ (dead-letter queue) in production for poison messages.

    App & Task

    Set up a Celery application and declare a task with common reliability options; this is the core unit you call from Django views, signals, or management commands.

    from celery import Celery, shared_task  # import the app factory and decorator

app = Celery("project", broker="redis://localhost:6379/0", backend="redis://localhost:6379/1")  # create app with broker/backend
app.config_from_object("django.conf:settings", namespace="CELERY")  # load Django settings with CELERY_* names

@shared_task(  # declare a task (registered as jobs.add)
    name="jobs.add",           # explicit task name for routing/CLI
    bind=False,                 # if True, first arg is 'self' (Task instance)
    ignore_result=False,        # store result in backend (True to skip storing)
    rate_limit=None,            # e.g., "10/m" to throttle per worker process
    soft_time_limit=None,       # graceful timeout (raises SoftTimeLimitExceeded)
    time_limit=None,            # hard timeout enforced by the worker
    autoretry_for=(),           # exceptions to auto-retry on
    retry_backoff=False,        # exponential backoff (bool or integer base)
    retry_kwargs=None           # extra retry args, e.g. {"max_retries": 3}
)
def add(x, y):                  # task body (keep side effects idempotent)
    return x + y                # return value is stored in result backend

    @task (bound to a Celery app) vs @shared_task

    @shared_task registers on the current app and is ideal for reusable Django apps (autodiscovered). @app.task binds to a specific Celery instance you create and control. Use it when you own the app module (microservice-style) or want per-app isolation.

    Docs: Tasks • Task ref
    from celery import Celery  # explicit app instance

app = Celery("project", broker="redis://localhost:6379/0", backend="redis://localhost:6379/1")

@app.task(                    # bind this task to 'app' (not the implicit current_app)
    name="jobs.mul",          # stable name for routing/ops
    bind=True,                 # access self (Task) → self.request, self.retry
    base=None,                 # custom Task base class if you subclass Task
    ignore_result=False,       # True to skip storing results
    rate_limit=None,           # e.g. "10/m" per worker process
    acks_late=False,           # at-least-once delivery; requires idempotent task
    soft_time_limit=None,      # graceful timeout (raises SoftTimeLimitExceeded)
    time_limit=None,           # hard kill enforced by worker
    autoretry_for=(ValueError,),  # exceptions to auto-retry on
    retry_backoff=True,        # 1s, 2s, 4s, ...
    retry_backoff_max=None,    # cap backoff seconds (optional)
    retry_jitter=True,         # add random jitter to backoff
    retry_kwargs={"max_retries": 3},  # cap retries
    max_retries=None,          # legacy alternative cap (optional)
    default_retry_delay=None,  # per-task default delay (optional)
    serializer=None            # override message serializer for this task
)
def mul(self, x, y):
    # access request context via self.request (id, retries, eta, etc.)
    if x == 0:
        # demonstrate manual retry with backoff
        raise self.retry(countdown=2 ** self.request.retries)
    return x * y

# When to use which:
# - Use @shared_task in Django reusable apps and when relying on autodiscovery.
# - Use @app.task in a single service or when you want strict coupling to a specific Celery() instance.

    Dispatch

    Send tasks to workers immediately or with fine-grained options like countdown, routing, priorities, and callbacks.

    # Shorthand fire-and-forget (no extra options)
res = add.delay(2, 2)  # returns AsyncResult immediately

# Full options via apply_async(...)
res = add.apply_async(
    args=(2, 2), kwargs={},              # positional/keyword args
    countdown=5, eta=None, expires=None, # delay, absolute schedule time, expiry
    queue="default", priority=None,     # route to a queue; priority 0 is highest (Redis)
    serializer=None, headers=None,       # override serializer; add custom headers
    link=None, link_error=None,          # success/error callbacks (signatures)
    retry=None, retry_policy=None,       # legacy retry flags (prefer task-level autoretry)
    time_limit=None, soft_time_limit=None# per-call timeouts
)

    delay vs apply_async

    MethodWhen to useOptions
    add.delay(...) Fire-and-forget, defaults OK None (no routing/ETA/headers)
    add.apply_async(...) Need control over countdown/ETA, queue, priority, headers, callbacks countdown, eta, expires, queue, priority, headers, link, link_error, timeouts

    Results

    Retrieve results, check status, and control lifecycle (revoke/forget) of individual task executions.

    res.get(timeout=10, propagate=True)  # wait up to 10s; raise task exception if failed
res.ready(); res.successful(); res.failed()  # boolean status helpers
res.state; res.result; res.traceback         # raw fields for introspection
res.revoke(terminate=False)                  # cancel future execution or stop if pending
res.forget()                                 # delete stored result from backend

    Canvas

    Compose tasks into pipelines (chain), fan-out (group), and fan-in (chord) workflows with signatures.

    from celery import chain, group, chord  # canvas primitives
sig = add.s(1, 2)    # mutable signature (downstream can alter args)
imm = add.si(1, 2)   # immutable signature (args locked)
chain(add.s(1,2), add.s(3)).delay()  # 1) add(1,2) -> 2) add(result, 3)
group(add.s(i, i) for i in range(3)).apply_async()  # parallel fan-out
chord(group(add.s(1,1), add.s(2,2)))(add.s(3))  # wait for group, then add(sum, 3)

    Retries

    Retry transient failures with self.retry for backoff and limits; bind=True gives access to the Task instance.

    from celery import shared_task

@shared_task(bind=True, max_retries=3)  # allow self.retry; cap retries at 3
def fetch_url(self, url):
    try:
        return http_get(url)  # your network call
    except Exception as exc:  # transient error
        # exponential backoff using attempt count
        raise self.retry(exc=exc, countdown=2 ** self.request.retries)

    Callbacks

    Attach success/error callbacks to a dispatch via link and link_error; use signatures.

    from celery import signature

on_ok = signature("jobs.add", args=(1, 1))     # runs on success
on_err = signature("jobs.add", args=(0, 0))    # runs on failure
add.apply_async(args=(2, 2), link=on_ok, link_error=on_err)

    Group & ResultSet

    Manage multiple results: wait for all values or inspect each child result as it completes.

    from celery import group

g = group(add.s(i, i) for i in range(5))()  # dispatch group
values = g.get(disable_sync_subtasks=False)  # wait for all -> list
for r in g.results:                          # each child AsyncResult
    print(r.id, r.state)

# res.children  # list of child results (may be nested)

    Signals

    Observe task lifecycle for logging/metrics without changing task code.

    from celery.signals import task_prerun, task_postrun, task_failure

@task_prerun.connect
def on_prerun(task_id, task, args, kwargs, **_):
    print(f"starting {task.name} {task_id}")

@task_postrun.connect
def on_postrun(task_id, task, retval, state, **_):
    print(f"finished {task.name} {task_id} -> {state}")

@task_failure.connect
def on_failure(task_id, exception, traceback, sender, **_):
    print(f"failed {sender.name} {task_id}: {exception}")

    Testing (Eager)

    Run tasks synchronously during tests by enabling eager mode; exceptions propagate immediately.

    from celery import Celery

app = Celery("project")
app.conf.update(task_always_eager=True, task_eager_propagates=True)

res = add.delay(2, 2)  # executes inline
assert res.result == 4

    Common pitfalls

    • Result backend: res.get() requires a configured result_backend; otherwise results aren’t stored.
    • Serialization: Task args/returns must be serializer-friendly (JSON by default). Send IDs/strings, not ORM objects.
    • Idempotency: Use acks_late=True only when tasks can safely be retried without double effects.
    • Time limits: Set soft_time_limit and handle SoftTimeLimitExceeded for clean shutdowns.
    • ETAs/clock: Beat/ETA scheduling assumes correct server time/timezone; set timezone in config.
