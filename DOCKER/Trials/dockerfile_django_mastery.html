    1) FROM - Base Image

    Docs: FROM instruction

    Specifies the base image for your Docker container.

    Full description

    FROM initializes a new build stage and sets the base image for subsequent instructions. For Django projects, this is typically a Python image. The image can be any valid image from Docker Hub, a private registry, or a previously built stage in a multi-stage build. This must be the first instruction in your Dockerfile (except for ARG instructions that can appear before FROM).

    Syntax

    FROM [--platform=<platform>] <image>[:<tag>] [AS <name>]
FROM [--platform=<platform>] <image>[@<digest>] [AS <name>]

    Parameters description

    • image: The base image name (e.g., python, alpine, ubuntu)
    • tag: Specific version of the image (e.g., 3.11, latest, slim). If omitted, defaults to 'latest'
    • digest: SHA256 hash for exact image identification (more secure than tags)
    • --platform: Target platform for multi-platform builds (e.g., linux/amd64, linux/arm64)
    • AS name: Assigns a name to this build stage for multi-stage builds

    Return value

    Sets up the filesystem and environment for the container. Returns the base layer that all subsequent instructions will build upon.

    2) WORKDIR - Working Directory

    Docs: WORKDIR instruction

    Sets the working directory for any RUN, CMD, ENTRYPOINT, COPY and ADD instructions that follow.

    Full description

    WORKDIR sets the working directory for subsequent instructions in the Dockerfile. If the directory doesn't exist, it will be created. You can use WORKDIR multiple times in a Dockerfile, and if a relative path is provided, it will be relative to the previous WORKDIR instruction. This is essential for Django projects to organize your application code properly within the container.

    Syntax

    WORKDIR /path/to/workdir

    Parameters description

    • path: Absolute or relative path to the working directory
    • Relative paths: Resolved relative to the previous WORKDIR instruction
    • Environment variables: Can be used in the path if set with ENV
    • Auto-creation: Directory is created automatically if it doesn't exist

    Return value

    Changes the current working directory context for all subsequent instructions. Also sets the working directory for the running container.

    5 short examples with Django project structure

    Step 1/1 : WORKDIR /app ---> Running in 1a2b3c4d5e6f Removing intermediate container 1a2b3c4d5e6f ---> f6e5d4c3b2a1 # Creates /app directory # All subsequent commands run from /app # Standard Django project location

    Example 2: Project-specific directory

    WORKDIR /opt/jobpost_project
    Step 1/1 : WORKDIR /opt/jobpost_project ---> Running in 2b3c4d5e6f7g Removing intermediate container 2b3c4d5e6f7g ---> g7f6e5d4c3b2 # Creates /opt/jobpost_project # Named after our JobPost Django model # Good for project identification

    3) COPY - Copy Files

    Docs: COPY instruction

    Copies files or directories from the build context to the container filesystem.

    Full description

    COPY transfers files and directories from your local build context (typically your project directory) into the Docker image. It's preferred over ADD for simple file copying because it's more predictable and explicit. COPY preserves file metadata and can copy multiple sources to a destination. For Django projects, this is used to copy your source code, requirements.txt, static files, and configuration files into the container.

    Syntax

    COPY [--chown=<user>:<group>] [--chmod=<perms>] <src>... <dest>
COPY [--chown=<user>:<group>] [--chmod=<perms>] ["<src>",... "<dest>"]

    Parameters description

    • src: Source file or directory path relative to build context
    • dest: Destination path in the container (absolute or relative to WORKDIR)
    • --chown: Set ownership of copied files (user:group format)
    • --chmod: Set permissions of copied files (octal format like 755)
    • Wildcards: Support for * and ? patterns in source paths
    • Multiple sources: Can copy multiple files/directories to one destination

    Return value

    Creates new filesystem layers with the copied files. Returns the modified container filesystem with your files added at the specified destination.

    5 short examples with Django files

    Example 1: Copy requirements.txt

    COPY requirements.txt .
    Step 1/1 : COPY requirements.txt . ---> 1a2b3c4d5e6f # Copied requirements.txt to WORKDIR # File: /app/requirements.txt (if WORKDIR is /app) # Use case: Install dependencies first for caching

    Example 2: Copy Django project

    COPY . /app/
    Step 1/1 : COPY . /app/ ---> 2b3c4d5e6f7g # Copied entire project to /app/ # Includes: models.py, views.py, settings.py, etc. # Contains JobPost, Skills, Author, Location models

    Example 3: Copy with ownership

    COPY --chown=django:django . /app/
    Step 1/1 : COPY --chown=django:django . /app/ ---> 3c4d5e6f7g8h # Copied with django user ownership # Owner: django:django # Prevents permission issues in production

    Example 4: Copy specific Django files

    COPY manage.py models.py views.py ./
    Step 1/1 : COPY manage.py models.py views.py ./ ---> 4d5e6f7g8h9i # Copied: manage.py, models.py (with JobPost model), views.py # Destination: current WORKDIR # Selective copying for minimal layers

    Example 5: Copy with permissions

    COPY --chmod=755 scripts/ /app/scripts/
    Step 1/1 : COPY --chmod=755 scripts/ /app/scripts/ ---> 5e6f7g8h9i0j # Copied scripts directory with execute permissions # Permissions: 755 (rwxr-xr-x) # Good for deployment and migration scripts

    4) RUN - Execute Commands

    Docs: RUN instruction

    Executes commands in a new layer on top of the current image and commits the results.

    Full description

    RUN executes any commands during the build process and creates a new layer with the results. Each RUN instruction creates a new layer, so it's important to combine commands efficiently. For Django projects, RUN is commonly used to install system packages, Python dependencies, run migrations, collect static files, and set up the application environment. Use shell form for complex commands and exec form for precise control.

    Syntax

    # Shell form
RUN <command>

# Exec form  
RUN ["executable", "param1", "param2"]

    Parameters description

    • Shell form: Commands run in a shell (/bin/sh -c on Linux). Supports shell features like pipes, variables
    • Exec form: Commands run directly without shell processing. More secure and predictable
    • Layer creation: Each RUN creates a new filesystem layer that can be cached
    • Command chaining: Use && to combine commands in one layer for efficiency
    • Cache invalidation: Changes in RUN commands invalidate Docker cache for subsequent layers

    Return value

    Creates a new image layer with the results of the command execution. Returns exit code 0 for success, non-zero for failure (stops build).

    5) ENV - Environment Variables

    Docs: ENV instruction

    Sets environment variables that persist when a container is run from the image.

    Full description

    ENV sets environment variables that are available during the build process and when the container runs. These variables can be used by subsequent instructions and are inherited by all child processes. For Django projects, ENV is crucial for setting database URLs, debug mode, secret keys, and other configuration that varies between environments. Variables set with ENV persist in the final image and can be overridden at runtime.

    Syntax

    ENV <key>=<value> ...
ENV <key> <value>

    Parameters description

    • key=value format: Multiple variables can be set in one instruction
    • key value format: Single variable, value can contain spaces
    • Variable substitution: Can reference previously set environment variables
    • Persistence: Variables persist in the final image and running containers
    • Override capability: Can be overridden with docker run -e or docker-compose environment

    Return value

    Sets environment variables in the image. Returns the modified environment context available to subsequent instructions and container runtime.

    Example 1: Django settings module

    6) EXPOSE - Expose Ports

    Docs: EXPOSE instruction

    Informs Docker that the container listens on the specified network ports at runtime.

    Full description

    EXPOSE serves as documentation between the person who builds the image and the person who runs the container about which ports are intended to be published. It doesn't actually publish the port - you still need to use -p flag with docker run or ports in docker-compose. For Django applications, this typically includes the web server port (8000 for development, 80/443 for production) and any additional services your application provides.

    Syntax

    EXPOSE <port> [<port>/<protocol>...]

    Parameters description

    • port: Port number that the container will listen on
    • protocol: TCP (default) or UDP. Format: port/protocol
    • Multiple ports: Can specify multiple ports in one instruction
    • Documentation purpose: Primarily serves as documentation, doesn't publish ports
    • Default protocol: TCP is assumed if no protocol is specified

    Return value

    Documents the intended port usage. Returns metadata about which ports the container expects to use for network communication.

    5 short examples with Django services

    Example 1: Django development server

    EXPOSE 8000
    Step 1/1 : EXPOSE 8000 ---> 1a2b3c4d5e6f # Documented port 8000 for Django dev server # Access JobPost admin at localhost:8000/admin # python manage.py runserver 0.0.0.0:8000

    Example 2: Production web server

    EXPOSE 80 443
    Step 1/1 : EXPOSE 80 443 ---> 2b3c4d5e6f7g # Port 80: HTTP traffic # Port 443: HTTPS traffic # Production Django with nginx/apache # JobPost API accessible via standard ports

    Example 3: Multiple Django services

    EXPOSE 8000 8001 8002
    Step 1/1 : EXPOSE 8000 8001 8002 ---> 3c4d5e6f7g8h # Port 8000: Main Django app (JobPost views) # Port 8001: Django admin interface # Port 8002: API endpoint for mobile apps # Multiple service endpoints

    Example 4: Custom protocol specification

    EXPOSE 8000/tcp 8001/udp
    Step 1/1 : EXPOSE 8000/tcp 8001/udp ---> 4d5e6f7g8h9i # Port 8000: TCP for HTTP Django server # Port 8001: UDP for custom real-time features # Mixed protocol support for advanced Django apps

    Example 5: Django with additional services

    EXPOSE 8000 5555 6379
    Step 1/1 : EXPOSE 8000 5555 6379 ---> 5e6f7g8h9i0j # Port 8000: Django web server (JobPost app) # Port 5555: Celery Flower monitoring # Port 6379: Redis for caching/sessions # Complete Django stack with monitoring

    7) CMD - Default Command

    Docs: CMD instruction

    Provides defaults for executing a container, including the executable and arguments.

    Full description

    CMD specifies the default command to run when the container starts. Unlike RUN which executes during build time, CMD runs when the container is started. There can only be one CMD instruction in a Dockerfile - if multiple are listed, only the last one takes effect. For Django projects, CMD typically starts the web server, runs migrations, or starts a specific Django management command. CMD can be overridden when running the container.

    Syntax

    # Exec form (preferred)
CMD ["executable","param1","param2"]

# Shell form
CMD command param1 param2

# As default parameters to ENTRYPOINT
CMD ["param1","param2"]

    Parameters description

    • Exec form: Recommended. Runs executable directly without shell processing
    • Shell form: Runs command in shell (/bin/sh -c). Supports shell features but less efficient
    • Parameter form: Provides default arguments to ENTRYPOINT
    • Override capability: Can be overridden by docker run arguments
    • Single instruction: Only the last CMD in Dockerfile takes effect

    Return value

    Sets the default command for container execution. Returns the process that will run when the container starts, unless overridden at runtime.

    8) ENTRYPOINT - Container Entry Point

    Docs: ENTRYPOINT instruction

    Configures a container that will run as an executable with a fixed entry point.

    Full description

    ENTRYPOINT allows you to configure a container that will run as an executable. Unlike CMD, ENTRYPOINT cannot be overridden by command line arguments to docker run, though it can be overridden with --entrypoint flag. When both ENTRYPOINT and CMD are present, CMD provides default arguments to ENTRYPOINT. For Django projects, ENTRYPOINT is useful for creating containers that always run Django-specific initialization or wrapper scripts before executing the main command.

    Syntax

    # Exec form (preferred)
ENTRYPOINT ["executable", "param1", "param2"]

# Shell form
ENTRYPOINT command param1 param2

    Parameters description

    • Exec form: Preferred form. Executable runs directly without shell
    • Shell form: Command runs in shell, but prevents CMD and run command line arguments from being used
    • Fixed execution: Cannot be overridden by docker run arguments (unlike CMD)
    • CMD interaction: CMD becomes default arguments to ENTRYPOINT when both are present
    • Override option: Can only be overridden with --entrypoint docker run flag

    Return value

    Sets the fixed entry point for container execution. Returns the executable that will always run first when the container starts.

    9) ARG - Build Arguments

    Docs: ARG instruction

    Defines variables that users can pass at build-time to the builder with docker build --build-arg.

    Full description

    ARG defines build-time variables that can be passed to the docker build command using --build-arg flag. Unlike ENV, ARG variables are only available during the build process and do not persist in the final image. For Django projects, ARG is useful for specifying Python versions, Django versions, environment-specific configurations, or build optimization flags that vary between different build scenarios.

    Syntax

    ARG <name>[=<default value>]

    Parameters description

    • name: Variable name that can be referenced with ${name} syntax
    • default value: Optional default value used if no --build-arg is provided
    • Build-time only: Variables are only available during build, not in running container
    • Multiple definitions: Can define multiple ARG instructions
    • Scope: ARG before FROM applies to FROM instruction only

    Return value

    Makes build-time variables available for use in subsequent instructions. Returns the variable value for use in FROM, RUN, COPY, and other instructions.

    5 short examples with Django build arguments

    Example 1: Python version selection

    ARG PYTHON_VERSION=3.11
FROM python:${PYTHON_VERSION}-slim
    $ docker build --build-arg PYTHON_VERSION=3.10 . Step 1/2 : ARG PYTHON_VERSION=3.11 Step 2/2 : FROM python:3.10-slim ---> Using Python 3.10 image # Build with Python 3.10 instead of default 3.11 # Allows testing JobPost models with different Python versions # Flexible base image selection

    Example 2: Django environment configuration

    ARG ENVIRONMENT=development

    $ docker build --build-arg ENVIRONMENT=production .     Example 3: Django version and extras

    10) ADD - Advanced Copy

    Copies files, directories, or remote URLs and can auto-extract compressed archives.

    Full description

    12) USER - Run as User

    • File permissions: Affects file creation and access permissions

    14) HEALTHCHECK - Health Monitoring

    Docs: HEALTHCHECK instruction

    Tells Docker how to test a container to check that it's still working.

    Full description

    HEALTHCHECK allows you to configure a check that Docker will run periodically to determine if your container is healthy. The health status is shown in docker ps and can be used by orchestrators like Docker Swarm or Kubernetes for automatic container restarts and load balancing decisions. For Django applications, health checks typically test if the web server is responding, the database is accessible, and critical application endpoints are working.

    Syntax

    • NONE: Disables health checking inherited from base image

    Return value

    Configures health monitoring for the container. Returns exit code 0 for healthy, 1 for unhealthy, 2 for reserved (don't use).

    15) Multi-stage Builds

    Docs: Multi-stage builds

    Use multiple FROM statements to create separate build stages and copy artifacts between them.

    Full description

    Multi-stage builds allow you to use multiple FROM instructions in a single Dockerfile, creating separate build environments and copying only the necessary artifacts to the final image. This dramatically reduces final image size by excluding build tools, dependencies, and intermediate files. For Django projects, this pattern is excellent for separating dependency installation, static file compilation, and runtime environments while keeping production images minimal and secure.

    Syntax

    FROM image:tag AS stage_name
# Build instructions for this stage

FROM another_image:tag AS another_stage  
# More build instructions

FROM final_image:tag
COPY --from=stage_name /path/to/artifact /final/path

    Parameters description

    • AS stage_name: Names a build stage for later reference
    • --from=stage_name: Copies files from a named stage
    • --from=index: Copies from stage by numeric index (0, 1, 2...)
    • Multiple stages: Each FROM starts a new build stage
    • Final stage: Last FROM determines the final image base
    • Stage independence: Each stage has its own filesystem and context

    Return value

    Creates optimized final image with only necessary artifacts. Returns a production-ready container without build dependencies and tools.

    16) Best Practices for Django Dockerfiles

    Docs: Docker best practices

    Essential practices for building secure, efficient, and maintainable Django containers.

    Full description

    Following Docker and Django best practices ensures your containers are secure, performant, and maintainable. Key areas include image optimization (layer caching, minimal base images), security practices (non-root users, minimal attack surface), Django-specific optimizations (static files, database connections), and operational considerations (health checks, logging, secrets management). These practices are critical for production Django deployments.

    Optimization Strategies

    • Layer caching: Order instructions from least to most frequently changing
    • Minimal base images: Use -slim or -alpine variants when possible
    • Multi-stage builds: Separate build and runtime environments
    • Dependency optimization: Copy requirements.txt first for better caching
    • Static file handling: Use collectstatic in build, serve with nginx in production

    Security Practices

    • Non-root execution: Always run Django as non-root user
    • Secrets management: Never embed secrets in images, use environment variables or secrets management
    • Minimal packages: Only install necessary system packages
    • Security scanning: Integrate vulnerability scanning in CI/CD
    • Read-only filesystem: Use read-only containers when possible

FROM python:3.11-slim AS security-scan

RUN safety check -r requirements.txt
RUN pip-audit -r requirements.txt

# Copy code and scan for security issues
COPY . /tmp/src
RUN bandit -r /tmp/src -f json -o /tmp/security-report.json

# Production stage
FROM python:3.11-slim AS production

# Minimal system packages
RUN apt-get update && apt-get install -y --no-install-recommends \
    postgresql-client \
    && rm -rf /var/lib/apt/lists/* \
    && apt-get purge -y --auto-remove

# Create restricted user
RUN groupadd -g 1001 django \
    && useradd -u 1001 -g django -s /sbin/nologin -d /app django

# Install Python packages as root, then switch
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt \
    && pip cache purge
