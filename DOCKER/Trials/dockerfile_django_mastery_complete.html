    1) Overview & Format

    Docs: Dockerfile Overview

    Docker can build images automatically by reading instructions from a Dockerfile.

    Full description

    A Dockerfile is a text document that contains all the commands a user could call on the command line to assemble an image. Using docker build users can create an automated build that executes several command-line instructions in succession. For Django applications, Dockerfiles define the complete runtime environment including Python version, dependencies, application code, and configuration needed to run your JobPost management system.

    Syntax

    # Comment
INSTRUCTION arguments

    Parameters description

    • Comments: Lines starting with # are treated as comments (except parser directives)
    • Instructions: Case-insensitive but convention is UPPERCASE
    • Arguments: Parameters passed to the instruction
    • Line continuation: Use backslash (\) at end of line to continue instruction on next line
    • Whitespace: Leading and trailing whitespace is ignored

    Return value

    Dockerfile parsing creates a build context and instruction sequence that Docker uses to build container images layer by layer.

    5 short examples with Django Dockerfile format

    Example 1: Basic format

    # Django JobPost Application
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
    # Valid Dockerfile format # Comment explains purpose # Instructions in logical order # Ready for JobPost Django app build

    Example 2: Line continuation

    RUN apt-get update && apt-get install -y \
    postgresql-client \
    redis-tools \
    && rm -rf /var/lib/apt/lists/*
    # Multi-line RUN instruction # Installs PostgreSQL client for JobPost database # Installs Redis tools for caching # Cleans up package lists

    Example 3: Multiple instructions

    ENV DJANGO_SETTINGS_MODULE=myproject.settings.production
ENV PYTHONUNBUFFERED=1
ENV DATABASE_URL=postgresql://user:pass@db:5432/jobpost_db
    # Multiple ENV instructions # Django settings module for production # Python unbuffered output # Database URL for JobPost models

    Example 4: Case insensitive

    from python:3.11-slim
copy requirements.txt .
run pip install -r requirements.txt
    # Lowercase instructions work but not recommended # Convention is UPPERCASE for better readability # Functions same as uppercase versions

    Example 5: Complex Django structure

    # Multi-service Django JobPost application
FROM python:3.11-slim AS base
LABEL maintainer="dev@company.com" \
      description="Django JobPost Management System"
ENV PYTHONUNBUFFERED=1
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
EXPOSE 8000
CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
    # Complete Django Dockerfile structure # Base image, metadata, environment # Dependencies, source code, networking # Default command to run JobPost server

    2) Parser Directives

    Docs: Parser Directives

    Special comment-like instructions that affect how the Dockerfile is parsed.

    Full description

    Parser directives are optional, and affect the way in which subsequent lines in a Dockerfile are handled. Parser directives do not add layers to the build, and will not be shown as a build step. Parser directives are written as a special type of comment in the form # directive=value. A single directive may only be used once. For Django projects, parser directives help configure build behavior, escape characters for Windows paths, and syntax compatibility.

    Syntax

    # directive=value

    Parameters description

    • directive: One of syntax, escape, or check
    • value: Configuration value for the directive
    • Position: Must appear before any builder instruction (including FROM)
    • Case insensitive: Both directive and value are case insensitive
    • Once only: Each directive can only be used once per Dockerfile

    Return value

    Configures Dockerfile parsing behavior. Returns modified parser state that affects how subsequent instructions are interpreted.

    syntax

    Example 1: BuildKit syntax

    # syntax=docker/dockerfile:1
FROM python:3.11-slim
    # Uses latest Dockerfile syntax features # Enables BuildKit advanced features # Better caching and parallel builds for Django

    Example 2: Specific syntax version

    # syntax=docker/dockerfile:1.4
FROM python:3.11-slim
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install django==4.2.0
    # Uses Dockerfile syntax version 1.4 # Enables cache mounts for pip # Faster Django dependency installation

    escape

    Example 3: Windows path escape

    # escape=`
FROM mcr.microsoft.com/windows/servercore:ltsc2019
COPY jobpost_app C:`\app`\jobpost
WORKDIR C:`\app
    # Uses backtick as escape character # Useful for Windows containers # Handles Windows path separators correctly

    Example 4: Alternative escape in Linux

    # escape=`
FROM python:3.11-slim
RUN echo "JobPost Django app" > /tmp/info.txt `
    && ls -la /tmp/
    # Backtick for line continuation # Alternative to backslash # Useful when backslash has special meaning

    check

    Example 5: Skip build checks

    # check=skip=SecretsUsedInArgOrEnv
ARG SECRET_KEY=default-secret-for-development
FROM python:3.11-slim
ENV SECRET_KEY=${SECRET_KEY}
    # Skips security check for secrets in ENV # Useful for development Dockerfiles # Production should use proper secrets management

    3) Environment Replacement

    Docs: Environment replacement

    Environment variables defined with ENV can be used in certain Dockerfile instructions.

    Full description

    Environment variables (declared with the ENV statement) can also be used in certain instructions as variables to be interpreted by the Dockerfile. Environment variables are notated in the Dockerfile either with $variable_name or ${variable_name}. The ${variable_name} syntax also supports standard bash modifiers. For Django applications, this is essential for configuring database URLs, debug settings, and other environment-specific values.

    Syntax

    $variable_name
${variable_name}
${variable_name:-default_value}
${variable_name:+value_if_set}

    Parameters description

    • $variable_name: Simple variable substitution
    • ${variable_name}: Explicit variable substitution (preferred)
    • :-default_value: Use default_value if variable is unset or empty
    • :+value_if_set: Use value_if_set if variable is set and non-empty
    • Supported instructions: ADD, COPY, ENV, EXPOSE, FROM, LABEL, STOPSIGNAL, USER, VOLUME, WORKDIR, ONBUILD

    Return value

    Substitutes environment variable values into instruction arguments during build time.

    5 short examples with Django environment variables

    Example 1: Django settings configuration

    ENV DJANGO_ENV=production
ENV SETTINGS_MODULE=myproject.settings.${DJANGO_ENV}
ENV DEBUG=False
    # SETTINGS_MODULE becomes: myproject.settings.production # Environment-specific Django settings # JobPost models use production configuration

    Example 2: Database URL with defaults

    ENV DB_HOST=localhost
ENV DB_NAME=jobpost_db
ENV DATABASE_URL=postgresql://user:pass@${DB_HOST}:5432/${DB_NAME:-django_db}
    # DATABASE_URL becomes: postgresql://user:pass@localhost:5432/jobpost_db # Uses DB_NAME if set, falls back to django_db # JobPost models connect to specified database

    Example 3: Conditional Django features

    ENV ENABLE_DEBUG_TOOLBAR=true
ENV DEBUG_APPS=${ENABLE_DEBUG_TOOLBAR:+debug_toolbar,}
ENV INSTALLED_APPS=django.contrib.admin,${DEBUG_APPS}myapp
    # If ENABLE_DEBUG_TOOLBAR is set: debug_toolbar, is added # INSTALLED_APPS becomes: django.contrib.admin,debug_toolbar,myapp # Conditional Django app installation for development

    Example 4: Port and workdir configuration

    ENV APP_PORT=8000
ENV APP_NAME=jobpost
WORKDIR /app/${APP_NAME}
EXPOSE ${APP_PORT}
    # WORKDIR becomes: /app/jobpost # EXPOSE becomes: 8000 # Environment-driven configuration for JobPost app

    Example 5: User and permissions

    ENV APP_USER=django
ENV APP_UID=1000
RUN groupadd -g ${APP_UID} ${APP_USER} && \
    useradd -u ${APP_UID} -g ${APP_USER} ${APP_USER}
USER ${APP_USER}
    # Creates user 'django' with UID 1000 # USER instruction switches to 'django' user # Configurable user settings for JobPost security

    4) .dockerignore File

    Docs: .dockerignore file

    Exclude files and directories from the Docker build context.

    Full description

    Before the docker CLI sends the context to the docker daemon, it looks for a file named .dockerignore in the root directory of the context. If this file exists, the CLI modifies the context to exclude files and directories that match patterns in it. This helps to avoid unnecessarily sending large or sensitive files and directories to the daemon. For Django projects, .dockerignore prevents unnecessary files from being sent to the build context, reducing build time and image size.

    Syntax

    # .dockerignore file patterns
pattern
!exception_pattern
# comment

    Parameters description

    • Glob patterns: Uses Go's filepath.Match rules for pattern matching
    • Wildcards: * matches any sequence of non-separator characters, ** matches directories
    • Negation: ! prefix negates the pattern (includes files that would otherwise be excluded)
    • Comments: Lines starting with # are treated as comments
    • Root context: Patterns are relative to the build context root

    Return value

    Reduces build context size by excluding specified files and directories from being sent to Docker daemon.

    5 short examples with Django project exclusions

    5) Shell and Exec Form

    Docs: Shell and exec form

    RUN, CMD and ENTRYPOINT instructions have two forms: shell form and exec form.

    Full description

    RUN, CMD, and ENTRYPOINT instructions can be written in two forms: shell form and exec form. The shell form runs commands in a shell (/bin/sh -c on Linux), while exec form runs commands directly without shell processing. For Django applications, understanding the difference is crucial for proper signal handling, environment variable behavior, and security considerations.

    Syntax

    # Shell form
RUN command param1 param2
CMD command param1 param2
ENTRYPOINT command param1 param2

# Exec form
RUN ["executable", "param1", "param2"]
CMD ["executable", "param1", "param2"]
ENTRYPOINT ["executable", "param1", "param2"]

    Parameters description

    • Shell form: Commands are run in a shell, allows shell features like pipes, variables, globbing
    • Exec form: Commands are run directly, no shell processing, better for signal handling
    • PID 1 process: Exec form makes the executable PID 1, important for proper signal handling
    • Environment variables: Shell form processes variables, exec form requires manual substitution
    • JSON array: Exec form must be valid JSON array format

    Return value

    Different process execution models affecting signal handling, environment variable processing, and container behavior.

    5 short examples with Django command forms

    Example 1: Shell form with variables

    ENV DJANGO_SETTINGS_MODULE=myproject.settings.production
CMD python manage.py runserver 0.0.0.0:$PORT
    $ docker run myapp # Shell processes environment variables # $PORT is expanded by shell # JobPost Django server starts with shell wrapping

    Example 2: Exec form for signals

    CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
    $ docker run myapp # Python process gets PID 1 # Proper signal handling (SIGTERM, SIGINT) # JobPost Django server responds to docker stop

    Example 3: Shell form with pipes

    RUN python manage.py dumpdata auth.User | gzip > /tmp/users.json.gz
    # Shell enables pipe and redirection # Django user data exported and compressed # Shell features work with JobPost data exports

    Example 4: Exec form with environment

    ENV PORT=8000
CMD ["sh", "-c", "python manage.py runserver 0.0.0.0:${PORT}"]
    # Exec form with explicit shell invocation # Environment variables processed by shell # Combines benefits of both forms for Django

    Example 5: Production ENTRYPOINT patterns

    # Shell form - simple
ENTRYPOINT python manage.py runserver 0.0.0.0:8000

# Exec form - production ready
ENTRYPOINT ["python", "manage.py"]
CMD ["runserver", "0.0.0.0:8000"]
    # Shell form: Simple but wrapped by shell # Exec form: Direct execution, proper signals # CMD can be overridden for different Django commands # JobPost app flexible deployment patterns

    6) FROM - Base Image

    Docs: FROM instruction

    Specifies the base image for your Docker container.

    Full description

    FROM initializes a new build stage and sets the base image for subsequent instructions. For Django projects, this is typically a Python image. The image can be any valid image from Docker Hub, a private registry, or a previously built stage in a multi-stage build. This must be the first instruction in your Dockerfile (except for ARG instructions that can appear before FROM).

    Syntax

    FROM [--platform=<platform>] <image>[:<tag>] [AS <name>]
FROM [--platform=<platform>] <image>[@<digest>] [AS <name>]

    Parameters description

    • image: The base image name (e.g., python, alpine, ubuntu)
    • tag: Specific version of the image (e.g., 3.11, latest, slim). If omitted, defaults to 'latest'
    • digest: SHA256 hash for exact image identification (more secure than tags)
    • --platform: Target platform for multi-platform builds (e.g., linux/amd64, linux/arm64)
    • AS name: Assigns a name to this build stage for multi-stage builds

    Return value

    Sets up the filesystem and environment for the container. Returns the base layer that all subsequent instructions will build upon.

    5 short examples with Django JobPost model

    Example 1: Standard Python image

    FROM python:3.11
    Step 1/1 : FROM python:3.11 ---> a7a6c2b8f0a1 Successfully built a7a6c2b8f0a1 # Base: Debian with Python 3.11 installed # Size: ~900MB # Use case: Development environment

    Example 2: Slim Python for production

    FROM python:3.11-slim
    Step 1/1 : FROM python:3.11-slim ---> b2c8f8f9e1d3 Successfully built b2c8f8f9e1d3 # Base: Minimal Debian with Python 3.11 # Size: ~150MB # Use case: Production Django app

    Example 3: Alpine for minimal size

    FROM python:3.11-alpine
    Step 1/1 : FROM python:3.11-alpine ---> c4d2e3f7a8b9 Successfully built c4d2e3f7a8b9 # Base: Alpine Linux with Python 3.11 # Size: ~50MB # Use case: Microservices, minimal footprint

    Example 4: Multi-stage with alias

    FROM python:3.11 AS builder
    Step 1/1 : FROM python:3.11 AS builder ---> a7a6c2b8f0a1 Successfully built a7a6c2b8f0a1 # Named stage for multi-stage builds # Can reference this stage later # Use case: Separate build and runtime environments

    Example 5: Platform-specific image

    FROM --platform=linux/amd64 python:3.11-slim
    Step 1/1 : FROM --platform=linux/amd64 python:3.11-slim ---> b2c8f8f9e1d3 Successfully built b2c8f8f9e1d3 # Forces AMD64 architecture # Ensures consistent builds across platforms # Use case: CI/CD with mixed architectures

    7) RUN - Execute Commands

    Docs: RUN instruction

    Executes commands in a new layer on top of the current image and commits the results.

    Full description

    RUN executes any commands during the build process and creates a new layer with the results. Each RUN instruction creates a new layer, so it's important to combine commands efficiently. For Django projects, RUN is commonly used to install system packages, Python dependencies, run migrations, collect static files, and set up the application environment. Advanced RUN options provide better caching, security, and build performance.

    Syntax

    # Shell form
RUN <command>

# Exec form  
RUN ["executable", "param1", "param2"]

# With mount options
RUN --mount=<mount-options> <command>

# With network options  
RUN --network=<network-mode> <command>

# With security options
RUN --security=<security-mode> <command>

    Parameters description

    • Shell form: Commands run in a shell (/bin/sh -c on Linux). Supports shell features like pipes, variables
    • Exec form: Commands run directly without shell processing. More secure and predictable
    • Layer creation: Each RUN creates a new filesystem layer that can be cached
    • Command chaining: Use && to combine commands in one layer for efficiency
    • Cache invalidation: Changes in RUN commands invalidate Docker cache for subsequent layers

    Return value

    Creates a new image layer with the results of the command execution. Returns exit code 0 for success, non-zero for failure (stops build).

    RUN --mount

    Cache mount for pip

    RUN --mount=type=cache,target=/root/.cache/pip \
    pip install django==4.2.0 psycopg2-binary
    # Pip cache persisted across builds # Faster Django dependency installation # JobPost requirements installed efficiently

    Secret mount for credentials

    RUN --mount=type=secret,id=pip_creds,target=/pip.conf \
    pip install --index-url private-repo django-extensions
    # Private repository credentials mounted securely # Credentials not stored in image layers # Private Django packages for JobPost app

    RUN --network

    No network access

    RUN --network=none \
    python manage.py collectstatic --noinput
    # Collects static files without network access # Prevents data exfiltration during build # JobPost static files processed securely

    Host network access

    RUN --network=host \
    curl -o requirements.txt http://localhost:8080/requirements.txt
    # Access to host network services # Download dependencies from local server # JobPost requirements from internal registry

    RUN --security

    Insecure mode for legacy tools

    RUN --security=insecure \
    apt-get update && apt-get install -y legacy-package
    # Allows elevated privileges if needed # Use sparingly for compatibility # Legacy dependencies for JobPost integrations

    8) CMD - Default Command

    Docs: CMD instruction

    Provides defaults for executing a container, including the executable and arguments.

    Full description

    CMD specifies the default command to run when the container starts. Unlike RUN which executes during build time, CMD runs when the container is started. There can only be one CMD instruction in a Dockerfile - if multiple are listed, only the last one takes effect. For Django projects, CMD typically starts the web server, runs migrations, or starts a specific Django management command. CMD can be overridden when running the container.

    Syntax

    # Exec form (preferred)
CMD ["executable","param1","param2"]

# Shell form
CMD command param1 param2

# As default parameters to ENTRYPOINT
CMD ["param1","param2"]

    Parameters description

    • Exec form: Recommended. Runs executable directly without shell processing
    • Shell form: Runs command in shell (/bin/sh -c). Supports shell features but less efficient
    • Parameter form: Provides default arguments to ENTRYPOINT
    • Override capability: Can be overridden by docker run arguments
    • Single instruction: Only the last CMD in Dockerfile takes effect

    Return value

    Sets the default command for container execution. Returns the process that will run when the container starts, unless overridden at runtime.

    17) ONBUILD - Trigger Instructions

    Docs: ONBUILD instruction

    Adds trigger instructions to be executed when the image is used as a base for another build.

    Full description

    ONBUILD adds a trigger instruction to the image that will be executed at a later time, when the image is used as the base for another build. The trigger will be executed in the context of the downstream build, as if it had been inserted immediately after the FROM instruction in the downstream Dockerfile. For Django projects, ONBUILD is useful for creating base images that automatically handle common setup tasks for child images.

    Syntax

    ONBUILD [INSTRUCTION]

    Parameters description

    • INSTRUCTION: Any Dockerfile instruction except FROM, MAINTAINER, or another ONBUILD
    • Trigger execution: Executes in child image builds, not in current build
    • Inheritance: Triggers are not inherited by grandchildren (child of child)
    • Build context: Executes in the context of the child build
    • Inspection: Can be viewed with docker inspect command

    Return value

    Adds trigger metadata to the image. Returns instructions that will execute when this image is used as a base.

    5 short examples with Django base images

    Example 1: Auto-install requirements

    # Base Django image
FROM python:3.11-slim
ONBUILD COPY requirements.txt .
ONBUILD RUN pip install -r requirements.txt
    # Child Dockerfile: FROM django-base:latest COPY . . # ONBUILD triggers execute here: # 1. COPY requirements.txt . # 2. RUN pip install -r requirements.txt # JobPost dependencies installed automatically

    Example 2: Django project structure

    FROM python:3.11-slim
WORKDIR /app
ONBUILD COPY . .
ONBUILD RUN python manage.py collectstatic --noinput
    # Child build automatically: # Copies entire project to /app # Collects Django static files # JobPost static assets ready for production

    Example 3: Environment setup

    FROM python:3.11-slim
ONBUILD ARG BUILD_ENV=production
ONBUILD ENV DJANGO_SETTINGS_MODULE=myproject.settings.${BUILD_ENV}
    # Child build receives: # BUILD_ENV argument for configuration # DJANGO_SETTINGS_MODULE set appropriately # JobPost app configured for target environment

    Example 4: Security setup

    FROM python:3.11-slim
ONBUILD RUN groupadd -r django && useradd -r -g django django
ONBUILD USER django
    # Child builds automatically get: # Django user and group created # Switched to non-root user # JobPost app runs securely by default

    Example 5: Complete Django base

    FROM python:3.11-slim AS django-base
RUN apt-get update && apt-get install -y postgresql-client
WORKDIR /app
ONBUILD COPY requirements.txt .
ONBUILD RUN pip install -r requirements.txt
ONBUILD COPY . .
ONBUILD RUN python manage.py collectstatic --noinput
ONBUILD EXPOSE 8000
ONBUILD CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
    # Child Dockerfile can be minimal: FROM django-base:latest # All setup happens automatically via ONBUILD: # Dependencies installed, code copied # Static files collected, port exposed # Default Django server command set # JobPost app ready with minimal configuration

    18) STOPSIGNAL - Stop Signal

    Docs: STOPSIGNAL instruction

    Sets the system call signal that will be sent to the container to exit.

    Full description

    STOPSIGNAL sets the system call signal that will be sent to the container to exit. This signal can be a signal name in the format SIG<NAME> or an unsigned number that matches a position in the kernel's syscall table. The default is SIGTERM. For Django applications, proper signal handling ensures graceful shutdown, allowing active requests to complete and database connections to close properly.

    Syntax

    STOPSIGNAL signal

    Parameters description

    • signal: Signal name (SIG<NAME>) or number to send for container shutdown
    • Default behavior: SIGTERM (15) is the default stop signal
    • Graceful shutdown: Applications should handle the signal gracefully
    • Timeout behavior: If process doesn't exit after timeout, SIGKILL is sent
    • Application specific: Different applications may prefer different signals

    Return value

    Configures the stop signal for container shutdown. Returns the signal that Docker will send to terminate the container process.

    5 short examples with Django signal handling

    Example 1: Default SIGTERM

    STOPSIGNAL SIGTERM
CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
    $ docker stop myapp # Sends SIGTERM to Django process # Django development server shuts down gracefully # Active JobPost requests complete before exit

    Example 2: Gunicorn with SIGTERM

    STOPSIGNAL SIGTERM
CMD ["gunicorn", "--bind", "0.0.0.0:8000", "myproject.wsgi:application"]
    $ docker stop myapp # Gunicorn receives SIGTERM # Workers finish current JobPost API requests # Graceful shutdown with connection cleanup

    Example 3: Custom signal for debugging

    STOPSIGNAL SIGUSR1

    19) SHELL - Default Shell

    Docs: SHELL instruction

    Allows the default shell used for the shell form of commands to be overridden.

    Full description

    SHELL allows the default shell used for the shell form of commands to be overridden. The default shell on Linux is ["/bin/sh", "-c"], and on Windows is ["cmd", "/S", "/C"]. The SHELL instruction must be written in JSON form in a Dockerfile. For Django projects, SHELL is useful when you need specific shell features, better error handling, or when working with different base images that have different default shells.

    Syntax

    SHELL ["executable", "parameters"]

    Parameters description

    • executable: Path to the shell executable (e.g., /bin/bash, /bin/sh, powershell)
    • parameters: Shell parameters and flags for shell behavior
    • JSON format: Must be written as JSON array
    • Shell form commands: Affects RUN, CMD, ENTRYPOINT in shell form
    • Multiple SHELL: Can be used multiple times to change shell for different layers

    Return value

    Changes the default shell for subsequent shell form commands. Returns the shell configuration for command execution.

    5 short examples with Django shell configuration

    20) Here-Documents

    Docs: Here-Documents

    Create multi-line files or run multi-line scripts without complex escaping.

    Full description

    Here-documents allow you to create files or run scripts with multiple lines of content without complex shell escaping. This feature makes it easier to embed configuration files, scripts, or multi-line content directly in your Dockerfile. For Django projects, here-documents are perfect for creating configuration files, database setup scripts, or complex initialization procedures.

    Syntax

    COPY <<EOF /path/to/file
content line 1
content line 2
EOF

RUN <<EOF
script line 1
script line 2
EOF

    Parameters description

    • EOF delimiter: Marks the start and end of here-document content
    • Custom delimiters: Can use any string as delimiter (not just EOF)
    • Indented content: Content can be indented for readability
    • Variable substitution: Environment variables are expanded in content
    • Multiple files: Can create multiple files in one instruction

    21) Multi-stage Builds

    Docs: Multi-stage builds

    Use multiple FROM statements to create separate build stages and copy artifacts between them.

    Full description

    Multi-stage builds allow you to use multiple FROM instructions in a single Dockerfile, creating separate build environments and copying only the necessary artifacts to the final image. This dramatically reduces final image size by excluding build tools, dependencies, and intermediate files. For Django projects, this pattern is excellent for separating dependency installation, static file compilation, and runtime environments while keeping production images minimal and secure.

    Syntax

    FROM image:tag AS stage_name
# Build instructions for this stage

FROM another_image:tag AS another_stage  
# More build instructions

FROM final_image:tag
COPY --from=stage_name /path/to/artifact /final/path

    Parameters description

    • AS stage_name: Names a build stage for later reference
    • --from=stage_name: Copies files from a named stage
    • --from=index: Copies from stage by numeric index (0, 1, 2...)
    • Multiple stages: Each FROM starts a new build stage
    • Final stage: Last FROM determines the final image base
    • Stage independence: Each stage has its own filesystem and context

    Return value

    Creates optimized final image with only necessary artifacts. Returns a production-ready container without build dependencies and tools.

    5 short examples with Django optimization

    Example 1: Basic Django build separation

RUN pip install Django==4.2.0

WORKDIR /app
CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
