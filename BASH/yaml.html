Pattern 2: Using Anchors for DRY

# database-config.yaml
---
defaults: &db_defaults
  adapter: postgresql
  pool: 5
  timeout: 5000
  encoding: unicode

development:
  <<: *db_defaults
  database: myapp_dev
  host: localhost

test:
  <<: *db_defaults
  database: myapp_test
  host: localhost

production:
  <<: *db_defaults
  database: myapp_prod
  host: db.example.com
  pool: 25
  ssl: true
Usage: Reuse common settings across environments with merge key

Pattern 3: Multiline Strings

# content.yaml
---
# Literal style - preserves newlines exactly
script: |
  #!/bin/bash
  echo "Starting deployment"
  ./deploy.sh --env production
  echo "Done"

# Folded style - newlines become spaces
description: >
  This is a long description that
  spans multiple lines but will be
  folded into a single paragraph
  when parsed.

# Quoted with escapes
message: "Line 1\nLine 2\nLine 3"
Literal (|): "#!/bin/bash\necho \"Starting...\"\n..."
Folded (>): "This is a long description that spans multiple lines..."
Quoted: "Line 1\nLine 2\nLine 3"

12. When to Use YAML

✅ Use YAML When

  • Configuration files (apps, servers)
  • Infrastructure as Code (K8s, Ansible)
  • CI/CD pipelines (GitHub Actions, GitLab)
  • Human-edited files
  • Files needing comments
  • Complex nested structures
  • Multiline text content
  • Reusable config with anchors

❌ Avoid YAML When

  • API responses (use JSON)
  • Browser/JavaScript (use JSON)
  • High-performance parsing needed
  • Machine-generated files
  • Untrusted user input
  • Simple key-value pairs (use .env)
  • Binary data storage
  • Strict schema validation needed

13. Best Practices Checklist

📝 Structure

  • ☑️ Use 2-space indentation consistently
  • ☑️ Never use tabs (spaces only)
  • ☑️ Keep nesting depth ≤ 4 levels
  • ☑️ Group related keys together
  • ☑️ Use blank lines between sections

💬 Comments

  • ☑️ Add file header with purpose
  • ☑️ Document non-obvious settings
  • ☑️ Include example values in comments
  • ☑️ Note units (seconds, MB, etc.)
  • ☑️ Reference external docs if needed

🔤 Values

  • ☑️ Quote strings with special chars
  • ☑️ Quote numeric strings ("01onal")
  • ☑️ Use explicit booleans (true/false)
  • ☑️ Use ISO 8601 for dates
  • ☑️ Use null for empty values

🔧 Advanced

  • ☑️ Use anchors for repeated values
  • ☑️ Validate before deployment
  • ☑️ Use schema validation (JSON Schema)
  • ☑️ Version control your YAML files
  • ☑️ Test with multiple parsers

16. Quick Conversion Snippets

Python: YAML ↔ Dict

import yaml

# Load YAML to dict
with open('config.yaml') as f:
    data = yaml.safe_load(f)

# Dict to YAML
yaml_str = yaml.dump(data, 
    default_flow_style=False,
    sort_keys=False)

# Save to file
with open('output.yaml', 'w') as f:
    yaml.dump(data, f)

JavaScript: YAML ↔ Object

import YAML from 'yaml';
import fs from 'fs';

// Load YAML to object
const file = fs.readFileSync('config.yaml', 'utf8');
const data = YAML.parse(file);

// Object to YAML
const yamlStr = YAML.stringify(data);

// Save to file
fs.writeFileSync('output.yaml', yamlStr);

CLI: YAML ↔ JSON

# YAML to JSON (Python)
python -c "import yaml,json,sys; \
  print(json.dumps(yaml.safe_load(sys.stdin)))" \
  < config.yaml > config.json

# JSON to YAML (Python)
python -c "import yaml,json,sys; \
  print(yaml.dump(json.load(sys.stdin)))" \
  < config.json > config.yaml

# Using yq (recommended)
yq -o=json config.yaml > config.json
yq -P config.json > config.yaml

Go: YAML ↔ Struct

import "gopkg.in/yaml.v3"

type Config struct {
    Server struct {
        Host string `yaml:"host"`
        Port int    `yaml:"port"`
    } `yaml:"server"`
}

// Load YAML
data, _ := os.ReadFile("config.yaml")
var config Config
yaml.Unmarshal(data, &config)

// Save YAML
out, _ := yaml.Marshal(&config)
os.WriteFile("out.yaml", out, 0644)

1. Introduction

Full Description

YAML is a human-friendly, cross-language, Unicode-based data serialization language. It is designed around native data structures of agile programming languages (mappings, sequences, scalars).

1.1 Goals

  • Easily readable by humans
  • Portable between programming languages
  • Matches native data structures of agile languages
  • Supports one-pass processing
  • Expressive and extensible
  • Easy to implement and use

1.2 Prior Art

YAML integrates concepts from: C, Java, Perl, Python, Ruby, RFC0822 (MAIL), RFC2045 (MIME), XML, SAX, and SOAP.

2.1 Collections

Full Description

YAML has two collection types: sequences (lists) and mappings (dictionaries). Both can be written in block or flow style.

Syntax

# Block Sequence
- item1
- item2

# Block Mapping
key1: value1
key2: value2

# Flow Sequence
[item1, item2, item3]

# Flow Mapping
{key1: value1, key2: value2}

Examples

# Example 1: Simple sequence (list of fruits)
fruits:
  - apple
  - banana
  - orange
Result: fruits = ["apple", "banana", "orange"]
# Example 2: Mapping (user profile)
user:
  name: Alice
  age: 30
  email: alice@example.com
Result: user = {name: "Alice", age: 30, email: "alice@example.com"}
# Example 3: Flow style collections
colors: [red, green, blue]
point: {x: 10, y: 20}
Result: colors = ["red", "green", "blue"], point = {x: 10, y: 20}
# Example 4: Nested collections
employees:
  - name: John
    department: Engineering
    skills:
      - Python
      - JavaScript
  - name: Jane
    department: Design
    skills:
      - Figma
      - CSS
Result: List of 2 employee objects with nested skills arrays
# Example 5: Mixed flow and block
servers:
  production: {host: prod.com, port: 443}
  staging: {host: stage.com, port: 8080}
  allowed_ips:
    - 192.168.1.1
    - 10.0.0.1
Result: Mapping with flow mappings and block sequence

2.2 Structures

Full Description

YAML uses document markers and comments to structure content. Multiple documents can exist in a single stream.

Syntax

---          # Document start
# Comment    # Comments start with #
...          # Document end (optional)

Examples

# Example 1: Single document
---
title: My Document
content: Hello World
...
Result: Single document with title and content
# Example 2: Multiple documents in one file
---
name: Document 1
type: config
---
name: Document 2
type: data
---
name: Document 3
type: log
Result: 3 separate documents in one stream
# Example 3: Comments for documentation
---
# Application Configuration
# Last updated: 2024-01-15

app:
  name: MyApp     # Application name
  version: 1.0.0  # Semantic versioning
  # debug: true   # Commented out for production
Result: Documented config (comments ignored in parsing)
# Example 4: Anchors and aliases (reuse data)
---
defaults: &defaults
  timeout: 30
  retries: 3

production:
  <<: *defaults
  server: prod.example.com

development:
  <<: *defaults
  server: localhost
Result: Both production and development inherit timeout: 30, retries: 3
# Example 5: Complex key (multiline key)
? - Detroit Tigers
  - Chicago Cubs

Result: Sequence ["Detroit Tigers", "Chicago Cubs"] as key, date as value

2.3 Scalars

Full Description

Scalars are single values: strings, numbers, booleans, null, dates. Can be plain, single-quoted, double-quoted, or block style.

Syntax

plain: unquoted string
single: 'preserves \n literally'
double: "interprets \n as newline"
literal: |
  preserves
  line breaks
folded: >
  folds lines
  into one

Examples

# Example 1: String types
plain_string: Hello World
single_quoted: 'Hello "World"'
double_quoted: "Hello\nWorld"
empty_string: ''
null_value: ~
Result: Various string representations, null = None/nil
# Example 2: Numeric types
integer: 42
negative: -17
float: 3.14159
scientific: 6.022e23
hex: 0xFF
octal: 0o755
infinity: .inf
not_a_number: .nan
Result: integer=42, float=3.14159, hex=255, octal=493
# Example 3: Boolean values
yes_bool: true
no_bool: false
also_yes: yes
also_no: no
on_value: on
off_value: off
Result: All resolve to boolean true/false
# Example 4: Literal block (preserves newlines)
poem: |
  Roses are red
  Violets are blue
  YAML is great
  And so are you
Result: "Roses are red\nViolets are blue\nYAML is great\nAnd so are you\n"

3. Processing YAML Information

YAML processing involves three stages: Representation → Serialization → Presentation (and reverse for loading).

3.1 Processes

3.1.1 Represent

Full Description: Converts native data structures (dict, list, string, int) to YAML representation using three node kinds: scalar, sequence, mapping.

Node Kinds:

scalar - Opaque data (strings, numbers, booleans, dates)

sequence - Ordered series of nodes (arrays/lists)

mapping - Unordered key-value pairs (hashes/dicts)

Return Value: YAML representation graph with tagged nodes.

Example 1: Dict to Mapping

# Python: {"name": "John", "age": 30}
name: John
age: 30
Result: Mapping node with 2 scalar children

Example 2: List to Sequence

# Python: ["apple", "banana", "orange"]
- apple
- banana
- orange
Result: Sequence node with 3 scalar children

Example 3: Nested Structure

# Python: {"users": [{"name": "Alice"}, {"name": "Bob"}]}
users:
  - name: Alice
  - name: Bob
Result: Mapping → Sequence → Mappings

Example 4: Complex Types

# Date, boolean, null representation
created: 2025-01-25
active: true
deleted: null
Result: Scalars with implicit tags (!!timestamp, !!bool, !!null)

Example 5: Binary Data

# Python: b'\x00\x01\x02' (bytes)
data: !!binary |
  AAEC
Result: Scalar with explicit !!binary tag, base64 encoded

3.1.2 Serialize

Full Description: Converts representation graph to serialization tree by imposing key order and replacing repeated nodes with anchors/aliases.

Syntax:

&anchor_name value    # Define anchor
*anchor_name          # Reference alias
<<: *anchor_name      # Merge anchor content

Parameters:

anchor - Identifier for the node (alphanumeric, starts with letter)

alias - Reference to a previously defined anchor

Return Value: Serialization tree with ordered keys and anchors.

Example 1: Simple Anchor/Alias

defaults: &defaults
  adapter: postgres
  host: localhost

development:
  database: dev_db
  <<: *defaults

production:
  database: prod_db
  <<: *defaults
Result:
development: {database: dev_db, adapter: postgres, host: localhost}
production: {database: prod_db, adapter: postgres, host: localhost}

Example 2: Reusing Complex Nodes

person: &person
  name: John
  address: &addr
    city: NYC
    zip: 10001

contact:
  <<: *person
  phone: 555-1234

shipping:
  destination: *addr
Result: Both contact and shipping reference the same address data

Example 3: Sequence with Anchors

base_packages: &base
  - nginx
  - curl
  - vim

web_server:
  packages:
    - *base
    - nodejs
    - npm
Result: web_server.packages includes all base packages plus nodejs and npm

Example 4: Override Merged Values

defaults: &defaults
  timeout: 30
  retries: 3

service:
  <<: *defaults
  timeout: 60  # Override the merged value
Result: service = {timeout: 60, retries: 3}

Example 5: Multiple Merge Keys

base1: &base1
  a: 1
  b: 2

base2: &base2
  c: 3
  d: 4

combined:
  <<: [*base1, *base2]
  e: 5
Result: combined = {a: 1, b: 2, c: 3, d: 4, e: 5}

3.1.3 Present

Full Description: Converts serialization tree to character stream with human-readable formatting including indentation, node styles, comments, and directives.

Style Choices:

Block Style - Uses indentation for structure (more readable)

Flow Style - Uses indicators [ ] { } (more compact)

Literal | - Preserves newlines in scalars

Folded > - Folds newlines into spaces

Return Value: YAML character stream (string/file).

Example 1: Block vs Flow Style

# Block style (readable)
fruits:
  - apple
  - banana

# Flow style (compact)
fruits: [apple, banana]
Result: Both produce same data: ["apple", "banana"]

Example 2: Indentation Levels

level1:
  level2:
    level3:
      value: deep
Result: 2-space indentation per level (standard)

Example 3: Adding Comments

# Database configuration
database:
  host: localhost  # Primary server
  port: 5432       # Default PostgreSQL port
  # Credentials below
  user: admin
Result: Comments preserved in presentation, ignored in data

Example 4: Literal Block for Code

script: |
  #!/bin/bash
  echo "Hello"
  exit 0
Result: "#!/bin/bash\necho \"Hello\"\nexit 0\n"

Example 5: Folded for Long Text

description: >
  This is a very long
  description that will
  be folded into a
  single line.
Result: "This is a very long description that will be folded into a single line.\n"

3.1.4 Parse

Full Description: Inverse of presentation - reads character stream and produces serialization events. Discards presentation details (comments, formatting).

Events Generated:

StreamStart/End - Begin/end of YAML stream

DocumentStart/End - Begin/end of document (---/...)

MappingStart/End - Begin/end of mapping

SequenceStart/End - Begin/end of sequence

Scalar - Scalar value with tag

Alias - Reference to anchor

Return Value: Stream of serialization events.

Example: Parse Events

---
name: John
items:
  - one
  - two
Events:
StreamStart → DocumentStart → MappingStart →
Scalar("name") → Scalar("John") →
Scalar("items") → SequenceStart →
Scalar("one") → Scalar("two") →
SequenceEnd → MappingEnd → DocumentEnd → StreamEnd

3.1.5 Compose

Full Description: Builds representation graph from serialization events, resolving aliases to their anchored nodes.

Return Value: Complete representation graph with resolved references.

Example: Composed Graph

shared: &shared
  timeout: 30
service:
  config: *shared
Graph:
Root (Mapping)
├── "shared" → Mapping {timeout: 30}
└── "service" → Mapping
└── "config" → [reference to same node as "shared"]

3.1.6 Construct

Full Description: Creates native data structures (Python dict, list, etc.) from representation graph based on tags.

Tag-to-Type Mapping:

YAML TagPython TypeExample
!!mapdict{key: value}
!!seqlist[1, 2, 3]
!!strstr"hello"
!!intint42
!!floatfloat3.14
!!boolbooltrue/false
!!nullNonenull, ~
!!timestampdatetime2025-01-25

Return Value: Native language data structure.

Example: Construction Result

user:
  name: Alice
  age: 30
  active: true
  joined: 2025-01-01
  tags: [admin, user]
Python:
{'user': {'name': 'Alice', 'age': 30, 'active': True,
'joined': datetime(2025, 1, 1), 'tags': ['admin', 'user']}}

3.2 Information Models

YAML uses three information models to represent data at different stages of processing.

3.2.1 Representation Graph

Full Description: A rooted, connected, directed graph of tagged nodes. This is YAML's abstract data model independent of serialization format.

Node Kinds:

Node KindDescriptionExample
ScalarOpaque datum - string, number, date"hello", 42, 2025-01-25
SequenceOrdered series of nodes[a, b, c]
MappingUnordered key-value pairs (unique keys){x: 1, y: 2}

Node Comparison:

Equality - Same tag and canonical content

Identity - Same memory address (same node instance)

Example 1: Graph Visualization

  employees:

      dept: &eng Engineering
    - name: Bob

3.2.2 Serialization Tree

Full Description: The representation graph converted to a tree by imposing key order and using anchors/aliases for repeated nodes.

Key Properties:

Keys Order - Mappings become ordered for sequential access

Anchors - First occurrence identified with &name

second: &data {x: 1, y: 2}  # Valid but unusual

3.2.3 Presentation Stream

Full Description: The final character stream with all presentation details: formatting, styles, comments, and directives.

Presentation Details:

DetailOptionsExample

CommentsLine comments# comment
Directives%YAML, %TAG%YAML 1.1

Example: Same Data, Different Presentations

  - name: Alice
    role: admin
  - name: Bob
    role: user
...

# Presentation 2: Flow style, compact
---

...
Result: Both produce identical representation graph

7.1 Directives

Full Description

Directives are instructions to the YAML processor that don't produce output data. They control document processing behavior. All directives start with % and must appear before the document start marker.

Syntax

%YAML - Specifies YAML version (1.0, 1.1, or 1.2). Must be the first directive in document.

7.1.2 %TAG Directive

%TAG - Defines shorthand tag handles (!handle! or !!). Maps short prefixes to full tag URIs.

Return Value

No output data. Affects how the processor interprets the document.

Examples

...

Result: Each document has its own directive scope

8.2 Node Tags (!)

Full Description

Tags explicitly specify the data type of a node, overriding automatic type detection. Use !!tag for standard YAML types or !tag for application-specific types.

Syntax

!!type value           # Standard YAML type
!local_tag value       # Local/application tag
!<full-uri> value     # Verbatim tag URI

Parameters

!!str - Force string type

!!int - Force integer type

!!float - Force floating-point type

!!bool - Force boolean type

!!null - Explicit null

!!binary - Base64-encoded binary data

!!timestamp - Date/time value

!!set - Set (unique keys, null values)

!!omap - Ordered mapping

Return Value

Node value converted to specified type. Type conversion errors may occur if incompatible.

Examples

# Example 1: Prevent number parsing (keep leading zeros)
zip_code: !!str 01234
phone: !!str 555-1234
version: !!str 1.0
id: !!str 007
Result: zip_code = "01234", phone = "555-1234", version = "1.0", id = "007" (all strings)
# Example 2: Force type conversion
string_num: !!str 42
int_string: !!int "42"
float_int: !!float 10
bool_string: !!bool "yes"
null_explicit: !!null ""
Result: string_num = "42", int_string = 42, float_int = 10.0, bool_string = true, null_explicit = null
# Example 3: Binary data (base64 encoded)
favicon: !!binary |
  R0lGODlhAQABAIAAAAAAAP///yH5BAEAAAAA
  LAAAAAABAAEAAAIBRAA7
certificate: !!binary |
  LS0tLS1CRUdJTiBDRVJUSUZJQ0FURS0tLS0t
Result: Binary data decoded from base64 representation
# Example 4: Set type (unique values only)
primary_colors: !!set
  red: null
  green: null
  blue: null
# Or shorthand:
tags: !!set {web, api, mobile}
Result: Sets containing unique values: {red, green, blue} and {web, api, mobile}
# Example 5: Timestamps
created: !!timestamp 2024-01-15
updated: !!timestamp 2024-01-15T14:30:00
with_tz: !!timestamp 2024-01-15T14:30:00-05:00
canonical: !!timestamp 2024-01-15T19:30:00.000Z
Result: Date/time objects parsed from ISO 8601 format

Result: Complex nested structure mixing all content types

8.4 Alias Nodes (*)

Full Description

Aliases reference previously anchored nodes using *anchor_name. The alias node becomes a reference to the original anchored content. Aliases must appear after their corresponding anchor.

Syntax

*anchor_name

Parameters

anchor_name - Name of previously defined anchor (case-sensitive, must exist)

Return Value

Reference to the anchored node's content. In most parsers, this creates a shared reference (not a copy).

Examples

# Example 1: Simple scalar alias
api_url: &url https://api.example.com/v1
users_endpoint: *url
posts_endpoint: *url
comments_endpoint: *url
Result: All four keys have value "https://api.example.com/v1"
# Example 2: Mapping merge with <<
base_config: &base
  timeout: 30
  retries: 3
  log_level: info

service_a:
  <<: *base
  name: ServiceA
  port: 8080

service_b:
  <<: *base
  name: ServiceB
  port: 8081
  timeout: 60  # Override
Result: service_a = {timeout: 30, retries: 3, log_level: "info", name: "ServiceA", port: 8080}
service_b = {timeout: 60, retries: 3, log_level: "info", name: "ServiceB", port: 8081}
# Example 3: Sequence alias
standard_ports: &ports
  - 22
  - 80
  - 443

server1:
  name: web1
  ports: *ports
server2:
  name: web2
  ports: *ports
Result: Both servers share ports array [22, 80, 443]
# Example 4: Multiple merges
db_defaults: &db
  driver: postgres
  pool: 5
cache_defaults: &cache
  backend: redis
  ttl: 3600

production:
  <<: [*db, *cache]
  db_host: prod-db.example.com
  cache_host: prod-cache.example.com
Result: production merges both db and cache defaults, then adds hosts
# Example 5: Self-referential structure (graph)
nodes:
  - &node1
    id: 1
    name: Node A
    links: []
  - &node2
    id: 2
    name: Node B
    links: [*node1]
  - id: 3
    name: Node C
    links: [*node1, *node2]
Result: Graph structure where Node C links to both Node A and Node B

9. Scalar Styles

9.1 Flow Scalar Styles

9.1.1 Double-Quoted Style (")

Full Description

Double-quoted scalars support all escape sequences and can span multiple lines. Backslash is the escape character. Use when you need special characters, Unicode, or control characters.

Syntax
key: "value with \"escapes\" and \n newlines"
Parameters (Escape Sequences)

\\ - Backslash

\" - Double quote

\n - Newline (LF)

\r - Carriage return (CR)

\t - Tab

\0 - Null character

\xNN - 8-bit Unicode (2 hex digits)

\uNNNN - 16-bit Unicode (4 hex digits)

\UNNNNNNNN - 32-bit Unicode (8 hex digits)

Return Value

String with escape sequences interpreted and converted to actual characters.

Examples
# Example 1: Line breaks and tabs
formatted: "Name:\tJohn\nAge:\t30\nCity:\tNYC"
Result: "Name: John\nAge: 30\nCity: NYC" (with actual tab and newline characters)
# Example 2: Quotes within string
message: "She said \"Hello World!\" to everyone"
json_str: "{\"name\": \"John\", \"age\": 30}"
Result: message = 'She said "Hello World!" to everyone'
# Example 3: File paths (Windows)
win_path: "C:\\Users\\john\\Documents\\file.txt"
regex: "^\\d{3}-\\d{4}$"
Result: win_path = "C:\Users\john\Documents\file.txt", regex = "^\d{3}-\d{4}$"
# Example 4: Unicode characters
emoji: "Hello \u263A World \u2665"
chinese: "\u4F60\u597D"
arrow: "\u2192"
Result: emoji = "Hello ☺ World ♥", chinese = "你好", arrow = "→"
# Example 5: Multi-line with explicit breaks
address: "123 Main Street\nApt 4B\nNew York, NY 10001"
sql: "SELECT * FROM users\nWHERE active = true\nORDER BY name"
Result: Multi-line strings with embedded newlines

9.1.2 Single-Quoted Style (')

Full Description

Single-quoted scalars are literal strings. No escape sequences except '' for apostrophe. Backslashes are literal. Use when you want exact literal content.

Syntax
key: 'literal string with ''apostrophe'''
Parameters

'' - Only escape: two single quotes become one apostrophe

Return Value

Literal string content (backslash-n is literally "\n", not newline).

Examples
# Example 1: Apostrophe in text
message: 'It''s a beautiful day'
quote: 'John''s car isn''t here'
Result: message = "It's a beautiful day", quote = "John's car isn't here"
# Example 2: Literal backslashes (no escape)
path: 'C:\Users\john\Documents'
regex: '^\d{3}-\d{4}$'
Result: path = "C:\Users\john\Documents" (backslashes literal)
# Example 3: Literal escape sequences (not interpreted)
literal_n: 'Line1\nLine2'
literal_t: 'Col1\tCol2'
Result: literal_n = "Line1\nLine2" (4 chars: \, n not newline)
# Example 4: Special YAML characters (safe in quotes)
colon_value: 'key: value'
bracket: '[not, a, list]'
brace: '{not: mapping}'
Result: All values are literal strings, not YAML structures
# Example 5: URLs and special strings
url: 'https://example.com/path?query=value&other=123'
pattern: '**/src/*.ts'
Result: Exact literal strings without interpretation

9.1.3 Plain (Unquoted) Style

Full Description

Plain scalars require no quotes for simple strings. However, certain starting characters and patterns trigger special interpretation (numbers, booleans, etc.).

Syntax
key: plain value without quotes
Parameters (Restrictions)

Cannot start with: - ? : , [ ] { } # & * ! | > ' " % @ `

Cannot contain: : # (followed by space)

Auto-typed values: true, false, yes, no, null, ~, numbers

Return Value

Auto-detected type: string, integer, float, boolean, or null.

Examples
# Example 1: Valid plain strings
name: John Doe
title: Senior Software Engineer
description: This is a plain text value
Result: All values are strings
# Example 2: Auto-typed as non-string (use quotes if string intended)
is_active: true      # boolean
count: 42            # integer
price: 19.99         # float
empty: null          # null
Result: true (bool), 42 (int), 19.99 (float), null
# Example 3: Must quote these (would be misinterpreted)
bool_string: "true"      # String, not boolean
num_string: "123"        # String, not integer
version: "1.0"           # String, not float
zip: "01234"             # String, preserves leading zero
Result: All quoted values remain strings
# Example 4: Colons and special chars need quotes
url: "http://example.com"    # Has colon
note: "# Not a comment"       # Starts with #
time: "10:30"                 # Time format
Result: Properly quoted to avoid YAML interpretation
# Example 5: Safe plain values
simple_path: /usr/local/bin
identifier: user_123
domain: example.com
Result: All are safe plain strings (no special chars)

9.2 Block Scalar Header

Full Description

Block scalars use indicators (| or >) followed by optional modifiers for indentation and chomping behavior.

9.2.1 Block Style Indicator

| (Literal) - Preserves line breaks exactly as newlines

> (Folded) - Converts line breaks to spaces (paragraph-style)

9.2.2 Indentation Indicator

Number after style indicator (1-9) specifies content indentation level. Required when content starts with spaces.

code: |2
    indented content
    starts here

9.2.3 Chomping Indicator

- (Strip) - Remove all trailing newlines

(default/Clip) - Keep single trailing newline

+ (Keep) - Keep all trailing newlines

Examples

  content
clip: |
  content
keep: |+
  content
Result: strip = "content", clip = "content\n", keep = "content\n\n"
# Example 2: Combined indicators
code: |2-
    def hello():
        print("Hi")
Result: Indentation 2, strip trailing newlines

Full Description

  line 2

Return Value

# Example 4: HTML template
template: |
  <!DOCTYPE html>
  <html>
  <head>
    <title>{{ title }}</title>
  </head>
  <body>
    <h1>{{ heading }}</h1>

# Example 5: ASCII art / diagram

Full Description
