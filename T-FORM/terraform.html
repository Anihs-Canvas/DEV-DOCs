Why Terraform for Django Applications?

Django applications like anihpj require complex infrastructure:

  • Compute: EC2 instances or ECS/EKS containers running Gunicorn/uWSGI
  • Database: RDS PostgreSQL with Multi-AZ, read replicas, automated backups
  • Storage: S3 for media files, CloudFront for static assets
  • Caching: ElastiCache Redis for sessions, database query caching
  • Load Balancing: ALB with health checks and SSL termination
  • Networking: VPC, subnets, security groups, NAT gateways
  • Secrets: Secrets Manager for database credentials, API keys
  • Monitoring: CloudWatch logs, metrics, alarms

Terraform lets you define all of this in code, deploy consistently across environments (dev/staging/prod), and avoid manual configuration errors.

1.1 Introduction to Terraform

What is Terraform?

Terraform is an open-source Infrastructure as Code (IaC) tool developed by HashiCorp. It allows you to define, provision, and manage cloud infrastructure using a declarative configuration language called HCL (HashiCorp Configuration Language). Instead of manually clicking through cloud consoles or writing imperative scripts, you describe your desired infrastructure state, and Terraform figures out how to achieve it.

Django Example: anihpj Job Portal

Throughout this documentation, we'll use the anihpj Job Portal Django application as our practical example. You'll learn to deploy its 4 models (Skills, Author, Location, JobPost) on AWS with EC2, RDS PostgreSQL, S3, and more—all managed with Terraform!

👀 Your First Look at Terraform Code

Here's a simple example creating an S3 bucket for Django static files:

main.tf 
# Configure the AWS Provider
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = "us-east-1"
}

# Create S3 bucket for Django static files
resource "aws_s3_bucket" "django_static" {
  bucket = "anihpj-jobportal-static-files"

  tags = {
    Name        = "Django Static Files"
    Project     = "anihpj"
    Environment = "production"
    ManagedBy   = "Terraform"
  }
}

# Output the bucket name
output "bucket_name" {
  value = aws_s3_bucket.django_static.id
}
📖
What This Code Does

Lines 2-9: Specifies we need the AWS provider version 5.x
Lines 11-13: Configures AWS to use the us-east-1 region
Lines 16-26: Creates an S3 bucket named "anihpj-jobportal-static-files" with tags
Lines 29-31: Outputs the bucket name after creation

1.2 Core Concepts

Understanding Terraform's core concepts is essential for effective infrastructure management. Let's explore each building block.

🧩 Terraform Building Blocks

🔌 Providers

🔌

What are Providers?

Providers are plugins that allow Terraform to interact with cloud platforms, SaaS providers, and other APIs. Each provider adds a set of resource types and data sources that Terraform can manage. The AWS provider, for example, lets you manage EC2, S3, RDS, and hundreds of other AWS services.

📚
Provider Registry
registry.terraform.io hosts 3000+ providers
🔢
Versioning
Pin versions with ~> 5.0 constraints
✅
Official vs Community
HashiCorp maintains official providers
🏷️
Aliases
Multiple instances for multi-region
providers.tf 
# Required providers block
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"  # Use version 5.x
    }
  }
  required_version = ">= 1.5.0"
}

# Primary AWS provider (us-east-1)
provider "aws" {
  region = "us-east-1"
  
  default_tags {
    tags = {
      Project   = "anihpj"
      ManagedBy = "Terraform"
    }
  }
}

# Secondary provider for different region (alias)
provider "aws" {
  alias  = "west"
  region = "us-west-2"
}

📦 Resources

📦

What are Resources?

Resources are the most important element in Terraform. Each resource block describes one or more infrastructure objects, such as EC2 instances, S3 buckets, or RDS databases. Resources have arguments (inputs), attributes (outputs), and can depend on other resources.

Resource Block Structure

ec2.tf - Django Server Resource 
# EC2 Instance for Django Application (anihpj Job Portal)
resource "aws_instance" "django_server" {
  ami                    = data.aws_ami.ubuntu.id
  instance_type          = var.instance_type
  subnet_id              = aws_subnet.private[0].id
  vpc_security_group_ids = [aws_security_group.django.id]
  iam_instance_profile   = aws_iam_instance_profile.django.name
  key_name               = aws_key_pair.deployer.key_name

  root_block_device {
    volume_size = 30
    volume_type = "gp3"
    encrypted   = true
  }

  user_data = templatefile("${path.module}/scripts/userdata.sh", {
    db_host     = aws_db_instance.postgres.address
    db_name     = "anihpj_db"
    bucket_name = aws_s3_bucket.static.id
  })

  tags = {
    Name        = "anihpj-django-server"
    Application = "JobPortal"
    Model       = "JobPost, Skills, Author, Location"
  }

  # Lifecycle rules
  lifecycle {
    create_before_destroy = true
  }
}

🔍 Data Sources

🔍

What are Data Sources?

Data Sources allow Terraform to query and use information from existing infrastructure that wasn't created by your Terraform configuration. Use them to fetch AMI IDs, look up existing VPCs, or get information about AWS services.

💡
Data Source vs Resource

Resource: Creates and manages infrastructure (CRUD operations)
Data Source: Only reads existing infrastructure (Read-only)

data.tf - Common Data Sources 
# Get latest Ubuntu AMI for Django server
data "aws_ami" "ubuntu" {
  most_recent = true
  owners      = ["099720109477"]  # Canonical

  filter {
    name   = "name"
    values = ["ubuntu/images/hvm-ssd/ubuntu-jammy-22.04-amd64-server-*"]
  }

  filter {
    name   = "virtualization-type"
    values = ["hvm"]
  }
}

# Get current AWS account info
data "aws_caller_identity" "current" {}

# Get available AZs in region
data "aws_availability_zones" "available" {
  state = "available"
}

# Look up existing VPC by tag (if not creating new)
data "aws_vpc" "existing" {
  filter {
    name   = "tag:Name"
    values = ["production-vpc"]
  }
}

# Usage in resources:
# ami = data.aws_ami.ubuntu.id
# account_id = data.aws_caller_identity.current.account_id

🔄 Variables & Outputs

📥

Input Variables

Parameters that allow customization of Terraform configurations without altering the source code. Define once, use across environments.

📤

Output Values

Export important values after apply—like instance IPs, database endpoints, or bucket names—for use in scripts or other configs.

variables.tf - Input Variables for anihpj 
# Environment variable
variable "environment" {
  description = "Deployment environment (dev, staging, production)"
  type        = string
  default     = "dev"

  validation {
    condition     = contains(["dev", "staging", "production"], var.environment)
    error_message = "Environment must be dev, staging, or production."
  }
}

# Instance type with different defaults per environment
variable "instance_type" {
  description = "EC2 instance type for Django server"
  type        = string
  default     = "t3.small"
}

# Database configuration as an object
variable "database" {
  description = "RDS PostgreSQL configuration"
  type = object({
    instance_class    = string
    allocated_storage = number
    db_name           = string
  })
  default = {
    instance_class    = "db.t3.micro"
    allocated_storage = 20
    db_name           = "anihpj_db"
  }
}

# Sensitive variable for secrets
variable "db_password" {
  description = "Database master password"
  type        = string
  sensitive   = true  # Won't show in logs
}
outputs.tf - Export Values 
# Output the Django server's public IP
output "django_server_ip" {
  description = "Public IP of the Django EC2 instance"
  value       = aws_instance.django_server.public_ip
}

# Output the RDS endpoint for Django settings.py
output "database_endpoint" {
  description = "RDS PostgreSQL endpoint"
  value       = aws_db_instance.postgres.endpoint
}

# Output S3 bucket for static files
output "static_bucket" {
  description = "S3 bucket name for Django static files"
  value       = aws_s3_bucket.static.id
}

# Sensitive output (hidden in CLI)
output "database_connection_string" {
  description = "Full database connection string"
  value       = "postgresql://${var.db_username}:${var.db_password}@${aws_db_instance.postgres.endpoint}/${var.database.db_name}"
  sensitive   = true
}

📊 State Management

📊

What is Terraform State?

State is Terraform's record of the infrastructure it manages. The state file (terraform.tfstate) maps your configuration to real-world resources, tracks metadata, and improves performance. Without state, Terraform wouldn't know which resources it created or how to update them.

State File Security

The state file contains sensitive information including passwords, keys, and resource IDs. Never commit terraform.tfstate to Git! Use remote backends with encryption (S3 + DynamoDB).

🔄 The Terraform Workflow (Write → Plan → Apply)

1️⃣ Write - Create Configuration

Define your infrastructure in .tf files using HCL. Organize code into logical files (main.tf, variables.tf, outputs.tf).

2️⃣ Init - Initialize Working Directory

Run terraform init to download providers, initialize backend, and prepare the working directory.

3️⃣ Validate - Check Syntax

Run terraform validate to check configuration syntax and internal consistency without accessing remote services.

4️⃣ Plan - Preview Changes

Run terraform plan to see what changes Terraform will make. Review carefully before proceeding!

5️⃣ Apply - Create/Update Resources

Run terraform apply to execute the plan and create/modify real infrastructure. Requires confirmation.

6️⃣ Destroy - Teardown (Optional)

Run terraform destroy to remove all resources. Use with caution in production!

💡
Pro Tip: Save Your Plan

Use terraform plan -out=tfplan to save the plan, then terraform apply tfplan to apply exactly what you reviewed. This prevents drift between plan and apply in CI/CD pipelines.

1.4 HCL Language Basics

HashiCorp Configuration Language (HCL) is Terraform's configuration language. It's designed to be both human-readable and machine-friendly.

📝

What is HCL?

HCL (HashiCorp Configuration Language) is a declarative language designed for defining infrastructure. It uses a block-based structure that's easier to read than JSON but maintains machine parseability. Terraform also supports JSON syntax, but HCL is preferred for human-written configurations.

📄 HCL vs JSON Syntax

Terraform accepts both HCL (.tf) and JSON (.tf.json) formats. Here's the same configuration in both:

main.tf (HCL - Recommended) 
resource "aws_s3_bucket" "static" {
  bucket = "anihpj-static"
  
  tags = {
    Name = "Django Static"
    Env  = "production"
  }
}
main.tf.json (JSON) 
{
  "resource": {
    "aws_s3_bucket": {
      "static": {
        "bucket": "anihpj-static",
        "tags": {
          "Name": "Django Static",
          "Env": "production"
        }
      }
    }
  }
}
💡
When to Use JSON?

Use JSON only for machine-generated configurations (e.g., from other tools). For human-written code, always use HCL for readability.

# S3 Bucket for Django Static Files
# ===========================================

terraform {
  required_version = ">= 1.5.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }

  region = var.aws_region

  default_tags {
    tags = {
      Project     = "anihpj"
      Environment = var.environment
      ManagedBy   = "Terraform"
    }
  }
}

resource "aws_s3_bucket" "django_static" {
  bucket = "${var.project_name}-static-${var.environment}"

  tags = {
    Name        = "Django Static Files"
    Application = "JobPortal"
  }

resource "aws_s3_bucket_versioning" "django_static" {
  bucket = aws_s3_bucket.django_static.id

  versioning_configuration {
    status = "Enabled"

}

# Enable server-side encryption
resource "aws_s3_bucket_server_side_encryption_configuration" "django_static" {
  bucket = aws_s3_bucket.django_static.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"
    }
  }
}

resource "aws_s3_bucket_public_access_block" "django_static" {
  bucket = aws_s3_bucket.django_static.id

  ignore_public_acls      = true

Common CLI Mistakes to Avoid
  • Running apply without plan: Always run terraform plan first to preview changes
  • Using -auto-approve in production: Reserve auto-approve for CI/CD only, never manual deployments
  • Ignoring state lock errors: Don't force-unlock unless you're certain no one else is running Terraform

  • Forgetting workspace context: Always check terraform workspace show before applying
  • Not saving plan files: Use -out=plan.tfplan to ensure apply matches plan exactly

🔓 terraform force-unlock (State Lock Management)

⚠️
Danger Zone: Use With Extreme Caution!

force-unlock should only be used when you're certain no other Terraform process is running. Force unlocking while another process is active can corrupt your state!

Terminal - force-unlock commands 
# When you see this error:
# Error: Error acquiring the state lock
# Lock Info:
#   ID:        12345678-1234-1234-1234-123456789012
#   Path:      terraform.tfstate
#   Operation: OperationTypePlan
#   Who:       user@hostname
#   Created:   2024-01-15 10:30:00.000000000 +0000 UTC

# Force release the lock (use the ID from error message)
terraform force-unlock 12345678-1234-1234-1234-123456789012

# Skip confirmation prompt (even more dangerous!)
terraform force-unlock -force 12345678-1234-1234-1234-123456789012
Scenario Safe to Force-Unlock? Recommended Action
CI/CD job crashed mid-run ✓ Usually safe Wait 5 min, then force-unlock
Terraform process killed (Ctrl+C) ✓ Safe Force-unlock immediately
Network timeout during apply ⚠️ Check state first Verify state, then unlock
Another team member is running ✗ Not safe Wait for them to finish
Lock from different machine/user ⚠️ Investigate first Contact lock owner first

# settings.py
import boto3
import json

# Initialize AWS clients
secrets_client = boto3.client('secretsmanager', region_name='us-east-1')
ssm_client = boto3.client('ssm', region_name='us-east-1')

def get_secret(secret_name):
    """Retrieve secret from Secrets Manager"""
    try:
        response = secrets_client.get_secret_value(SecretId=secret_name)
        return json.loads(response['SecretString'])
    except Exception as e:
        print(f"Error retrieving secret: {e}")
        return None

def get_parameter(parameter_name):
    """Retrieve parameter from SSM Parameter Store"""
    try:
        response = ssm_client.get_parameter(
            Name=parameter_name,
            WithDecryption=True
        )
        return response['Parameter']['Value']
    except Exception as e:
        print(f"Error retrieving parameter: {e}")
        return None

def get_parameters_by_path(path):
    """Retrieve multiple parameters by path"""
    try:
        response = ssm_client.get_parameters_by_path(
            Path=path,
            Recursive=True,
            WithDecryption=True
        )
        return {p['Name'].split('/')[-1]: p['Value'] for p in response['Parameters']}
    except Exception as e:
        print(f"Error retrieving parameters: {e}")
        return {}

ENVIRONMENT = os.environ.get('ENVIRONMENT', 'development')

# Get Django secret key from Secrets Manager
secret_key_data = get_secret(f'anihpj/{ENVIRONMENT}/django/secret-key')

# Get database credentials
db_credentials = get_secret(f'anihpj/{ENVIRONMENT}/rds/credentials')

    DATABASES = {
        'default': {
            'ENGINE': 'django.db.backends.postgresql',
            'NAME': db_credentials['dbname'],

            'PASSWORD': db_credentials['password'],
            'HOST': db_credentials['host'].split(':')[0],

4.1 Module Basics

📦

What is a Module?

A module is a container for multiple resources used together. Every Terraform configuration has at least one module (root module). Child modules can be called from other modules to create reusable components.

🏠

Root Module

Your main working directory with .tf files

👶

Child Module

Reusable module called from other modules

📂

Local Module

Module in a local directory path

🌐

Remote Module

Module from registry, Git, or S3

5. State Management

Terraform state is the source of truth for your infrastructure. Master state management to work safely in teams.

5.1 Understanding State

📊

What is Terraform State?

State is a JSON file that maps your configuration to real-world resources. It tracks resource IDs, metadata, and dependencies to determine what changes need to be made.

🔄 State's Role in Terraform Workflow

⚠️
State Contains Sensitive Data

State files contain all resource attributes including secrets, passwords, and keys in plain text. Never commit state to Git! Always use remote backends with encryption.

import os

from botocore.exceptions import ClientError

def get_secret(secret_name: str) -> dict:
    """Retrieve secrets from AWS Secrets Manager."""
    session = boto3.session.Session()
    client = session.client(
        service_name='secretsmanager',
        region_name=os.environ.get('AWS_REGION', 'us-east-1')
    )

    try:
        response = client.get_secret_value(SecretId=secret_name)
        return json.loads(response['SecretString'])
    except ClientError as e:
        raise Exception(f"Failed to retrieve secret: {e}")

18.5 Drift Detection

# Scheduled drift detection with EventBridge + Lambda
.github/workflows/drift.yml - Scheduled Drift Check 
# Scheduled drift detection workflow
name: Drift Detection

on:
  schedule:
    - cron: '0 8 * * *'  # Daily at 8 AM UTC
  workflow_dispatch:

jobs:
  drift-check:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        environment: [staging, production]

    permissions:
      id-token: write
      contents: read
      issues: write

    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
          aws-region: us-east-1

      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v3

      - name: Terraform Init
        working-directory: terraform/environments/${{ matrix.environment }}
        run: terraform init

      - name: Check for Drift
        id: drift
        working-directory: terraform/environments/${{ matrix.environment }}
        run: |
          terraform plan -refresh-only -detailed-exitcode -no-color 2>&1 | tee plan.txt
          echo "exitcode=$?" >> $GITHUB_OUTPUT
        continue-on-error: true

      - name: Create Issue on Drift
        if: steps.drift.outputs.exitcode == '2'
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const plan = fs.readFileSync('terraform/environments/${{ matrix.environment }}/plan.txt', 'utf8');

            github.rest.issues.create({
              owner: context.repo.owner,
              repo: context.repo.repo,
              title: `🔄 Drift Detected: ${{ matrix.environment }}`,
              body: `Infrastructure drift detected in **${{ matrix.environment }}**.\n\n\`\`\`\n${plan.slice(0, 60000)}\n\`\`\``,
              labels: ['drift', '${{ matrix.environment }}']
            });

C.1 Terraform Terminology

Term Definition
Resource A single infrastructure object (e.g., aws_instance, aws_s3_bucket)
Data Source Read-only query to fetch information from providers
Provider Plugin that manages resources for a service (AWS, Azure, GCP)
Module Reusable container for multiple resources
State JSON file mapping configuration to real resources
Backend Where state is stored (local, S3, Terraform Cloud)
Workspace Isolated state instance within same configuration
Plan Preview of changes Terraform will make
Apply Execute planned changes to infrastructure
Drift Difference between state and actual infrastructure
Taint Mark resource for recreation (deprecated, use -replace)
Provisioner Scripts that run during resource creation/destruction
Local Values Named expressions for reuse within a module (locals)
Output Values Return values from a module or root configuration
Input Variables Parameters that customize module behavior
