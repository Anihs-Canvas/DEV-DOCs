    About YAML syntax for workflows

    Docs: GitHub Actions Workflow Syntax

    GitHub Actions workflows are defined in YAML files stored in the .github/workflows directory of your repository.

    Full description

    YAML (YAML Ain't Markup Language) is a human-readable data serialization standard used by GitHub Actions to define workflows. YAML files use indentation and key-value pairs to structure data. GitHub Actions workflow files must be valid YAML and follow specific syntax rules for defining automated processes.

    Syntax

    # Basic YAML structure for workflows
key: value
list:
  - item1
  - item2
nested:
  subkey: subvalue

    Parameters description

    Indentation: YAML uses spaces (not tabs) for indentation. Use 2 spaces per level.
    Key-value pairs: Use colon and space (key: value) syntax.
    Lists: Use dash and space (- item) for list items.
    Strings: Can be unquoted, single-quoted, or double-quoted.
    Comments: Use hash symbol (#) for comments.

    Return value

    YAML files are parsed by GitHub Actions into workflow definitions that execute automated processes.

    Examples

    1. Basic Django test workflow

    name: Django Tests
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Test models
        run: python manage.py test jobpost
    ✅ Workflow validates Django models syntax

    2. Environment variables

    env:
  DJANGO_SETTINGS_MODULE: myproject.settings.test
  DATABASE_URL: sqlite:///test.db
    🔧 Sets environment for Django testing

    3. Matrix strategy

    strategy:
  matrix:
    python-version: [3.9, 3.10, 3.11]
    django-version: [3.2, 4.1, 4.2]
    🔄 Tests 9 combinations of Python/Django versions

    4. Conditional execution

    - name: Deploy to production
  if: github.ref == 'refs/heads/main'
  run: python manage.py migrate
    🚀 Deploys only from main branch

    5. Multi-line commands

    - name: Setup Django
  run: |
    pip install -r requirements.txt
    python manage.py migrate
    python manage.py collectstatic --noinput
    📦 Installs deps, migrates DB, collects static files

    The name of the workflow displayed in the GitHub Actions tab.

    Full description

    The name field defines a human-readable name for your workflow. This name appears in the GitHub Actions tab of your repository, in pull request status checks, and in the workflow run history. If omitted, GitHub uses the workflow filename as the name.

    Syntax

    name: "Your Workflow Name"

    Parameters description

    String value: Any descriptive text to identify your workflow.
    Optional: If not provided, GitHub uses the filename.
    Best practice: Use descriptive names that clearly indicate the workflow's purpose.

    Return value

    Displays the specified name in GitHub's UI for easy workflow identification.

    run-name

    Docs: workflow run name

    Customizes the name of individual workflow runs with dynamic expressions.

    Full description

    The run-name field allows you to customize the display name for each workflow run using GitHub Actions expressions. This helps distinguish between different runs and provides context about what triggered the workflow or what it's processing.

    Syntax

    run-name: "Custom run name with ${{ expression }}"

    Parameters description

    Expression support: Can include GitHub Actions expressions using ${{ }} syntax.
    Context variables: Access to github, env, vars, and inputs contexts.
    Dynamic content: Can include commit info, branch names, or input parameters.

    Return value

    Displays the custom name for the workflow run in the GitHub Actions interface.

    Examples

    1. Include committer info

    run-name: "JobPost tests by ${{ github.actor }}"
on: [push, pull_request]
    🏃 Run name: "JobPost tests by john-doe"

    2. Branch-specific naming

    run-name: "Deploy ${{ github.ref_name }} branch"
on:
  push:
    branches: [main, staging]
    🏃 Run name: "Deploy main branch"

    3. Event-based naming

    run-name: "PR #${{ github.event.number }} - ${{ github.event.pull_request.title }}"
on: [pull_request]
    🏃 Run name: "PR #42 - Add JobPost model validation"

    4. Include commit message

    run-name: "${{ github.event.head_commit.message }}"
on: [push]
    🏃 Run name: "Fix JobPost author relationship"

    5. Environment deployment

    run-name: "Deploy to ${{ inputs.environment }} - ${{ github.sha }}"
on:
  workflow_dispatch:
    inputs:
      environment:
        type: choice
        options: [staging, production]
    🏃 Run name: "Deploy to staging - abc1234"

    on

    Docs: workflow triggers

    Defines the events that trigger the workflow to run.

    Full description

    The on field specifies which repository events will trigger your workflow. You can configure workflows to run on push, pull requests, issues, releases, schedules, and many other GitHub events. Multiple triggers can be combined to create flexible automation rules.

    Syntax

    on: event_name
# or
on: [event1, event2]
# or
on:
  event_name:
    # event configuration

    Parameters description

    Single event: on: push for one trigger.
    Multiple events: on: [push, pull_request] for multiple triggers.
    Configured events: Use object syntax for event-specific configuration.
    Common events: push, pull_request, schedule, workflow_dispatch, release.

    Return value

    Triggers workflow execution when specified events occur in the repository.

    Examples

    1. Basic push trigger

    name: "Test JobPost Models"
on: push
jobs:
  test:
    runs-on: ubuntu-latest
    🔄 Triggers on every push to any branch

    2. Multiple events

    name: "Django CI"
on: [push, pull_request, workflow_dispatch]
jobs:
  test:
    runs-on: ubuntu-latest
    🔄 Triggers on push, PRs, and manual dispatch

    3. Scheduled workflow

    name: "Daily JobPost Cleanup"
on:
  schedule:
    - cron: '0 0 * * *'
jobs:
  cleanup:
    runs-on: ubuntu-latest
    🔄 Triggers daily at midnight UTC

    4. Release automation

    name: "Deploy on Release"
on:
  release:
    types: [published]
jobs:
  deploy:
    runs-on: ubuntu-latest
    🔄 Triggers when a release is published

    5. Issue automation

    name: "Auto-label JobPost issues"
on:
  issues:
    types: [opened, labeled]
jobs:
  label:
    runs-on: ubuntu-latest
    🔄 Triggers when issues are opened or labeled

    on.<pull_request|pull_request_target>.<branches|branches-ignore>

    Docs: branch filtering

    Filters pull request events based on target branch names.

    Full description

    When using pull request triggers, you can filter which branches will trigger the workflow based on the target branch (base branch) of the pull request. Use branches to include specific branches or branches-ignore to exclude them.

    Syntax

    on:
  pull_request:
    branches: [branch1, branch2]
    # or
    branches-ignore: [branch1, branch2]

    Parameters description

    branches: Include only PRs targeting these branches.
    branches-ignore: Exclude PRs targeting these branches.
    Wildcards: Support glob patterns like feature/*.
    Mutually exclusive: Cannot use both branches and branches-ignore.

    Return value

    Workflow runs only for pull requests targeting the specified branches.

    Examples

    1. Main branch PRs only

    name: "JobPost Main PR Tests"
on:
  pull_request:
    branches: [main]
jobs:
  test:
    runs-on: ubuntu-latest
    🎯 Runs only for PRs targeting main branch

    2. Production branches

    name: "Production PR Review"
on:
  pull_request:
    branches: [main, production, release/*]
jobs:
  security-scan:
    runs-on: ubuntu-latest
    🎯 Runs for PRs to main, production, or release/* branches

    3. Ignore development branches

    name: "JobPost Stable Tests"
on:
  pull_request:
    branches-ignore: [dev, feature/*, experimental]
jobs:
  test:
    runs-on: ubuntu-latest
    🚫 Skips PRs to dev, feature/*, experimental branches

    4. Multiple stable branches

    name: "JobPost Release Tests"
on:
  pull_request:
    branches: [main, staging, v*.x]
jobs:
  integration-test:
    runs-on: ubuntu-latest
    🎯 Runs for PRs to main, staging, or version branches

    tags: Include pushes of specific tags.
    branches-ignore: Exclude pushes to specific branches.
    tags-ignore: Exclude pushes of specific tags.
    Glob patterns: Use wildcards for flexible matching.

    Return value

    Workflow runs only for pushes matching the specified branch or tag patterns.

  push:

    Filter pattern cheat sheet

    Docs: pattern matching

    Common patterns for filtering branches, tags, and paths in workflows.

    Full description

    GitHub Actions supports glob patterns for filtering branches, tags, and file paths. These patterns allow flexible matching rules to trigger workflows only when specific conditions are met. Understanding these patterns is crucial for effective workflow automation.

    Syntax

    # Wildcards
*         # matches any character except /
**        # matches any character including /
?         # matches any single character
[abc]     # matches any character in the set
[!abc]    # matches any character not in the set

    Parameters description

    Single asterisk (*): Matches any string except slash.
    Double asterisk (**): Matches any string including slashes.
    Question mark (?): Matches exactly one character.
    Brackets []: Character class matching.
    Negation [!]: Matches characters not in the set.

    Return value

    Boolean match result determining if the pattern matches the target string.

    Examples

    1. JobPost model files

    on:
  push:
    paths: 
      - 'jobpost/models.py'
      - 'jobpost/models/**'
      - '**/models.py'
    📁 Matches JobPost models and any models.py files

    2. Version branches

    on:
  push:
    branches:
      - 'main'
      - 'release-*'
