    About events that trigger workflows

    Docs: GitHub Actions Events Documentation

    GitHub Actions workflows are triggered by events that occur in your repository or organization, enabling powerful automation for your JobPost application.

    Full description

    GitHub Actions workflows are triggered by events that occur in your repository or organization. These events can be related to code changes, issues, pull requests, deployments, schedules, and more. Understanding these events is essential for automating your CI/CD pipelines and customizing workflow behavior for your JobPost Django application.

    Event Categories

    Repository Events: push, pull_request, issues, fork, watch
    Workflow Events: workflow_dispatch, workflow_call, workflow_run
    Scheduled Events: schedule (cron-based triggers)
    External Events: repository_dispatch, deployment, release
    Check Events: check_run, check_suite, status

    Syntax

    on:
  # Single event
  push
  
  # Multiple events
  [push, pull_request]
  
  # Event with configuration
  push:
    branches: [main]
    paths: ['jobpost/**']

    Parameters description

    Event filters: Control when workflows run with branches, paths, and types filters.
    Activity types: Many events support specific activity types (opened, closed, etc.).
    Webhook payloads: Each event provides context data via github.event.
    Security considerations: Some events have restricted permissions for external contributors.

    Return value

    Triggers workflow execution with event-specific context data available via expressions.

    Examples

    1. JobPost code changes

    on:
  push:
    branches: [main, develop]
    paths: ['jobpost/**', 'requirements.txt']
    🔄 Triggers on JobPost app changes or dependency updates

    2. Multiple event triggers

    on: [push, pull_request, issues]
jobs:
  analyze:
    if: github.event_name == 'push'
    ⚡ Responds to multiple event types with conditional logic

    3. JobPost deployment events

    on:
  release:
    types: [published]
  deployment:
    types: [created]
    🚀 Automates JobPost deployments on releases and deployment events

    4. Scheduled JobPost maintenance

    on:
  schedule:
    - cron: '0 2 * * *'  # Daily at 2 AM
    - cron: '0 0 * * 0'  # Weekly on Sunday
    ⏰ Runs JobPost maintenance tasks on scheduled intervals

    5. External JobPost automation

    on:
  repository_dispatch:
    types: [jobpost-cleanup, data-sync]
  workflow_dispatch:
    inputs:
      job_count:
        type: number
    🎯 Enables external systems to trigger JobPost workflows

    branch_protection_rule

    Docs: branch_protection_rule event

    Triggers workflows when branch protection rules are created, edited, or deleted.

    Full description

    The branch_protection_rule event triggers workflows when branch protection rules change in your repository. This is useful for automating security compliance checks, updating CI/CD configurations, or notifying teams about protection changes that affect your JobPost deployment workflow.

    Syntax

    on:
  branch_protection_rule:
    types: [created, edited, deleted]

    Parameters description

    created: A new branch protection rule was created.
    edited: An existing branch protection rule was modified.
    deleted: A branch protection rule was removed.
    Payload: Contains rule details in github.event.rule.

    Return value

    Workflow runs with access to branch protection rule details and changes.

    Examples

    1. JobPost protection monitoring

    on:
  branch_protection_rule:
    types: [created, edited, deleted]

jobs:
  notify-team:
    runs-on: ubuntu-latest
    steps:
      - name: Notify JobPost team
        run: |
          echo "Branch protection changed: ${{ github.event.action }}"
          echo "Rule: ${{ github.event.rule.pattern }}"
    🛡️ Notifies JobPost team of protection rule changes

    2. Security compliance check

    on:
  branch_protection_rule:
    types: [edited, deleted]

jobs:
  compliance-check:
    if: github.event.rule.pattern == 'main'
    runs-on: ubuntu-latest
    steps:
      - name: Validate JobPost main branch protection
        run: |
          if [ "${{ github.event.action }}" = "deleted" ]; then
            echo "❌ Main branch protection removed - JobPost security compromised"
            exit 1
          fi
    🔒 Validates JobPost main branch remains protected

    3. Update deployment workflow

    on:
  branch_protection_rule:
    types: [created]

jobs:
  update-deployment:
    if: contains(github.event.rule.pattern, 'release')
    runs-on: ubuntu-latest
    steps:
      - name: Update JobPost deployment rules
        run: |
          echo "New protected branch: ${{ github.event.rule.pattern }}"
          # Update deployment configuration
    📝 Updates JobPost deployment config for new protected branches

    4. Audit logging

    on:
  branch_protection_rule

jobs:
  audit-log:
    runs-on: ubuntu-latest
    steps:
      - name: Log JobPost protection changes
        run: |
          echo "$(date): ${{ github.actor }} ${{ github.event.action }} protection rule"
          echo "Pattern: ${{ github.event.rule.pattern }}"
          echo "Required reviews: ${{ github.event.rule.required_pull_request_reviews.required_approving_review_count }}"
    📋 Creates audit log of JobPost branch protection changes

    5. CI configuration sync

    on:
  branch_protection_rule:
    types: [created, edited]

jobs:
  sync-ci:
    runs-on: ubuntu-latest
    steps:
      - name: Sync JobPost CI with protection rules
        run: |
          PATTERN="${{ github.event.rule.pattern }}"
          if [[ "$PATTERN" == *"jobpost"* ]]; then
            echo "Updating CI for JobPost-related branch: $PATTERN"
            # Update CI configuration
          fi
    ⚙️ Syncs JobPost CI configuration with new protection rules

    pull_request_comment (use issue_comment)

    Docs: issue_comment event

    Note: There is no separate pull_request_comment event. Use issue_comment instead, which triggers for both issue and pull request comments.

    Full description

    GitHub does not have a separate pull_request_comment event. Comments on pull requests are handled by the issue_comment event, which triggers for comments on both issues and pull requests. You can differentiate between the two by checking if github.event.issue.pull_request exists in your workflow.

    Syntax

    on:
  issue_comment:
    types: [created, edited, deleted]

jobs:
  handle-pr-comment:
    if: github.event.issue.pull_request
    # This job only runs for pull request comments

    Parameters description

    Unified event: issue_comment handles both issue and PR comments.
    Detection: Use github.event.issue.pull_request to identify PR comments.
    Same functionality: All comment actions work the same way.
    Context access: Full access to comment content and PR details.

    Return value

    Use the issue_comment event to handle pull request comments with proper filtering.

    Examples

    1. JobPost PR comment handler

    on:
  issue_comment:
    types: [created]

jobs:
  handle-pr-comment:
    if: github.event.issue.pull_request
    runs-on: ubuntu-latest
    steps:
      - name: Handle JobPost PR comment
        run: |
          echo "Comment on JobPost PR #${{ github.event.issue.number }}"
          echo "Comment by: ${{ github.event.comment.user.login }}"
          echo "Comment: ${{ github.event.comment.body }}"
    💬 Handles JobPost pull request comments using issue_comment event

    2. Differentiate comments

    on:
  issue_comment:
    types: [created]

jobs:
  route-comment:
    runs-on: ubuntu-latest
    steps:
      - name: Route JobPost comment
        run: |
          if [ "${{ github.event.issue.pull_request }}" != "" ]; then
            echo "This is a JobPost PR comment"
          else
            echo "This is a JobPost issue comment"
          fi
    🔀 Differentiates between JobPost issue and PR comments

    3. PR-specific commands

    on:
  issue_comment:
    types: [created]

jobs:
  pr-commands:
    if: |
      github.event.issue.pull_request && 
      startsWith(github.event.comment.body, '/deploy')
    runs-on: ubuntu-latest
    steps:
      - name: Handle JobPost deploy command
        run: |
          echo "Deploy command for JobPost PR #${{ github.event.issue.number }}"
          echo "Command: ${{ github.event.comment.body }}"
    🚀 Handles JobPost deployment commands in PR comments

    4. Access PR details from comment

    on:
  issue_comment:
    types: [created]

jobs:
  pr-info:
    if: github.event.issue.pull_request
    runs-on: ubuntu-latest
    steps:
      - name: Get JobPost PR info
        run: |
          echo "JobPost PR URL: ${{ github.event.issue.pull_request.url }}"
          echo "PR title: ${{ github.event.issue.title }}"
          echo "PR state: ${{ github.event.issue.state }}"
    📋 Accesses JobPost PR details from comment events

    5. Combined issue and PR handling

    on:
  issue_comment:
    types: [created]

jobs:
  unified-handler:
    runs-on: ubuntu-latest
    steps:
      - name: Handle JobPost comment
        run: |
          TYPE=${{ github.event.issue.pull_request && 'PR' || 'issue' }}
          echo "JobPost $TYPE comment: ${{ github.event.comment.body }}"
          echo "On: ${{ github.event.issue.title }}"
    🎯 Unified handler for both JobPost issue and PR comments
