The check_run event triggers workflows when check runs are created, completed, re-requested, or requested for action. rerequested: Someone requested to re-run the check.

Return value

Workflow runs with access to check run status, conclusion, and output details.

    About events that trigger workflows

    Docs: GitHub Actions Events Documentation

    GitHub Actions workflows are triggered by events that occur in your repository or organization, enabling powerful automation for your JobPost application.

    Full description

    GitHub Actions workflows are triggered by events that occur in your repository or organization. These events can be related to code changes, issues, pull requests, deployments, schedules, and more. Understanding these events is essential for automating your CI/CD pipelines and customizing workflow behavior for your JobPost Django application.

    Event Categories

    Repository Events: push, pull_request, issues, fork, watch
    Workflow Events: workflow_dispatch, workflow_call, workflow_run
    Scheduled Events: schedule (cron-based triggers)
    External Events: repository_dispatch, deployment, release
    Check Events: check_run, check_suite, status

    Syntax

    on:
  # Single event
  push
  
  # Multiple events
  [push, pull_request]
  
  # Event with configuration
  push:
    branches: [main]
    paths: ['jobpost/**']

    Parameters description

    Event filters: Control when workflows run with branches, paths, and types filters.
    Activity types: Many events support specific activity types (opened, closed, etc.).
    Webhook payloads: Each event provides context data via github.event.
    Security considerations: Some events have restricted permissions for external contributors.

    Return value

    Triggers workflow execution with event-specific context data available via expressions.

    Examples

    1. JobPost code changes

    on:
  push:
    branches: [main, develop]
    paths: ['jobpost/**', 'requirements.txt']
    🔄 Triggers on JobPost app changes or dependency updates

    branch_protection_rule

    Docs: branch_protection_rule event

    Triggers workflows when branch protection rules are created, edited, or deleted.

    Full description

    The branch_protection_rule event triggers workflows when branch protection rules change in your repository. This is useful for automating security compliance checks, updating CI/CD configurations, or notifying teams about protection changes that affect your JobPost deployment workflow.

    Syntax

    on:
  branch_protection_rule:
    types: [created, edited, deleted]

    Parameters description

    created: A new branch protection rule was created.
    edited: An existing branch protection rule was modified.
    deleted: A branch protection rule was removed.
    Payload: Contains rule details in github.event.rule.

    Return value

    Workflow runs with access to branch protection rule details and changes.

    Examples

    1. JobPost protection monitoring

    on:
  branch_protection_rule:
    types: [created, edited, deleted]

jobs:
  notify-team:
    runs-on: ubuntu-latest
    steps:
      - name: Notify JobPost team
        run: |
          echo "Branch protection changed: ${{ github.event.action }}"
          echo "Rule: ${{ github.event.rule.pattern }}"
    🛡️ Notifies JobPost team of protection rule changes

    2. Security compliance check

    on:

    types: [edited, deleted]

jobs:
