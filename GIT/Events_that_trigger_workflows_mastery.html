    branch_protection_rule

    Docs: branch_protection_rule event

    Triggers workflows when branch protection rules are created, edited, or deleted.

    Full description

    The branch_protection_rule event triggers workflows when branch protection rules change in your repository. This is useful for automating security compliance checks, updating CI/CD configurations, or notifying teams about protection changes that affect your JobPost deployment workflow.

    Syntax

    on:
  branch_protection_rule:
    types: [created, edited, deleted]

    Parameters description

    created: A new branch protection rule was created.
    edited: An existing branch protection rule was modified.
    deleted: A branch protection rule was removed.
    Payload: Contains rule details in github.event.rule.

    Return value

    Workflow runs with access to branch protection rule details and changes.

    Examples

    1. JobPost protection monitoring

    on:
  branch_protection_rule:
    types: [created, edited, deleted]

jobs:
  notify-team:
    runs-on: ubuntu-latest
    steps:
      - name: Notify JobPost team
        run: |
          echo "Branch protection changed: ${{ github.event.action }}"
          echo "Rule: ${{ github.event.rule.pattern }}"
    🛡️ Notifies JobPost team of protection rule changes

    2. Security compliance check

    on:
  branch_protection_rule:
    types: [edited, deleted]

jobs:
  compliance-check:
    if: github.event.rule.pattern == 'main'
    runs-on: ubuntu-latest
    steps:
      - name: Validate JobPost main branch protection
        run: |
          if [ "${{ github.event.action }}" = "deleted" ]; then
            echo "❌ Main branch protection removed - JobPost security compromised"

    🔒 Validates JobPost main branch remains protected

    check_run

    Docs: check_run event

    Triggers workflows when check run activities occur on the repository.

    Full description

    The check_run event triggers workflows when check runs are created, completed, re-requested, or requested for action. This is useful for creating workflows that respond to the results of other CI checks, implement complex approval processes, or perform follow-up actions based on JobPost test results.

    Syntax

    on:
  check_run:
    types: [created, completed, rerequested, requested_action]

    Parameters description

    created: A new check run was created.
    completed: A check run has completed (success, failure, neutral, cancelled).
    rerequested: Someone requested to re-run the check.
    requested_action: An action button in the check run was clicked.
    Payload: Check run details available in github.event.check_run.

    Return value

    Workflow runs with access to check run status, conclusion, and output details.

    Examples

    1. JobPost test completion handler

    on:
  check_run:
    types: [completed]

jobs:
  handle-test-completion:
    if: github.event.check_run.name == 'JobPost Tests'
    runs-on: ubuntu-latest
    steps:
      - name: Handle JobPost test results
        run: |
          echo "Test status: ${{ github.event.check_run.conclusion }}"
          if [ "${{ github.event.check_run.conclusion }}" = "success" ]; then
            echo "✅ JobPost tests passed - triggering deployment"
          fi
    ✅ Handles JobPost test completion and triggers follow-up actions

    Full description

    The check_suite event triggers when check suites are created, completed, requested, or re-requested. This enables coordination of multiple related checks, batch processing of JobPost validation workflows, and comprehensive quality gate management for your Django application.

    Syntax

    on:
  check_suite:
    types: [completed, requested, rerequested]

    Parameters description

    completed: All checks in the suite have finished running.
    requested: A new check suite was requested for a commit.
    rerequested: Someone requested to re-run the entire check suite.
    Suite coordination: Manage multiple related checks as a group.
    Quality gates: Implement comprehensive validation workflows.

    Return value

    Workflow runs with access to check suite status and all contained check runs.

    Examples

    on:
  check_suite:
    types: [completed]

jobs:

    Full description

    The create event triggers when someone creates a branch or tag in your JobPost repository. This is useful for automating branch setup, initializing new feature environments, or responding to new release tags with deployment workflows.

    Syntax

  create

    ref_type: Either 'branch' or 'tag' to identify what was created.
    ref: The name of the created reference.

    Context access: Full access to repository and reference information.

    Return value

    Workflow runs with information about the newly created branch or tag.

    on:
  create

jobs:
  setup-feature-branch:
    if: github.event.ref_type == 'branch' && startsWith(github.event.ref, 'feature/')

    steps:
      - name: Setup JobPost feature branch
        run: |
          echo "🌿 Setting up JobPost feature branch"
          echo "Branch: ${{ github.event.ref }}"
          echo "Created by: ${{ github.event.sender.login }}"
          echo "Default branch: ${{ github.event.master_branch }}"
          
          # Create development environment
          echo "Creating development environment for JobPost feature"
          echo "Branch name: ${{ github.event.ref }}"
          
          # Setup branch protection if needed
          echo "Configuring branch settings for JobPost development"
    🌿 Automatically sets up JobPost feature branch environments

    Docs: deployment event

    Triggers workflows when deployments are created via the GitHub API.

    Full description

    The deployment event triggers workflows when deployments are created through the GitHub Deployments API. This event is useful for handling deployment requests from external systems, implementing custom deployment orchestration, and coordinating complex deployment processes for your JobPost application.

    Syntax

    on:
  deployment

    Parameters description

    No activity types: The deployment event doesn't use activity types.
    API-triggered only: Only triggered via GitHub Deployments API, not by pushes or releases.
    Environment info: Contains deployment environment and configuration.
    Payload access: Deployment details available in github.event.deployment.

    Return value

    Workflow runs with access to deployment environment, payload, and configuration.

    Examples

    1. JobPost deployment orchestration

    on:
  deployment

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy JobPost application
        run: |
          echo "🚀 JobPost deployment requested"
          echo "Environment: ${{ github.event.deployment.environment }}"
          echo "Ref: ${{ github.event.deployment.ref }}"
          echo "Task: ${{ github.event.deployment.task }}"
          echo "Payload: ${{ toJson(github.event.deployment.payload) }}"
          
          # Execute deployment based on environment
          if [ "${{ github.event.deployment.environment }}" = "production" ]; then
            echo "Deploying JobPost to production"
          fi
    🚀 Orchestrates JobPost deployments based on API requests

    About workflow commands

    Docs: Workflow commands documentation

    Use workflow commands to communicate with the GitHub Actions runner and enhance your workflow logs.

    Full description

    Workflow commands allow you to communicate with the runner machine to set environment variables, output values, add debug messages, and more. Most workflow commands use the echo command in a specific format. They can be used within workflow files or shell scripts run during a workflow. Workflow commands provide a way to interact with the Actions runner without requiring custom actions or additional tooling.

    Syntax

    ::workflow-command parameter1={data},parameter2={data}::{command value}

    Parameters description

    workflow-command: The command name (e.g., debug, notice, error, group)
    parameter1, parameter2: Optional parameters specific to each command
    command value: The message or value for the command
    Note: Command and parameter names are case-insensitive. Command values are case-sensitive.

    Understanding the order of precedence when the same variable is defined at multiple levels.

    Full description

    When you define a configuration variable with the same name at multiple levels (organization, repository, and environment), GitHub Actions uses a precedence hierarchy to determine which value to use. The variable at the most specific level takes precedence, allowing you to set organization-wide defaults while overriding them for specific repositories or environments. This is particularly useful for JobPost deployments across different environments (development, staging, production) where certain settings need to vary.

    Syntax

    # Access configuration variables (automatically resolves precedence)

  deploy:
    environment: production  # Uses production environment variables first
    steps:
      - run: echo "${{ vars.API_URL }}"

    Parameters description

    Precedence order (highest to lowest):
    1. Environment variable - Defined in a specific environment (production, staging, etc.)
    2. Repository variable - Defined for the specific repository
    3. Organization variable - Defined at organization level (lowest priority)
    Rules:
    • More specific levels override less specific levels
    • If variable not found at environment level, checks repository level
    • If variable not found at repository level, checks organization level
    • Variables are resolved at workflow execution time

    Return value

    Returns the value from the highest precedence level where the variable is defined. If not found at any level, the expression evaluates to an empty string.

          echo "Deploying JobPost to: ${{ vars.JOBPOST_DEPLOY_URL }}"

    Supported contexts

    Docs: Context availability

    Different contexts are available at different points in a workflow. Understanding where you can use vars, env, and other contexts.

    Full description

    Variables can be accessed through different contexts depending on where they're defined and where you're accessing them in your workflow. Each context has specific availability rules within workflows. The vars context provides access to configuration variables and is available in most workflow locations. The env context provides access to environment variables but only after they are set. Understanding context availability is crucial for building robust JobPost workflows that reference variables correctly at each stage.

    Syntax

    # vars context - available throughout workflow
${{ vars.VARIABLE_NAME }}

# env context - available after environment variables are set
${{ env.VARIABLE_NAME }}

# Access in shell (after variables are defined)
echo "$VARIABLE_NAME"

    Parameters description

    vars context:
    • Available in: workflow-level env, job-level if, step-level run
    • Available in: conditionals, expressions, action inputs
    • Not available in: workflow triggers (on), workflow name
    env context:
    • Available in: step-level run commands (after env is set)
    • Available in: expressions within steps (after variable is defined)
    • Not available in: step-level if conditionals (use shell conditionals instead)
    secrets context:
    • Available in: workflow file, limited in log outputs (masked)
    • Similar availability to vars context
    github context:
    • Available throughout: entire workflow at all levels
    • Automatically populated with workflow metadata

    Return value

    Contexts return string values when accessed via expressions. Availability depends on context type and workflow location. Accessing unavailable contexts results in empty strings or errors.
