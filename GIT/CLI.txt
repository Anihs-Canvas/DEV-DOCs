1. Setup & Configuration
git config
•	Description: Set author name/email or other preferences.
•	Syntax:
•	git config [--global] <name> "<value>"
•	Example:
•	git config --global user.name "Alice Doe"
•	git config --global user.email "alice@example.com"
git init
•	Description: Create a new Git repo in the current folder.
•	Syntax:
•	git init [<directory>]
•	Example:
•	cd ~/projects
•	git init my-django-app
•	cd my-django-app
git clone
•	Description: Copy a remote repo to your machine.
•	Syntax:
•	git clone <repository-url> [<directory>]
•	Example:
•	git clone git@github.com:acme/my-django-app.git

2. Viewing Status & Changes
git status
•	Description: See which files are staged, modified, or untracked.
•	Syntax:
•	git status [--short | --branch]
•	Example:
•	git status
git diff
•	Description: Show line-by-line changes between working copy and index or commits.
•	Syntax:
•	git diff [<commit>..<commit>] [--] [<path>…]
•	Example:
•	git diff                # unstaged changes
•	git diff --staged       # staged but uncommitted
•	git diff HEAD~1 HEAD    # compare last two commits
git log
•	Description: Browse commit history.
•	Syntax:
•	git log [--oneline] [--graph] [<path>…]
•	Example:
•	git log --oneline --graph --decorate

•	Description: Stage changes (new files, modifications, deletions).
•	Syntax:
•	git add [<pathspec>]

•	git add manage.py requirements.txt
•	git add .

•	Description: Remove a file from working tree and stage deletion.
•	Syntax:
•	git rm [--cached] <file>

•	Syntax:
•	git mv <oldpath> <newpath>

•	Syntax:

•	Syntax:

•	Description: List, create, or delete branches.

6. Inspection & Debugging
git show
•	Description: Show various types of objects (commits, trees, tags).
•	Syntax:
•	git show [<object>]
•	Example:
•	git show HEAD~2
git blame
•	Description: Show who last modified each line of a file.
•	Syntax:
•	git blame [<options>] <file>
•	Example:
•	git blame apps/users/models.py

•	Description: Binary search to find the commit that introduced a bug.
•	Syntax:
•	git bisect start
•	git bisect bad               # current is bad
•	git bisect good <commit>
•	Example:
•	git bisect start
•	git bisect bad HEAD
•	git bisect good v1.0.0

7. Undoing & Cleaning
git reset
•	Description: Move HEAD and (optionally) reset index/worktree.
•	Syntax:
•	git reset [--soft | --mixed | --hard] <commit>
•	Example:
•	git reset --soft HEAD~1   # undo last commit, keep changes staged
•	git reset --hard HEAD     # discard all local changes
git revert
•	Description: Create a new commit that undoes changes from a past commit.
•	Syntax:
•	git revert <commit>
•	Example:
•	git revert 1a2b3c4d
git clean
•	Description: Remove untracked files or directories.
•	Syntax:
•	git clean -n| -f [-d]
•	Example:
•	git clean -fd    # force delete untracked dirs
git stash
•	Description: Temporarily save (stash) changes in working directory.
•	Syntax:
•	git stash [push] [-m <message>]
•	git stash list
•	git stash apply [<stash>]
•	git stash pop
•	Example:
•	git stash push -m "WIP: fix signup form"
•	git stash list
•	git stash apply stash@{0}

•	Syntax:
•	git tag [-a | -m <msg>] <tagname> [<commit>]
•	Example:

•	git push origin --tags

Repository setup and cloning
•	git config – Configure user identity and preferences (e.g. name and email) at the global or local level.
Syntax: git config [--global] user.name "Your Name"
Example: Set up your identity before committing:
o	git config --global user.name "Alice Smith"
o	git config --global user.email "alice@example.com"
•	git init – Create a new Git repository in the current directory. This initializes a .git folder to track changes.
Syntax: git init [directory]
Example: Starting a new Django project:
o	mkdir mysite
o	cd mysite
o	git init
•	git clone – Copy an existing remote repository to a new local folder. This creates a local copy (with an automatic origin remote) of, e.g., a GitHub repo.
Syntax: git clone <repository_url>
o	Example: Cloning your fork of Django:
o	git clone https://github.com/YourName/django.git
o	cd django
Branching and merging
•	git branch – List, create, or delete branches. Branches let you isolate work (features, bugfixes) from the main code line.
Syntax: git branch [<branch_name>] (with no name, lists branches; with a name, creates it)
o	Example: Create a branch for a new feature:
o	git branch add-login
o	git branch  # shows "add-login" in list
•	git checkout – Switch branches or restore files. (In Git ≥2.23, git switch <branch> is a more focused way to change branches.)
Syntax:
	Switch: git checkout <branch_name> (or git switch <branch_name>)
	Create+switch: git checkout -b <new_branch> (or git switch -c <new_branch>)
	Restore file: git checkout <branch> -- <file>
Example: Create and switch to add-login:
o	git checkout -b add-login
•	git merge – Merge another branch into the current branch. This integrates changes, creating a fast-forward or merge commit as needed.
Syntax: git merge <branch_name>
Example: Merge add-login into main:
o	git checkout main
o	git merge add-login
•	git merge --abort – Abort an in-progress merge and revert to the pre-merge state. Use this if you want to cancel a merge after conflicts occur (no citation needed).
Commit operations
•	git add – Stage changes (add files to the index) in preparation for a commit.
Syntax: git add <file> (or git add . to stage all changes)
o	Example: Stage all modified files in your Django app:
o	git add .
•	git commit – Record staged changes as a new commit with a message. Each commit is a snapshot in history.
Syntax: git commit -m "commit message"
o	Example: Commit changes after updating a Django view:
o	git commit -m "Add login view and form validation"
•	git commit --amend – Amend the most recent commit. This replaces the last commit with a new one (combining staged changes and/or a new message).
o	Syntax: git commit --amend [-m "new message"]
o	Example: Fix a typo in the last commit message:
o	git commit --amend -m "Correct typo in login view commit message"
•	git revert – Revert a commit by creating a new commit that undoes its changes. Unlike reset, this preserves history while “undoing” a change.
o	Syntax: git revert <commit_hash>
o	Example: If commit abc123 introduced a bug, revert it:
o	git revert abc123
Viewing history and diffs
•	git log – Show the commit history of the current branch (with commit IDs, authors, dates, and messages).
o	Syntax: git log [--oneline] [--graph]
o	Example: View a concise log graph:
o	git log --oneline --graph
•	git diff – Show changes between commits, branches, or your working directory and the index. Helps inspect code differences.
o	Syntax:
	git diff (changes in working directory vs. last commit)
	git diff <branch1>..<branch2> (differences between two branches or commits)
o	Example: Compare current edits to the last commit:
o	git diff
Stashing changes
•	git stash – Stash (shelve) uncommitted changes to clean the working directory. This saves work-in-progress without committing.
o	Syntax: git stash (or git stash push)
o	Example: You were editing views and need to switch branches; stash your changes:
o	git stash
•	git stash pop – Reapply the most recently stashed changes and remove them from the stash list.
o	Syntax: git stash pop
o	Example: After switching back, restore the stashed work:
o	git stash pop
Rebasing and cherry-picking
•	git rebase – Reapply commits from one branch onto another, rewriting history. Useful to incorporate upstream changes or to squash commits (with -i) for a linear history.
o	Syntax: git rebase <base_branch> (often used as git rebase main while on your feature branch)
o	Example: On branch add-login, incorporate the latest main:
o	git checkout add-login
o	git rebase main
•	git rebase -i – (Advanced) Interactive rebase. Let’s you edit, squash or reorder commits during a rebase (no extra citation needed). Example: git rebase -i HEAD~3 to squash the last 3 commits.
•	git cherry-pick – Apply a specific commit from another branch onto the current branch. This “picks” one commit out of history.
o	Syntax: git cherry-pick <commit_hash>
o	Example: If commit def456 on another branch fixes a bug needed here:
o	git cherry-pick def456
Tagging
•	git tag – Create a tag to mark a specific commit (often used for releases). Tags are immutable references.
o	Syntax: git tag <tagname> (lightweight) or git tag -a <tagname> (annotated, with message).
o	Example: Tag version 1.0 of the current state:
o	git tag -a v1.0
•	git push --tags – Push all local tags to the remote repository. By default, git push does not transfer tags.
o	Syntax: git push origin --tags
o	Example: Publish the new v1.0 tag:
o	git push origin --tags
Working with remotes
•	git remote add – Add a remote repository reference. Commonly used to name an upstream source or a fork’s origin.
o	Syntax: git remote add <name> <url>
o	Example: After forking Django on GitHub, add the main Django repo as upstream:
o	git remote add upstream https://github.com/django/django.git
•	git fetch – Download commits, branches, and tags from a remote without merging. Updates remote-tracking branches (e.g. origin/main).
o	Syntax: git fetch <remote>
o	Example: Fetch all updates from origin:
o	git fetch origin
•	git pull – Fetch from the remote and immediately merge into the current branch. (Equivalent to git fetch then git merge.)
o	Syntax: git pull <remote> <branch>
o	Example: Pull changes from origin’s main into your local main:
o	git pull origin main
•	git push – Push local commits to a remote branch.
o	Syntax: git push <remote> <branch> (add -u to set upstream).
o	Example: Push your feature branch to GitHub:
o	git push origin add-login
Ignoring files (.gitignore)
•	.gitignore file – Specifies files/patterns for Git to ignore (not track). In Django projects, ignore generated or environment files (virtualenvs, caches, compiled files) so they aren’t committed.
o	Example patterns:
o	*.pyc
o	__pycache__/
o	db.sqlite3
o	/media/
o	/venv/
o	.env
These entries keep logs, database files, virtual environments, etc., out of the repository.
Conflict resolution
•	When merging or rebasing, conflicts can occur. Run git status to identify unmerged files. For example, Git will report something like “both modified: filename”.
•	Edit the conflicted files to resolve issues (conflict sections are marked <<<<<<<, =======, >>>>>>>). After fixing, mark them resolved with git add <file>, then commit.
o	Example: If merge.txt has a conflict, fix its contents, then:
o	git add merge.txt
o	git commit -m "Resolve merge conflict in merge.txt"
•	To abort a merge altogether, use git merge --abort. (This resets to pre-merge state.)
Pull requests and collaboration
•	In collaborative workflows (e.g. GitHub), developers typically create branches, push them to a fork or origin, and open Pull Requests (PRs). When ready, run:
•	git push origin <branch>
Then create a PR via the platform’s interface. Good practice is one logical change per commit and clear messages.
•	To synchronize with an upstream repo, fetch and rebase often:
•	git fetch upstream
•	git rebase upstream/main
(This keeps your branch up-to-date with the main project.)
•	To review or integrate someone else’s PR locally, you can fetch it. For example, Django’s docs show:
•	git fetch upstream
•	git checkout -b pull_x123 upstream/main
•	git pull origin pull/123/head
Or apply the PR patch with git am:
curl -L https://github.com/django/django/pull/123.patch | git am
These commands create a local branch for the PR and apply its commits.
