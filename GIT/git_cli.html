1. Setup & Configuration

Initialize repositories and configure Git settings for your Django projects.

git config

Full Description:

The git config command allows you to configure Git settings at different levels (system, global, or local repository). You can set your identity (name and email), configure your default editor, set up aliases, define merge/diff tools, and customize hundreds of other Git behaviors. These configurations are stored in plain text files and can be edited directly or via this command.

Syntax:

git config [--global | --system | --local] [--add | --get | --unset] <section>.<key> [<value>]

Parameters:

--global: Applies the configuration to all repositories for the current user. Stored in ~/.gitconfig

--system: Applies the configuration system-wide for all users. Stored in /etc/gitconfig

--local: Applies the configuration only to the current repository. Stored in .git/config (default if no scope specified)

--list: Lists all configuration settings

--add: Adds a new line to a multi-valued option

--get: Gets the value for a given key

--unset: Removes a configuration variable

section.key: The configuration variable name (e.g., user.name, core.editor)

value: The value to set for the configuration variable

Return Value:

Returns 0 on success. When getting a value, outputs the configuration value to stdout. When setting a value, returns 0 if successful, non-zero if the configuration couldn't be written.

Examples:

Example 1: Configure user identity for JobPost author

# Set identity for John Doe from ABC Corp (Author 1)
git config --global user.name "John Doe"
git config --global user.email "user1@example.com"

Result:

  • Configuration Applied: Git identity set for all future commits
  • Name: John Doe (Author 1 from ABC Corp)
  • Email: user1@example.com
  • Saved To: ~/.gitconfig (global configuration)
  • Scope: Applied to anihpj project and all other repositories
  • Usage: Every commit you make will be attributed to John Doe

Example 2: Set default editor for commit messages

# Configure VS Code as the default editor for writing commit messages
# when adding JobPost records
git config --global core.editor "code --wait"

Result:

  • Editor Set: Visual Studio Code configured as default Git editor
  • Command: code --wait
  • Behavior: Git opens VS Code when commit message needed (e.g., git commit without -m)
  • --wait Flag: Git waits for editor to close before proceeding
  • Use Case: Write detailed commit messages when adding JobPost records or modifying Skills model

Example 3: View all configuration settings for the anihpj project

# List all Git configurations
git config --list

# View specific configuration
git config user.name

Result:

All Configurations Displayed:

  • user.name=John Doe
  • user.email=user1@example.com
  • core.editor=code --wait
  • ...plus all other configured values

Scopes Shown: System-wide, global (user), and local (repository) settings

Specific Query: git config user.name returns just "John Doe"

Example 4: Configure project-specific settings for jobpost app

# Set a different email just for the anihpj/jobpost project
cd ~/anihpj
git config --local user.email "jobpost-dev@abc-corp.com"

# Verify the local override
git config user.email

Result:

  • Project-Specific Email: jobpost-dev@abc-corp.com
  • Scope: Local to anihpj repository only
  • Override: Supersedes global setting (user1@example.com)
  • Stored In: .git/config (repository level)
  • Effect: Commits in anihpj use jobpost-dev@abc-corp.com
  • Other Repos: Still use global email user1@example.com
  • Verification: git config user.email returns jobpost-dev@abc-corp.com

Example 5: Create alias for common Django development commands

# Create shortcuts for frequently used commands in Django projects
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.cm "commit -m"

# Now you can use shorter commands
git st  # instead of git status
git cm "Added Skills model to jobpost"  # instead of git commit -m

Result:

Aliases Created:

  • git st → git status
  • git co → git checkout
  • git br → git branch
  • git cm → git commit -m

Benefits:

  • Save typing time during development
  • Faster workflow when committing Skills, Author, Location, JobPost changes
  • Available globally in all Git repositories
  • Can create custom aliases for any Git command or combination

git init

Full Description:

The git init command creates a new Git repository in a directory. It initializes a hidden .git subdirectory containing all necessary repository metadata including object database, refs, template files, and HEAD pointer. This transforms a regular directory into a Git-tracked project where you can start tracking changes, creating commits, branches, and managing version history. You can initialize a repository in a new directory or convert an existing project to use Git.

Syntax:

git init [<directory>] [--bare] [--initial-branch=<branch-name>]

Parameters:

directory: Optional name of the directory to create and initialize. If not specified, initializes the current directory

--bare: Creates a bare repository (without a working directory), typically used for central repositories

--initial-branch: Sets the name of the initial branch (default is usually 'master' or 'main')

--template: Specifies a template directory from which to copy files

--shared: Specifies that the Git repository is to be shared amongst several users with proper permissions

Return Value:

Returns 0 on successful initialization. Creates a .git directory and outputs "Initialized empty Git repository in /path/to/project/.git/". Returns non-zero if initialization fails (e.g., insufficient permissions).

Examples:

Example 1: Initialize new Django project for jobpost application

# Create and initialize a new project directory
mkdir anihpj
cd anihpj
git init

Result:

Output: "Initialized empty Git repository in ~/anihpj/.git/"

.git Directory Created With:

  • objects/ - Git object database
  • refs/ - Branch and tag references
  • HEAD - Points to current branch
  • config - Repository configuration
  • description - Repository description
  • hooks/ - Git hook scripts
  • info/ - Exclude patterns

Ready To: Add JobPost models and track changes

Example 2: Initialize repository with a specific initial branch name

# Initialize with 'main' as the default branch instead of 'master'
git init --initial-branch=main anihpj-jobpost

Result:

  • Directory Created: anihpj-jobpost/
  • Default Branch: main (not master)
  • Convention: Follows modern Git best practices
  • Use Case: Setting up repositories for Skills, User, Author, Location, and JobPost models
  • Benefit: Aligns with GitHub and GitLab modern defaults

Example 3: Initialize existing Django project with models

# Navigate to existing project containing models.py
cd ~/projects/anihpj
ls  # Shows: manage.py, jobpost/, anihpj/, models.py

# Initialize Git tracking
git init

Result:

Existing Project Converted to Git Repository:

  • Files Preserved: All existing files remain untouched
    • manage.py - Django management script
    • settings.py - Configuration
    • models.py - Skills, Author, Location, JobPost models
  • .git/ Folder Created: Repository metadata added
  • Django Functionality: Completely unaffected
  • Now Trackable: Can version control all changes

Example 4: Create bare repository for team collaboration

# Create a central repository on a shared server
mkdir /shared/repos/anihpj-central.git
cd /shared/repos
git init --bare anihpj-central.git

Result:

Output: "Initialized empty Git repository in /shared/repos/anihpj-central.git/"

  • Repository Type: Bare (no working directory)
  • Contents: Git data only (objects, refs, HEAD)
  • Purpose: Central collaboration hub
  • Usage: Multiple developers clone and push
  • Team Workflow: Shared repository for JobPost application development
  • Convention: .git suffix indicates bare repository

Example 5: Initialize with template for Django projects

# Create project with pre-configured Git hooks and ignore files
git init --template=/home/user/.git-templates/django anihpj

# Verify initialization
cd anihpj
ls -la .git/hooks/  # Shows pre-commit, pre-push hooks for Django
cat .gitignore      # Shows *.pyc, __pycache__/, db.sqlite3, etc.

Result:

Repository Initialized with Django Template:

  • Pre-configured .gitignore:
    • __pycache__/ directories
    • *.pyc compiled files
    • Virtual environment folders
    • db.sqlite3 database
    • Optional migration files
  • Git Hooks: Pre-commit linting and testing
  • Benefit: Clean commits without bytecode or database files
  • Team Standard: Consistent setup across all developers

git clone

Full Description:

The git clone command creates a complete copy of a remote repository on your local machine. It downloads all files, commit history, branches, and tags from the remote repository and sets up a local working directory. The command automatically configures the remote repository as "origin" and checks out the default branch. This is typically the first command you run when joining an existing project or deploying code to a new environment.

Syntax:

git clone [--branch <branch>] [--depth <depth>] <repository-url> [<directory>]

Parameters:

repository-url: The URL of the remote repository (HTTPS or SSH format)

directory: Optional name for the local directory. Defaults to repository name

--branch: Clone a specific branch instead of the default branch

--depth: Create a shallow clone with limited commit history

--recursive: Clone including all submodules

--single-branch: Clone only one branch, not all branches

Return Value:

Returns 0 on successful clone. Creates a new directory containing the repository with full Git history. Outputs progress including object counting, compression, and file transfer statistics. Returns non-zero if clone fails (network error, authentication failure, etc.).

Examples:

Example 1: Clone anihpj Django project with JobPost models

# Clone from GitHub using HTTPS
git clone https://github.com/username/anihpj.git

# Navigate into cloned project
cd anihpj
ls  # Shows: manage.py, jobpost/, anihpj/, models.py

Result:

  • Repository Downloaded: Complete anihpj project
  • Directory Created: anihpj/ with full project structure
  • Models Included: Skills, Author, Location, JobPost in jobpost/ app
  • History: All commits, branches, and tags downloaded
  • Remote Configured: origin → https://github.com/username/anihpj.git
  • .git Directory: Contains complete repository history
  • Ready To: Start development immediately

Example 2: Clone with custom directory name

# Clone and rename directory to django-jobpost-app
git clone https://github.com/username/anihpj.git django-jobpost-app

cd django-jobpost-app
git remote -v  # Verify origin is set

Result:

  • Directory Name: django-jobpost-app (custom)
  • Repository: Same as anihpj (complete clone)
  • Models Available: Skills, User, Author, Location, JobPost
  • Git History: Full history preserved
  • Use Case: Local naming convention differs from repository name
  • Functionality: Identical to cloning as "anihpj"

Example 3: Clone specific branch for feature development

# Clone only the feature-skills-enhancement branch
git clone --branch feature-skills-enhancement --single-branch \
    https://github.com/username/anihpj.git

cd anihpj
git branch  # Shows only feature-skills-enhancement

Result:

  • Branch Cloned: feature-skills-enhancement only
  • Other Branches: Not downloaded
  • Benefits:
    • Faster download time
    • Less disk space usage
    • Focused on specific feature
  • History: Full commit history for this branch
  • Use Case: Working on Skills model enhancement without needing main

Example 4: Shallow clone for deployment

# Clone with only the last commit (no history)
git clone --depth 1 https://github.com/username/anihpj.git

cd anihpj
git log --oneline  # Shows only the most recent commit

Result:

  • Clone Type: Shallow (latest commit only)
  • History Depth: 1 commit
  • Benefits:
    • Extremely fast download
    • Minimal disk space
    • Perfect for production deployment
  • Use Case: Deploy current JobPost application code
  • Trade-off: Cannot view historical commits with git log
  • Ideal For: CI/CD pipelines, production servers

Example 5: Clone with submodules for complex Django apps

# Clone repository including all Git submodules
git clone --recursive https://github.com/username/anihpj.git

cd anihpj
ls jobpost/  # Shows models.py and any submodule dependencies
git submodule status  # Lists all initialized submodules

Result:

  • Main Repository: anihpj cloned completely
  • Submodules: All external Git repositories automatically cloned
  • Initialization: Submodules initialized and checked out
  • Use Case: JobPost app with external Django package dependencies
  • Without --recursive: Submodule directories would be empty
  • Benefit: Complete project with all dependencies in one command

2. Viewing Status & Changes

Check repository status, view differences, and browse commit history.

git status

Full Description:

The git status command displays the current state of your working directory and staging area. It shows which files have been modified but not staged, which files are staged for commit, which files are untracked (not yet added to Git), and which branch you're on. It also indicates if your branch is ahead or behind the remote branch. This is one of the most frequently used Git commands for understanding what changes are pending.

Syntax:

git status [--short] [--branch] [--porcelain] [<pathspec>]

Parameters:

--short or -s: Displays output in compact format with two-letter status codes

--branch or -b: Shows branch information even in short format

--porcelain: Machine-readable output format for scripts

--untracked-files: Controls how untracked files are shown (all, no, normal)

pathspec: Optional path to check status of specific files or directories

Return Value:

Returns 0 if successful. Outputs current branch name, tracking status versus remote, lists of staged files (green), unstaged modifications (red), and untracked files. No output changes are made to the repository - this is a read-only operation.

Examples:

Example 1: Check status after modifying JobPost model

# After editing models.py to add a field to JobPost
cd ~/anihpj
git status

Result:

On branch main
Your branch is up to date with 'origin/main'.

Changes not staged for commit:
  (use "git add ..." to update what will be committed)
  (use "git restore ..." to discard changes in working directory)
        modified:   jobpost/models.py

no changes added to commit (use "git add" and/or "git commit -a")
  • Modified File: jobpost/models.py
  • Status: Changes not staged for commit
  • Next Step: Use git add to stage or git restore to discard
  • Context: JobPost model field added but not yet tracked

Example 2: Check status with new Skills migration file

# After running makemigrations for Skills model
python manage.py makemigrations
git status

Result:

On branch main
Changes not staged for commit:
        modified:   jobpost/models.py

Untracked files:
  (use "git add ..." to include in what will be committed)
        jobpost/migrations/0002_skills_level.py
  • Modified: jobpost/models.py (unstaged)
  • Untracked: jobpost/migrations/0002_skills_level.py
  • Interpretation: Migration generated but not added to Git
  • Action Needed: Stage both files to commit together

Example 3: Short status format for quick overview

# Use compact status output
git status -s
# Or with branch info
git status -sb

Result:

 M jobpost/models.py
?? jobpost/migrations/0002_skills_level.py
?? jobpost/__pycache__/
  • Format: Compact two-letter status codes
  • M (Modified): File has unstaged changes
  • ?? (Untracked): File not in Git yet
  • Benefit: Quick overview when working with multiple models
  • Use Case: Fast scan of changes across Skills, Author, Location, JobPost

Example 4: Check status of specific file or directory

# Check only the jobpost directory
git status jobpost/

# Check only models.py
git status jobpost/models.py

Result:

On branch main
Changes not staged for commit:
        modified:   jobpost/models.py
  • Filtered Output: Only shows specified path
  • Use Case: Focus on jobpost app in large Django projects
  • Benefit: Ignore changes in other apps or directories
  • Example: Check jobpost/ without seeing anihpj/ settings changes

Example 5: Status showing branch ahead of remote

# After committing JobPost changes locally
git add jobpost/models.py
git commit -m "Add salary_range field to JobPost"
git status

Result:

On branch main
Your branch is ahead of 'origin/main' by 1 commit.
  (use "git push" to publish your local commits)

nothing to commit, working tree clean
  • Branch Status: Ahead of origin/main by 1 commit
  • Local Commit: JobPost salary_range field added
  • Working Tree: Clean (no pending changes)
  • Action: Use git push to publish to GitHub

git diff

Full Description:

The git diff command shows the differences between various states of your repository. It displays line-by-line changes in a unified diff format, showing additions (prefixed with +), deletions (prefixed with -), and context lines. You can compare working directory vs staging area, staging area vs last commit, between commits, or between branches. This is essential for reviewing changes before committing, understanding what was modified in a commit, or seeing differences between branches during merges.

Syntax:

git diff [<options>] [<commit>] [--] [<path>...]

Parameters:

(no arguments): Shows unstaged changes in working directory vs staging area

--staged or --cached: Shows staged changes (staging area vs last commit)

HEAD: Compares working directory + staging area vs last commit

commit1..commit2: Shows changes between two commits

branch1..branch2: Shows changes between two branches

--stat: Shows summary of changes (file names and line counts) instead of full diff

--name-only: Lists only the names of changed files

path: Limits diff to specific files or directories

Return Value:

Returns 0 if differences exist or if no differences exist. Outputs unified diff format showing file paths, change locations (@@), removed lines (- in red), added lines (+ in green), and unchanged context lines. Exit code 0 even when no changes - check output length to determine if differences exist.

Examples:

Example 1: View unstaged changes to Skills model

# After editing Skills model but before staging
git diff jobpost/models.py

Result:

diff --git a/jobpost/models.py b/jobpost/models.py
index a3b4c5d..e7f8g9h 100644
--- a/jobpost/models.py
+++ b/jobpost/models.py
@@ -5,6 +5,7 @@ class Skills(models.Model):
     name = models.CharField(max_length=200)
+    level = models.CharField(max_length=50)
  • Changed File: jobpost/models.py
  • Addition: level field added to Skills model
  • Format: + prefix indicates new line (green in terminal)
  • Purpose: Review changes before staging with git add

Example 2: Review staged changes before committing

# After staging Author and Location model updates
git add jobpost/models.py
git diff --staged

Result:

diff --git a/jobpost/models.py b/jobpost/models.py
index a3b4c5d..m9n8o7p 100644
--- a/jobpost/models.py
+++ b/jobpost/models.py
@@ -12,6 +12,7 @@ class Author(models.Model):
     designation = models.CharField(max_length=200)
+    bio = models.TextField(blank=True)
  • Shows: What will be included in next commit
  • Flag: --staged shows staged changes only
  • Change: bio field added to Author model
  • Purpose: Verify correct changes before committing

Example 3: Compare working directory with last commit

# Show all changes (staged + unstaged) vs last commit
git diff HEAD jobpost/models.py

# Or for entire repository
git diff HEAD

Result:

  • Shows: Combined view of staged + unstaged changes
  • Comparison: Working directory vs last commit (HEAD)
  • Example: JobPost changes (unstaged) + Skills changes (staged) both shown
  • Benefit: Complete picture of work-in-progress
  • Use Case: Review all modifications regardless of staging status

Example 4: See summary of changes across multiple models

# After modifying multiple files (models, views, migrations)
git diff --stat

# For specific directory
git diff --stat jobpost/

Result:

jobpost/migrations/0004.py|  8 ++++++

  • Symbols: + for additions, - for deletions
  • Files Changed: 3 (models, views, migrations)
  • Total Changes: 38 additions, 8 deletions
  • Use Case: Quick overview when modifying multiple models

Example 5: Compare feature branch with main for JobPost enhancements

# See what changes are in feature-jobpost-salary branch
git diff main..feature-jobpost-salary

# Or just filenames
git diff --name-only main..feature-jobpost-salary

Result:

Full diff shows: All differences between main and feature-jobpost-salary branches

With --name-only flag:

jobpost/views.py
jobpost/tests.py

  • Use Case: Code reviews and understanding branch divergence

git log

Full Description:

The git log command displays the commit history of your repository. It shows a chronological list of commits with details including commit hash (SHA), author, date, and commit message. You can filter by date range, author, file path, or search commit messages. The log helps you understand project evolution, find when bugs were introduced, review what changed, track who made changes, and navigate through different versions of your code. It's essential for debugging, code archaeology, and understanding project history.

Syntax:

git log [<options>] [<revision-range>] [[--] <path>...]

Parameters:

--oneline: Shows each commit on a single line with abbreviated hash and message

--graph: Draws an ASCII graph showing branch structure and merges

--all: Shows commits from all branches, not just current branch

-n <number> or -<number>: Limits output to specified number of commits

--author=<pattern>: Shows only commits by specific author

--since / --after: Shows commits after a specific date

--until / --before: Shows commits before a specific date

--grep=<pattern>: Filters commits by message content

path: Shows only commits that modified specific files or directories

Return Value:

Returns 0 on success. Outputs list of commits in reverse chronological order (newest first). Each commit shows full SHA hash, author name and email, date, and full commit message. Press 'q' to quit the pager. Returns non-zero if invalid options or revision range specified.

Examples:

Example 1: View compact history of JobPost model changes

# One-line format for quick overview
git log --oneline jobpost/models.py

Result:

a3b4c5d Add salary_range field to JobPost model
e7f8g9h Create JobPost model with relationships
m9n0o1p Add Author and Location models
q2r3s4t Add Skills model for job requirements
u5v6w7x Initial commit with Django project structure
  • Format: Abbreviated commit hash (7 chars) + first line of message
  • File: History filtered to jobpost/models.py only
  • Models: Shows evolution of Skills, Author, Location, JobPost
  • Benefit: Quick timeline of model changes

Example 2: Visual branch structure with merge history

# See branch relationships and merges
git log --oneline --graph --all --decorate

Result: * a3b4c5d (HEAD -> main) Merge feature-jobpost-enhancements
|\\
| * e7f8g9h (feature-jobpost-enhancements) Add salary range validation
| * m9n0o1p Enhance JobPost with salary fields
|/
* q2r3s4t Update Skills model with proficiency levels
* u5v6w7x Add Author model with user relationship

ASCII graph shows branch structure. Vertical lines (|) show branch paths, asterisks (*) are commits, and merge points connect branches. Helps visualize parallel development on Skills and JobPost features.

Example 3: Find all commits by specific developer

# See John Doe's contributions to anihpj project
git log --author="John Doe" --oneline

# Or with full details
git log --author="user1@example.com"

Result:

a3b4c5d Add salary_range field to JobPost model
q2r3s4t Update Skills model with proficiency levels
y8z9a0b Configure Author model with company details
  • Filter: Only commits by John Doe (Author #1)
  • Search By: Name or email address
  • Use Case: Code reviews, individual contributions tracking
  • Context: Shows which models this developer worked on

Example 4: Review recent changes in the last week

# Commits from last 7 days
git log --since="1 week ago" --oneline

# Or specific date range
git log --since="2025-10-27" --until="2025-11-03"

Result:

a3b4c5d Add salary_range field to JobPost model
e7f8g9h Create JobPost model with relationships
m9n0o1p Add Author and Location models
  • Date Filter: Last 7 days or specific range
  • Natural Language: Accepts "1 week ago", "2 weeks ago"
  • ISO Format: Also accepts "2025-10-27" format
  • Use Case: Sprint reviews, daily standups, recent activity tracking
  • Models: Shows recent changes to Skills, Author, Location, JobPost

Example 5: Search commit messages for bug fixes

# Find commits mentioning "fix" or "bug"
git log --grep="fix" --oneline

# Case-insensitive search for JobPost
git log --grep="JobPost" -i --oneline

Result:

a3b4c5d Fix JobPost salary field validation
p7q8r9s Fix Author model OneToOne relationship
t0u1v2w Bugfix: Location zip code max length

  • Keywords: "fix", "bug", "JobPost" in messages
  • Case-Insensitive: -i flag matches jobpost/JobPost/JOBPOST
  • Use Case: Track bug fixes, understand issue resolution history
  • Models: Find fixes for JobPost, Author, Location models

3. Staging & Committing

Stage files and record changes to your Django project repository.

git add

Full Description:

The git add command stages changes from your working directory to the staging area (also called the index). Staging prepares files for the next commit - only staged changes will be included when you commit. You can stage new files, modifications to existing files, and file deletions. This command doesn't affect the repository or commit history until you run git commit. It's a crucial step in Git's three-stage workflow: working directory → staging area → repository.

Syntax:

git add [--all | -A] [--update | -u] [--patch | -p] [--] [<pathspec>...]

Parameters:

pathspec: Files, directories, or patterns to stage (e.g., models.py, jobpost/, *.py)

.: Stages all changes in current directory and subdirectories (new, modified, deleted files)

--all or -A: Stages all changes in entire repository regardless of current directory

--update or -u: Stages only modifications and deletions, ignoring new untracked files

--patch or -p: Interactive mode - review and stage changes in chunks (hunks)

--force or -f: Allows staging files ignored by .gitignore

Return Value:

Returns 0 on success. No visible output unless there's an error or warning. Files are moved to staging area and will appear in "Changes to be committed" section when running git status. Returns non-zero if pathspec doesn't match any files or there are permission errors.

Examples:

Example 1: Stage new Skills model file

# After creating the Skills model in models.py
git add jobpost/models.py

# Verify it's staged
git status

Result:

Full Description:

The git checkout command switches between branches or restores working tree files from a specific commit. When checking out a branch, it updates HEAD to point to the new branch, updates the index, and modifies working directory files to match the branch's snapshot. git merge

Full Description:

The git merge command integrates changes from one branch into another by combining their commit histories. It creates a new "merge commit" with two parent commits (unless fast-forward is possible), preserving the complete history of both branches. Merge is the standard way to combine completed features, bug fixes, or releases back into main development branches. Git automatically handles non-conflicting changes, but when the same lines are modified in both branches, it requires manual conflict resolution. Merge preserves branch context and is ideal for collaborative workflows where maintaining history visibility is important.

Syntax:

git merge [options] <branch> [-m <message>]

Parameters:

branch: The branch name to merge into the current branch

--no-ff: Create a merge commit even if fast-forward is possible, preserving branch history

--ff-only: Only merge if fast-forward is possible, abort otherwise

--squash: Combine all commits from branch into one commit, don't create merge commit

-m <message>: Custom merge commit message (auto-generated if omitted)

--abort: Abort the merge process and return to pre-merge state

--continue: Continue merge after resolving conflicts

--no-commit: Perform merge but don't create commit automatically

git rebase

Full Description:

The git rebase command moves or reapplies a series of commits to a new base commit, effectively rewriting commit history. Unlike merge which preserves both branch histories, rebase creates a linear history by transplanting commits from your branch onto the tip of another branch. Each commit is replayed one-by-one, potentially with conflict resolution at each step. Rebase is powerful for cleaning up commit history, maintaining a linear project timeline, and preparing feature branches before merging. WARNING: Never rebase commits that have been pushed to shared/public branches - it rewrites history and causes problems for collaborators.

Syntax:

git rebase [options] [<upstream> [<branch>]]

Parameters:

upstream: The branch to rebase onto (e.g., main, develop). Defaults to upstream tracking branch

-i or --interactive: Interactive mode - choose what to do with each commit (edit, squash, reword, drop)

--continue: Continue rebase after resolving conflicts

--abort: Cancel the rebase and return to original state

--skip: Skip current commit and continue with next (if commit is no longer needed)

--onto <newbase>: Rebase onto a different branch than the upstream

--autosquash: Automatically squash commits marked with fixup! or squash! prefixes

--preserve-merges: Preserve merge commits during rebase (deprecated, use --rebase-merges)

Return Value:

Returns 0 on successful rebase. Output shows: "Successfully rebased and updated refs/heads/branch". Each commit is replayed with new SHA hashes. Working directory ends at the tip of rebased branch. Returns non-zero if conflicts occur during replay - must resolve conflicts, git add files, then git rebase --continue. Can abort at any time with git rebase --abort.

git fetch

Full Description:

The git fetch command downloads commits, files, and references from a remote repository into your local repository, but does not modify your working directory or current branch. It updates remote-tracking branches (like origin/main, origin/develop) to reflect the current state of the remote repository. Fetch is a safe operation - it only retrieves data and lets you review changes before integrating them. You can inspect fetched changes with git log origin/main, then decide to merge or rebase. This is in contrast to git pull which fetches and immediately merges. Fetch is essential for staying synchronized with team members while maintaining control over when changes are integrated.

Syntax:

Parameters:

remote: Name of remote to fetch from (e.g., origin, upstream). Defaults to origin if omitted

branch: Specific branch to fetch. If omitted, fetches all branches from the remote

--all: Fetch from all configured remotes simultaneously

--prune or -p: Delete local references to remote branches that no longer exist

--tags: Fetch all tags from the remote

--no-tags: Don't fetch any tags

--depth <n>: Fetch only n commits deep (shallow fetch for large repositories)

--dry-run: Show what would be fetched without actually fetching

Returns 0 on success. Outputs "From " followed by list of fetched branches/tags. Shows " -> origin/" for each updated reference. Remote-tracking branches are updated but local branches and working directory remain unchanged. Returns non-zero if remote doesn't exist or network issues occur. git pull

Full Description:

The git pull command fetches changes from a remote repository and immediately integrates them into the current local branch. It's essentially git fetch followed by either git merge (default) or git rebase (with --rebase flag). Pull is the quickest way to synchronize your local branch with remote changes, commonly used when starting work to ensure you have the latest code. However, it can create unexpected merge commits or conflicts if your local branch has diverged from remote. For more control, many developers prefer separate git fetch and git merge commands. Pull is most reliable when working directory is clean and no local commits exist.

Syntax:

git pull [<remote>] [<branch>] [--rebase]

Parameters:

remote: Remote repository to pull from (e.g., origin). Defaults to current branch's upstream

branch: Branch to pull from remote. Defaults to current branch's tracking branch

--rebase: Use rebase instead of merge to integrate changes (creates linear history)

--no-rebase: Force merge even if pull.rebase is configured globally

--ff-only: Only fast-forward, abort if merge commit would be needed

--no-commit: Fetch and merge but don't create merge commit automatically

--squash: Combine all pulled commits into one

-p or --prune: Prune remote-tracking branches that no longer exist remotely

Return Value:

Returns 0 on successful pull. For fast-forward: "Updating a1b2c3d..e5f6g7h. Fast-forward." For merge: "Merge made by 'strategy'". Shows file change statistics. Updates current branch and working directory. Returns non-zero if conflicts occur - must resolve conflicts, git add files, then git commit (for merge) or git rebase --continue (for rebase pull). If local changes would be overwritten, pull fails and asks you to commit or stash first.

Examples:

Example 1: Pull latest changes on main branch

# Pull updates from origin/main

Result:

  • No Merge Commit: Local branch simply moved forward

  • Tracking: If configured, use just git pull without arguments
  • Use Case: Quick way to stay current before starting new work

git push

Full Description:

The git push command uploads local repository commits to a remote repository (like GitHub, GitLab, or Bitbucket). It transfers commits, files, and references from your local branch to the corresponding remote branch, making your work visible to teammates and backing up your code to the cloud. Push updates the remote branch pointer to match your local branch. This is essential for collaboration, code sharing, continuous integration, and maintaining project synchronization across team members. Always pull before pushing to avoid conflicts.

Full Description:

The git blame command shows which commit and author last modified each line of a file, displaying the commit SHA, author name, timestamp, and line number alongside the code. This is invaluable for code archaeology - understanding why a line was written, who to ask about it, and when it was introduced. Blame helps track down when bugs were introduced, find the rationale behind design decisions (by viewing commit messages), and understand code ownership. git bisect

Full Description:

The git bisect command uses binary search to efficiently find which commit introduced a bug or regression. Instead of manually checking hundreds of commits, bisect automatically checks out commits in the middle of the range, asks you to test them, and narrows down the search based on your responses. This logarithmic approach means finding the bad commit in 1000 commits takes only about 10 tests (log2(1000)). Bisect is invaluable when tests passed before but fail now, and you don't know which change broke things. Can be automated with scripts for continuous integration. Works by dividing the commit range in half repeatedly until the exact problematic commit is isolated.

Syntax:

git bisect start
git bisect bad [<commit>]
git bisect good <commit>
[test and mark good/bad]
git bisect reset

Parameters:

start: Begin a bisect session

bad [commit]: Mark a commit as containing the bug. Defaults to HEAD if no commit specified

good commit: Mark a commit as working correctly (before bug was introduced)

skip: Skip current commit (if it can't be tested, e.g., doesn't compile)

reset: End bisect session and return to original branch

run <script>: Automate bisect by running script that returns 0 for good, 1-127 (except 125) for bad

log: Show bisect log for reproducibility

replay <file>: Replay a bisect log

Return Value:

Bisect checks out commits and waits for you to test and mark them. When the problematic commit is found, outputs: " is the first bad commit" with full commit details. Returns 0 when done. After bisect, use git show to examine the bad commit's changes. git bisect reset returns to original HEAD. Can use git bisect visualize to see current search state graphically.

7. Undoing & Cleaning

Revert changes, clean workspace, and manage stashed modifications.

git reset

Full Description:

The git reset command moves the current branch pointer (HEAD) to a different commit and optionally modifies the staging area (index) and working directory. It has three modes: --soft (move HEAD only), --mixed (move HEAD and reset index, default), and --hard (move HEAD, reset index, and discard working directory changes). Reset is powerful for undoing commits, unstaging files, or completely discarding work. WARNING: --hard is destructive and permanently deletes uncommitted changes. Unlike revert which creates new commits, reset rewrites history. Never reset commits that have been pushed to shared branches - use revert instead for public commits.

Syntax:

git reset [--soft | --mixed | --hard] [<commit>]

Parameters:

--soft: Move HEAD to commit, keep index and working directory unchanged (files stay staged)

--mixed: Move HEAD and reset index, keep working directory (files unstaged but preserved). Default

--hard: Move HEAD, reset index, discard working directory changes (DESTRUCTIVE!)

commit: Target commit to reset to (e.g., HEAD~1, a1b2c3d). Defaults to HEAD

HEAD~n: Go back n commits (HEAD~1 = previous commit, HEAD~3 = 3 commits back)

file: When no commit specified, unstage specific file from index

--merge: Reset but keep uncommitted changes if possible

--keep: Reset but abort if working directory would be affected

Return Value:

Returns 0 on success. Output shows which files changed between current and target commit. --soft has no output. --mixed shows \"Unstaged changes after reset\". --hard shows \"HEAD is now at \". Returns non-zero if commit doesn't exist or if there are conflicts between reset and local changes.

git revert

Full Description:

The git revert command creates a new commit that undoes the changes introduced by a previous commit, effectively applying the inverse of that commit's changes. Unlike reset which moves branch pointers and rewrites history, revert preserves history by adding a new "revert commit" that negates the original changes. This makes revert safe for shared/public branches - it doesn't alter existing commits, so collaborators aren't affected. Revert is the standard way to undo published commits. It can handle merge commits and multiple commits. If conflicts occur during revert, Git pauses for manual resolution just like merges.

Syntax:

git revert [options] <commit>...

Parameters:

commit: The commit(s) to revert. Can be SHA, branch name, or HEAD~n

-n or --no-commit: Apply revert but don't create commit automatically (allows batch reverts)

-m parent-number: For merge commits, specify which parent to revert to (usually -m 1)

--continue: Continue revert after resolving conflicts

--abort: Cancel the revert and return to pre-revert state

--skip: Skip current commit and continue reverting others

-e or --edit: Edit the revert commit message before committing

Return Value:

Returns 0 on successful revert. Creates new commit with message: "Revert ''" and details explaining what was reverted. Shows file change statistics. Returns non-zero if conflicts occur - must resolve conflicts, git add files, then git revert --continue. Preserves full history - both original commit and revert commit remain visible in git log.

Examples:

Example 1: Revert problematic Skills model change

Result:

  • Action: Creates new commit that undoes changes from c3d4e5f
  • Output: "Revert 'Broke Skills validation'. This reverts commit c3d4e5f6g7h8..."
  • Commit Message: Automatically generated
  • History: Git log now shows revert commit on top, followed by normal history
  • Code State: Skills model returns to working state before c3d4e5f, all history preserved
  • Collaboration: Teammates pulling from remote get the fix automatically
  • Safety: Safer than reset for public branches
  • Flexibility: Can later re-revert if needed

git stash

Full Description:

The git stash command temporarily saves uncommitted changes (both staged and unstaged) and reverts the working directory to match HEAD, giving you a clean workspace. Stashed changes are saved in a stack that you can reapply later with git stash pop or git stash apply. This is essential when you need to switch branches urgently but aren't ready to commit current work, or when pulling changes that would conflict with local modifications. Stash includes a message for identification, can be listed with git stash list, and individual stashes referenced as stash@{0}, stash@{1}, etc. Unlike commits, stashes are local-only and not pushed to remotes. Can stash untracked files with -u flag, and can create branch from stash for complex work.

Syntax:

git stash [push] [-m "message"] [-u] [-a]
git stash list
git stash show [stash@{n}]
git stash apply [stash@{n}]
git stash pop [stash@{n}]
git stash drop [stash@{n}]
git stash clear
git stash branch <branchname> [stash@{n}]

Parameters:

git stash or git stash push: Save current changes to stash stack. Default command

-m "message": Add descriptive message to stash entry. Helps identify stash later

-u or --include-untracked: Also stash untracked files (not just modified tracked files)

-a or --all: Stash untracked AND ignored files. Rarely used

list: Show all stash entries with messages and commit references

show [stash@{n}]: Show diff of stash. Without index, shows latest stash@{0}

apply [stash@{n}]: Apply stash changes but keep stash in stack. Safe for trying out stash

pop [stash@{n}]: Apply stash changes and remove from stack. Most common usage

drop [stash@{n}]: Delete specific stash without applying

clear: Remove ALL stashes. Permanent deletion, use carefully

branch <name> [stash@{n}]: Create new branch from stash commit, apply stash to it

Return Value:

Stash returns 0 on success. Shows \"Saved working directory and index state WIP on <branch>: <commit> <message>\". List shows format: \"stash@{0}: WIP on feature-branch: a1b2c3d Added JobPost validation\". Apply/pop may show merge conflicts if stash doesn't apply cleanly - resolve like normal merge. Pop removes stash after successful apply. Show displays diff stats and patch. Stashes stored in .git/refs/stash and can be recovered with reflog if accidentally deleted. Max stash entries unlimited but keep it manageable.

Examples:

Example 1: Stash work in progress to switch branches

Result:

  • Action: Stash saves JobPost validation work
  • Output: "Saved working directory and index state WIP on feature/jobpost-validation: a1b2c3d Started JobPost validation"
  • Working Directory: Now clean, can switch branches freely
  • Context Switch: Fixed Location bug on main branch without losing JobPost progress
  • Restore: After switching back, git stash pop restores JobPost changes
  • Pop Output: "On branch feature/jobpost-validation. Changes not staged: jobpost/models.py. Dropped refs/stash@{0}"
  • Workflow: Work continues seamlessly
  • Common Usage: Most common stash workflow for context switching

8. Tagging & Releases

Mark specific commits as releases or important milestones.

git tag

Full Description:

The git tag command creates, lists, and deletes tags - named references to specific commits that mark important points in project history. Tags are used for releases (v1.0.0, v2.1.3), milestones, or any commit you want to reference easily. There are two types: lightweight tags (simple pointers to commits) and annotated tags (full Git objects with tagger name, email, date, and message - recommended for releases). Unlike branches, tags don't move when new commits are added. Tags must be explicitly pushed to remotes with git push origin <tagname> or git push --tags. Common in Django projects for marking production deployments, stable releases, or migration checkpoints. Can checkout code at any tag to see historical state.

Syntax:

git tag                          # List tags
git tag <tagname>                # Create lightweight tag
git tag -a <tagname> -m "msg"    # Create annotated tag
git tag <tagname> <commit>       # Tag specific commit
git tag -d <tagname>             # Delete local tag
git push origin <tagname>        # Push tag to remote
git push origin --tags           # Push all tags
git push origin --delete <tag>  # Delete remote tag

Parameters:

<tagname>: Name of the tag. Convention: v1.0.0 for semantic versioning (vMAJOR.MINOR.PATCH)

-a: Create annotated tag (stores metadata). Recommended for releases

-m "message": Tag message (with -a flag). Describes release or milestone

<commit>: Specific commit SHA to tag. If omitted, tags current HEAD

-l "pattern": List tags matching pattern. Example: git tag -l "v1.*"

-d: Delete local tag

-f: Force tag creation (replace existing tag). Dangerous, avoid

--sort=<key>: Sort tags by version, date, etc. Example: --sort=-version:refname

Return Value:

Returns 0 on success. Listing tags shows all tags in alphabetical order (v1.0.0, v1.1.0, v2.0.0). Creating tag shows no output on success. Annotated tag with -a opens editor for message if -m not provided. Git push origin <tagname> shows \"* [new tag] v1.0.0 -> v1.0.0\". Deleting shows \"Deleted tag 'v1.0.0'\". Tags stored in .git/refs/tags/. Can view tag details with git show <tagname>. Annotated tags show tagger info and message, lightweight tags just show commit.

9. Submodules

Manage external dependencies and nested repositories.

git submodule add

Full Description:

The git submodule add command embeds an external Git repository inside your project as a subdirectory, while maintaining it as a separate repository. Submodules are useful for including shared libraries, reusable Django apps, or third-party components that have their own development lifecycle. The parent repo (anihpj) stores a reference to a specific commit in the submodule, not the actual files. This keeps submodule code isolated and versioned independently. After adding, submodule info is stored in .gitmodules file and .git/modules/ directory. Must commit .gitmodules and submodule path to parent repo. Team members must run git submodule update --init after cloning to populate submodules.

Syntax:

git submodule add <repository-url> [<path>]
git submodule add -b <branch> <repository-url> [<path>]

Parameters:

<repository-url>: Git URL of submodule repo. Can be HTTPS or SSH

<path>: Directory to place submodule. If omitted, uses repo name

-b <branch>: Track specific branch in submodule (instead of detached HEAD)

--name <name>: Logical name for submodule (different from path)

--force: Add submodule even if path exists in index

--depth <n>: Shallow clone with limited history

Return Value:

Returns 0 on success. Creates submodule directory, clones repository into it, creates/updates .gitmodules file with submodule config, and stages changes for commit. Output: "Cloning into ''... done". Git status shows new files: .gitmodules and submodule path. Must commit these to finalize submodule addition. Submodule is locked to specific commit SHA. Changes inside submodule directory don't affect parent repo unless you commit submodule pointer update.

Examples:

Example 1: Add reusable Django authentication app as submodule

git commit -m "Add django-auth-lib as submodule for User authentication"

Result: Clones django-auth-lib into lib/auth directory. .gitmodules file created with submodule configuration. Submodule appears as a single entry in git status (not individual files). Parent repo stores commit SHA that lib/auth submodule should be at. Can now use authentication features from lib/auth in anihpj project for User model authentication. git submodule update

Full Description:

The git submodule update command synchronizes submodule working directories to the commit SHA recorded in the parent repository. When you pull changes in parent repo that update submodule pointers, submodule directories become out of sync until you run update. The --init flag initializes uninitialized submodules (like after fresh clone), and --recursive processes nested submodules. Use --remote to fetch latest from submodule's tracked branch instead of parent's recorded commit. Critical for keeping development environment consistent with team - ensures everyone uses same versions of auth library, widgets, and shared Django apps. Run after pulling parent repo changes or switching branches that have different submodule commits.

Syntax:

git submodule update [--init] [--recursive] [--remote] [<path>]

Parameters:

--init: Initialize submodules that aren't initialized yet. Essential after clone

--recursive: Update nested submodules (submodules within submodules)

--remote: Fetch latest commit from submodule's tracked branch instead of parent's recorded commit

<path>: Update only specific submodule instead of all. Example: lib/auth

--force: Discard local changes in submodule. Dangerous

--checkout: Checkout recorded commit (default behavior)

--rebase: Rebase current branch onto recorded commit

--merge: Merge recorded commit into submodule's current branch

Return Value:

Returns 0 on success. With --init, clones missing submodules. Without --init, checks out recorded commit in each submodule. Output: "Submodule path 'lib/auth': checked out 'a1b2c3d'". Shows commit SHA each submodule updated to. With --remote, may show "Submodule 'lib/auth' (https://...) registered" and fetch progress. Submodules in detached HEAD state after update (not on branch). Use git status in submodule to see "HEAD detached at a1b2c3d". Normal - submodules track specific commits not branches.

# Verify

Output shows registration of lib/auth, lib/notifications, lib/search submodules, cloning progress, and checkout of specific commits. --init flag essential for first-time setup. --recursive handles nested submodules. Useful Git Aliases

Add these to your global Git config for faster workflow:

# Set up aliases
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.ci commit
git config --global alias.unstage 'reset HEAD --'
git config --global alias.last 'log -1 HEAD'
git config --global alias.visual 'log --oneline --graph --decorate --all'

# Now you can use:
git st         # instead of git status
git co main    # instead of git checkout main
git visual     # beautiful commit graph
