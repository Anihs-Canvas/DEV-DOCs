🏭 Factory Class

The Factory class is the core building block of FactoryBoy. It allows you to define templates for creating instances of your Django models with predictable, customizable test data. Each Factory subclass corresponds to a specific model and declares how to generate values for its fields.

Factory Class - Overview

The Factory class provides a declarative way to create model instances for testing and development.

📖 Full Description

The Factory class is the foundation of FactoryBoy's test data generation capabilities. It provides a declarative interface for defining how to create instances of your Django models with realistic, consistent test data.

Key characteristics of the Factory class:

  • Declarative Syntax: Define field values using class attributes rather than imperative code
  • Inheritance Support: Factory classes can inherit from other factories, enabling code reuse
  • Strategy Pattern: Supports multiple object creation strategies (BUILD, CREATE, STUB)
  • Lazy Evaluation: Field values can be computed dynamically at object creation time
  • Relationship Handling: Built-in support for ForeignKey, OneToOne, and ManyToMany relationships
  • Post-Generation Hooks: Execute custom logic after object creation
  • Sequence Management: Automatic sequence tracking for unique values

The Factory class works by defining a Meta inner class that specifies the target model, and then declaring class attributes that define how each model field should be populated. When you call the factory, it generates an instance with all declared fields properly populated.

Syntax

import factory
from factory.django import DjangoModelFactory

class MyModelFactory(DjangoModelFactory):
    """Factory for creating MyModel instances."""
    
    class Meta:
        model = MyModel  # Required: The Django model to create
        abstract = False  # Optional: Whether this is an abstract base factory
        inline_args = ()  # Optional: Positional arguments for model __init__
        exclude = ()  # Optional: Fields to exclude from model instantiation
        rename = {}  # Optional: Rename factory attributes to model fields
        strategy = factory.CREATE_STRATEGY  # Optional: Default creation strategy
    
    # Field declarations
    field_name = factory.Faker('provider_name')
    another_field = factory.Sequence(lambda n: f'value_{n}')
    related_field = factory.SubFactory(RelatedFactory)
    
    # Class methods
    @classmethod
    def _create(cls, model_class, *args, **kwargs):
        """Override to customize object creation."""
        return model_class.objects.create(*args, **kwargs)
    
    @classmethod
    def _build(cls, model_class, *args, **kwargs):
        """Override to customize object building."""
        return model_class(*args, **kwargs)

📋 Meta Options (Parameters Description)

The Meta inner class configures the Factory's behavior:

  • model - The Django model class that this factory will create instances of. This is required for all concrete factories. Example: model = User
  • abstract - Boolean flag indicating whether this factory is abstract (a base class for other factories). Abstract factories don't require a model and cannot be instantiated directly. Default is False. Example: abstract = True
  • inline_args - Tuple of field names to pass as positional arguments to the model's __init__ method instead of keyword arguments. Useful for models with required positional parameters. Example: inline_args = ('name', 'email')
  • exclude - Tuple of factory attribute names that should not be passed to the model during instantiation. These attributes exist only in the factory for computing other values. Example: exclude = ('temp_value',)
  • rename - Dictionary mapping factory attribute names to model field names. Useful when factory attributes need different names than model fields (e.g., to avoid Python keyword conflicts). Example: rename = {'class_name': 'class'}
  • strategy - Default creation strategy for this factory. Can be BUILD_STRATEGY, CREATE_STRATEGY, or STUB_STRATEGY. Default is CREATE_STRATEGY. Example: strategy = factory.BUILD_STRATEGY

🔧 Attributes

Factory class attributes define how model fields are populated:

  • _meta - Internal attribute containing the factory's configuration (read-only)
  • _options_class - The class used for Meta options (can be overridden)
  • Field declarations - Class attributes that define how to generate values for model fields using FactoryBoy declarations like Faker, Sequence, SubFactory, etc.

⚡ Methods (with Syntax and Parameters Description)

__call__(**kwargs)

Creates an instance using the factory's default strategy.

Syntax:

instance = MyFactory(**kwargs)

Parameters:

  • **kwargs - Keyword arguments to override factory-declared field values

Behavior: Uses the factory's default strategy (typically CREATE) to generate an instance.

build(**kwargs)

Builds an instance without saving it to the database.

Syntax:

instance = MyFactory.build(**kwargs)

Parameters:

  • **kwargs - Keyword arguments to override factory-declared field values

Behavior: Creates an instance by calling the model's __init__ without persisting to database.

create(**kwargs)

Creates and saves an instance to the database.

Syntax:

instance = MyFactory.create(**kwargs)

Parameters:

  • **kwargs - Keyword arguments to override factory-declared field values

Behavior: Creates and persists an instance to the database using the model's save method or objects.create().

create_batch(size, **kwargs)

Creates multiple instances and saves them to the database.

Syntax:

instances = MyFactory.create_batch(size, **kwargs)

Parameters:

  • size - Integer, the number of instances to create
  • **kwargs - Keyword arguments to override factory-declared field values for all instances

Behavior: Creates multiple instances efficiently, each with unique sequence values.

build_batch(size, **kwargs)

Builds multiple instances without saving them.

Syntax:

instances = MyFactory.build_batch(size, **kwargs)

Parameters:

  • size - Integer, the number of instances to build
  • **kwargs - Keyword arguments to override factory-declared field values
stub(**kwargs)

Creates a stub object with attributes but no database interaction.

Syntax:

stub = MyFactory.stub(**kwargs)

Parameters:

  • **kwargs - Keyword arguments to override factory-declared field values

Behavior: Returns a simple object with attributes but no database functionality.

reset_sequence(value=0, force=False)

Resets the factory's sequence counter.

Syntax:

MyFactory.reset_sequence(value=0, force=False)

Parameters:

  • value - Integer, the value to reset the sequence to (default: 0)
  • force - Boolean, if True, resets even if the factory has a custom _setup_next_sequence method

↩️ Return Value

The Factory class methods return different types of values depending on the strategy used:

  • create() / __call__(): Returns a fully persisted Django model instance saved to the database with a primary key
  • build(): Returns a Django model instance that has not been saved to the database (no primary key unless manually set)
  • stub(): Returns a StubObject with attributes matching the factory declarations but no model functionality
  • create_batch() / build_batch(): Returns a list of model instances
  • reset_sequence(): Returns None

Example 4: Advanced Factory with JobPost Model and All Relationships

This example demonstrates creating a complete JobPost factory that handles multiple relationships (ForeignKey for Author and Location, ManyToMany for Skills) and uses various field generation strategies including Faker, Sequence, SubFactory, and LazyAttribute.

import factory
from factory.django import DjangoModelFactory
from jobpost.models import JobPost, Author, Location, Skills
from django.utils.text import slugify
import datetime

class JobPostFactory(DjangoModelFactory):
    """Factory for creating complete JobPost instances with all relationships."""
    
    class Meta:
        model = JobPost
    
    # Basic fields
    company_name = factory.Faker('company')
    company_email = factory.Faker('company_email')
    job_title = factory.Faker('job')
    
    # Generate slug from job title
    slug = factory.LazyAttribute(lambda obj: slugify(obj.job_title))
    
    # Text fields
    job_description = factory.Faker('text', max_nb_chars=500)
    requirements = factory.Faker('text', max_nb_chars=300)
    
    # Relationships
    author = factory.SubFactory(AuthorFactory)
    location = factory.SubFactory(LocationFactory)
    
    # Date fields
    application_start = factory.LazyFunction(datetime.date.today)
    application_end = factory.LazyAttribute(
        lambda obj: obj.application_start + datetime.timedelta(days=30)
    )
    
    # ManyToMany relationship - handled in post-generation
    @factory.post_generation
    def skills_list(self, create, extracted, **kwargs):
        """Add skills to the job post."""
        if not create:
            return
        
        if extracted:
            # If skills_list was provided, use those
            for skill in extracted:
                self.skills_list.add(skill)
        else:
            # Otherwise, create and add 3 random skills
            for _ in range(3):
                skill = SkillsFactory.create()
                self.skills_list.add(skill)

Example 5: Meta Options, Exclude, and Strategy Variations

This example demonstrates advanced Meta options including abstract factories, exclude fields, inline_args, and different strategies. We'll create base factories and extend them, showing inheritance and customization patterns.

import factory
from factory.django import DjangoModelFactory
from jobpost.models import JobPost, Author, Location

# Abstract base factory
class BaseJobPostFactory(DjangoModelFactory):
    """Abstract factory with common fields for JobPost."""
    
    class Meta:
        model = JobPost
        abstract = True  # Can't be instantiated directly
    
    company_name = factory.Faker('company')
    company_email = factory.Faker('company_email')
    author = factory.SubFactory(AuthorFactory)
    location = factory.SubFactory(LocationFactory)

# Concrete factory extending abstract base
class RemoteJobFactory(BaseJobPostFactory):
    """Factory for remote job posts."""
    
    class Meta:
        model = JobPost
        exclude = ('is_remote',)  # Exclude from model creation
    
    # Helper attribute (excluded from model)
    is_remote = True
    
    job_title = factory.LazyAttribute(
        lambda obj: f"Remote {factory.Faker('job').generate({})}"
    )
    slug = factory.LazyAttribute(lambda obj: slugify(obj.job_title))
    job_description = "This is a fully remote position."
    requirements = "Must have home office setup."
    
    # Override location to indicate remote
    location = factory.SubFactory(LocationFactory, city='Remote', state='Anywhere')

# Factory with BUILD strategy
class JobPostDraftFactory(DjangoModelFactory):
    """Factory that builds but doesn't save JobPost instances."""
    
    class Meta:
        model = JobPost
        strategy = factory.BUILD_STRATEGY  # Default to build

    company_email = factory.Faker('company_email')
    job_title = factory.Faker('job')
    slug = factory.LazyAttribute(lambda obj: slugify(obj.job_title))
    job_description = factory.Faker('text')
    requirements = factory.Faker('text')
    author = factory.SubFactory(AuthorFactory)
    location = factory.SubFactory(LocationFactory)

⚙️ Meta Options Configuration

Meta options control factory behavior and configuration. These options are specified in the inner Meta class and control everything from which model to use, to how arguments are passed, to which fields are excluded from generation.

model - Target Model Specification

Specify the Django model this factory creates instances for.

📖 Full Description

The model Meta option is the cornerstone of every DjangoModelFactory, defining which Django model the factory will generate instances for. This required setting establishes the connection between your factory class and the database model it represents, enabling FactoryBoy to understand the model's structure, relationships, and constraints.

Key aspects of the model Meta option:

  • Required Setting: Every concrete DjangoModelFactory must specify a model
  • Direct Model Reference: Takes the actual model class, not a string
  • Manager Integration: Uses the model's default manager for database operations
  • Field Introspection: FactoryBoy can auto-discover model fields for validation
  • Relationship Handling: Enables proper handling of ForeignKey, OneToOne, and ManyToMany

Syntax

import factory
from factory.django import DjangoModelFactory
from jobpost.models import JobPost, Skills, Location, Author
from django.contrib.auth.models import User

# Basic model specification
class JobPostFactory(DjangoModelFactory):
    class Meta:
        model = JobPost  # Required: direct reference to model class
    
    title = factory.Faker('job')
    description = factory.Faker('text')

# Multiple factories for different models
class UserFactory(DjangoModelFactory):
    class Meta:
        model = User
    
    username = factory.Sequence(lambda n: f'user_{n}')

class SkillsFactory(DjangoModelFactory):
    class Meta:
        model = Skills
    
    skill_name = factory.Faker('job')

📋 Parameters Description

The model Meta option accepts a single parameter:

  • model class - The Django model class that this factory will create instances of. Must be an actual class reference (e.g., User, JobPost), not a string. The model must be imported and accessible in the factory's scope. FactoryBoy will use this model's objects.create() manager method for the CREATE strategy and the model's __init__() for the BUILD strategy.

🔧 Attributes

When the model Meta option is set, FactoryBoy internally configures:

  • _meta.model - Stores the reference to the specified model class
  • _meta.model._meta - Access to Django's model metadata for field introspection
  • _meta.django_get_or_create - Related setting for get_or_create behavior

⚡ Methods

The model setting influences these factory methods:

  • _create(model_class, *args, **kwargs) - Creates and saves instance using the specified model
    • model_class - The model class from Meta.model
    • **kwargs - Field values to pass to model creation
    • Returns: Saved model instance
  • _build(model_class, *args, **kwargs) - Builds instance without saving
    • model_class - The model class from Meta.model
    • **kwargs - Field values to pass to model __init__
    • Returns: Unsaved model instance

↩️ Return Value

Setting the model option doesn't directly return a value, but it determines what type of instances the factory will create. When you call MyFactory.create(), it returns an instance of the specified model class.

abstract - Abstract Base Factory

Define factories that cannot be instantiated directly but can be inherited.

📖 Full Description

Setting abstract = True in the Meta class creates an abstract factory that cannot be used to create objects directly. Abstract factories are useful as base classes that define common declarations which can be inherited by concrete factories. This promotes DRY (Don't Repeat Yourself) principles in test factories.

Key features and benefits of abstract factories:

  • Code Reusability: Define common field declarations once and reuse across multiple factories
  • DRY Principle: Avoid repetition of similar factory code
  • Inheritance Hierarchy: Create factory class hierarchies that mirror your model hierarchies
  • Cannot Be Instantiated: Abstract factories raise an error if you try to call them directly
  • Override Flexibility: Child factories can override parent declarations selectively

Syntax

import factory
from django.contrib.auth.models import User
from jobpost.models import Author

class BaseFactory(factory.django.DjangoModelFactory):
    """Abstract base factory - cannot be instantiated"""
    class Meta:
        abstract = True  # Marks this factory as abstract
    
    # Common declarations for all child factories
    created_at = factory.Faker('date_time_this_year')
    updated_at = factory.Faker('date_time_this_year')

class BaseUserFactory(factory.django.DjangoModelFactory):
    """Abstract base factory with common user fields"""
    class Meta:
        model = User
        abstract = True  # Cannot instantiate this factory directly
    
    username = factory.Sequence(lambda n: f'user_{n}')
    email = factory.LazyAttribute(lambda obj: f'{obj.username}@example.com')
    first_name = factory.Faker('first_name')
    last_name = factory.Faker('last_name')
    is_active = True

class RegularUserFactory(BaseUserFactory):
    """Concrete factory for regular users"""
    class Meta:
        model = User
    
    is_staff = False
    is_superuser = False

class AdminUserFactory(BaseUserFactory):
    """Concrete factory for admin users"""
    class Meta:
        model = User
    
    is_staff = True
    is_superuser = True

# Usage - abstract factories cannot be called
try:
    user = BaseUserFactory.create()  # ERROR
except factory.FactoryError:
    print("Cannot instantiate abstract factory")

# Concrete factories work normally
regular = RegularUserFactory.create()  # OK
admin = AdminUserFactory.create()  # OK

Parameters Description

abstract (bool): A boolean value set in the factory's Meta class. When set to True, the factory becomes abstract and cannot be instantiated directly. The factory can only be used as a base class for inheritance. When set to False or omitted (default), the factory is concrete and can be instantiated normally.

The abstract parameter is particularly useful when you have multiple factories sharing common field definitions, allowing you to extract shared logic into a base factory class. Child factories inherit all declarations from the abstract parent and can override them as needed.

Attributes

  • _meta.abstract (bool): Returns True if the factory is abstract, False otherwise
  • _meta.base_declarations (dict): Dictionary of all declarations defined in the factory class, including inherited ones
  • _meta.model: The model class specified in Meta (can be None for abstract factories)
  • __mro__: Method resolution order showing the inheritance chain from child to abstract parents

Methods

Inherited Methods: Abstract factories cannot be called directly, but child factories inherit standard factory methods:

  • build(**kwargs): Create an instance without saving (inherited by children)
  • create(**kwargs): Create and save an instance (inherited by children)
  • build_batch(size, **kwargs): Build multiple instances (inherited by children)
  • create_batch(size, **kwargs): Create multiple instances (inherited by children)

Attempting to call any instantiation method on an abstract factory will raise a factory.FactoryError.

Return Value

Abstract factories themselves cannot return values since they cannot be instantiated. However, child factories that inherit from abstract factories return model instances as normal. The return value is the same as any concrete factory:

  • For build(): Returns an unsaved model instance
  • For create(): Returns a saved model instance
  • For build_batch() and create_batch(): Returns a list of instances

inline_args - Positional Argument Specification

Define which factory declarations should be passed as positional arguments during object instantiation.

📖 Full Description

The inline_args Meta option specifies a tuple of field names that should be passed as positional arguments to the model's __init__ method instead of keyword arguments. This is useful for models that require positional arguments or for optimizing instantiation when you know the expected argument order.

Key features and use cases of inline_args:

  • Positional Requirement: Some models require certain arguments in a specific positional order
  • Legacy Code Support: Work with older code that expects positional arguments
  • API Compatibility: Match third-party library initialization signatures
  • Order Preservation: Arguments are passed in the order specified in the tuple
  • Mixed Arguments: Can combine inline_args (positional) with regular kwargs

Syntax

import factory
from django.contrib.auth.models import User

class CustomModel:
    """Model that expects positional arguments"""
    def __init__(self, name, category, *args, **kwargs):
        self.name = name
        self.category = category
        self.description = kwargs.get('description', '')
        super().__init__(*args, **kwargs)

class CustomModelFactory(factory.Factory):
    class Meta:
        model = CustomModel
        inline_args = ('name', 'category')  # Pass these as positional args
    
    name = factory.Faker('word')
    category = factory.Faker('word')
    description = factory.Faker('text')  # This remains a keyword arg

# When creating: CustomModel('word1', 'word2', description='text...')

# For Django models with natural key requirements
class UserFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = User
        inline_args = ('username',)  # Pass username as first positional arg
    
    username = factory.Sequence(lambda n: f'user_{n}')
    email = factory.LazyAttribute(lambda obj: f'{obj.username}@example.com')
    first_name = factory.Faker('first_name')
    last_name = factory.Faker('last_name')

Parameters Description

inline_args (tuple): A tuple of string field names that specifies which factory declarations should be passed as positional arguments to the model's __init__ method. The order of field names in the tuple determines the order of positional arguments. Fields not listed in inline_args are passed as keyword arguments after all positional arguments.

When a factory with inline_args is called, FactoryBoy will extract the values for the specified fields and pass them as positional arguments in the exact order listed. All other field values are passed as keyword arguments. This allows you to match any __init__ signature that requires specific positional arguments.

Attributes

  • _meta.inline_args (tuple): Returns the tuple of field names to be passed as positional arguments
  • _meta.base_declarations (dict): Dictionary containing all field declarations, including those specified in inline_args
  • _meta.model: The target model class that will receive the positional arguments

Methods

Factories with inline_args support all standard factory methods, with positional argument handling applied automatically:

  • build(**kwargs): Create instance without saving, passing inline_args as positional parameters
  • create(**kwargs): Create and save instance, passing inline_args as positional parameters
  • build_batch(size, **kwargs): Build multiple instances, each with inline_args as positional
  • create_batch(size, **kwargs): Create multiple instances, each with inline_args as positional

When overriding values via kwargs in these methods, values for inline_args fields are still passed positionally in the correct order.

Return Value

Factories with inline_args return the same types as standard factories:

  • build(): Returns an unsaved model instance created with positional arguments
  • create(): Returns a saved model instance created with positional arguments
  • build_batch() and create_batch(): Return lists of instances

exclude - Helper Field Exclusion

Declare helper fields that should not be passed to the model during instantiation.

📖 Full Description

The exclude Meta option accepts a tuple of field names that are used as helper values during factory execution but should not be passed to the model's constructor. This is perfect for parameters, intermediate calculations, or values used by LazyAttribute that aren't actual model fields.

Key features and use cases of exclude:

  • Helper Parameters: Define intermediate values used in calculations without cluttering the model
  • Conditional Logic: Create flags or parameters that drive Maybe, LazyAttribute, or post_generation behavior
  • Clean Separation: Separate factory logic from model attributes
  • Parameter Passing: Accept parameters during factory calls that affect generation but aren't model fields
  • Computation Helpers: Store multipliers, flags, or temporary values used in complex field calculations

Syntax

import factory
from django.contrib.auth.models import User
from jobpost.models import JobPost, Skills, Author, Location

    salary_multiplier = 1.0
    is_remote = False
    experience_level = 'mid'  # Helper: 'junior', 'mid', 'senior'
    
    # Use helper fields in LazyAttribute
    salary_min = factory.LazyAttribute(
        lambda obj: int(50000 * obj.salary_multiplier)
    )
    salary_max = factory.LazyAttribute(
        lambda obj: int(100000 * obj.salary_multiplier)
    )
    
    # Use is_remote helper in Maybe
    location = factory.Maybe(
        'is_remote',
        yes_declaration=None,  # Remote jobs have no location

    )
    
    author = factory.SubFactory(AuthorFactory)

    salary_multiplier=2.0,  # Helper affects salary calculation
    experience_level='senior'
)

    is_remote=True  # Helper determines location=None
)

Parameters Description

exclude (tuple): A tuple of string field names that specifies which factory declarations should be excluded from being passed to the model's __init__ method. These fields exist only within the factory's build context and are available for use in LazyAttribute, Maybe, post_generation, and other factory declarations, but they never reach the model itself.

Excluded fields can still be accessed within the factory's scope using obj.field_name in LazyAttribute callbacks or by referencing them in conditional declarations like Maybe. They are evaluated normally during factory execution but are stripped out before instantiating the model.

Attributes

  • _meta.exclude (tuple): Returns the tuple of field names that are excluded from model instantiation
  • _meta.base_declarations (dict): Dictionary containing ALL declarations including excluded ones
  • _meta.model: The target model class that receives only non-excluded fields
  • _meta.pre_declarations (dict): Declarations evaluated before model instantiation, includes excluded fields

Methods

Factories with exclude support all standard factory methods:

  • build(**kwargs): Create instance without saving; excluded fields are available during build but not passed to model. Parameters:
    • **kwargs: Override any field including excluded ones (excluded remain internal)
  • create(**kwargs): Create and save instance; excluded fields guide generation but aren't saved. Parameters:
    • **kwargs: Override any field including excluded ones
  • build_batch(size, **kwargs): Build multiple instances with excluded fields. Parameters:
    • size (int): Number of instances to create
    • **kwargs: Overrides applied to all instances
  • create_batch(size, **kwargs): Create multiple instances with excluded fields. Parameters:
    • size (int): Number of instances to create
    • **kwargs: Overrides applied to all instances

Return Value

Factories with exclude return model instances that do NOT contain the excluded fields:

  • build(): Returns an unsaved model instance without excluded fields
  • create(): Returns a saved model instance without excluded fields
  • build_batch() and create_batch(): Return lists of instances without excluded fields

The returned instances will not have attributes for excluded fields, even though those fields were used during generation.

rename - Field Name Mapping

Map factory declaration names to different model field names.

📖 Full Description

The rename Meta option provides a dictionary mapping factory field names to model field names. This is useful when you want to use a different name in your factory (perhaps to avoid Python reserved keywords like 'class', 'from', 'type', or 'import') but pass it to the model with the correct field name.

Key features and use cases of rename:

  • Keyword Avoidance: Use valid Python identifiers in factories even when model uses reserved keywords
  • API Compatibility: Match external API field names that conflict with Python syntax
  • Legacy Support: Work with models that have unconventional field naming
  • Code Clarity: Use more descriptive names in factories that map to shorter model field names
  • Multiple Mappings: Support multiple field renames in a single factory

Syntax

import factory
from django.contrib.auth.models import User

class Message:
    """Model with 'from' field (Python reserved keyword)"""
    def __init__(self, from_user, to_user, content, type_info):
        self.from_user = from_user
        self.to_user = to_user
        self.content = content
        self.type_info = type_info

class MessageFactory(factory.Factory):
    class Meta:
        model = Message
        rename = {
            'from_': 'from_user',  # Map from_ to from_user
            'type_': 'type_info'   # Map type_ to type_info
        }
    
    from_ = factory.SubFactory(UserFactory)  # Use from_ in factory
    to_user = factory.SubFactory(UserFactory)
    content = factory.Faker('text')
    type_ = factory.Iterator(['email', 'sms', 'notification'])

# Creates: Message(from_user=..., to_user=..., content=..., type_info='email')

# For Django models
class UserFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = User
        rename = {'klass': 'class'}  # If model had a 'class' field
    
    username = factory.Sequence(lambda n: f'user_{n}')
    email = factory.Faker('email')
    klass = factory.Iterator(['A', 'B', 'C'])  # Use klass in factory

Parameters Description

rename (dict): A dictionary mapping factory field names (keys) to model field names (values). When the factory builds an object, it will use the keys internally but pass the values to the model's __init__ method. The dictionary allows multiple field mappings, enabling you to rename as many fields as needed in a single factory.

The rename mapping applies during the final instantiation step. Throughout factory execution, the factory uses the original field names (dictionary keys). Only when calling the model's __init__ method does FactoryBoy substitute the mapped names (dictionary values).

Attributes

  • _meta.rename (dict): Returns the dictionary of field name mappings
  • _meta.base_declarations (dict): Contains declarations using factory field names (not renamed)
  • _meta.model: The target model class that receives the renamed field names
  • _meta.pre_declarations (dict): Shows factory field names before renaming is applied

Methods

Factories with rename support all standard factory methods, with name mapping applied automatically:

  • build(**kwargs): Create instance without saving, applying rename mappings. Parameters:
    • **kwargs: Use factory field names (not model names) when overriding
  • create(**kwargs): Create and save instance with rename mappings. Parameters:
    • **kwargs: Use factory field names for overrides
  • build_batch(size, **kwargs): Build multiple instances with renaming. Parameters:
    • size (int): Number of instances to create
    • **kwargs: Overrides using factory field names
  • create_batch(size, **kwargs): Create multiple instances with renaming. Parameters:
    • size (int): Number of instances to create
    • **kwargs: Overrides using factory field names

Return Value

Factories with rename return model instances with the renamed fields:

  • build(): Returns unsaved model instance with renamed field names as attributes
  • create(): Returns saved model instance with renamed field names as attributes
  • build_batch() and create_batch(): Return lists of instances with renamed fields

The returned instances will have attributes matching the model field names (renamed values), not the factory field names (keys).

strategy - Default Creation Strategy

Set the default strategy (BUILD, CREATE, or STUB) for factory invocation.

📖 Full Description

The strategy Meta option sets the default creation strategy when the factory is called without an explicit strategy method. The strategy determines how objects are created and whether they interact with the database.

The three available strategies and their use cases:

  • CREATE_STRATEGY: Default strategy that creates and saves objects to the database using the _create() method. Best for integration tests requiring persisted data.
  • BUILD_STRATEGY: Creates objects without saving to database using the _build() method. Ideal for unit tests that don't need persistence or for performance-critical test suites.
  • STUB_STRATEGY: Creates lightweight stub objects (StubObject) without database interaction. Perfect for testing logic that only accesses object attributes without requiring a real model instance.

Key features of the strategy option:

  • Default Behavior: Sets what happens when you call the factory directly without build()/create()/stub()
  • Override Flexibility: Can always explicitly call build(), create(), or stub() regardless of default strategy
  • Performance Tuning: BUILD and STUB strategies avoid database I/O for faster tests
  • Test Isolation: BUILD_STRATEGY helps with test isolation by not persisting data
  • Mock Testing: STUB_STRATEGY provides lightweight objects for pure logic testing

Syntax

import factory
from django.contrib.auth.models import User
from jobpost.models import Skills, Author

# Default strategy: CREATE (most common)
class UserFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = User
        # strategy = factory.CREATE_STRATEGY  # This is the default
    
    username = factory.Sequence(lambda n: f'user_{n}')
    email = factory.Faker('email')

# Override default to BUILD
class SkillsFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = Skills
        strategy = factory.BUILD_STRATEGY  # Default to build (no save)
    
    name = factory.Faker('word')
    level = factory.Iterator(['Beginner', 'Intermediate', 'Advanced'])

# Override default to STUB
class AuthorStubFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = Author
        strategy = factory.STUB_STRATEGY  # Default to stub
    
    fullname = factory.Faker('name')
    slug = factory.LazyAttribute(lambda obj: obj.fullname.lower().replace(' ', '-'))

# Usage examples
user1 = UserFactory()  # CREATE (saved to DB)
skill1 = SkillsFactory()  # BUILD (not saved)
author1 = AuthorStubFactory()  # STUB (StubObject)

# Can always override default strategy
user2 = UserFactory.build()  # BUILD despite CREATE default
skill2 = SkillsFactory.create()  # CREATE despite BUILD default
author2 = AuthorStubFactory.create()  # CREATE despite STUB default

Parameters Description

strategy (factory constant): One of three factory strategy constants that determines the default behavior when calling the factory:

  • factory.CREATE_STRATEGY: Creates and persists objects to the database. Calls the factory's _create() method, which typically calls Model.objects.create() or saves the instance. This is the default for DjangoModelFactory.
  • factory.BUILD_STRATEGY: Creates model instances without saving them. Calls the factory's _build() method, which instantiates the model but never calls save(). Useful for testing model methods and properties without database overhead.
  • factory.STUB_STRATEGY: Creates StubObject instances that have the declared attributes but are not actual model instances. No database interaction occurs. Perfect for testing code that only accesses object attributes.

The strategy can be set to control the default factory behavior, but explicit calls to build(), create(), or stub() always override this default.

Attributes

  • _meta.strategy: Returns the default strategy constant for the factory
  • _meta.model: The target model class (None for STUB_STRATEGY in pure stub factories)
  • _meta.base_factory: Reference to the base factory class

Methods

All factories support these methods regardless of default strategy:

  • __call__(**kwargs): Invokes factory using the default strategy set in Meta. Parameters:
    • **kwargs: Field overrides passed to the factory
    Returns: Object created using default strategy
  • build(**kwargs): Explicitly use BUILD_STRATEGY regardless of default. Parameters:
    • **kwargs: Field overrides
    Returns: Unsaved model instance
  • create(**kwargs): Explicitly use CREATE_STRATEGY regardless of default. Parameters:
    • **kwargs: Field overrides
    Returns: Saved model instance
  • stub(**kwargs): Explicitly use STUB_STRATEGY regardless of default. Parameters:
    • **kwargs: Field overrides
    Returns: StubObject with declared attributes
  • build_batch(size, **kwargs): Build multiple instances. Parameters:
    • size (int): Number of instances
    • **kwargs: Field overrides for all instances
    Returns: List of unsaved instances
  • create_batch(size, **kwargs): Create multiple instances. Parameters:
    • size (int): Number of instances
    • **kwargs: Field overrides for all instances
    Returns: List of saved instances
  • stub_batch(size, **kwargs): Stub multiple instances. Parameters:
    • size (int): Number of instances
    • **kwargs: Field overrides for all instances
    Returns: List of StubObjects

Return Value

The return value depends on the strategy used:

  • CREATE_STRATEGY: Returns a fully initialized and saved model instance with a primary key
  • BUILD_STRATEGY: Returns an initialized model instance without a primary key (not saved)
  • STUB_STRATEGY: Returns a StubObject with attributes matching the factory declarations but not a real model instance

  • String providers: Return strings (name, email, company, text, etc.)

LazyAttributeSequence - Combined Context and Sequence

Generate values based on both object attributes and sequence counter.

📖 Full Description

The LazyAttributeSequence combines the features of LazyAttribute and Sequence, providing access to both the object being built and an auto-incrementing sequence counter. This powerful declaration enables you to generate unique values that also depend on other field values in the same factory.

Key characteristics of LazyAttributeSequence:

  • Dual Context: Access to both object context (obj) and sequence number (n)
  • Automatic Uniqueness: Sequence counter ensures unique values across instances
  • Context-Aware Generation: Can reference other field values during computation
  • Flexible Computation: Supports complex logic combining multiple factors
  • Lazy Evaluation: Evaluated at object creation time, ensuring all dependencies are available

LazyAttributeSequence is ideal for generating unique identifiers that incorporate meaningful prefixes or patterns, creating sequential usernames based on names, or building unique URLs with path components.

Syntax

import factory

class MyFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = MyModel
    
    # LazyAttributeSequence with lambda
    field_name = factory.LazyAttributeSequence(lambda obj, n: f'{obj.prefix}{n}')
    
    # LazyAttributeSequence with function
    def generate_unique_code(obj, n):
        return f'{obj.category_code}-{n:05d}'
    
    unique_code = factory.LazyAttributeSequence(generate_unique_code)

📋 Parameters Description

  • method - A callable that accepts two positional arguments: obj (the factory stub object with all previously declared attributes) and n (an integer sequence counter starting from 0), and returns the computed field value. Can be a lambda function or a regular function.

🔧 Attributes

LazyAttributeSequence instances have the following internal attributes:

  • function - The callable method used to generate values
  • sequence - Reference to the factory's sequence tracker

⚡ Methods

LazyAttributeSequence uses these internal methods during evaluation:

  • evaluate(instance, step, extra) - Main evaluation method that computes the value
    • instance - The object being built with its current attribute values
    • step - The factory's build step object containing sequence information
    • extra - Additional context or overrides
    • Returns: The computed value based on both instance context and sequence

↩️ Return Value

Returns the value computed by the callable function, which can be any Python type (string, integer, object, etc.) based on both the object context and the sequence number. The return type depends entirely on what your function returns.

Iterator - Cycle Through Values

Use successive values from an iterable for field generation.

📖 Full Description

The Iterator declaration cycles through successive values from a provided iterable (list, tuple, generator, or any iterable object). Each time a new factory instance is created, it retrieves the next value from the iterable. When the iterable is exhausted, it can either restart from the beginning (if cycle=True) or raise an error (if cycle=False).

Key features of Iterator:

  • Sequential Access: Values are retrieved in order from the iterable
  • Cycling Control: Can repeat values indefinitely or stop after one pass
  • Getter Transformation: Apply a function to transform values before use
  • Manual Reset: Can reset iteration to start from the first value
  • Shared State: All instances of the factory share the same iterator position

Iterator is particularly useful for rotating through predefined categories, assigning round-robin values, or creating test data with controlled variation patterns.

Syntax

import factory

class MyFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = MyModel
    
    # Basic iterator with list
    field_name = factory.Iterator(['value1', 'value2', 'value3'])
    
    # Iterator with tuple and getter
    role = factory.Iterator(
        [('admin', 'Administrator'), ('user', 'Regular User')],
        getter=lambda x: x[0]  # Extract first element
    )
    
    # Non-cycling iterator (stops after exhausting values)
    country = factory.Iterator(['US', 'UK', 'FR'], cycle=False)
    
    # Iterator with generator
    status_codes = factory.Iterator(range(200, 206))

# Reset iterator manually
MyFactory.field_name.reset()

📋 Parameters Description

  • iterable - Required. The sequence of values to iterate through. Can be a list, tuple, generator, range object, or any Python iterable. Values are consumed in the order they appear.
  • cycle - Optional boolean (default: True). If True, the iterator restarts from the beginning when exhausted. If False, raises StopIteration error when all values have been used.
  • getter - Optional callable function that transforms each value before it's assigned. Receives the current iterated value and returns the transformed value. Useful for extracting specific elements from tuples or performing transformations.

🔧 Attributes

Iterator instances have the following internal attributes:

  • iterator - The Python iterator object wrapping the iterable
  • cycle - Boolean flag indicating whether the iterator should cycle
  • getter - The transformation function (if provided)
  • _original_iterable - Reference to the original iterable for reset purposes

⚡ Methods

  • reset() - Resets the iterator to start from the first value again
    • Parameters: None
    • Returns: None
    • Usage: Call on the factory class attribute (e.g., MyFactory.field_name.reset())
  • evaluate(instance, step, extra) - Internal evaluation method that retrieves the next value
    • instance - The object being built
    • step - The factory's build step
    • extra - Additional context
    • Returns: The next value from the iterator (optionally transformed by getter)

↩️ Return Value

Returns the next value from the iterable sequence. If a getter function is provided, returns the transformed value. The return type depends on what's in the iterable and what the getter function returns (if applicable).

Dict and List - Complex Data Structures

Generate dictionary and list fields with factory declarations.

📖 Full Description

The Dict and List declarations enable you to create complex nested data structures where values can utilize any factory declaration (Faker, Sequence, LazyAttribute, SubFactory, etc.). These declarations are essential for working with JSON fields, configuration objects, metadata, or any structured data that requires dynamic generation.

Key features of Dict and List:

  • Nested Declarations: Use any FactoryBoy declaration within dict values or list items
  • Dynamic Structures: Generate complex nested hierarchies with lazy evaluation
  • Field Override: Override specific dict keys or list indices using double underscore syntax
  • Parent Access: Reference factory-level fields using SelfAttribute with '..' notation
  • Custom Types: Support for OrderedDict, tuple, set, and other collection types
  • Mixed Content: Combine static values with dynamic declarations

Dict and List are particularly valuable for modern applications that store structured data in JSON fields, configuration management systems, or when testing APIs that accept complex nested payloads.

Syntax

import factory
from collections import OrderedDict

class MyFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = MyModel
    
    username = 'testuser'
    is_superuser = False
    
    # Dict with factory declarations
    metadata = factory.Dict({
        'created_at': factory.Faker('date_time'),
        'version': factory.Sequence(lambda n: f'1.{n}'),
        'is_admin': factory.SelfAttribute('..is_superuser'),  # Reference parent field
        'nested': {
            'key': 'static_value',
            'dynamic': factory.Faker('word')
        }
    })
    
    # List with mixed static and dynamic values
    permissions = factory.List([
        'read',
        'write',
        factory.Iterator(['admin', 'user']),
        factory.Faker('word')
    ])
    
    # OrderedDict example
    ordered_config = factory.Dict({
        'first': 'value1',
        'second': 'value2'
    }, dict_factory=OrderedDict)
    
    # List with custom factory (tuple)
    coordinates = factory.List([
        factory.Faker('latitude'),
        factory.Faker('longitude')
    ], list_factory=tuple)

# Usage with overrides
obj = MyFactory(
    metadata__version='2.0',  # Override dict key
    permissions__2='superadmin'  # Override list index 2
)

📋 Parameters Description

Dict Parameters:

  • params - Required dictionary where keys are field names and values can be static values or factory declarations. Supports nested dictionaries for hierarchical structures.
  • dict_factory - Optional callable (default: dict). Specifies the dictionary type to use. Can be OrderedDict, ChainMap, or any callable that returns a dict-like object. Useful when order matters or custom dict behavior is needed.

List Parameters:

  • items - Required sequence of values. Items can be static values or factory declarations. Each declaration is evaluated independently at creation time.
  • list_factory - Optional callable (default: list). Specifies the sequence type to use. Can be tuple, set, frozenset, or any callable that returns a sequence. Useful for immutable collections or set operations.

🔧 Attributes

Dict and List declarations have these internal attributes:

  • params/items - The original dictionary or list specification
  • dict_factory/list_factory - The callable used to construct the collection
  • _declarations - Internal registry of factory declarations within the structure

⚡ Methods

  • evaluate(instance, step, extra) - Main evaluation method that builds the collection
    • instance - The object being built
    • step - The factory's build step context
    • extra - Additional context including override values (e.g., metadata__key=value)
    • Returns: The constructed dictionary or list with all declarations evaluated

↩️ Return Value

Dict returns a dictionary (or dict_factory type) with all keys and values evaluated. List returns a list (or list_factory type) with all items evaluated. Nested declarations are recursively evaluated, and overrides specified via double underscore notation are applied.

Maybe - Conditional Field Values

Choose between two declarations based on a boolean field.

📖 Full Description

The Maybe declaration provides conditional field generation by selecting between two different declarations based on the truthiness of a "decider" field. This powerful feature enables factories to generate contextually appropriate data where one field's value determines how other fields should be populated.

Key characteristics of Maybe:

  • Boolean Logic: Evaluates decider field for truthiness (not just True/False)
  • Dual Declarations: Supports any FactoryBoy declaration for both yes and no branches
  • Lazy Evaluation: Only evaluates the selected declaration branch
  • Static Values: Can use static values (None, strings, numbers) or dynamic declarations
  • Field Dependencies: Creates logical dependencies between related fields

Maybe is particularly useful for modeling business logic where certain fields should only have values under specific conditions, such as deactivation dates for inactive accounts, premium features for paid users, or manager assignments for senior positions.

Syntax

import factory

class MyFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = MyModel
    
    is_active = True
    is_premium = False
    
    # Basic Maybe with static values
    status = factory.Maybe(
        'is_active',
        yes_declaration='active',
        no_declaration='inactive'
    )
    
    # Maybe with factory declarations
    deactivation_date = factory.Maybe(
        'is_active',
        yes_declaration=None,  # Active users have no deactivation date
        no_declaration=factory.Faker('past_date')  # Inactive users have past date
    )
    
    # Maybe with complex declarations
    subscription_level = factory.Maybe(
        'is_premium',
        yes_declaration=factory.Iterator(['gold', 'platinum', 'diamond']),
        no_declaration='free'
    )

# Usage - decider field controls dependent fields
obj = MyFactory(is_active=False)  # deactivation_date will be a past date

📋 Parameters Description

  • decider - Required string specifying the name of the boolean field (or any truthy/falsy field) that determines which declaration to use. The field is evaluated at object creation time.
  • yes_declaration - Required declaration or value used when the decider field evaluates to True or any truthy value. Can be a static value (None, string, number) or any FactoryBoy declaration (Faker, Sequence, SubFactory, etc.).
  • no_declaration - Required declaration or value used when the decider field evaluates to False or any falsy value (including 0, empty string, empty list). Can be a static value or any FactoryBoy declaration.

🔧 Attributes

Maybe instances have these internal attributes:

  • decider - The field name used for conditional evaluation
  • yes_declaration - The declaration for truthy conditions
  • no_declaration - The declaration for falsy conditions

⚡ Methods

  • evaluate(instance, step, extra) - Evaluation method that selects and evaluates the appropriate declaration
    • instance - The object being built with current attribute values
    • step - The factory's build step context
    • extra - Additional context or overrides
    • Returns: The value from yes_declaration if decider is truthy, otherwise value from no_declaration

↩️ Return Value

Returns the evaluated value from either yes_declaration or no_declaration based on the decider field's truthiness. The return type depends on what the selected declaration produces (can be any Python type).

Transformer - Transform Input Values

Apply a transformation function to field values.

📖 Full Description

The Transformer declaration applies a transformation function to input values, enabling consistent data normalization, format conversion, and validation enforcement across factory-generated instances. It processes both default values and values passed at creation time through the same transformation pipeline.

Key capabilities of Transformer:

  • Universal Transformation: Applies transformation to both default and overridden values
  • Lambda Support: Accepts any callable including lambdas, built-in functions, or custom methods
  • Bypass Mechanism: Provides Transformer.Force() to skip transformation when needed
  • Data Sanitization: Ideal for normalizing user input, trimming whitespace, or enforcing formats
  • Type Conversion: Can convert between types (string to int, lowercase to uppercase, etc.)
  • Validation Enforcement: Apply bounds checking, clamping, or constraint enforcement

Transformer is particularly useful when you need consistent data formatting regardless of how values are provided, ensuring data integrity and standardization across test datasets.

Syntax

import factory

class UserFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = User
    
    # Transform with built-in function
    username = factory.Transformer('john', transform=str.upper)
    
    # Transform with lambda
    email = factory.Transformer(
        'user@example.com',
        transform=lambda x: x.lower().strip()
    )
    
    # Chain multiple transformations
    normalized_name = factory.Transformer(
        '  John Doe  ',
        transform=lambda x: x.strip().title()
    )
    
    # Numeric transformations with bounds
    age = factory.Transformer(25, transform=lambda x: max(18, min(x, 100)))
    
    # Custom transformation function
    def format_phone(value):
        digits = ''.join(filter(str.isdigit, str(value)))
        return f"({digits[:3]}) {digits[3:6]}-{digits[6:10]}"
    
    phone = factory.Transformer('5551234567', transform=format_phone)

# Usage - all values are transformed
user = UserFactory()  # username = 'JOHN' (default transformed)
user = UserFactory(username='jane')  # username = 'JANE' (override transformed)

# Bypass transformation with Force
user = UserFactory(username=factory.Transformer.Force('jane'))
# username = 'jane' (not transformed)

📋 Parameters Description

  • default_value - Required positional argument specifying the default value to be transformed when no override is provided at creation time. Can be any Python type that the transform function can process.
  • transform - Required keyword-only parameter accepting a callable (function, lambda, or method) that takes a single value argument and returns the transformed result. The callable is applied to both default values and any values passed during instance creation.

🔧 Attributes

Transformer instances have these internal attributes:

  • default_value - The default value to be transformed
  • transform - The transformation callable
  • Force - Inner class used to bypass transformation

⚡ Methods

  • evaluate(instance, step, extra) - Evaluation method that applies the transformation
    • instance - The object being built with current attribute values
    • step - The factory's build step context
    • extra - Additional context including potential value overrides
    • Returns: The transformed value unless wrapped in Transformer.Force()

↩️ Return Value

Returns the result of applying the transform callable to the input value. Return type depends on what the transformation function produces. If the value is wrapped in Transformer.Force(), returns the unwrapped value without transformation.

SelfAttribute - Reference Other Fields

Reference and extract values from other fields in the factory or parent factories.

📖 Full Description

The SelfAttribute declaration creates dynamic references to other fields within the same factory or across parent-child factory hierarchies, enabling powerful data relationships and context-aware field generation. It uses dot notation for attribute traversal and special double-dot notation for parent factory navigation.

Key features of SelfAttribute:

  • Dot Navigation: Use '.' to access nested attributes through relationships
  • Parent Traversal: Use '..' to reference fields in parent factory contexts
  • Dynamic Resolution: Evaluates references at object creation time, ensuring current values
  • Cross-Factory References: Access fields from SubFactory relationships seamlessly
  • Deep Nesting: Chain multiple dots to traverse complex object hierarchies
  • Lazy Evaluation: Values resolved only when needed, respecting dependency order

SelfAttribute is particularly powerful when combined with SubFactory, allowing child factories to reference parent factory attributes, creating cohesive and contextually related data sets without manual parameter passing.

Syntax

import factory

class UserFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = User
    
    first_name = factory.Faker('first_name')
    last_name = factory.Faker('last_name')
    
    # Reference another field in same factory
    email_username = factory.SelfAttribute('first_name')
    email = factory.LazyAttribute(
        lambda obj: f"{obj.email_username}@company.com".lower()
    )

class ProfileFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = Profile
    
    user = factory.SubFactory(UserFactory, first_name='John')
    
    # Reference subfactory field with dot notation
    display_name = factory.SelfAttribute('user.first_name')
    profile_url = factory.LazyAttribute(
        lambda obj: f"/users/{obj.user.first_name}".lower()
    )

class CompanyFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = Company
    
    country_code = 'US'
    timezone = 'America/New_York'
    
    owner = factory.SubFactory(
        UserFactory,
        # Use .. to reference parent factory fields
        country = factory.SelfAttribute('..country_code'),
        timezone = factory.SelfAttribute('..timezone')
    )
    
    # Multiple levels of parent traversal
    manager = factory.SubFactory(
        ProfileFactory,
        user__country = factory.SelfAttribute('...country_code')  # Three dots for grandparent
    )

# Usage examples
user = UserFactory(first_name='Jane')
# email_username = 'Jane', email = 'jane@company.com'

profile = ProfileFactory()
# display_name = 'John', profile_url = '/users/john'

company = CompanyFactory(country_code='UK')
# owner.country = 'UK' (referenced from parent)

📋 Parameters Description

  • dotted_path - Required string argument specifying the attribute path using dot notation. Single dots ('.') traverse into attributes of the current object or sub-factories. Each leading pair of dots ('..') moves up one level to the parent factory context. For example: 'field_name' (current factory), 'subfactory.field' (subfactory's field), '..parent_field' (parent factory), '...grandparent_field' (grandparent factory).

🔧 Attributes

SelfAttribute instances have these internal attributes:

  • attribute_name - The dotted path string to resolve
  • depth - Number of parent levels to traverse (count of leading '..')

⚡ Methods

  • evaluate(instance, step, extra) - Evaluation method that resolves the attribute path
    • instance - The object being built with current attribute values
    • step - The factory's build step context providing parent access
    • extra - Additional context or overrides
    • Returns: The value found at the specified dotted path

↩️ Return Value

Returns the value found at the specified dotted path in the object being built or its parent factory context. Return type depends on the referenced field. Raises AttributeError if the path cannot be resolved.

🔧 Factory Extension Points

Factory Extension Points provide powerful customization hooks. These class methods allow you to control the factory's behavior at different stages of object creation, enabling advanced customization beyond simple declarations.

_adjust_kwargs - Modify Creation Arguments

Hook to modify or adjust arguments before object instantiation.

📖 Full Description

The _adjust_kwargs class method is a powerful extension hook that is called after all factory declarations have been evaluated but immediately before the object is instantiated. It receives the complete dictionary of keyword arguments that will be passed to the model constructor and provides an opportunity to inspect, modify, add, or remove arguments. This method is particularly useful for implementing cross-field validation, applying business rules, normalizing data formats, computing derived values, or enforcing constraints that span multiple fields. The method is called at the perfect time in the factory lifecycle - after all Faker values are generated, after all SubFactory relationships are resolved, but before the actual model instance is created. This makes it ideal for final adjustments that depend on the complete set of field values. Common use cases include ensuring numeric ranges are valid, normalizing text fields, setting default values based on other fields, generating computed identifiers, and enforcing business logic constraints.

Syntax

import factory
from jobpost.models import ModelName

class ModelFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = ModelName
    
    # Factory field declarations
    field_name = factory.Faker('provider')
    
    @classmethod
    def _adjust_kwargs(cls, **kwargs):
        """
        Modify keyword arguments before object instantiation.
        
        Args:
            **kwargs: Dictionary of all evaluated field values
            
        Returns:
            dict: Modified keyword arguments dictionary
        """
        # Perform modifications
        if some_condition:
            kwargs['field_name'] = modified_value
        
        return kwargs

📋 Parameters Description

**kwargs (dict) - A dictionary containing all keyword arguments that will be used to instantiate the model object. This dictionary includes all values from factory declarations after they have been fully evaluated. The dictionary keys are field names (strings) and values are the resolved field values. Any SubFactory relationships will already be resolved to actual model instances. Faker values will already be generated. The kwargs dictionary can be freely modified - you can change existing values, add new keys, or remove keys entirely. The modified dictionary is what will ultimately be passed to the model constructor.

🏷️ Attributes

cls - Reference to the factory class itself (classmethod decorator). This allows access to factory-level metadata and configuration.

⚙️ Methods

This is a class method that should be decorated with @classmethod. It does not call other methods but is itself called by FactoryBoy during the object creation process.

↩️ Return Value

Returns a dictionary of keyword arguments (potentially modified) that will be passed to the model's __init__ method. The returned dictionary must contain valid field names and appropriate values for the model being instantiated. If you don't want to make any modifications, simply return the kwargs dictionary unchanged. FactoryBoy will use the returned dictionary as the final set of arguments for creating the model instance.

import factory
import re
from jobpost.models import Location

class LocationFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = Location
    
    street_address = factory.Faker('street_address')
    city = factory.Faker('city')
    state = factory.Faker('state_abbr')
    postal_code = factory.Faker('zipcode')
    country = 'USA'
    
    @classmethod
    def _adjust_kwargs(cls, **kwargs):
        """
        Format postal code based on country standards.
        - USA: Force 5-digit or 5+4 format (XXXXX or XXXXX-XXXX)
        - Canada: Force format A1A 1A1
        - Other: No formatting
        """
        country = kwargs.get('country', 'USA')
        postal = str(kwargs.get('postal_code', ''))
        
        if country == 'USA':
            # Remove all non-digits
            digits = re.sub(r'\D', '', postal)
            if len(digits) >= 9:
                # Format as XXXXX-XXXX
                kwargs['postal_code'] = f"{digits[:5]}-{digits[5:9]}"
            elif len(digits) >= 5:
                # Format as XXXXX
                kwargs['postal_code'] = digits[:5]
            else:
                # Pad with zeros if needed
                kwargs['postal_code'] = digits.zfill(5)
                
        elif country == 'Canada':
            # Remove spaces and format as A1A 1A1

_build & _create - Object Instantiation Hooks

Hooks to customize how objects are instantiated and saved.

📖 Full Description

The _build and _create class methods control how objects are instantiated. _build creates objects without saving them to the database (BUILD strategy), while _create creates and saves them (CREATE strategy). Override these methods to customize instantiation logic, such as using custom managers or handling special creation requirements.

Syntax

import factory
from anihpj.models import User

class UserFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = User
    
    username = factory.Sequence(lambda n: f'user_{n}')
    email = factory.Faker('email')
    
    @classmethod
    def _build(cls, model_class, *args, **kwargs):
        """Custom build logic - just instantiate without saving"""
        return model_class(*args, **kwargs)
    
    @classmethod
    def _create(cls, model_class, *args, **kwargs):
        """Custom create logic - use create_user manager method"""
        # Extract password if present
        password = kwargs.pop('password', 'defaultpass123')
        
        # Use Django's create_user method for proper password hashing
        manager = cls._get_manager(model_class)
        if hasattr(manager, 'create_user'):
            return manager.create_user(*args, password=password, **kwargs)
        else:
            return manager.create(*args, **kwargs)

# Usage
built_user = UserFactory.build()  # Uses _build (not saved)
created_user = UserFactory.create()  # Uses _create (saved with hashed password)

📋 Parameters

  • model_class - The model class being instantiated (from Meta.model)
  • *args - Positional arguments for instantiation (if Meta.inline_args is defined)
  • **kwargs - Keyword arguments for instantiation (after _adjust_kwargs processing)

↩️ Return Value

Returns an instance of the model class (unsaved for _build, saved for _create).

_after_postgeneration - Final Cleanup Hook

Hook called after all post-generation hooks have executed.

📖 Full Description

The _after_postgeneration class method is called as the final step of object creation, after all PostGeneration hooks have been executed. This is useful for final validation, saving modified objects, or cleaning up temporary state. It receives the created object and information about the creation strategy.

Syntax

🎯 Strategies & StubObject

FactoryBoy supports three different creation strategies. These strategies control how objects are instantiated and whether they interact with the database, providing flexibility for different testing scenarios.

BUILD_STRATEGY - Create Without Saving

Instantiate objects without saving them to the database.

📖 Full Description

The BUILD strategy creates model instances by calling the model's __init__ method but does NOT save them to the database. This is useful for unit tests that don't require database interaction or when you need to inspect objects before saving. Uses the factory's _build class method.

Syntax

import factory
from anihpj.models import User

class UserFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = User
    
    username = factory.Sequence(lambda n: f'user_{n}')
    email = factory.Faker('email')

# Usage
user = UserFactory.build()  # Explicit BUILD strategy
print(user.pk)  # Output: None (not saved to database)
print(user.username)  # Output: user_1 (object exists in memory)

# Can modify before saving
user.is_staff = True
user.save()
print(user.pk)  # Output: 1 (now has primary key)

💡 Use Case: BUILD strategy is ideal for fast unit tests that don't need database persistence, testing validation logic before save, or when you need to customize objects before committing to the database.

CREATE_STRATEGY - Create and Save

Instantiate objects and save them to the database (default strategy).

📖 Full Description

The CREATE strategy is the default strategy for DjangoModelFactory. It creates model instances and immediately saves them to the database by calling the model manager's create method (or using the _create class method). This is essential for integration tests and when testing relationships that require database constraints.

Syntax

import factory
from jobpost.models import JobPost, Author

class JobPostFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = JobPost
        strategy = factory.CREATE_STRATEGY  # Default, can be omitted
    
    title = factory.Faker('job')
    description = factory.Faker('text')
    author = factory.SubFactory('jobpost.factories.AuthorFactory')
    location = factory.SubFactory('jobpost.factories.LocationFactory')

# Usage
job = JobPostFactory.create()  # Explicit CREATE
print(job.pk)  # Output: 1 (saved with primary key)
print(JobPost.objects.count())  # Output: 1

# Can also use shorthand
job2 = JobPostFactory()  # Uses default CREATE_STRATEGY
print(JobPost.objects.count())  # Output: 2

⚠️ Database Impact: CREATE strategy persists objects to the database, which may be slower for large test suites. Use BUILD or STUB strategies when database persistence isn't required.

STUB_STRATEGY - Create Stub Objects

Create lightweight stub objects without database interaction.

📖 Full Description

The STUB strategy creates StubObject instances instead of actual model instances. StubObjects are lightweight containers that have the same attributes as the model but don't interact with the database at all. They don't have a primary key and can't be saved. This is perfect for extremely fast tests that only need object attributes without any model behavior or database constraints.

Syntax

import factory

  • Has all the attributes defined in the factory
  • Does NOT have model methods (save, delete, etc.)
  • Does NOT have a primary key or database ID
  • Cannot be used with SubFactory relationships that require database operations
  • Extremely lightweight and fast to create

💡 Perfect For: Algorithm testing that only uses object attributes, serializer tests, presentation layer tests, or any scenario where you need blazing-fast test execution without database overhead.

class UserFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = User
    
    username = factory.Sequence(lambda n: f'user_{n}')
    email = factory.Faker('email')
    
    @factory.use_strategy(factory.BUILD_STRATEGY)
    @staticmethod
    def password_build(obj, create, extracted, **kwargs):
        """For BUILD: use simple plaintext password"""
        return 'password123'
    
    @factory.use_strategy(factory.CREATE_STRATEGY)
    @staticmethod
    def password_create(obj, create, extracted, **kwargs):
        """For CREATE: use properly hashed password"""
        obj.set_password('password123')
        obj.save()
    
    password = factory.PostGeneration(password_build)

# Usage
built_user = UserFactory.build()
print(built_user.password)  # Output: password123 (plaintext)

created_user = UserFactory.create()
print(created_user.check_password('password123'))  # Output: True (hashed)

FactoryBoy provides convenient module-level functions for common operations. These functions offer shortcuts for building and creating objects without defining full factory classes.

make_factory - Dynamic Factory Creation

Create factory classes programmatically at runtime.

📖 Full Description

The make_factory function allows you to create factory classes dynamically without explicitly defining a class. Useful for quick tests or when factory structure needs to be determined at runtime.

Syntax

from factory import make_factory, Faker, Sequence
from jobpost.models import Skills

# Create a factory dynamically
SkillsFactory = make_factory(
    Skills,
    skills=Faker('job')
)

# Use it like a normal factory
skill = SkillsFactory.create()
print(skill.skills)

build() and create() - Standalone Functions

Create instances using inline factory definitions.

📖 Full Description

FactoryBoy provides standalone build() and create() functions that can be used without defining a factory class. These are useful for one-off instance creation in tests.

Syntax

from factory import build, create, Faker
from jobpost.models import Skills

# Build without saving
skill = build(Skills, skills=Faker('job').generate({}))

# Create and save
skill = create(Skills, skills='Python Development')
