    1.2  Kubernetes Defined
         - Official definition simplified: "An open-source system for automating
           deployment, scaling, and management of containerized applications"
         - K8s = Kubernetes (K + 8 letters + s)
         - Born at Google (Borg → Kubernetes), donated to CNCF in 2014
         - Written in Go language
         [CARD: Quick facts — Release year, creator, language, license]

    1.3  Key Benefits of Kubernetes
         - Automatic scaling (more users → more pods, fewer users → fewer pods)
         - Self-healing (pod crashes → Kubernetes replaces it automatically)
         - Load balancing (traffic distributed evenly across all healthy pods)
         - Automated rollouts and rollbacks (deploy new versions safely)
         - Secret and configuration management (separate config from code)
         - Service discovery and DNS (pods find each other by name)
         - Storage orchestration (attach storage systems automatically)
         - Batch execution (run one-off/scheduled tasks like Django migrations)

    3.2  Control Plane Components (The Brain)
         a) kube-apiserver
            - The front door to Kubernetes (all communication goes through it)
            - REST API — every kubectl command hits this
            - Authentication, authorization, admission control
            [DIAGRAM: API Server as the central hub]

         b) etcd
            - Distributed key-value store (the cluster's database)
            - Stores ALL cluster state: what pods exist, their configs, secrets
            - Raft consensus algorithm for consistency
            - Why backing up etcd is critical
            [ILLUSTRATION: etcd as a filing cabinet]

         c) kube-scheduler
            - Watches for unscheduled pods
            - Decides WHICH node to place a pod on
            - Considers: resource requests, affinity, taints/tolerations
            - Example: "anihpj pod needs 256Mi RAM → Node 2 has space → assign"
            [FLOWCHART: Scheduler decision process]

         d) kube-controller-manager
            - Runs controller loops (watch → compare → act)
            - Deployment Controller: ensures desired replica count
            - Node Controller: monitors node health
            - Job Controller: manages job completion
            - Endpoint Controller: populates endpoint objects
            [DIAGRAM: Controller loop — Desired state vs Actual state]

         e) cloud-controller-manager
            - Integrates with cloud provider APIs (AWS, GCP, Azure)
            - Manages: LoadBalancers, Routes, Node lifecycle
            - Not present in local clusters (Minikube, kind)

    3.3  Worker Node Components (The Muscles)
         a) kubelet
            - Agent running on every node
            - Receives pod specs from API server
            - Ensures containers are running and healthy
            - Reports node status back to control plane
            [DIAGRAM: kubelet managing containers on a node]

         b) kube-proxy
            - Network proxy on every node
            - Maintains network rules (iptables/IPVS)
            - Enables Service abstraction (ClusterIP, NodePort)
            - Routes traffic to the correct pod
            [DIAGRAM: kube-proxy routing traffic]

         c) Container Runtime
            - Software that actually runs containers
            - containerd (most common, default in modern K8s)
            - CRI-O (lightweight alternative)
            - Docker Engine (deprecated as runtime since K8s 1.24)
            [TABLE: Runtime comparison]

    3.4  How a Pod Gets Created — The Full Journey
         [NUMBERED FLOWCHART with 8 steps]
         1. You run: kubectl apply -f anihpj-deployment.yaml
         2. kubectl sends request to kube-apiserver
         3. API server validates, authenticates, stores in etcd
         4. Deployment controller sees new deployment, creates ReplicaSet
         5. ReplicaSet controller creates Pod object
         6. Scheduler assigns Pod to a node
         7. kubelet on that node pulls the anihpj image
         8. Container runtime starts the container
         → User can now access anihpj at the Service endpoint

    4.4  Deployments — Managing Application Updates
         - The most common workload resource

         - Deployment → ReplicaSet → Pod (the hierarchy)
         - Strategies:

           • Recreate: kill all, then create new
         - Rollbacks: undo to any previous revision
         - Scaling: change replica count on the fly

--------------------------------------------------------------------------------
    7.1  Why YAML? (And Not JSON or XML?)
         - Human-readable and writable
         - Less verbose than JSON (no braces/quotes everywhere)
         - Kubernetes native (API accepts both YAML and JSON)
         - Easy to version control with Git
         [COMPARISON: Same config in JSON vs YAML vs XML — line count]

         [SIDE-BY-SIDE FORMAT COMPARISON — 3 code blocks]
         YAML (5 lines)       |  JSON (12 lines)        |  XML (15 lines)
         ──────────────────  |  ────────────────────  |  ───────────────────
         name: anihpj         |  {                      |  <pod>
         replicas: 3          |    "name": "anihpj",    |    <name>anihpj</name>
         image: anihpj:v1     |    "replicas": 3,       |    <replicas>3</replicas>
         port: 8000           |    "image": "anihpj:v1",|    <image>anihpj:v1</image>
                              |    "port": 8000         |    <port>8000</port>
                              |  }                      |  </pod>
         ✅ Clean, readable     |  🟡 More brackets       |  ❌ Very verbose

         [WINNER BADGE: YAML ⭐ — "60% fewer characters, 100% more readable"]

              - containerPort: 8000

         e) Comments (# prefix)

            enabled: true

Chapter 10: Secrets — Managing Sensitive Data

    10.1  What are Secrets?
          - Like ConfigMaps but for sensitive data
          - Base64 encoded (NOT encrypted by default!)
          - Stored in etcd (can be encrypted at rest)
          - Access controlled via RBAC
          [WARNING BOX: Base64 != encryption. Anyone can decode it.]

          [DANGER CARD — visual warning with icon]
          ┌─────────────────────────────────────────────────────┐
          │ ⚠️  COMMON MISCONCEPTION                            │
          │                                                     │
          │  Base64 is NOT encryption!                           │
          │                                                     │
          │  echo 'c3VwZXJzZWNyZXQ=' | base64 -d                │
          │  → "supersecret"   😱 Anyone can decode this!        │
          │                                                     │
          │  Without extra security:                             │
          │  Base64 ≠ 🔐  |  Base64 = 📦 (just packaging)          │
          │                                                     │
          │  → Enable encryption at rest for real security       │
          └─────────────────────────────────────────────────────┘

          [SECRET vs CONFIGMAP COMPARISON CARD]
          | Feature        | ConfigMap         | Secret                |

          | Encoding       | Plain text         | Base64                |

          | Mount as       | Env var / Volume   | Env var / Volume      |

    11.2  Volume Types Overview
          [CATEGORIZED TABLE]
          Temporary:
          - emptyDir: scratch space, shared between containers in a pod
          - hostPath: mounts from the node's filesystem (dev only!)

          Network/Cloud:
          - nfs: Network File System mount
          - awsElasticBlockStore, gcePersistentDisk, azureDisk
          - csi: Container Storage Interface (universal)

          - PersistentVolume (PV): cluster-wide storage resource
          - PersistentVolumeClaim (PVC): request for storage by a pod

          Configuration:
          - configMap: mounts ConfigMap as files

          - projected: combines multiple sources

Chapter 13: Services Deep Dive
--------------------------------------------------------------------------------
    13.1  Why Services Exist — The Problem
          - Pods are mortal: they get created, die, are replaced
          - Pod IPs change on every restart
          - You can't point your frontend to a pod IP
          - Services = stable address + load balancing
          [PROBLEM DIAGRAM: Pod A calls Pod B → B dies, new IP → A can't connect
           SOLUTION: Pod A calls Service B → routes to any healthy B pod]

          └───────┘             └──┬────┘   └───────┘         └───┬───┘

    16.1  Why Health Checks Matter
          [SCENARIO CARD]
          Without probes: Django app connects to wrong DB hostname → app runs but
          returns 500 errors → Kubernetes thinks it's healthy → traffic keeps routing

          With probes: K8s detects failure → removes from service → restarts pod

    16.8  Common Probe Mistakes
          [WARNING BOX]
          ❌ Making liveness probe depend on external service (DB, API)
             → External failure causes unnecessary pod restarts
          ❌ Setting timeoutSeconds too low for readiness

Chapter 17: Resource Management
--------------------------------------------------------------------------------
    17.1  Why Resource Management Matters
          - Without limits: one pod can consume all node resources
          - Without requests: scheduler can't make good placement decisions
          - anihpj scenario: memory leak fills the node → ALL pods crash
          [DIAGRAM: One "hungry" pod starving others on the same node]

    17.2  Resource Requests and Limits
          a) Requests: minimum guaranteed resources
             - Used by scheduler for pod placement
             - "I need at least 256Mi RAM and 250m CPU"
          b) Limits: maximum allowed resources
             - Enforced by kubelet
             - "Never use more than 512Mi RAM and 500m CPU"
          c) CPU Units
             - 1 CPU = 1000 millicores (m)
             - 100m = 0.1 CPU = 10% of a core
             - When limit exceeded: throttled (not killed)
          d) Memory Units
             - Mi (mebibytes), Gi (gibibytes)
             - When limit exceeded: OOMKilled (pod is killed!)
          [DIAGRAM: Request vs Limit — guaranteed zone vs burst zone]
          [CODE: Resource specs in Deployment YAML]

Chapter 21: StatefulSets — Stateful Applications
--------------------------------------------------------------------------------
    21.1  Stateless vs Stateful Applications
          - Stateless: any pod is interchangeable (anihpj Django app)
          - Stateful: each pod has unique identity/state (PostgreSQL, Redis)
          [COMPARISON TABLE]
          | Aspect        | Stateless (Deployment) | Stateful (StatefulSet)    |
          | Pod identity  | Random name            | Ordered name (app-0,1,2) |
          | Storage       | Shared or none         | Each pod gets own PVC     |
          | Network       | Random IP              | Stable DNS per pod        |
          | Scaling       | Any order              | Ordered (0→1→2)           |
          | Deletion      | Any order              | Reverse order (2→1→0)     |

    21.2  StatefulSet Features
         a) Stable Pod Names
            - Pods named: <statefulset>-0, <statefulset>-1, <statefulset>-2
            - Names survive across restarts
         b) Stable Network Identity
            - Each pod gets a DNS: <pod>.<headless-svc>.<ns>.svc.cluster.local
            - Example: postgres-0.postgres-svc.default.svc.cluster.local
         c) Stable Storage
            - volumeClaimTemplates: each pod gets its own PVC
            - PVCs persist even if pod is deleted
         d) Ordered Operations
            - Created sequentially: 0 must be Ready before 1 starts
            - Deleted in reverse: 2 first, then 1, then 0
         [DIAGRAM: 3 StatefulSet pods with individual PVCs and DNS]

         [STATEFULSET ARCHITECTURE ILLUSTRATION]

         │  StatefulSet: postgres   (Headless Service)           │
         │                                                       │

         │  │  postgres-0   │  │  postgres-1   │  │  postgres-2   │ │

    24.1  Security Contexts — Container-Level Security
          a) runAsUser / runAsGroup: which OS user runs the process
             - Don't run as root (UID 0)!
             - Example: run Django as user 1000
          b) runAsNonRoot: true — prevent root execution
          c) readOnlyRootFilesystem: true — container can't write to its filesystem
             - Use emptyDir for temp files
          d) capabilities: drop ALL, add only what's needed
             - Drop: NET_RAW, MKNOD, SYS_ADMIN
             - Rarely need to add any
          e) allowPrivilegeEscalation: false — prevent setuid

    25.1  Built-in Secrets Limitations
          - Stored as base64 in etcd (easily decoded)

          - No audit trail of secret access by default
          - No automatic rotation
          [WARNING BOX: K8s Secrets are NOT encrypted by default]

          - AES-CBC or AES-GCM encryption providers

             - Dynamic secrets, rotation, audit logs

          b) AWS Secrets Manager + External Secrets Operator

    25.4  Secret Rotation
          - Why: compromised secrets, compliance requirements
          - How: update Secret → rolling restart pods
          - Automation: External Secrets Operator auto-sync

    25.5  Best Practices Summary
          [COMPREHENSIVE CHECKLIST]
          ✅ Enable encryption at rest
          ✅ Use RBAC to restrict secret access
          ✅ Use external secret management in production

    28.4  Scenario-Based Debugging for anihpj
          [SCENARIO CARDS — each with Problem → Investigate → Solution]

          Scenario 1: CrashLoopBackOff
          - Problem: anihpj pod keeps restarting
          - Investigate: kubectl logs anihpj-xxx → "ModuleNotFoundError: No module named 'gunicorn'"
          - Solution: Add gunicorn to requirements.txt, rebuild image

          Scenario 2: ImagePullBackOff
          - Problem: pod stuck pulling image
          - Investigate: kubectl describe pod → "repository not found"
          - Solution: Fix image name or add imagePullSecret

          Scenario 3: Pending Pod
          - Problem: pod stays in Pending state
          - Investigate: kubectl describe pod → "Insufficient memory"
          - Solution: Reduce resource requests or add more nodes

          Scenario 4: Service Not Reachable
          - Problem: curl to service returns "connection refused"
          - Investigate: kubectl describe svc → no endpoints
          - Solution: Fix Service selector to match pod labels

          Scenario 5: Database Connection Error
          - Problem: "could not connect to server" in logs
          - Investigate: Check Secret values, check postgres pod status
          - Solution: Fix DB_HOST to postgres service name

Chapter 29: Production Checklist
--------------------------------------------------------------------------------
    29.1  Pre-Production Readiness Assessment
          [COMPREHENSIVE CHECKLIST — grouped by category]

          a) Application Readiness
             □ Health check endpoints implemented (/healthz, /readyz)
             □ Graceful shutdown handling (SIGTERM signal)
             □ 12-Factor App principles followed
             □ Stateless design (session storage externalized)
             □ Database migrations strategy defined

          b) Container Readiness
             □ Minimal base image (distroless / alpine / slim)
             □ Non-root user in Dockerfile
             □ .dockerignore file optimized
             □ Multi-stage build for smaller images
             □ Image vulnerability scan passing (Trivy)
             □ Specific image tags (no :latest)

          c) Kubernetes Readiness
             □ Resource requests AND limits set for every container
             □ Liveness, readiness, startup probes configured
             □ Pod Disruption Budgets defined
             □ Pod anti-affinity for spread across nodes
             □ Security contexts applied (non-root, read-only root fs)
             □ Network Policies isolating pods
             □ RBAC roles and bindings configured
             □ Secrets managed externally (Vault, AWS SM)
             □ ConfigMaps for environment-specific config
             □ Horizontal Pod Autoscaler configured

          d) Operational Readiness
             □ Logging centralized (EFK / Loki)
             □ Monitoring and alerting set up (Prometheus / Grafana)
             □ Backup strategy for persistent data
             □ CI/CD pipeline configured and tested
             □ Rollback procedure documented and tested
             □ On-call runbook created

    29.2  High Availability (HA) Setup
          - Multiple replicas for every component (min 3)
          - Pod anti-affinity: spread across nodes
          - Multiple nodes across availability zones

Appendix C: Common Errors and Solutions
--------------------------------------------------------------------------------
    [TROUBLESHOOTING GUIDE — 20+ common errors]

    C.1   Pod Errors
          | Error                    | Likely Cause                | Fix                              |
          | ImagePullBackOff         | Wrong image name or registry auth | Check image name, create imagePullSecret |
          | CrashLoopBackOff         | App crashes on start        | Check logs: kubectl logs <pod> --previous |
          | OOMKilled                | Container exceeded memory limit | Increase memory limit or optimize app |
          | CreateContainerConfigError | Missing ConfigMap or Secret | Check CM/Secret exists in namespace |
          | RunContainerError        | Missing volume or bad command | Check volumeMounts and container command |
          | Pending                  | No node has enough resources | Check node resources, add nodes, adjust requests |
          | Evicted                  | Node under resource pressure | Right-size requests, add nodes |
          | Init:Error               | Init container failed       | Check init container logs |

    C.2   Service / Networking Errors
          | Symptom                  | Likely Cause                | Fix                              |
          | Cannot reach service     | Selector doesn't match pod labels | Verify labels match selector |
          | 503 from Ingress         | Backend pods not ready      | Check readiness probes           |
          | DNS resolution fails     | CoreDNS issue               | Check CoreDNS pods, restart if needed |
          | Connection refused       | Wrong port/targetPort       | Verify ports match               |

    C.3   Deployment Errors
          | Symptom                  | Likely Cause                | Fix                              |
          | Rollout stuck            | New pods failing probes     | Check events, fix probes or app  |
          | Insufficient quota       | ResourceQuota exceeded      | Increase quota or reduce requests |
          | Forbidden                | RBAC permissions missing    | Add missing Role/ClusterRole     |

    C.4   Django-Specific Errors in Kubernetes
          | Error                    | Cause                       | Fix                              |
          | DisallowedHost           | ALLOWED_HOSTS missing K8s hostname | Add '*' or service name to ALLOWED_HOSTS |
          | OperationalError: DB     | Database not reachable      | Check DB service/endpoint, network policy |
          | Static files 404         | STATIC_ROOT not set, no collectstatic | Add collectstatic to init container |
          | Migration errors         | Migrations not run          | Add init container: python manage.py migrate |

    C.5   Debugging Decision Flowchart
          [FLOWCHART]
          Pod not starting?
          ├─ Status: Pending → Check node resources / scheduler
          ├─ Status: ImagePullBackOff → Check image name / registry auth
          ├─ Status: CrashLoopBackOff → Check logs (--previous flag)
          ├─ Status: Running but not working → Check probes / service selector
          └─ Status: Evicted → Node pressure → right-size or add nodes

          [VISUAL DEBUGGING DECISION TREE]
          ┌───────────────────────────────────────────────────┐
          │  QUICK DEBUG CHEAT SHEET                            │
          │                                                     │
          │  1️⃣  kubectl get pods          → What's the status? │
          │  2️⃣  kubectl describe pod X    → What events fired?  │
          │  3️⃣  kubectl logs X             → What does app say? │
          │  4️⃣  kubectl logs X --previous  → What crashed it?   │
          │  5️⃣  kubectl exec -it X -- sh   → Look inside!       │
          │  6️⃣  kubectl get events         → Cluster timeline   │
          │                                                     │
          │  99% of problems solved by steps 1-4 ✅              │
          └───────────────────────────────────────────────────┘
