Official Docs: ReplicationController v1 API | API Version: v1

ReplicationController represents the configuration of a replication controller that ensures a specified number of pod replicas are running at any given time.

Full Description

ReplicationController is one of the oldest workload resources in Kubernetes, designed to ensure that a specified number of identical pods are running at all times. When deploying your anihpj Django application, a ReplicationController will monitor your Django pods and automatically create new ones if any fail or are deleted, ensuring high availability for your jobpost application.

While ReplicationController has largely been superseded by Deployment (which provides additional features like rolling updates), it's still fundamental to understand as it forms the basis for more advanced workload resources. In production Django deployments, you'll typically use Deployments, but ReplicationController is perfect for understanding basic pod replication concepts.

Syntax

# Create a ReplicationController from YAML file
kubectl create -f replicationcontroller.yaml
# List all ReplicationControllers (long form)
kubectl get replicationcontrollers
# List all ReplicationControllers (short form)
kubectl get rc
# Get detailed information about a specific ReplicationController
kubectl describe replicationcontroller {name}
# Get detailed information about a specific ReplicationController (short form)
kubectl describe rc {name}
# Scale ReplicationController to specified number of replicas
kubectl scale replicationcontroller {name} --replicas={count}
# Scale ReplicationController to specified number of replicas (short form)
kubectl scale rc {name} --replicas={count}
# Delete a specific ReplicationController
kubectl delete replicationcontroller {name}
# Delete a specific ReplicationController (short form)
kubectl delete rc {name}

YAML Structure

apiVersion: v1
kind: ReplicationController
metadata:
  name: anihpj-django-rc
  namespace: default
  labels:
    app: anihpj
    component: web
spec:
  replicas: 3
  selector:
    app: anihpj
    component: web
  template:
    metadata:
      labels:
        app: anihpj
        component: web
    spec:
      containers:
      - name: django-web
        image: anihpj/jobpost-app:v1.0.0
        ports:
        - containerPort: 8000
        env:
        - name: DJANGO_SETTINGS_MODULE
          value: "anihpj.settings"
        resources:
          requests:
            memory: "256Mi"
            cpu: "250m"
          limits:
            memory: "512Mi"
            cpu: "500m"

Parameters Description

  • apiVersion: Always "v1" for ReplicationController. This indicates the API version being used.
  • kind: Must be "ReplicationController" to specify the resource type.
  • metadata.name: Unique name for the ReplicationController within the namespace. Use descriptive names like "anihpj-django-rc".
  • metadata.namespace: Kubernetes namespace where the ReplicationController will be created. Defaults to "default".
  • metadata.labels: Key-value pairs for organizing and selecting ReplicationControllers. Useful for grouping Django app components.
  • spec.replicas: Desired number of pod replicas. For Django apps, start with 2-3 for high availability.
  • spec.selector: Label selector to identify which pods belong to this ReplicationController. Must match template labels.
  • spec.template: Pod template specification that defines how new pods should be created.
  • spec.template.metadata.labels: Labels applied to created pods. Must include selector labels.
  • spec.template.spec: Pod specification including containers, volumes, and other pod-level configurations.
  • spec.minReadySeconds: Minimum seconds a pod must be ready before considered available. Useful for Django apps with startup time.

Return Value

Returns the ReplicationController object with its current state, including status information about replicas, ready replicas, and conditions indicating the controller's health.

  • 200 OK: ReplicationController successfully created/updated
  • 201 Created: ReplicationController successfully created
  • 400 Bad Request: Invalid ReplicationController specification
  • 401 Unauthorized: Authentication required
  • 403 Forbidden: Insufficient permissions
  • 404 Not Found: Namespace not found
  • 409 Conflict: ReplicationController already exists
  • 422 Unprocessable Entity: Invalid field values

ReplicationControllerSpec

ReplicationControllerSpec defines the desired behavior of the replication controller. For Django applications, carefully consider the replica count based on expected traffic, resource requirements, and high availability needs.

Syntax
spec:
  replicas: 3
  selector:
    app: anihpj
    component: web
  template:
    metadata:
      labels:
        app: anihpj
        component: web
    spec:
      containers:
      - name: django-app
        image: anihpj/jobpost-app:latest
        ports:
        - containerPort: 8000
  minReadySeconds: 30
Parameters Description
  • selector (map[string]string): Label query over pods that should match the replicas count. If empty, defaults to labels on Pod template. For anihpj Django apps, use labels like app=anihpj, component=web.
  • template (PodTemplateSpec): Defines the pod that will be created if insufficient replicas are detected. Must include restartPolicy: Always for ReplicationController.
  • replicas (int32): Number of desired replicas. For Django production, typically 2-5 replicas for web apps, 1-3 for workers.
  • minReadySeconds (int32): Minimum seconds for a pod to be ready before considered available. Important for Django apps with database connections.

ReplicationControllerStatus

ReplicationControllerStatus represents the current status of a replication controller, providing visibility into the health and state of your Django application deployment.

Syntax
status:
  replicas: 3
  readyReplicas: 3
  availableReplicas: 3
  fullyLabeledReplicas: 3
  observedGeneration: 1
  conditions:
  - type: ReplicaFailure
    status: "False"
    lastTransitionTime: "2025-09-08T10:30:00Z"
    reason: PodsRunning
    message: "All pods are running successfully"
Parameters Description
  • replicas (int32, required): Most recently observed number of replicas. Monitor this to ensure your Django app maintains desired capacity.
  • availableReplicas (int32): Number of available replicas (ready for at least minReadySeconds). Critical for Django app availability.
  • readyReplicas (int32): Number of ready replicas. Should match desired replicas for healthy Django deployment.
  • fullyLabeledReplicas (int32): Pods with labels matching the pod template labels.
  • conditions ([]ReplicationControllerCondition): Latest observations of the controller's current state.
  • observedGeneration (int64): Generation of the most recently observed replication controller.

ReplicationControllerCondition

ReplicationControllerCondition describes the state of a replication controller at a certain point in time. This provides detailed insights into why your Django application might be experiencing issues.

Syntax
conditions:
- type: "ReplicaFailure"
  status: "True"
  lastTransitionTime: "2025-09-08T10:30:00Z"
  reason: "FailedCreate"
  message: "Error creating: pods \"anihpj-skills-rc-\" is forbidden"
- type: "Progressing"
  status: "True"
  lastTransitionTime: "2025-09-08T10:25:00Z"
  reason: "NewReplicaSetAvailable"
  message: "ReplicationController has successfully progressed"
Parameters Description
  • type (string, required): Type of replication controller condition. Common types include "ReplicaFailure", "Progressing"
  • status (string, required): Status of the condition, one of "True", "False", "Unknown"
  • lastTransitionTime (Time): The last time the condition transitioned from one status to another
  • reason (string): Machine-readable, UpperCamelCase text indicating the reason for the condition's last transition
  • message (string): Human-readable message indicating details about the transition
Return Value

Conditions are read-only fields populated by the Kubernetes system. They provide status information and are part of the ReplicationController status object.

Operations

Operations Overview

Read - GET single ReplicationController details
List - GET multiple ReplicationControllers
Create - POST new ReplicationController
Update - PUT/PATCH existing ReplicationController
Delete - DELETE ReplicationController resources
Status - Manage ReplicationController status subresource
Patch - Partially update ReplicationController
Watch - Monitor ReplicationController changes

Operations allow you to interact with ReplicationController resources through the Kubernetes API. These operations enable you to manage Django application deployments, scale services, and monitor health status.

Read ReplicationController

HTTP Request

Method - GET
Path - /api/v1/namespaces/{namespace}/replicationcontrollers/{name}
Response - ReplicationController object
Use Case - Retrieve Django app deployment details

Read the specified ReplicationController to get complete configuration and status information for your Django application deployment.

Syntax
kubectl get replicationcontroller {name} -o yaml
kubectl get rc {name} -o json
kubectl describe replicationcontroller {name}
HTTP API Syntax
GET /api/v1/namespaces/{namespace}/replicationcontrollers/{name}
Accept: application/json
Authorization: Bearer {token}
Parameters
  • name (string, required): Name of the ReplicationController
  • namespace (string, required): Object namespace (default: "default")
  • pretty (string): Pretty-print the output
Return Value
  • 200 (ReplicationController): OK - Returns the complete ReplicationController object
  • 401: Unauthorized - Authentication required

Deployment

Deployment Overview

DeploymentSpec - Desired state specification
DeploymentStatus - Current observed status
DeploymentList - Collection of deployments
Operations - API operations for management
Official Docs: Deployment v1 API | API Version: apps/v1

Deployment enables declarative updates for Pods and ReplicaSets, providing advanced deployment strategies for Django applications.

Full Description

Deployment is the recommended way to deploy stateless applications in Kubernetes. It provides declarative updates for Pods and ReplicaSets, enabling rolling updates, rollbacks, and scaling for your anihpj Django application. Unlike ReplicationController, Deployments support advanced deployment strategies and are the modern standard for managing stateless workloads in production environments.

For Django applications, Deployments are ideal for web servers, API services, and background workers that don't require persistent state. They automatically manage ReplicaSets and provide features like rolling updates, which allow you to update your jobpost application without downtime.

Syntax

# Create a Deployment from YAML file
kubectl create -f deployment.yaml
# Create a Deployment directly with kubectl
kubectl create deployment {name} --image={image}
# List all Deployments
kubectl get deployments
# List all Deployments (short form)
kubectl get deploy
# Get detailed information about a specific Deployment
kubectl describe deployment {name}
# Get detailed information about a specific Deployment (short form)
kubectl describe deploy {name}
# Scale Deployment to specified number of replicas
kubectl scale deployment {name} --replicas={count}
# Check the rollout status of a Deployment
kubectl rollout status deployment/{name}
# View the rollout history of a Deployment
kubectl rollout history deployment/{name}
# Rollback to previous version of a Deployment
kubectl rollout undo deployment/{name}
# Rollback to specific revision of a Deployment
kubectl rollout undo deployment/{name} --to-revision={number}
# Pause a deployment rollout
kubectl rollout pause deployment/{name}
# Resume a paused deployment rollout
kubectl rollout resume deployment/{name}
# Delete a specific Deployment
kubectl delete deployment {name}
# Delete a specific Deployment (short form)
kubectl delete deploy {name}

Parameters Description

  • apiVersion: Always "apps/v1" for Deployment. This indicates the API version being used.
  • kind: Must be "Deployment" to specify the resource type.
  • metadata.name: Unique name for the Deployment within the namespace. Use descriptive names like "anihpj-django-deployment".
  • metadata.namespace: Kubernetes namespace where the Deployment will be created. Defaults to "default".
  • metadata.labels: Key-value pairs for organizing and selecting Deployments. Useful for grouping Django app components.
  • spec.replicas: Number of desired pod replicas. For Django apps, start with 2-3 for high availability.
  • spec.selector.matchLabels: Label selector to identify which pods belong to this Deployment. Must match template labels.
  • spec.template: Pod template specification that defines how new pods should be created.
  • spec.strategy: Deployment strategy (RollingUpdate or Recreate). RollingUpdate is recommended for Django apps.
  • spec.strategy.rollingUpdate.maxUnavailable: Maximum number of pods that can be unavailable during update.
  • spec.strategy.rollingUpdate.maxSurge: Maximum number of pods that can be created above desired number during update.
  • spec.minReadySeconds: Minimum seconds a pod must be ready before considered available. Important for Django apps with startup time.
  • spec.revisionHistoryLimit: Number of old ReplicaSets to retain for rollback. Defaults to 10.
  • spec.progressDeadlineSeconds: Maximum time in seconds for deployment to make progress before considered failed.

Return Value

Returns the Deployment object with its current state, including status information about replicas, conditions, and deployment progress.

  • 200 OK: Deployment successfully created/updated
  • 201 Created: Deployment successfully created
  • 400 Bad Request: Invalid Deployment specification
  • 401 Unauthorized: Authentication required
  • 403 Forbidden: Insufficient permissions
  • 404 Not Found: Namespace not found
  • 409 Conflict: Deployment already exists
  • 422 Unprocessable Entity: Invalid field values

    DeploymentSpec

    DeploymentSpec is a description of the deployment specification. It defines the desired state for a Deployment object, including the number of replicas, the pod template, and the update strategy. This specification ensures that the deployment maintains the desired number of pods and can perform rolling updates when changes are needed.

    API Definition

    Group: apps
    Version: v1
    Kind: DeploymentSpec
    Scope: Namespaced

    Syntax

    # Create deployment with custom spec from YAML file
kubectl apply -f deployment-spec.yaml

# Update deployment spec directly with kubectl patch
kubectl patch deployment my-deployment -p '{"spec":{"replicas":5}}'

# Get deployment spec details in YAML format
kubectl get deployment my-deployment -o yaml | grep -A 20 "spec:"

# Scale deployment by modifying spec replicas
kubectl scale deployment my-deployment --replicas=3

# Set deployment spec strategy (RollingUpdate/Recreate)
kubectl patch deployment my-deployment -p '{"spec":{"strategy":{"type":"RollingUpdate"}}}'

    YAML Structure

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: example-deployment
  namespace: default
spec:
  replicas: 3                    # Number of desired pods
  selector:                      # Label selector for pods
    matchLabels:
      app: example-app
  template:                      # Pod template specification
    metadata:
      labels:
        app: example-app
    spec:
      containers:
      - name: app-container
        image: nginx:1.21
        ports:
        - containerPort: 80
  strategy:                      # Update strategy
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1
      maxSurge: 1
  revisionHistoryLimit: 10       # Number of old ReplicaSets to retain
  progressDeadlineSeconds: 600   # Maximum time for deployment to progress
  paused: false                  # Whether deployment is paused

    Parameters

    replicas integer

    Number of desired pods. This is a pointer to distinguish between explicit zero and not specified. Defaults to 1.

    selector LabelSelector

    Label selector for pods. Existing ReplicaSets whose pods are selected by this will be affected by this deployment.

    template PodTemplateSpec

    Template describes the pods that will be created. This is required.

    strategy DeploymentStrategy

    The deployment strategy to use to replace existing pods with new ones.

    revisionHistoryLimit integer

    Number of old ReplicaSets to retain to allow rollback. This is a pointer to distinguish between explicit zero and not specified. Defaults to 10.

    progressDeadlineSeconds integer

    The maximum time in seconds for a deployment to make progress before it is considered to be failed.

    paused boolean

    Indicates that the deployment is paused. Defaults to false.

    minReadySeconds integer

    Minimum number of seconds for which a newly created pod should be ready without any of its container crashing.

    Return Value

    Returns a DeploymentSpec object containing the deployment specification with all configured parameters for pod management and update strategies.

    Operations

    This section covers the available operations for Deployment resources in Kubernetes. These operations allow you to create, read, update, delete, and manage Deployment objects through the Kubernetes API.

    Create Deployment

    Create a new Deployment resource in the specified namespace. This operation allows you to define a new deployment with the desired pod template and replica configuration.

    HTTP Request

    POST /apis/apps/v1/namespaces/{namespace}/deployments

    kubectl Syntax

    # Create deployment from YAML file
kubectl apply -f deployment.yaml

# Create deployment directly with kubectl create
kubectl create deployment my-deployment --image=nginx:1.21 --replicas=3

# Create deployment with specific namespace
kubectl create deployment my-deployment --image=nginx:1.21 -n my-namespace

# Create deployment with resource limits
kubectl create deployment my-deployment --image=nginx:1.21 --replicas=3 --requests=cpu=100m,memory=128Mi

# Create deployment with environment variables
kubectl create deployment my-deployment --image=nginx:1.21 --env="ENV=production"

    Update Deployment

    Update a Deployment resource by replacing the entire resource definition. This triggers a rolling update of the pods managed by the deployment.

    HTTP Request

    PUT /apis/apps/v1/namespaces/{namespace}/deployments/{name}

    kubectl Syntax

    # Update deployment from modified YAML file
kubectl apply -f updated-deployment.yaml

# Update deployment image
kubectl set image deployment/my-deployment container-name=new-image:tag

# Update deployment with environment variables
kubectl set env deployment/my-deployment ENV=production

# Update deployment resources
kubectl set resources deployment my-deployment --limits=cpu=200m,memory=256Mi --requests=cpu=100m,memory=128Mi

# Replace deployment entirely
kubectl replace -f deployment.yaml

    Django Example

    Patch Deployment

    Partially update a Deployment resource by modifying specific fields without replacing the entire resource definition.

    HTTP Request

    PATCH /apis/apps/v1/namespaces/{namespace}/deployments/{name}

    kubectl Syntax

    # Patch deployment replicas
kubectl patch deployment my-deployment -p '{"spec":{"replicas":5}}'

# Patch deployment strategy
kubectl patch deployment my-deployment -p '{"spec":{"strategy":{"type":"RollingUpdate","rollingUpdate":{"maxSurge":2}}}}'

# Patch deployment with merge strategy
kubectl patch deployment my-deployment --type merge -p '{"spec":{"template":{"spec":{"containers":[{"name":"app","image":"new-image:tag"}]}}}}'

# Patch deployment annotations
kubectl patch deployment my-deployment -p '{"metadata":{"annotations":{"deployment.kubernetes.io/revision":"3"}}}'

# Patch deployment with JSON file
kubectl patch deployment my-deployment --patch-file patch.json

    Django Example

    Patch anihpj Django deployment to modify specific configuration without full replacement.

    # Patch anihpj jobpost deployment replicas
kubectl patch deployment anihpj-jobpost-deployment -p '{"spec":{"replicas":6}}' -n anihpj-services

# Patch anihpj API deployment strategy
kubectl patch deployment anihpj-api-deployment -p '{"spec":{"strategy":{"rollingUpdate":{"maxUnavailable":"0"}}}}' -n anihpj-services

    Delete Deployment

    Delete a Deployment resource and its associated ReplicaSets and Pods. This operation removes the deployment and all resources it manages.

    HTTP Request

    DELETE /apis/apps/v1/namespaces/{namespace}/deployments/{name}

    kubectl Syntax

    # Delete deployment
kubectl delete deployment my-deployment

# Delete deployment with grace period
kubectl delete deployment my-deployment --grace-period=30

# Delete deployment immediately
kubectl delete deployment my-deployment --force --grace-period=0

# Delete deployment from file
kubectl delete -f deployment.yaml

# Delete multiple deployments
kubectl delete deployment deployment1 deployment2 deployment3

    Scale Deployment

    Scale a Deployment by changing the number of replicas. This operation modifies the replica count and triggers the creation or deletion of pods as needed.

    HTTP Request

    PATCH /apis/apps/v1/namespaces/{namespace}/deployments/{name}/scale

    kubectl Syntax

    # Scale deployment to specific replica count
kubectl scale deployment my-deployment --replicas=5

# Scale deployment with condition
kubectl scale deployment my-deployment --replicas=3 --current-replicas=5

# Scale multiple deployments
kubectl scale deployment deployment1 deployment2 --replicas=3

# Auto-scale deployment (requires HPA)
kubectl autoscale deployment my-deployment --min=1 --max=10 --cpu-percent=80

# Scale to zero (stop all pods)
kubectl scale deployment my-deployment --replicas=0

    Django Example

    Scale anihpj Django deployments based on traffic and resource requirements.

    # Scale anihpj web deployment for high traffic
kubectl scale deployment anihpj-web-deployment --replicas=8 -n anihpj-services

# Scale down anihpj staging deployment
kubectl scale deployment anihpj-staging-deployment --replicas=1 -n anihpj-staging
    deployment.apps/anihpj-web-deployment scaled deployment.apps/anihpj-staging-deployment scaled

    Rollout Management

    Manage Deployment rollouts including checking status, history, pausing, resuming, and rolling back to previous versions.

    kubectl Syntax

    # Check rollout status
kubectl rollout status deployment/my-deployment

# View rollout history
kubectl rollout history deployment/my-deployment

# Pause rollout
kubectl rollout pause deployment/my-deployment

# Resume rollout
kubectl rollout resume deployment/my-deployment

# Rollback to previous version
kubectl rollout undo deployment/my-deployment

# Rollback to specific revision
kubectl rollout undo deployment/my-deployment --to-revision=2

# Restart deployment (rolling restart)
kubectl rollout restart deployment/my-deployment
