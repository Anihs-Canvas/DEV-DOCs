ReplicationController

Official Docs: ReplicationController v1 API | API Version: v1

ReplicationController represents the configuration of a replication controller that ensures a specified number of pod replicas are running at any given time.

Full Description

ReplicationController is one of the oldest workload resources in Kubernetes, designed to ensure that a specified number of identical pods are running at all times. When deploying your anihpj Django application, a ReplicationController will monitor your Django pods and automatically create new ones if any fail or are deleted, ensuring high availability for your jobpost application.

While ReplicationController has largely been superseded by Deployment (which provides additional features like rolling updates), it's still fundamental to understand as it forms the basis for more advanced workload resources. In production Django deployments, you'll typically use Deployments, but ReplicationController is perfect for understanding basic pod replication concepts.

Syntax

apiVersion: v1
kind: ReplicationController
metadata:
  name: string
  namespace: string
  labels:
    key: value
spec:
  replicas: int32
  selector:
    matchLabels:
      key: value
  template:
    metadata:
      labels:
        key: value
    spec:
      containers:
      - name: string
        image: string
        ports:
        - containerPort: int32

Parameters Description

  • apiVersion: Always "v1" for ReplicationController. This indicates the API version being used.
  • kind: Must be "ReplicationController" to specify the resource type.
  • metadata.name: Unique name for the ReplicationController within the namespace. Use descriptive names like "anihpj-django-rc".
  • metadata.namespace: Kubernetes namespace where the ReplicationController will be created. Defaults to "default".
  • metadata.labels: Key-value pairs for organizing and selecting ReplicationControllers. Useful for grouping Django app components.
  • spec.replicas: Desired number of pod replicas. For Django apps, start with 2-3 for high availability.
  • spec.selector: Label selector to identify which pods belong to this ReplicationController. Must match template labels.
  • spec.template: Pod template specification that defines how new pods should be created.
  • spec.template.metadata.labels: Labels applied to created pods. Must include selector labels.
  • spec.template.spec: Pod specification including containers, volumes, and other pod-level configurations.
  • spec.minReadySeconds: Minimum seconds a pod must be ready before considered available. Useful for Django apps with startup time.

Return Value

When successfully created, returns a ReplicationController object containing the complete specification and current status. The status includes current replica count, ready replicas, and conditions indicating the controller's health.

ReplicationControllerSpec

ReplicationControllerSpec Fields

replicas - Desired pod count (default: 1)
selector - Label query for pod selection
template - Pod template specification
minReadySeconds - Minimum ready time (default: 0)

ReplicationControllerSpec defines the desired behavior of the replication controller. For Django applications, carefully consider the replica count based on expected traffic, resource requirements, and high availability needs.

Syntax
spec:
  replicas: 3
  selector:
    app: anihpj
    component: web
  template:
    metadata:
      labels:
        app: anihpj
        component: web
    spec:
      containers:
      - name: django-app
        image: anihpj/django:latest
        ports:
        - containerPort: 8000
  minReadySeconds: 30
Parameters Description
  • selector (map[string]string): Label query over pods that should match the replicas count. If empty, defaults to labels on Pod template. For anihpj Django apps, use labels like app=anihpj, component=web.
  • template (PodTemplateSpec): Defines the pod that will be created if insufficient replicas are detected. Must include restartPolicy: Always for ReplicationController.
  • replicas (int32): Number of desired replicas. For Django production, typically 2-5 replicas for web apps, 1-3 for workers.
  • minReadySeconds (int32): Minimum seconds for a pod to be ready before considered available. Important for Django apps with database connections.

availableReplicas - Available replicas count
conditions - Current state conditions

ReplicationControllerStatus represents the current status of a replication controller, providing visibility into the health and state of your Django application deployment.

  • replicas (int32, required): Most recently observed number of replicas. Monitor this to ensure your Django app maintains desired capacity.

  • readyReplicas (int32): Number of ready replicas. Should match desired replicas for healthy Django deployment.

ReplicationControllerCondition

ReplicationControllerCondition Fields

type - Type of replication controller condition
status - Status of the condition (True, False, Unknown)
lastTransitionTime - Time condition changed
reason - Machine-readable reason for condition
message - Human-readable message about condition

ReplicationControllerCondition describes the state of a replication controller at a certain point in time. This provides detailed insights into why your Django application might be experiencing issues.

Syntax
conditions:
- type: "ReplicaFailure"
  status: "True"
  lastTransitionTime: "2025-09-08T10:30:00Z"
  reason: "FailedCreate"
  message: "Error creating: pods \"anihpj-skills-rc-\" is forbidden"
- type: "Progressing"
  status: "True"
  lastTransitionTime: "2025-09-08T10:25:00Z"
  reason: "NewReplicaSetAvailable"
  message: "ReplicationController has successfully progressed"
Parameters Description
  • type (string, required): Type of replication controller condition. Common types include "ReplicaFailure", "Progressing"
  • status (string, required): Status of the condition, one of "True", "False", "Unknown"
  • lastTransitionTime (Time): The last time the condition transitioned from one status to another
  • reason (string): Machine-readable, UpperCamelCase text indicating the reason for the condition's last transition
  • message (string): Human-readable message indicating details about the transition
Return Value

Conditions are read-only fields populated by the Kubernetes system. They provide status information and are part of the ReplicationController status object.

Update Status ReplicationController

HTTP Request

Method - PUT
Path - /api/v1/namespaces/{namespace}/replicationcontrollers/{name}/status
Body - ReplicationController object
Response - Updated ReplicationController

Replace status of the specified ReplicationController. This operation is typically used by controllers, not end users.

HTTP API Syntax
PUT /api/v1/namespaces/{namespace}/replicationcontrollers/{name}/status
Content-Type: application/yaml
Authorization: Bearer {token}

{ReplicationController-status-object}
Parameters
  • name (string, required): Name of the ReplicationController
  • namespace (string, required): Object namespace
  • body (ReplicationController, required): ReplicationController object
  • dryRun (string): Dry run mode
  • fieldManager (string): Field manager name
  • fieldValidation (string): Field validation mode
  • pretty (string): Pretty-print the output
Return Value

Returns the updated ReplicationController object with status information reflecting the current state of replicas.

  • 200 OK: ReplicationController status successfully updated
  • 400 Bad Request: Invalid ReplicationController status object
  • 401 Unauthorized: Authentication required
  • 403 Forbidden: Insufficient permissions
  • 404 Not Found: ReplicationController not found
  • 409 Conflict: Conflict with current state
  • 422 Unprocessable Entity: Invalid field values

# This is typically done by Kubernetes controllers, not users
curl -X PUT \

  -d '{
    "status": {
      "replicas": 3,
      "readyReplicas": 2,
      "availableReplicas": 2
    }
  }'
HTTP/1.1 200 OK { "kind": "ReplicationController", "metadata": {"name": "anihpj-django-rc"}, "status": { "replicas": 3, "readyReplicas": 2, "availableReplicas": 2 } }

Delete ReplicationController

HTTP Request

Method - DELETE
Path - /api/v1/namespaces/{namespace}/replicationcontrollers/{name}
Response - Status of deletion
Options - Cascade deletion settings

Delete a ReplicationController and optionally its managed pods. Critical for Django application cleanup and resource management.

Syntax
kubectl delete rc {name}
kubectl delete rc {name} --cascade=orphan
kubectl delete -f replicationcontroller.yaml
HTTP API Syntax
DELETE /api/v1/namespaces/{namespace}/replicationcontrollers/{name}
Content-Type: application/json
Authorization: Bearer {token}

{
  "kind": "DeleteOptions",
  "propagationPolicy": "Foreground"
}
Parameters
  • propagationPolicy (string): "Foreground", "Background", or "Orphan"
  • gracePeriodSeconds (integer): Grace period for pod termination
  • orphanDependents (boolean): Deprecated. Use propagationPolicy
Return Value

Returns the status of the deletion operation with information about the ReplicationController removal.

  • 200 OK: ReplicationController successfully deleted
  • 400 Bad Request: Invalid delete options
  • 401 Unauthorized: Authentication required
  • 403 Forbidden: Insufficient permissions to delete
  • 404 Not Found: ReplicationController not found
  • 409 Conflict: Conflict with current state during deletion
Example 1: Delete Django RC and Pods
kubectl delete rc anihpj-django-rc
replicationcontroller "anihpj-django-rc" deleted
kubectl get pods -l app=anihpj,component=web
No resources found in default namespace.
Example 2: Delete RC but Keep Pods Running

Example 4: Force Delete Stuck ReplicationController

Delete Collection ReplicationController

HTTP Request

Method - DELETE
Path - /api/v1/namespaces/{namespace}/replicationcontrollers
Body - DeleteOptions
Response - Status

Delete collection of ReplicationController resources matching specified criteria. Useful for bulk cleanup of Django deployments.

Syntax
kubectl delete rc -l {label-selector}
kubectl delete rc --all
kubectl delete rc -l app=anihpj
HTTP API Syntax
DELETE /api/v1/namespaces/{namespace}/replicationcontrollers
Content-Type: application/json
Authorization: Bearer {token}

{
  "kind": "DeleteOptions",
  "propagationPolicy": "Foreground"
}
Parameters
  • namespace (string, required): Object namespace
  • body (DeleteOptions): Delete options
  • continue (string): Continue token for pagination
  • dryRun (string): Dry run mode
  • fieldSelector (string): Field selector for filtering
  • gracePeriodSeconds (integer): Grace period for termination
  • ignoreStoreReadErrorWithClusterBreakingPotential (boolean): Ignore store read errors
  • labelSelector (string): Label selector for filtering
  • limit (integer): Maximum number to delete
  • pretty (string): Pretty-print the output
  • propagationPolicy (string): Propagation policy
  • resourceVersion (string): Resource version
  • resourceVersionMatch (string): Resource version match strategy
  • sendInitialEvents (boolean): Send initial events
  • timeoutSeconds (integer): Timeout for the operation
Example 1: Delete All Django ReplicationControllers
kubectl delete rc -l app=anihpj
replicationcontroller "anihpj-django-rc" deleted replicationcontroller "jobpost-worker-rc" deleted

✅ Best Practices for Django Applications:
  • Use meaningful names like "anihpj-web-rc" or "jobpost-worker-rc"
  • Set appropriate resource requests/limits for Django containers
  • Include health checks for Django application endpoints
  • Use ConfigMaps for Django settings and Secrets for sensitive data
  • Consider using Deployments instead of ReplicationController for production
