Pod

Docs: Kubernetes Pod v1 API

The smallest deployable unit in Kubernetes, representing one or more containers that share storage and network.

Full description

A Pod is the atomic unit of deployment in Kubernetes. For anihpj Django applications, a Pod typically contains your Django web server container, and optionally sidecar containers for logging, monitoring, or caching. Pods share an IP address, storage volumes, and lifecycle. When you deploy your anihpj jobpost Django application, each Pod runs your Django server and can access your models like Skills, Author, Location, and JobPost from the jobpost app. Pods are ephemeral - they can be created, destroyed, and recreated by higher-level controllers like Deployments.

Syntax

apiVersion: v1
kind: Pod
metadata:
  name: pod-name
  namespace: namespace-name
  labels:
    app: app-name
    component: component-name
spec:
  containers:
  - name: container-name
    image: image:tag
    ports:
    - containerPort: port-number
    env:
    - name: ENV_VAR
      value: "value"
    resources:
      requests:
        memory: "memory-request"
        cpu: "cpu-request"
      limits:
        memory: "memory-limit"
        cpu: "cpu-limit"

Parameters description

  • apiVersion: Always "v1" for Pod resources. Specifies the Kubernetes API version to use for this resource.
  • kind: Must be "Pod" to indicate this is a Pod resource definition.
  • metadata.name: Unique name for the Pod within the namespace. Use descriptive names like "jobpost-web-pod" or "django-app-worker".
  • metadata.namespace: Kubernetes namespace where the Pod will be created. Defaults to "default" if not specified. Use separate namespaces for different environments.
  • metadata.labels: Key-value pairs for organizing and selecting Pods. Essential for Service discovery and Deployment management. Use consistent labeling like app=jobpost, component=web.
  • spec.containers: Array of container specifications. Most Django applications need only one container, but you can add sidecars for Redis, logging, or monitoring.
  • spec.containers[].name: Name of the container within the Pod. Use descriptive names like "django-web" or "jobpost-api".
  • spec.containers[].image: Docker image to run. Always include a specific tag for version control (e.g., "jobpost-django:v2.1.0").
  • spec.containers[].ports: Ports that the container exposes. Django typically uses port 8000 for development or 80/443 for production.
  • spec.containers[].env: Environment variables passed to the container. Include Django settings, database URLs, Redis URLs, and secret keys.
  • spec.containers[].resources: CPU and memory resource requests and limits. Requests guarantee resources, limits prevent overconsumption.

Return value

Pod object: Returns the created Pod resource with system-assigned fields including IP address, node placement, creation timestamp, and current status. The Pod will initially show phase "Pending" while being scheduled, then "Running" when containers start successfully, or "Failed" if there are issues.

PodSpec

Docs: Kubernetes PodSpec Documentation

Defines the desired state and configuration for a Pod running Django applications.

Full description

PodSpec is the specification section of a Pod resource that defines how your Django application should run. It includes container definitions, volumes, networking, scheduling constraints, and security contexts. The PodSpec is crucial for Django deployments as it controls resource allocation, environment variables, persistent storage for Django media files, and database connections. Understanding PodSpec is essential for optimizing Django application performance in Kubernetes.

Syntax

spec:
  containers:
  - name: container-name
    image: image:tag
    ports:
    - containerPort: 8000
  volumes:
  - name: volume-name
    persistentVolumeClaim:
      claimName: pvc-name
  restartPolicy: Always
  nodeSelector:
    disktype: ssd
  affinity:
    podAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchLabels:
            app: database

Parameters description

  • containers: Array of container specifications running in the Pod. Each Django application typically needs one main container plus optional sidecars.
  • volumes: Array of volumes that containers can mount. Essential for Django static files, media uploads, and database persistence.
  • restartPolicy: Defines when containers should be restarted. Options: Always (default), OnFailure, Never. Use Always for Django web servers.
  • nodeSelector: Labels to constrain Pod scheduling to specific nodes. Useful for Django apps requiring SSD storage or GPU resources.
  • affinity: Advanced scheduling rules for Pod placement. Use to co-locate Django apps with databases or separate them for fault tolerance.
  • tolerations: Allows Pods to schedule on nodes with matching taints. Important for dedicated Django node pools.
  • securityContext: Pod-level security settings. Set user IDs, filesystem groups, and capabilities for Django containers.
  • serviceAccountName: Service account for Pod authentication. Required for Django apps accessing Kubernetes APIs or cloud resources.

Return value

PodSpec object: Configuration specification that Kubernetes uses to create and manage the Pod. Changes to PodSpec require Pod recreation.

Full description

NodeAffinity defines rules for scheduling Django Pods on specific nodes based on node labels and properties. This enables deployment of Django applications on nodes with specific characteristics like SSD storage, GPU capabilities, geographic location, or instance types optimized for Django workloads.

Syntax

  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: disktype
          operator: In
          values: ["ssd"]
    preferredDuringSchedulingIgnoredDuringExecution:
    - weight: 100
      preference:
        matchExpressions:
        - key: zone
          operator: In
          values: ["us-west-1a"]

Parameters description

  • requiredDuringSchedulingIgnoredDuringExecution: Hard constraints that must be satisfied for Pod scheduling
  • preferredDuringSchedulingIgnoredDuringExecution: Soft preferences with weights for Pod scheduling
  • nodeSelectorTerms: Node selection criteria based on node labels
  • matchExpressions: Label selector expressions with key, operator, and values
  • weight: Priority weight for preferred scheduling rules (1-100)

Return value

NodeAffinity configuration that influences Pod placement decisions based on node characteristics.

Binding establishes a relationship between Kubernetes objects, most commonly used by the scheduler to bind Pods to Nodes.

Full description

Binding is a Kubernetes API resource that creates a relationship between two objects. The most common use case is when the Kubernetes scheduler creates a Binding to tie a Pod to a specific Node. This process is fundamental to how Django applications get scheduled and placed on worker nodes in your cluster. When you deploy a Django application, the scheduler evaluates resource requirements, node capacity, and scheduling constraints to determine the best node for your Pod, then creates a Binding to make this assignment official.

target:

  name: worker-node-1

  • target.kind: Type of the target object (commonly "Node" for Pod-to-Node bindings).

Binding object: Establishes the relationship between source and target objects, confirming the scheduling decision for Django Pods.

ReplicationController

Docs: Kubernetes ReplicationController v1

ReplicationController represents the configuration of a replication controller.

📋 Full Description

A ReplicationController ensures that a specified number of pod replicas are running at any one time. In other words, a ReplicationController makes sure that a pod or a homogeneous set of pods is always up and available. If there are too many pods, the ReplicationController terminates the extra pods. If there are too few, the ReplicationController starts more pods.

For Django applications, ReplicationControllers provide basic scaling and self-healing capabilities. However, modern Kubernetes deployments typically use Deployment resources instead, which provide more advanced features like rolling updates and rollback capabilities.

💡 Note: While ReplicationControllers are still supported, it's recommended to use Deployments for new applications as they provide better management capabilities and are the preferred way to ensure your Django applications are properly replicated and updated.

Deployment

Docs: Kubernetes Deployment v1

Deployment enables declarative updates for Pods and ReplicaSets.

🎯 Overview

Deployment provides declarative updates for Pods and ReplicaSets. You describe a desired state in a Deployment, and the Deployment Controller changes the actual state to the desired state at a controlled rate. You can define Deployments to create new ReplicaSets, or to remove existing Deployments and adopt all their resources with new Deployments.

For Django applications, Deployments are the recommended way to manage your application containers. They provide rolling updates, rollback capabilities, and declarative configuration management. Unlike direct Pod or ReplicaSet management, Deployments handle the complexity of updating your Django application without downtime.

💡 Note: Don't manage ReplicaSets owned by a Deployment. Consider opening an issue in the main Kubernetes repository if your use case is not covered below.

Deployment

Docs: Kubernetes Deployment v1 API

Provides declarative updates for Pods and ReplicaSets, enabling rolling updates, rollbacks, and scaling for Django applications.

Full description

A Deployment provides declarative updates for Pods and ReplicaSets. It's the recommended way to deploy Django applications in Kubernetes. Deployments handle rolling updates, rollbacks, and scaling operations automatically. When you update your Django application (like changing JobPost model logic or adding new Author features), Deployments ensure zero-downtime updates by gradually replacing old Pods with new ones. They also maintain ReplicaSets to ensure your desired number of Django application replicas are always running.

Parameters description

  • apiVersion: Always "apps/v1" for Deployment resources. Specifies the Kubernetes API version to use for this resource.

📋 Full Description

A ReplicaSet ensures that a specified number of pod replicas are running at any given time. In other words, a ReplicaSet makes sure that a pod or a homogeneous set of pods is always up and available. If there are too many pods, the ReplicaSet terminates the extra pods. If there are too few, the ReplicaSet starts more pods.

For Django applications, ReplicaSets provide the foundation for maintaining your application's availability and scaling. ReplicaSets use more sophisticated label selectors compared to ReplicationControllers, supporting set-based selection requirements.

🎯 Overview

StatefulSet represents a set of pods with consistent identities. Identities are defined as:

  • Network: A single stable DNS and hostname
  • Storage: As many VolumeClaims as requested

The StatefulSet guarantees that a given network identity will always map to the same storage identity.

For Django applications, StatefulSets are essential when you need stable, persistent identities for your pods. 💡 Note: StatefulSets are valuable for Django applications that need persistent data, ordered operations, or stable network identities. Common use cases include Django applications with local file storage, distributed caches, or applications requiring specific initialization order.

DaemonSet

Docs: Kubernetes DaemonSet v1

DaemonSet represents the configuration of a daemon set.

🎯 Overview

DaemonSet represents the configuration of a daemon set. A DaemonSet ensures that all (or some) Nodes run a copy of a Pod. As nodes are added to the cluster, Pods are added to them. As nodes are removed from the cluster, those Pods are garbage collected. Deleting a DaemonSet will clean up the Pods it created.

Some typical uses of a DaemonSet are:

  • Running a cluster storage daemon on every node
  • Running a logs collection daemon on every node
  • Running a node monitoring daemon on every node

For Django applications, DaemonSets are essential for infrastructure-level services that need to run on every node in your cluster. This includes monitoring agents, log collectors, network plugins, and security scanners that support your Django application infrastructure. Unlike Deployments that manage a specific number of replicas, DaemonSets ensure one pod per node, making them perfect for node-level services.

💡 Note: DaemonSets are valuable for Django applications when you need to run infrastructure services on every node, such as monitoring Django application metrics, collecting logs from all Django pods, or running security agents across your cluster nodes.

Specification of a DaemonSet defining pod template, update strategy, and node selection criteria.

Full description

DaemonSetSpec defines the desired behavior of a DaemonSet for Django applications requiring node-level services. Parameters description

  • selector (LabelSelector, required): A label query over pods that are managed by the daemon set. Must match in order to be controlled. It must match the pod template's labels. More info: https://kubernetes.io/docs/concepts/overview/working-with-objects/labels/#label-selectors

    🎯 Overview

    Job runs one or more Pods to completion and ensures that a specified number of them successfully terminate. Unlike Deployments or ReplicaSets that maintain long-running services, Jobs are designed for batch processing, one-time tasks, and finite workloads. Jobs are perfect for Django maintenance tasks, data migrations, report generation, or cleanup operations.

    For Django applications, Jobs are ideal for running periodic maintenance tasks like data processing, backup operations, sending batch emails, or performing database migrations. When your Django JobPost application needs to process user data, generate reports, or perform cleanup tasks, Jobs provide reliability with retry logic and completion tracking.

    💡 Note: Jobs are different from CronJobs - Jobs run once, while CronJobs run Jobs on a schedule. Use Jobs for immediate one-time tasks and CronJobs for recurring scheduled tasks.

    JobStatus

    Current state of a Job including execution progress, completion status, and pod tracking information.

    Full description

    JobStatus represents the current state of a Job providing real-time information about job execution progress and completion status.     Parameters description

    • conditions: Latest available observations of the job's current state. Includes "Complete", "Failed", or "Suspended" conditions.
    • uncountedTerminatedPods: UIDs of Pods that have terminated but haven't been accounted for in status counters yet.
