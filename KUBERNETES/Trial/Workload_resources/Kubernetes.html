Pod v1

Pod Overview

A Pod is the smallest deployable unit in Kubernetes, wrapping one or more containers with shared storage, network, and lifecycle.

📚 Official Documentation: https://kubernetes.io/docs/reference/kubernetes-api/workload-resources/pod-v1/

Full Description

A Pod represents a group of one or more containers that share storage, network, and a specification for how to run them. In Django applications, Pods typically contain your Django web server along with optional sidecar containers for logging, monitoring, caching, or background processing tasks.

Basic Syntax

apiVersion: v1
kind: Pod
metadata:
  name: django-jobpost-pod
  labels:
    app: jobpost-api
spec:
  containers:
  - name: django-app
    image: django-jobpost:latest
    ports:
    - containerPort: 8000

Return Values

Pod Creation Returns:
  • metadata.name - Pod name
  • metadata.uid - Unique identifier
  • status.phase - Current phase (Pending, Running, Succeeded, Failed)
  • status.podIP - Assigned IP address
  • status.containerStatuses - Container status array

LifecycleHandler

Lifecycle Handler Configuration

Handlers for container lifecycle events that execute at specific points in the container lifecycle.

Full Description

LifecycleHandler defines actions that are executed at specific points in a container's lifecycle. These handlers allow you to run custom logic immediately after a container starts (postStart) or just before it terminates (preStop). For Django applications, this is useful for database migrations, cache warming, graceful shutdowns, and cleanup operations. The handlers can execute commands inside the container or make HTTP requests to specific endpoints.

Syntax

apiVersion: v1
kind: Pod
metadata:
  name: django-lifecycle-pod
spec:
  containers:
  - name: django-app
    image: django-jobpost:latest
    lifecycle:
      postStart:
        exec:
          command: ["/bin/sh", "-c", "python manage.py migrate"]
      preStop:
        httpGet:
          path: /shutdown
          port: 8000

Parameters Description

  • postStart: Handler executed immediately after container creation but before the container's main process starts. Use for initialization tasks like database migrations, cache warming, or setting up required files. Can be exec (run command) or httpGet (HTTP request).
  • preStop: Handler executed immediately before a container is terminated. Use for graceful shutdown procedures, saving state, or cleanup operations. Gives the container time to finish processing before termination.
  • exec: Executes a command inside the container. Specify the command as an array of strings. Commonly used for Django management commands or shell scripts.
  • httpGet: Performs an HTTP GET request to a specific path and port. Use this for triggering application-specific shutdown or initialization endpoints.

Return Value

LifecycleHandler executions return success/failure status. PostStart handlers must complete successfully for the container to be considered started. PreStop handlers are given a grace period to complete before forceful termination. If handlers fail, appropriate events are recorded in the pod status.

NodeAffinity

Node Affinity Rules

Advanced node selection rules for controlling pod placement on specific cluster nodes.

Full Description

NodeAffinity provides advanced scheduling rules for placing pods on specific nodes based on node labels and properties. Unlike simple nodeSelector, NodeAffinity supports complex expressions, required and preferred rules, and operator-based matching. For Django applications, use NodeAffinity to schedule database-heavy workloads on high-memory nodes, place web servers on nodes with SSD storage, or separate production and development environments. It ensures your Django JobPost application runs on nodes with the right hardware specifications and geographic location.

Syntax

apiVersion: v1
kind: Pod
metadata:
  name: django-node-affinity-pod
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: kubernetes.io/arch
            operator: In
            values: ["amd64"]
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 100
        preference:
          matchExpressions:
          - key: node-type
            operator: In
            values: ["compute-optimized"]
  containers:
  - name: django-app
    image: django-jobpost:latest

Parameters Description

  • requiredDuringSchedulingIgnoredDuringExecution: Hard constraint that must be satisfied for pod scheduling. If no nodes match these rules, the pod remains in Pending state. Use for critical requirements like specific CPU architecture or hardware features.
  • preferredDuringSchedulingIgnoredDuringExecution: Soft preferences that the scheduler tries to satisfy but can ignore if necessary. Each preference has a weight (1-100) to prioritize multiple preferences. Use for performance optimizations.
  • nodeSelectorTerms: List of node selector requirements. Pod is scheduled if any term matches. Each term contains matchExpressions or matchFields.
  • matchExpressions: Label-based selection criteria using key, operator (In, NotIn, Exists, DoesNotExist), and values. More flexible than simple key-value matching.
  • weight: Priority weight (1-100) for preferred rules. Higher weights are given more preference during scheduling decisions.

Return Value

NodeAffinity influences pod scheduling decisions. The scheduler evaluates all rules and places the pod on the best matching node. Returns pod placement information including the selected node name, node labels, and scheduling events. If required rules cannot be satisfied, pod remains unscheduled with appropriate error events.
