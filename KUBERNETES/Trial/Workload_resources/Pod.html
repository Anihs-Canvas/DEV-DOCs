Pod Basics

A Pod is the smallest deployable unit in Kubernetes, representing a group of one or more containers that share storage and network. In Django applications, Pods typically contain your web application container, and optionally sidecar containers for logging, monitoring, or data processing.

🚀 Pod Core Concepts

Single Container - Most Django apps use one container per Pod
Shared Network - All containers share the same IP and port space
Shared Storage - Volumes are accessible to all containers
Atomic Unit - Pods are created, scheduled, and destroyed together
Ephemeral - Pods are mortal; they can be killed and recreated
Unique IP - Each Pod gets its own cluster-internal IP address
💡 Django Best Practice: Run your Django application in a single container Pod with Redis or PostgreSQL in separate Pods, connected via Services for better scalability and maintenance.

Create Pod

API: Kubernetes Pod v1 API

Creates a new Pod in the specified namespace to run your Django application containers.

Full description

Pod creation is the fundamental operation for deploying Django applications in Kubernetes. A Pod encapsulates your Django application container(s), defines resource requirements, environment variables, and networking configuration. When you create a Pod for your JobPost application, you're essentially telling Kubernetes to schedule and run your containerized Django app on a worker node.

Syntax

apiVersion: v1
kind: Pod
metadata:
  name: pod-name
  namespace: namespace-name
  labels:
    app: application-name
spec:
  containers:
  - name: container-name
    image: image:tag
    ports:
    - containerPort: port-number
    env:
    - name: ENV_VAR
      value: "value"

Parameters description

  • apiVersion: Always "v1" for Pod resources. This specifies which version of the Kubernetes API to use.
  • kind: Must be "Pod" to indicate this is a Pod resource definition.
  • metadata.name: Unique name for the Pod within the namespace. Use descriptive names like "jobpost-web-pod" or "anihpj-django-app".
  • metadata.namespace: Kubernetes namespace to create the Pod in. Defaults to "default" if not specified.
  • metadata.labels: Key-value pairs for organizing and selecting Pods. Essential for Service discovery and management.
  • spec.containers: Array of container specifications. Most Django apps need only one container.
  • spec.containers[].name: Name of the container within the Pod. Use descriptive names like "django-app" or "jobpost-web".
  • spec.containers[].image: Docker image to run. Include tag for version control (e.g., "my-django-app:v1.2.3").
  • spec.containers[].ports: Ports to expose from the container. Django typically uses port 8000 or 80.
  • spec.containers[].env: Environment variables to pass to the container. Include database URLs, secret keys, etc.

Return value

Pod object: Returns the created Pod resource with assigned IP address, node placement, and current status. The Pod will have a phase of "Pending" initially, then transition to "Running" when containers start successfully.

5 short examples with Django models

Example 1: Basic Django JobPost app Pod
apiVersion: v1
kind: Pod
metadata:
  name: jobpost-django-pod
  labels:
    app: jobpost
    component: web
spec:
  containers:
  - name: django-app
    image: jobpost-django:latest
    ports:
    - containerPort: 8000
    env:
    - name: DJANGO_SETTINGS_MODULE
      value: "anihpj.settings"
pod/jobpost-django-pod created Name: jobpost-django-pod Status: Running IP: 10.244.1.15
Example 2: Django app with database connection
apiVersion: v1
kind: Pod
metadata:
  name: anihpj-web-pod
  labels:
    app: anihpj
    tier: web
spec:
  containers:
  - name: django-web
    image: anihpj-django:v1.0
    ports:
    - containerPort: 8000
    env:
    - name: DATABASE_URL
      value: "postgresql://user:pass@postgres-service:5432/anihpj_db"
    - name: REDIS_URL
      value: "redis://redis-service:6379/0"
pod/anihpj-web-pod created Name: anihpj-web-pod Status: Running IP: 10.244.2.23 Node: worker-node-1
Example 3: JobPost API Pod with resource limits
apiVersion: v1
kind: Pod
metadata:
  name: jobpost-api-pod
  labels:
    app: jobpost
    component: api
spec:
  containers:
  - name: django-api
    image: jobpost-api:v2.1
    ports:
    - containerPort: 8000
    resources:
      requests:
        memory: "256Mi"
        cpu: "250m"
      limits:
        memory: "512Mi"
        cpu: "500m"
pod/jobpost-api-pod created Name: jobpost-api-pod Status: Running CPU: 250m/500m Memory: 256Mi/512Mi
Example 4: Multi-container Pod with Django + Redis sidecar
apiVersion: v1
kind: Pod
metadata:
  name: jobpost-with-cache
  labels:
    app: jobpost
    type: web-with-cache
spec:
  containers:
  - name: django-app
    image: jobpost-django:latest
    ports:
    - containerPort: 8000
    env:
    - name: REDIS_URL
      value: "redis://localhost:6379/0"
  - name: redis-cache
    image: redis:7-alpine
    ports:
    - containerPort: 6379
pod/jobpost-with-cache created Containers: 2/2 Running Django: Ready on port 8000 Redis: Ready on port 6379
Example 5: Production Django Pod with health checks
apiVersion: v1
kind: Pod
metadata:
  name: jobpost-production
  labels:
    app: jobpost
    env: production
spec:
  containers:
  - name: django-prod
    image: jobpost-django:v3.0-prod
    ports:
    - containerPort: 8000
    livenessProbe:
      httpGet:
        path: /health/
        port: 8000
      initialDelaySeconds: 30
      periodSeconds: 10
    readinessProbe:
      httpGet:
        path: /ready/
        port: 8000
      initialDelaySeconds: 5
      periodSeconds: 5
pod/jobpost-production created Health checks: Configured Liveness: /health/ every 10s Readiness: /ready/ every 5s Status: Running and Ready
