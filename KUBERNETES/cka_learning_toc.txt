    1.2  What is Kubernetes? (The Simple Answer)
         - "Kubernetes is a container orchestrator"
         - It manages WHERE your containers run, WHEN they restart,
           HOW they scale, and WHO can access them
         - You tell Kubernetes "I want 3 copies of anihpj running"
           → Kubernetes makes it happen and KEEPS it happening
         - Name origin: Greek for "helmsman" (κυβερνήτης) — steers the ship
         [ANALOGY CARD: Kubernetes = Air Traffic Controller for containers]

         [THE KUBERNETES ELEVATOR PITCH]
         ┌──────────────────────────────────────────────────────┐
         │  You DECLARE what you want:                           │
         │    "Run 3 copies of anihpj, always"                  │
         │                                                       │
         │  Kubernetes DELIVERS:                                  │
         │    ✅ Starts 3 containers across available servers    │
         │    ✅ Restarts any that crash                         │
         │    ✅ Replaces any on a failed server                 │
         │    ✅ Scales up/down based on traffic                 │
         │    ✅ Routes network traffic to healthy instances     │
         │    ✅ Rolls out new versions with zero downtime       │
         │                                                       │
         │  🎯 "Desired State" → Kubernetes → "Actual State"    │
         └──────────────────────────────────────────────────────┘

    1.3  The CKA Perspective: Administrator vs Developer
         - CKAD: "I deploy and configure apps on a cluster"
         - CKA:  "I BUILD, MANAGE, and TROUBLESHOOT the cluster itself"
         - As a CKA admin, you are responsible for:
           • Installing Kubernetes clusters from scratch
           • Upgrading cluster versions safely
           • Backing up and restoring the cluster database (etcd)
           • Managing certificates and security
           • Troubleshooting broken nodes and failed components
           • Ensuring high availability and disaster recovery

    1.4  Kubernetes Core Concepts — The Big Picture
         - Cluster: a set of machines (nodes) managed by Kubernetes
         - Control Plane: the "brain" that makes decisions
         - Worker Nodes: the "muscles" that run your containers
         - Pod: smallest deployable unit (one or more containers)
         - Desired State vs Actual State: declarative magic

Chapter 2: Docker and Containers — The Building Blocks
--------------------------------------------------------------------------------
    2.1  Why Containers? The "Works on My Machine" Problem
         - Without containers: "it works on my laptop but not in production"
         - Containers package EVERYTHING: code + dependencies + config
         - Like a shipping container: same box works on any ship/truck/train
         [ANALOGY CARD: Shipping container = Docker container]

         [THE "WORKS ON MY MACHINE" PROBLEM]

         │                                                         │
         │  WHY? Different Python version, missing packages,       │
         │  different OS, missing environment variables...          │
         │                                                         │
         │  SOLUTION: Pack EVERYTHING into a container 📦           │
         │  ┌─────────────────────────────┐                        │

    3.2  Control Plane Components — Deep Dive for Admins
         🎯 [CKA EXAM TOPIC]
         a) kube-apiserver
            - The ONLY entry point to the cluster
            - All kubectl commands go through the API server
            - Validates and processes REST requests
            - Communicates with etcd
            - As admin: you configure its flags, certificates, RBAC
            [DIAGRAM: All paths lead to the API server]

         b) etcd
            - Distributed key-value store — THE cluster database
            - Stores ALL cluster state: pods, services, secrets, config
            - Uses Raft consensus protocol for consistency
            - As CKA admin: you MUST know backup and restore
            - Default port: 2379 (client), 2380 (peer)
            [DIAGRAM: etcd replication across 3 nodes]
            [WARNING: If etcd is lost without backup, the cluster is GONE]

         c) kube-scheduler
            - Decides WHICH node a new pod should run on
            - Considers: resource availability, affinity rules, taints, node conditions
            - Scoring algorithm: filters nodes → scores them → picks the best
            - As admin: you configure scheduler policies and priority classes
            [FLOWCHART: New pod → Filter → Score → Bind to best node]

         d) kube-controller-manager
            - Runs multiple controllers in a single process
            - Each controller watches one thing and corrects drift
            - Key controllers:
              • Node Controller: detects when nodes go down
              • ReplicaSet Controller: maintains correct pod count
              • Endpoint Controller: connects Services to Pods
              • ServiceAccount Controller: creates default accounts
            [DIAGRAM: Controller loop — Observe → Diff → Act]

            [THE CONTROLLER LOOP — heart of Kubernetes]
            ┌────────────────────────────────────────────┐
            │  CONTROLLER LOOP (runs continuously)        │
            │                                              │
            │  ┌──────────┐                               │
            │  │ OBSERVE  │ ← Watch actual state           │
            │  │ (watch)  │   "3 pods running"             │
            │  └────┬─────┘                               │
            │       │                                      │
            │       ▼                                      │
            │  ┌──────────┐                               │
            │  │ COMPARE  │ ← Compare to desired state     │
            │  │ (diff)   │   "spec says 5 replicas"       │
            │  └────┬─────┘                               │
            │       │                                      │
            │       ▼                                      │
            │  ┌──────────┐                               │
            │  │   ACT    │ ← Take corrective action       │
            │  │ (reconcile)│  "create 2 more pods"        │
            │  └────┬─────┘                               │
            │       │                                      │
            │       └───────────── loop forever ──────────┘│
            └────────────────────────────────────────────┘

         e) cloud-controller-manager (optional)
            - Integrates with cloud provider APIs (AWS, GCP, Azure)
            - Manages: LoadBalancers, Routes, Node lifecycle
            - Only present in cloud-hosted clusters

    3.3  Worker Node Components — Deep Dive for Admins
         🎯 [CKA EXAM TOPIC]
         a) kubelet
            - Agent running on EVERY worker node
            - Receives pod specifications from API server
            - Ensures containers are running and healthy
            - Reports node status back to control plane
            - Manages: container lifecycle, probes, volume mounts
            - Config: /var/lib/kubelet/config.yaml
            - As admin: you configure kubelet flags and troubleshoot it
            [NOTE: kubelet is NOT a pod — it's a systemd service on the node]

         b) kube-proxy
            - Network proxy running on every node
            - Implements Service networking rules
            - Modes: iptables (default), IPVS (high performance), nftables
            - Routes traffic to correct pod endpoints
            - As admin: you choose the mode and troubleshoot connectivity

         c) Container Runtime
            - Runs the actual containers
            - containerd (default for most K8s installations)
            - CRI-O (used by OpenShift)
            - Must implement Kubernetes CRI (Container Runtime Interface)

    3.5  Cluster Networking Overview (Admin Perspective)
         🎯 [CKA EXAM TOPIC]

         - All Pods can talk to all other Pods (flat network)
         - CNI plugins implement the networking: Calico, Cilium, Flannel

           • Pod-to-Service: via kube-proxy rules

    6.4  Imperative vs Declarative — When to Use Each
         🎯 [CKA EXAM TOPIC]

         [IMPERATIVE vs DECLARATIVE — visual comparison]
         ┌────────────────────────────────────────────────────────┐
         │                                                        │
         │  🏃 IMPERATIVE: "Do this NOW"                          │
         │  ┌──────────────────────────────────────────────┐     │
         │  │ $ kubectl run anihpj --image=anihpj:v1       │     │
         │  │ $ kubectl scale deployment anihpj --replicas=5│     │
         │  │ $ kubectl expose deployment anihpj --port=80  │     │
         │  │                                                │     │
         │  │  Like giving step-by-step cooking instructions │     │
         │  │  "chop onions, then fry them, then add salt"   │     │
         │  └──────────────────────────────────────────────┘     │
         │  ✅ Faster for one-off tasks & CKA exam speed         │
         │  ❌ No record of what you did, hard to reproduce      │
         │                                                        │
         │  📝 DECLARATIVE: "Make it look like THIS"              │
         │  ┌──────────────────────────────────────────────┐     │
         │  │ $ kubectl apply -f deployment.yaml            │     │
         │  │                                                │     │
         │  │  deployment.yaml defines the desired state:    │     │
         │  │  replicas: 5, image: anihpj:v1, port: 80      │     │
         │  │                                                │     │
         │  │  Like ordering from a menu:                    │     │
         │  │  "I want pasta" — the kitchen figures it out   │     │
         │  └──────────────────────────────────────────────┘     │
         │  ✅ Repeatable, version-controlled, auditable         │
         │  ✅ Production standard, GitOps friendly               │
         └────────────────────────────────────────────────────────┘

         Imperative (quick, one-off):
         $ kubectl run anihpj --image=anihpj:v1 --port=8000
         $ kubectl create deployment anihpj --image=anihpj:v1 --replicas=3
         $ kubectl expose deployment anihpj --port=80 --target-port=8000

         Declarative (production, repeatable):
         $ kubectl apply -f deployment.yaml

         [COMPARISON TABLE]
         | Approach     | When to use           | CKA exam   | Production  |
         | Imperative   | Quick tasks, exam speed| ✅ Faster   | ❌ Avoid   |
         | Declarative  | Real environments     | ✅ Required | ✅ Always  |

         Pro tip for CKA exam: Use imperative to GENERATE YAML quickly:
         $ kubectl run anihpj --image=anihpj:v1 --dry-run=client -o yaml > pod.yaml

    7.1  YAML Basics for Kubernetes
         - YAML = "YAML Ain't Markup Language"
         - Indentation matters (spaces only, NO tabs!)
         - Key-value pairs, lists, nested objects
         - Every K8s manifest has 4 required fields:
           apiVersion, kind, metadata, spec

         [YAML SYNTAX — visual cheat sheet]
         ┌────────────────────────────────────────────────────────┐

         │  ┌─────────────────────────────┐                       │
         │  │ name: anihpj               │  key: value            │
         │  │ replicas: 3                │  key: number           │
         │  │ debug: true                │  key: boolean          │
         │  └─────────────────────────────┘                       │
         │                                                         │
         │  2️⃣ LIST (array):                                       │
         │  ┌─────────────────────────────┐                       │
         │  │ ports:                      │  each "- " is an item │
         │  │ - 8000           ◄── item 1 │                       │
         │  │ - 8080           ◄── item 2 │                       │
         │  │ - 443            ◄── item 3 │                       │
         │  └─────────────────────────────┘                       │

         │  └─────────────────────────────┘                       │

         │  ✅ Spaces only (2 per indent level)                   │

         │  ✅ Colons need a space after them: key: value         │
         │  ❌ NOT key:value (missing space!)                     │
         │  ✅ Strings can be unquoted: name: anihpj              │

Chapter 15: Resource Management
--------------------------------------------------------------------------------
    🎯 [CKA EXAM TOPIC]

    15.1  Resource Requests and Limits
          - Requests: guaranteed minimum resources (used for scheduling)
          - Limits: maximum resources (enforced by kubelet)
          - CPU: measured in millicores (1000m = 1 CPU core)
          - Memory: measured in bytes (Mi, Gi)
          [CODE: Pod with resources.requests and resources.limits]

          [REQUESTS vs LIMITS — CKA must-know]
          ┌─────────────────────────────────────────────────┐
          │  RESOURCE REQUESTS & LIMITS                       │
          │                                                   │
          │  resources:                                       │
          │    requests:        ← "I need at least this much" │
          │      cpu: "250m"   ← 0.25 CPU cores              │
          │      memory: "256Mi" ← 256 MB RAM                │
          │    limits:          ← "Don't use more than this"  │
          │      cpu: "500m"   ← 0.5 CPU cores               │
          │      memory: "512Mi" ← 512 MB RAM                │
          │                                                   │
          │  CPU exceeded limit?    → Throttled (slowed) ⏳   │
          │  Memory exceeded limit? → OOMKilled (killed) 💀   │
          │                                                   │
          │  No requests set?  → Scheduler doesn't know needs │
          │  No limits set?    → Pod can consume ALL node RAM  │
          │  ⚠️ ALWAYS set both in production!                │
          └─────────────────────────────────────────────────┘

    15.2  QoS Classes — Quality of Service
          🎯 [CKA EXAM TOPIC]
          - Guaranteed: requests == limits (both set and equal)
          - Burstable: requests < limits (at least one set)
          - BestEffort: no requests or limits set
          - Order of eviction: BestEffort (first) → Burstable → Guaranteed (last)
          [TABLE: QoS class, criteria, eviction priority]

          [QOS CLASSES — eviction order]
          | QoS Class    | When                      | Eviction priority | anihpj? |
          | Guaranteed   | requests == limits        | Last (safest)     | ✅ Prod |
          | Burstable    | requests < limits         | Middle            | ⚠️ Dev  |
          | BestEffort   | No requests or limits     | First (risky!)    | ❌ Never|

    15.3  LimitRange — Namespace Default Limits
          🎯 [CKA EXAM TOPIC]
          - Sets default requests/limits for containers in a namespace
          - Prevents pods from running without limits
          [CODE: LimitRange YAML for anihpj-dev namespace]

    15.4  ResourceQuota — Namespace Capacity Control
          🎯 [CKA EXAM TOPIC]
          - Limits total resource consumption per namespace
          - Prevents one team from using all cluster resources
          [CODE: ResourceQuota YAML — max 10 pods, 4 CPU, 8Gi memory]

          [RESOURCE CONTROL HIERARCHY]
          ┌─────────────────────────────────────────────────┐
          │  RESOURCE CONTROL — 3 levels                     │
          │                                                   │
          │  Level 1: ResourceQuota (namespace total)        │
          │  "anihpj-dev can use max 4 CPUs total"           │

Chapter 17: Kubernetes Networking Fundamentals
--------------------------------------------------------------------------------
    🎯 [CKA EXAM TOPIC]

    17.1  The Kubernetes Networking Model
          - Every Pod gets its own unique IP address
          - All Pods can reach all other Pods without NAT
          - All Nodes can reach all Pods without NAT
          - The IP a Pod sees itself as = the IP others see it as
          [DIAGRAM: Flat networking — any pod can talk to any pod]

          [KUBERNETES NETWORKING RULES]
          ┌────────────────────────────────────────────────────────┐
          │  THE 4 FUNDAMENTAL NETWORKING RULES                     │
          │                                                          │
          │  Rule 1: Every Pod gets a unique IP                     │
          │  ┌──────┐  ┌──────┐  ┌──────┐                          │
          │  │Pod A │  │Pod B │  │Pod C │                          │
          │  │10.1.1│  │10.1.2│  │10.2.1│                          │
          │  └──────┘  └──────┘  └──────┘                          │
          │                                                          │
          │  Rule 2: Pod-to-Pod — direct communication              │
          │  Pod A (10.1.1) → Pod C (10.2.1) ✅ No NAT needed     │
          │                                                          │
          │  Rule 3: Node-to-Pod — direct communication             │
          │  Node 1 → Pod C (10.2.1) on Node 2 ✅ No NAT          │
          │                                                          │
          │  Rule 4: Pod sees its own real IP                       │
          │  Inside Pod A: "my IP is 10.1.1"                        │
          │  From Pod B:   "Pod A is 10.1.1"  ✅ Same IP           │
          └────────────────────────────────────────────────────────┘

Chapter 21: Volumes and Persistent Storage
--------------------------------------------------------------------------------
    🎯 [CKA EXAM TOPIC]

    21.1  The Problem: Container Storage is Ephemeral
          - Container filesystem is temporary — dies with the container
          - Pod restart = all data inside the container GONE
          - anihpj users upload resumes, profile photos → must persist!
          [DIAGRAM: Container dies → data lost vs Volume → data survives]

          [WHY VOLUMES? — THE DATA LOSS PROBLEM]

          │  WITHOUT VOLUMES:              WITH VOLUMES:            │
          │                                                         │
          │  ┌────────────┐               ┌────────────┐           │

          │  └──────┬─────┘               └──────┬─────┘  │       │

          [TABLE: Node maintenance commands]

Chapter 31: Debugging Pods and Containers
--------------------------------------------------------------------------------
    🎯 [CKA EXAM TOPIC — 30% weight section]

    31.1  Pod Lifecycle and Status Codes
          [TABLE: Pod phases and their meaning]
          | Phase        | Meaning                                         |
          | Pending      | Accepted but not yet running (scheduling/pull)   |
          | Running      | At least one container is running                |
          | Succeeded    | All containers terminated successfully           |
          | Failed       | All containers terminated, at least one failed   |
          | Unknown      | Cannot determine pod state (node comm issue)     |

          [TABLE: Common container states]
          | State                  | Common Cause                            |
          | Waiting: ErrImagePull  | Wrong image name or tag                 |
          | Waiting: CrashLoopBack | Container starts then crashes repeatedly|
          | Waiting: CreateContErr | Security context / volume mount issue   |
          | Waiting: ContainerCrea | Image pull or resource issue            |

          │ STEP-BY-STEP WALKTHROUGH:                             │
          │                                                       │
          │ 1. Identify: $ kubectl get nodes                      │
          │    → worker01  NotReady  <roles>  <age>  <version>   │
          │                                                       │
          │ 2. Describe:  $ kubectl describe node worker01        │
          │    → Look at Conditions section for clues             │
          │    → MemoryPressure, DiskPressure, PIDPressure?      │
          │                                                       │
          │ 3. SSH to node: $ ssh worker01                        │
          │                                                       │
          │ 4. Check kubelet:                                     │
          │    $ systemctl status kubelet                          │
          │    → If "inactive/dead": systemctl start kubelet     │
          │    → If errors: journalctl -u kubelet -f             │
          │                                                       │
          │ 5. Common fixes:                                       │
          │    - Kubelet not running → systemctl enable kubelet   │
          │      && systemctl start kubelet                        │
          │    - Wrong kubelet config → check /var/lib/kubelet/   │

          │    - Swap enabled → swapoff -a                         │
          │                                                       │

          │    → worker01 should show Ready                       │

          │    a) "Insufficient cpu/memory"                       │

          │       → Add toleration to pod spec OR remove taint   │

    34.2  Troubleshooting Methodology Checklist
          [CHECKLIST: The 5-Step Troubleshooting Method]
          1️⃣ IDENTIFY: What is the symptom? (get → describe → events)
          2️⃣ GATHER: Collect info (logs, describe, exec)
          3️⃣ ANALYZE: What changed? What's different?
          4️⃣ FIX: Apply the smallest possible fix
          5️⃣ VERIFY: Confirm the fix worked (get, test, curl)

    35.4  etcd Quorum and Leader Election
          - etcd requires a MAJORITY of members to function (quorum)
          - 3 members: tolerates 1 failure (quorum = 2)
          - 5 members: tolerates 2 failures (quorum = 3)
          - NEVER use even numbers for etcd cluster size
          [DIAGRAM: etcd quorum math]
          ┌────────────────────────────────────────────────────────┐
          │  ETCD QUORUM FORMULA: (N/2) + 1                       │
          │                                                        │
          │  Members │ Quorum │ Tolerated Failures │ Recommended? │
          │  ────────┼────────┼────────────────────┼──────────────│
          │    1     │   1    │         0           │ Dev only ❌   │
          │    2     │   2    │         0           │ Never ❌      │
          │    3     │   2    │         1           │ Standard ✅   │
          │    5     │   3    │         2           │ Large prod ✅ │
          │    7     │   4    │         3           │ Rare ⚠️       │
          └────────────────────────────────────────────────────────┘

    37.6  Helm vs Kustomize — When to Use Each
          [COMPARISON TABLE]
          | Feature          | Helm                | Kustomize              |
          | Approach         | Templates + values   | Patches on base YAML   |
          | Learning curve   | Moderate             | Low                     |
          | Package sharing  | ✅ Charts repos      | ❌ Not designed for     |
          | Complexity mgmt  | Go templates         | Strategic merge patches |
          | Built into kubectl| No (separate binary) | Yes (kubectl apply -k) |
          | Best for         | Shared/public charts | Internal config mgmt   |
          
          [TIP: You can use BOTH together — Helm renders, Kustomize patches]

Appendix B: CKA YAML Templates
--------------------------------------------------------------------------------

    B.1  Pod Template (with all common fields)
         [YAML TEMPLATE WITH FIELD ANNOTATIONS]
         apiVersion: v1                    # Always v1 for pods
         kind: Pod                         # Object type
         metadata:
           name: anihpj-web               # Required: unique within namespace
           namespace: anihpj              # Omit for default namespace
           labels:                         # Key-value pairs for selection
             app: anihpj
             tier: frontend
         spec:
           containers:                     # List of containers (at least 1)
           - name: web                     # Container name
             image: anihpj:v1             # Image[:tag]
             ports:
             - containerPort: 8000        # Informational, does NOT expose
             env:                          # Environment variables
             - name: DEBUG
               value: "False"
             resources:                    # ALWAYS set in production
               requests:                   # Scheduling guarantee
                 memory: "128Mi"
                 cpu: "250m"
               limits:                     # Hard ceiling
                 memory: "256Mi"
                 cpu: "500m"
             readinessProbe:               # When pod is ready for traffic
               httpGet:
                 path: /health
                 port: 8000
               initialDelaySeconds: 5
             livenessProbe:                # When to restart container
               httpGet:
                 path: /health
                 port: 8000
               initialDelaySeconds: 15
           restartPolicy: Always           # Always | OnFailure | Never

    B.2  Deployment Template (with strategy)
         [YAML TEMPLATE WITH FIELD ANNOTATIONS]
         apiVersion: apps/v1
         kind: Deployment
         metadata:
           name: anihpj-web
         spec:
           replicas: 3                     # Desired pod count
           strategy:
             type: RollingUpdate           # RollingUpdate | Recreate
             rollingUpdate:
               maxSurge: 1                 # Extra pods during update
               maxUnavailable: 0           # Zero-downtime update
           selector:
             matchLabels:                  # MUST match template labels
               app: anihpj
           template:                       # Pod template (same as B.1)
             metadata:
               labels:
                 app: anihpj
             spec:
               containers:
               - name: web
                 image: anihpj:v1

    B.3  Service Templates (ClusterIP, NodePort, LoadBalancer)
         [THREE YAML TEMPLATES]
         # ClusterIP (internal only — default)
         apiVersion: v1
         kind: Service
         spec:
           type: ClusterIP                 # Omit type = ClusterIP
           selector:
             app: anihpj                  # MUST match pod labels
           ports:
           - port: 80                     # Service port
             targetPort: 8000             # Container port

         # NodePort (external via node IP:port)
         spec:
           type: NodePort
           ports:
           - port: 80
             targetPort: 8000
             nodePort: 30080              # 30000-32767 range

         # LoadBalancer (cloud provider LB)
         spec:
           type: LoadBalancer
           ports:
           - port: 80
             targetPort: 8000

    B.4  PersistentVolume + PVC Template
         [YAML TEMPLATE WITH FIELD ANNOTATIONS]
         # PV (admin creates)
         apiVersion: v1
         kind: PersistentVolume
         spec:
           capacity:
             storage: 10Gi               # Storage amount
           accessModes:
           - ReadWriteOnce                # RWO | ROX | RWX
           persistentVolumeReclaimPolicy: Retain  # Retain | Delete | Recycle
           storageClassName: manual       # MUST match PVC
           hostPath:                       # Dev only! Use cloud storage in prod
             path: /data/db

         # PVC (developer creates)
         apiVersion: v1
         kind: PersistentVolumeClaim
         spec:
           accessModes:
           - ReadWriteOnce
           resources:
             requests:
               storage: 10Gi             # Must be <= PV capacity
           storageClassName: manual       # Must match PV

Appendix C: Common Errors and Solutions
--------------------------------------------------------------------------------

    [TABLE: Quick reference — error → cause → fix]
    | Error                           | Cause                         | Fix                        |
    | CrashLoopBackOff                | App crash / missing config    | Check logs, fix config     |
    | ImagePullBackOff                | Wrong image / no registry auth| Fix image or add secret    |
    | Pending (pod)                   | No resources / no matching PV | Add resources or create PV |
    | Pending (PVC)                   | No matching PV or SC          | Create PV or fix SC        |
    | NodeNotReady                    | kubelet down / resource issue | SSH, check kubelet, restart|
    | FailedScheduling                | Taints / affinity mismatch    | Check taints, tolerations  |
    | Forbidden                       | RBAC insufficient permissions | Add Role/ClusterRole       |
    | connection refused              | Service/pod not running       | Check target pod status    |
    | certificate has expired         | Cert past validity period     | kubeadm certs renew        |
    | etcd cluster is unavailable     | etcd down / cert issue        | Check etcd pods/certs      |
    | unable to connect to server     | kubeconfig wrong / API down   | Fix kubeconfig / check API |
