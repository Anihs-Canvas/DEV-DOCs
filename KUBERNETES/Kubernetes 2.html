ServiceAccount

ServiceAccount Overview

ServiceAccountList - Collection of ServiceAccounts

Official Docs: ServiceAccount v1 API | API Version: v1

ServiceAccount binds together a name, authentication principal, and secrets for pod identity management.

Full Description

ServiceAccount provides an identity for processes that run in a Pod, enabling your anihpj Django application to authenticate with the Kubernetes API server. Each pod in your jobpost application can use a ServiceAccount to access cluster resources securely, making API calls, reading ConfigMaps, or accessing Secrets.

By default, Kubernetes automatically creates a default ServiceAccount in every namespace and mounts an API token into each pod. However, for production Django applications, you should create dedicated ServiceAccounts with minimal permissions following the principle of least privilege. This ensures your Skills, Author, Location, and JobPost services only have access to the resources they need.

Syntax

# Create a ServiceAccount from YAML file
kubectl create -f serviceaccount.yaml
# Create a ServiceAccount imperatively
kubectl create serviceaccount {name}
# List all ServiceAccounts
kubectl get serviceaccounts
# List all ServiceAccounts (short form)
kubectl get sa
# Get detailed information about a specific ServiceAccount
kubectl describe serviceaccount {name}
# Get detailed information about a specific ServiceAccount (short form)
kubectl describe sa {name}
# Get ServiceAccount in YAML format
kubectl get serviceaccount {name} -o yaml
# Delete a specific ServiceAccount
kubectl delete serviceaccount {name}
# Delete a specific ServiceAccount (short form)
kubectl delete sa {name}

YAML Structure

apiVersion: v1
kind: ServiceAccount
metadata:
  name: anihpj-django-sa
  namespace: default
  labels:
    app: anihpj
    component: authentication
automountServiceAccountToken: true
imagePullSecrets:
- name: docker-registry-secret
secrets:
- name: anihpj-api-token

Parameters Description

  • apiVersion: string - API version, always "v1" for ServiceAccount
  • kind: string - Resource type, always "ServiceAccount"
  • metadata: ObjectMeta - Standard object metadata including name, namespace, labels, and annotations
  • automountServiceAccountToken: boolean - Whether pods using this ServiceAccount should automatically mount API tokens
  • imagePullSecrets: []LocalObjectReference - List of secrets for pulling container images
  • secrets: []ObjectReference - List of secrets that pods can use with this ServiceAccount

Return Value

  • 200 (ServiceAccount): OK - Request successful, ServiceAccount returned
  • 201 (ServiceAccount): Created - ServiceAccount successfully created
  • 202 (ServiceAccount): Accepted - Request accepted for processing
  • 401: Unauthorized - Authentication required

ServiceAccountList

Official Docs: ServiceAccountList v1 API | API Version: v1

ServiceAccountList is a collection of ServiceAccount objects returned by list operations.

Full Description

ServiceAccountList represents a collection of ServiceAccounts in your Kubernetes cluster. When you query for multiple ServiceAccounts using kubectl or the Kubernetes API, the response is returned as a ServiceAccountList object. This is particularly useful for managing authentication across all services in your anihpj Django application, allowing you to view and manage ServiceAccounts for Skills, User, Author, Location, and JobPost models simultaneously.

Parameters Description

  • apiVersion: string - API version, always "v1"
  • kind: string - Resource type, always "ServiceAccountList"
  • metadata: ListMeta - Standard list metadata including resourceVersion and continue token
  • items: []ServiceAccount (required) - Array of ServiceAccount objects

Return Value

  • 200 (ServiceAccountList): OK - Request successful, list of ServiceAccounts returned
  • 401: Unauthorized - Authentication required

TokenRequest Overview

Official Docs: TokenRequest v1 API | API Version: authentication.k8s.io/v1

TokenRequest requests a token for a given service account, enabling secure authentication for Django applications.

Full Description

TokenRequest provides a mechanism to request authentication tokens for ServiceAccounts in your anihpj Django application. This resource is essential for implementing secure pod-to-API communication, allowing your Django services to authenticate with the Kubernetes API server or other services that require token-based authentication. TokenRequest creates temporary, scoped tokens that can be bound to specific objects and have configurable expiration times.

In the context of the anihpj Django application, TokenRequest enables secure communication between your jobpost services, user management components, and external APIs. The token mechanism ensures that only authorized pods can access sensitive resources, making it ideal for microservices architectures where different Django components need to authenticate with each other or with external services.

Syntax

# Create a token for a ServiceAccount
kubectl create token <serviceaccount-name>

# Create a token with specific audience
kubectl create token <serviceaccount-name> --audience=<audience>

# Create a token with expiration time
kubectl create token <serviceaccount-name> --duration=<duration>

# Create a token bound to a specific pod
kubectl create token <serviceaccount-name> --bound-object-kind=Pod --bound-object-name=<pod-name>

YAML Structure

apiVersion: authentication.k8s.io/v1
kind: TokenRequest
metadata:
  name: anihpj-token-request
  namespace: anihpj-prod
spec:
  audiences:
    - "https://kubernetes.default.svc"
    - "https://anihpj-api.example.com"
  expirationSeconds: 3600
  boundObjectRef:
    kind: Pod
    name: anihpj-jobpost-pod
    uid: "12345678-1234-1234-1234-123456789abc"

Parameters Description

  • audiences - Array of intended audiences for the token, must include identifiers that recipients will validate against
  • expirationSeconds - Requested duration of token validity in seconds, actual duration may vary based on server policy
  • boundObjectRef - Reference to an object that the token will be bound to, token only valid while bound object exists
  • boundObjectRef.kind - Type of object to bind to, valid values are 'Pod' and 'Secret'
  • boundObjectRef.name - Name of the object to bind the token to
  • boundObjectRef.uid - Unique identifier of the bound object for additional security

Return Value

  • 200 OK - TokenRequest successfully created, returns token and expiration details
  • 201 Created - TokenRequest resource created successfully
  • 202 Accepted - TokenRequest accepted for processing
  • 401 Unauthorized - Authentication failed, invalid or missing credentials
  • 403 Forbidden - Authorization failed, insufficient permissions for ServiceAccount
