📚 Summary & Quick Start

🥬 What is Celery?

Celery is a distributed task queue for Python. It executes tasks asynchronously (in the background) outside the HTTP request-response cycle.

❌ Without Celery (Blocking)

1. Introduction

1.1 What is Celery?

Celery is an asynchronous task queue/job queue based on distributed message passing. It focuses on real-time operation but supports scheduling as well.

TypeDistributed Task Queue
LanguagePython 3.8+
ProtocolAMQP, Redis, SQS
LicenseBSD-3-Clause
Current Version5.4.x (as of 2026)
Key Concept: Celery doesn't store tasks itself - it uses a message broker (Redis/RabbitMQ) to send tasks and a result backend to store results.

1.2 What is a Task Queue?

A task queue manages work that needs to be done asynchronously. Instead of executing work immediately, tasks are placed in a queue and processed by workers.

┌─────────────────────────────────────────────────────────────────┐
│                        TASK QUEUE CONCEPT                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   REQUEST                    QUEUE                   WORKERS    │
│   ───────                    ─────                   ───────    │
│                                                                  │
│   User clicks      ┌─────┐  ┌──────────────┐  ┌─────┐          │
│   "Generate        │Task │  │ ■ ■ ■ ■ ■ ■  │  │ W1  │──▶ Done  │
│    Report"    ───▶ │  A  │─▶│              │─▶│ W2  │──▶ Done  │
│                    └─────┘  │   FIFO Queue │  │ W3  │──▶ Done  │
│   Response:                 └──────────────┘  └─────┘          │
│   "Processing..."                                               │
│   (instant)                                                     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

Why Use a Task Queue?

  • ⚡ Non-blocking responses - Users don't wait for slow operations
  • 📈 Scalability - Add more workers to handle more tasks
  • 🔄 Reliability - Failed tasks can be retried automatically
  • ⏰ Scheduling - Run tasks at specific times or intervals

1.3 What is a Message Broker?

The message broker is the transport layer - it receives tasks from producers (Django) and delivers them to workers.

Broker URL Format Pros Cons
Redis ⭐ redis://localhost:6379/0 Fast, simple, dual-use (broker + backend) In-memory (data loss on crash)
RabbitMQ amqp://guest:guest@localhost:5672// Reliable, feature-rich, persistent More complex setup
Amazon SQS sqs://aws_key:aws_secret@ Managed, scalable, no maintenance Higher latency, AWS cost
Recommendation: Use Redis for most Django projects. It's simple, fast, and can serve as both broker and result backend.

1.5 When to Use Celery

✅ Good Use Cases

  • 📧 Sending emails/notifications
  • 📊 Generating reports/PDFs
  • 🖼️ Image/video processing
  • 🔄 Data import/export (CSV, Excel)
  • 🌐 API calls to external services
  • 🧹 Database cleanup/maintenance
  • 📈 Analytics processing
  • ⏰ Scheduled tasks (daily reports)

❌ Don't Use For

  • Simple CRUD operations
  • Tasks that must complete before response
  • Operations under 100ms
  • Single-server hobby projects
  • Tasks requiring immediate user feedback

        raise self.retry(exc=exc, countdown=60)  # Retry in 60s

4.9.2 Automatic Retry (autoretry_for)

@shared_task(
    autoretry_for=(ConnectionError, TimeoutError, requests.HTTPError),
    max_retries=5,
    default_retry_delay=30
)
def fetch_data(endpoint):
    """Automatically retries on specified exceptions"""
    response = requests.get(endpoint, timeout=10)
    response.raise_for_status()
    return response.json()

4.9.3 Exponential Backoff

@shared_task(
    bind=True,
    autoretry_for=(Exception,),
    retry_backoff=True,        # Enable exponential backoff
    retry_backoff_max=600,     # Max 10 minutes between retries
    retry_jitter=True,         # Add randomness to prevent thundering herd
    max_retries=5
)
def resilient_task(self, data):
    """
    Retry delays with backoff:
    Attempt 1: immediate
    Attempt 2: ~2 seconds (with jitter)
    Attempt 3: ~4 seconds
    Attempt 4: ~8 seconds
    Attempt 5: ~16 seconds
    Capped at retry_backoff_max (600s)
    """
    process(data)

4.9.4 Retry with Different Arguments

@shared_task(bind=True, max_retries=3)
def process_with_fallback(self, url):
    try:
        return fetch(url)
    except ConnectionError as exc:
        # Retry with a different URL
        fallback_url = url.replace('primary', 'backup')
        raise self.retry(exc=exc, args=[fallback_url])

4.9.5 Max Retries & Handling

from celery.exceptions import MaxRetriesExceededError

@shared_task(bind=True, max_retries=3)
def task_with_final_handler(self, data):
    try:
        return risky_operation(data)
    except Exception as exc:
        try:
            raise self.retry(exc=exc, countdown=60)
        except MaxRetriesExceededError:
            # All retries exhausted - handle gracefully
            logger.error(f"Task failed after max retries: {exc}")
            notify_admin(f"Task permanently failed: {data}")
            return None

5.1 Signatures

A signature wraps a task call with its arguments, allowing you to pass tasks around and combine them.

5.1.1 Creating Signatures

from celery import signature
from jobpost.tasks import send_job_alert, count_active_jobs

# Full signature function
sig = signature('jobpost.tasks.send_job_alert', args=(job_id,))

# Or from task directly
sig = send_job_alert.signature((job_id,), countdown=60)

5.1.2 signature() / s()

# .s() shortcut (most common)

# Execute later

# or

# Normal signature receives result from previous task in chain

# Immutable signature ignores previous result
sig = add.si(2, 3)  # Always add(2, 3), ignores chain input

5.1.4 Partial Arguments

# Create partial signature

# Complete it later
sig = sig.clone(args=(email,))  # Add email
result = sig.delay()

5.2 Primitives Overview

┌─────────────────────────────────────────────────────────────────────────────┐
│                         CELERY CANVAS PRIMITIVES                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  CHAIN ─────▶  [Task A] ──▶ [Task B] ──▶ [Task C]  (Sequential)            │
│                  │            │            │                                │
│                  └────────────┴────────────┘                                │
│                     Results flow forward                                    │
│                                                                              │
│  GROUP ─────▶  ┌─ [Task A] ──┐                                              │
│                ├─ [Task B] ──┼──▶ [Results]  (Parallel)                    │
│                └─ [Task C] ──┘                                              │
│                                                                              │
│  CHORD ─────▶  ┌─ [Task A] ──┐                                              │
│                ├─ [Task B] ──┼──▶ [Callback Task]  (Parallel + Callback)   │
│                └─ [Task C] ──┘         │                                    │
│                                        │                                    │
│                              Receives all results                           │
│                                                                              │
│  MAP ───────▶  [Task(item) for item in iterable]  (One task per item)      │
│                                                                              │
│  STARMAP ───▶  [Task(*args) for args in iterable]  (Unpack arguments)      │
│                                                                              │
│  CHUNKS ────▶  [Task(chunk) for chunk in split(items, n)]  (Batched)       │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
Primitive Import Pattern
chain from celery import chain Sequential A → B → C
group from celery import group Parallel [A, B, C]
chord from celery import chord Parallel + final callback
map task.map(items) Apply to each item
starmap task.starmap(items) Apply with *args
chunks task.chunks(items, n) Split into n-sized batches

5.3 chain()

Execute tasks sequentially - each task receives the result of the previous.

5.3.1 Basic Chain

from celery import chain

result = workflow.delay()

# Alternative using chain()
workflow = chain(
    validate_job.s(job_id),
    process_job.s(),
    notify_users.s()
)
result = workflow.apply_async()

5.3.2 Passing Results

# Results flow from one task to the next
@shared_task
def step1(job_id):
    job = JobPost.objects.get(id=job_id)
    return {'job_id': job_id, 'title': job.title}

@shared_task
def step2(data):  # Receives result from step1
    return {**data, 'validated': True}

@shared_task
def step3(data):  # Receives result from step2
    return f"Processed: {data['title']}"

# Chain execution
chain(step1.s(123), step2.s(), step3.s()).delay()
# step1(123) → step2(step1_result) → step3(step2_result)

# Error in any task stops the chain
workflow = chain(
    step1.s(job_id),
    step2.s(),
    step3.s()
).on_error(error_handler.s())  # Called if any task fails

# Or use link_error on specific tasks
workflow = step1.s(job_id) | step2.s() | step3.s()
result = workflow.apply_async(link_error=error_handler.s())

5.5 chord()

Group of tasks with a callback that receives all results.

5.5.1 Header & Callback

from celery import chord

# - Task duration
# - Number of workers
# - Memory per task

# Rule of thumb: chunks = num_items / (num_workers * 2)

chunk_size = len(items) // (num_workers * 2) or 1
process_batch.chunks(items, chunk_size)

6.1 Result Object

When you call a task asynchronously, you get an AsyncResult object to track its status.

CELERY_BEAT_SCHEDULER = 'django_celery_beat.schedulers:DatabaseScheduler'

❌ Don't

# Don't run multiple beat instances
celery -A proj beat  # On server 1
celery -A proj beat  # On server 2  # BAD!

# Don't schedule tasks too frequently
'every-second': {
    'schedule': 1.0,  # Too aggressive!
}

# Don't ignore timezone settings
# Always set:
CELERY_TIMEZONE = 'UTC'
CELERY_ENABLE_UTC = True
