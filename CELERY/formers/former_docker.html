Docker Fundamentals for Django

🚀 Docker Basics & Installation

Docker revolutionizes application deployment by containerizing your Django applications with all their dependencies. This ensures consistency across development, testing, and production environments.

🐳

What is Docker?

Docker is a platform that uses containerization to package applications and their dependencies into lightweight, portable containers that can run consistently across different environments.

📦

Key Benefits

  • Environment consistency
  • Easy deployment
  • Scalability
  • Resource efficiency
⚡

Django Advantages

Docker eliminates "works on my machine" issues, simplifies dependency management, and enables easy scaling of Django applications in production environments.

Installation Steps

1
Install Docker
curl -fsSL https://get.docker.com -o get-docker.sh && sh get-docker.sh
2
Install Docker Compose
sudo curl -L "https://github.com/docker/compose/releases/download/v2.23.0/docker-compose-$(uname -s)-$(uname -m)" -o /usr/local/bin/docker-compose
3
Verify Installation
docker --version && docker-compose --version

📄 Dockerfile Creation

A Dockerfile is a text file containing instructions to build a Docker image. Here's how to create an optimized Dockerfile for Django applications.

Basic Django Dockerfile

Dockerfile 
# Use Python 3.11 slim image for smaller size
FROM python:3.11-slim

# Set environment variables
ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    PIP_NO_CACHE_DIR=1 \
    PIP_DISABLE_PIP_VERSION_CHECK=1

# Install system dependencies
RUN apt-get update && apt-get install -y \
    build-essential \
    libpq-dev \
    && rm -rf /var/lib/apt/lists/*

# Set work directory
WORKDIR /app

# Install Python dependencies
COPY requirements.txt .
RUN pip install -r requirements.txt

# Copy project
COPY . .

# Create non-root user
RUN useradd --create-home --shell /bin/bash app && \
    chown -R app:app /app
USER app

# Expose port
EXPOSE 8000

# Run the application
CMD ["gunicorn", "--bind", "0.0.0.0:8000", "jobportal.wsgi:application"]
💡 Pro Tip: Dockerfile Optimization

Order your Dockerfile instructions from least to most likely to change. This maximizes Docker's layer caching efficiency, reducing build times significantly.

Advanced Dockerfile with Build Arguments

Dockerfile.advanced 
# Build arguments for flexibility
ARG PYTHON_VERSION=3.11
ARG DEBIAN_VERSION=slim

FROM python:${PYTHON_VERSION}-${DEBIAN_VERSION}

# Metadata labels
LABEL maintainer="your-email@example.com" \
      version="1.0" \
      description="Django Job Portal Application"

# Build arguments
ARG APP_ENV=production
ARG DEBUG=0

# Environment variables
ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    PIP_NO_CACHE_DIR=1 \
    APP_ENV=${APP_ENV} \
    DEBUG=${DEBUG}

# Install system dependencies in single layer
RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential \
    libpq-dev \
    postgresql-client \
    curl \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /app

# Copy and install Python dependencies
COPY requirements.txt requirements-dev.txt ./
RUN if [ "$APP_ENV" = "development" ]; then \
        pip install -r requirements-dev.txt; \
    else \
        pip install -r requirements.txt; \
    fi

# Copy application code
COPY . .

# Create app user and set permissions
RUN groupadd -r app && useradd -r -g app app && \
    chown -R app:app /app && \
    chmod +x /app/entrypoint.sh

USER app

EXPOSE 8000

# Health check
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
    CMD curl -f http://localhost:8000/health/ || exit 1

ENTRYPOINT ["/app/entrypoint.sh"]
CMD ["gunicorn", "--bind", "0.0.0.0:8000", "jobportal.wsgi:application"]

🖼️ Image Management

Effective image management is crucial for maintaining clean, efficient Docker environments. Learn the essential commands and best practices.

Essential Docker Commands

Command Description Example
docker build Build an image from Dockerfile docker build -t myapp:latest .
docker images List all images docker images --filter dangling=false
docker rmi Remove images docker rmi $(docker images -q)
docker tag Tag an image docker tag myapp:latest myapp:v1.0
docker push Push to registry docker push myregistry/myapp:latest

Image Optimization Techniques

Build Commands 
# Build with build args
docker build --build-arg PYTHON_VERSION=3.11 --build-arg APP_ENV=production -t myapp:prod .

# Build with cache optimization
docker build --cache-from myapp:latest -t myapp:new .

# Build and push in one step
docker build -t myregistry/myapp:$(git rev-parse --short HEAD) . && docker push myregistry/myapp:$(git rev-parse --short HEAD)

# Multi-platform build
docker buildx build --platform linux/amd64,linux/arm64 -t myapp:latest --push .

# Analyze image layers
docker history myapp:latest --no-trunc
⚠️ Image Size Warning

Large images increase deployment time and storage costs. Use docker image ls --format "table {{.Repository}}\t{{.Tag}}\t{{.Size}}" to monitor image sizes and optimize accordingly.

🏗️ Multi-stage Builds

Multi-stage builds allow you to create smaller, more secure production images by separating build dependencies from runtime requirements.

Production-Optimized Multi-stage Dockerfile

Dockerfile.multistage 
# ================================
# Build Stage - Contains build tools
# ================================
FROM python:3.11-slim as builder

ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    PIP_NO_CACHE_DIR=1

WORKDIR /app

# Install build dependencies
RUN apt-get update && apt-get install -y \
    build-essential \
    libpq-dev \
    git \
    && rm -rf /var/lib/apt/lists/*

# Install Python packages
COPY requirements.txt .
RUN pip install --user --no-warn-script-location -r requirements.txt

# ================================
# Production Stage - Minimal runtime
# ================================
FROM python:3.11-slim as production

ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    PATH=/home/app/.local/bin:$PATH

# Create app user
RUN groupadd -r app && useradd -r -g app app

# Install only runtime dependencies
RUN apt-get update && apt-get install -y \
    libpq5 \
    curl \
    && rm -rf /var/lib/apt/lists/*

# Copy Python packages from builder stage
COPY --from=builder /root/.local /home/app/.local

# Set up application directory
WORKDIR /home/app
COPY --chown=app:app . .

# Collect static files and set permissions
RUN python manage.py collectstatic --noinput && \
    chown -R app:app /home/app

USER app

EXPOSE 8000

# Health check
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
    CMD curl -f http://localhost:8000/health/ || exit 1

CMD ["gunicorn", "--bind", "0.0.0.0:8000", "--workers", "4", "jobportal.wsgi:application"]

# ================================
# Development Stage - Full toolset
# ================================
FROM builder as development

ENV DEBUG=1

# Install development dependencies
COPY requirements-dev.txt .
RUN pip install --user -r requirements-dev.txt

# Copy source code
COPY . .

# Create non-root user
RUN useradd --create-home --shell /bin/bash app && \
    chown -R app:app /app

USER app

CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]

Building Specific Stages

Build Commands 
# Build development stage
docker build --target development -t myapp:dev .

# Build production stage (default)
docker build --target production -t myapp:prod .

# Build and compare image sizes
docker build --target development -t myapp:dev . && \
docker build --target production -t myapp:prod . && \
docker images myapp
✅ Multi-stage Benefits
  • Significantly smaller production images
  • Improved security (fewer attack vectors)
  • Faster deployment and startup times
  • Clear separation of concerns

Container Operations

🔄 Container Lifecycle Management

Understanding container states and transitions is essential for effective Docker container management.

Container States Diagram

📊 Container State Flow 
Created → Running → Paused → Stopped → Removed
    ↓        ↓        ↓         ↓
   Start   Pause   Unpause   Start

Lifecycle Management Commands

Container Management 
# Create container without starting
docker create --name django-app -p 8000:8000 myapp:latest

# Start existing container
docker start django-app

# Run new container (create + start)
docker run -d --name django-app -p 8000:8000 myapp:latest

# Pause/unpause container
docker pause django-app
docker unpause django-app

# Stop container gracefully (SIGTERM)
docker stop django-app

# Kill container forcefully (SIGKILL)
docker kill django-app

# Restart container
docker restart django-app

# Remove container
docker rm django-app

# Remove running container forcefully
docker rm -f django-app

# View container status
docker ps -a --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}\t{{.RunningFor}}"

Advanced Container Operations

Advanced Operations 
# Execute commands in running container
docker exec -it django-app bash
docker exec django-app python manage.py migrate

# Copy files to/from container
docker cp local-file.txt django-app:/app/
docker cp django-app:/app/logs/error.log ./

# View container logs
docker logs django-app --follow --tail 100

# Inspect container configuration
docker inspect django-app

# View container resource usage
docker stats django-app

# Export container as image
docker commit django-app myapp:snapshot

# Update container configuration
docker update --memory="1g" --cpus="1.5" django-app

🌐 Networking & Port Management

Docker networking enables communication between containers and external systems. Understanding port mapping and custom networks is crucial for Django applications.

Port Mapping Strategies

Port Mapping Examples 
# Basic port mapping (host:container)
docker run -p 8000:8000 myapp:latest

# Multiple port mappings
docker run -p 8000:8000 -p 8001:8001 -p 5432:5432 myapp:latest

# Bind to specific interface
docker run -p 127.0.0.1:8000:8000 myapp:latest

# Random host port assignment
docker run -P myapp:latest

# UDP port mapping
docker run -p 8000:8000/udp myapp:latest

# Port range mapping
docker run -p 8000-8010:8000-8010 myapp:latest

Custom Network Configuration

Network Management 
# Create custom bridge network
docker network create --driver bridge \
    --subnet=172.20.0.0/16 \
    --gateway=172.20.0.1 \
    --opt com.docker.network.bridge.name=django-br \
    django-network

# Create network with custom DNS
docker network create \
    --driver bridge \
    --subnet=172.20.0.0/16 \
    --dns=8.8.8.8 \
    --dns=8.8.4.4 \
    django-network

# Run containers on custom network
docker run -d --name django-web \
    --network django-network \
    --ip 172.20.0.10 \
    -p 8000:8000 \
    myapp:latest

docker run -d --name postgres-db \
    --network django-network \
    --ip 172.20.0.20 \
    postgres:15-alpine

# Connect existing container to network
docker network connect django-network existing-container

# Disconnect from network
docker network disconnect django-network container-name

# Inspect network configuration
docker network inspect django-network

# List networks
docker network ls

# Remove network
docker network rm django-network
🔗 Network Communication

Containers on the same custom network can communicate using container names as hostnames. For example, Django can connect to PostgreSQL using postgres-db:5432 as the database host.

🔧 Environment Variables & Configuration

Environment variables are essential for configuring Django applications in containerized environments. Learn best practices for managing sensitive configuration data.

Environment Variable Methods

Environment Configuration 
# Single environment variable
docker run -e DEBUG=1 -e SECRET_KEY=mysecretkey myapp:latest

# Multiple environment variables
docker run \
    -e DEBUG=0 \
    -e SECRET_KEY=production-secret-key \
    -e DATABASE_URL=postgres://user:pass@db:5432/dbname \
    -e ALLOWED_HOSTS=yourdomain.com,www.yourdomain.com \
    myapp:latest

# Environment file
docker run --env-file .env myapp:latest

# Environment file with overrides
docker run --env-file .env.base --env-file .env.local -e DEBUG=1 myapp:latest

# Read environment from host
docker run -e SECRET_KEY=${SECRET_KEY} myapp:latest

Environment File Examples

.env.example 
# Django Configuration
DEBUG=0

ALLOWED_HOSTS=localhost,127.0.0.1,yourdomain.com

# Redis Configuration

# Email Configuration

EMAIL_HOST_USER=your-email@gmail.com
EMAIL_HOST_PASSWORD=your-app-password

# Static/Media Files

# Security Settings

.env.development 
# Development Environment

ALLOWED_HOSTS=*

# Development Database (SQLite)
DATABASE_URL=sqlite:///db.sqlite3

# Development Email (Console backend)

🔒 Security Warning

Never store sensitive data like passwords or API keys in environment files that are committed to version control. Use Docker secrets or external secret management systems for production deployments.

💾 Volumes & Data Persistence

Containers are ephemeral by default. Volumes provide persistent storage for your Django application data, databases, and user uploads.

Volume Types Comparison

Volume Type Use Case Performance Portability
Named Volumes Database data, persistent storage High High
Bind Mounts Development, config files High Low
tmpfs Mounts Temporary data, caches Very High N/A

Named Volumes for Production

Named Volume Commands 
# Create named volumes
docker volume create postgres_data
docker volume create django_media
docker volume create django_static

# Use named volumes in containers
docker run -d \
    --name postgres-db \
    -v postgres_data:/var/lib/postgresql/data \
    -e POSTGRES_DB=jobportal \
    postgres:15-alpine

docker run -d \
    --name django-web \
    -v django_media:/app/media \
    -v django_static:/app/staticfiles \
    -p 8000:8000 \
    myapp:latest

# Inspect volume details
docker volume inspect postgres_data

# List all volumes
docker volume ls

# Remove unused volumes
docker volume prune

# Backup volume data
docker run --rm \
    -v postgres_data:/data \
    -v $(pwd):/backup \
    alpine tar czf /backup/postgres_backup.tar.gz -C /data .

# Restore volume data
docker run --rm \
    -v postgres_data:/data \
    -v $(pwd):/backup \
    alpine tar xzf /backup/postgres_backup.tar.gz -C /data

Bind Mounts for Development

Bind Mount Examples 
# Development with source code sync
docker run -d \
    --name django-dev \
    -v $(pwd):/app \
    -v /app/node_modules \
    -p 8000:8000 \
    myapp:dev

# Configuration file binding
docker run -d \
    --name nginx-proxy \
    -v $(pwd)/nginx.conf:/etc/nginx/nginx.conf:ro \
    -v $(pwd)/ssl:/etc/ssl/certs:ro \
    -p 80:80 -p 443:443 \
    nginx:alpine

# Log directory binding
docker run -d \
    --name django-app \
    -v $(pwd)/logs:/app/logs \
    -v $(pwd)/media:/app/media \
    myapp:latest

# Read-only bind mounts
docker run -d \
    --name django-app \
    -v $(pwd)/config:/app/config:ro \
    -v $(pwd)/templates:/app/templates:ro \
    myapp:latest

Advanced Volume Management

Advanced Volume Operations 
# Create volume with specific driver
docker volume create \
    --driver local \
    --opt type=nfs \
    --opt o=addr=192.168.1.100,rw \
    --opt device=:/path/to/dir \
    nfs_volume

# Volume with size limit (requires specific storage driver)
docker volume create \
    --driver local \
    --opt type=tmpfs \
    --opt device=tmpfs \
    --opt o=size=1g \
    temp_volume

# Copy data between volumes
docker run --rm \
    -v source_volume:/source \
    -v dest_volume:/dest \
    alpine cp -r /source/. /dest/

# Volume permissions fix
docker run --rm \
    -v django_media:/app/media \
    alpine chown -R 1000:1000 /app/media

# Inspect volume usage
docker system df -v
💡 Volume Best Practices
  • Use named volumes for production data persistence
  • Use bind mounts for development and configuration files
  • Regular backup of critical volume data
  • Monitor volume disk usage with docker system df
  • Use appropriate file permissions (avoid root ownership)

Docker Compose Mastery

🎼 Docker Compose Fundamentals

Docker Compose simplifies multi-container application management by defining services, networks, and volumes in a single YAML file.

Basic Django Compose Configuration

docker-compose.yml 
version: '3.8'

services:
  # PostgreSQL Database
  db:
    image: postgres:15-alpine
    container_name: jobportal_postgres
    restart: unless-stopped
    environment:
      POSTGRES_DB: jobportal
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: ${DB_PASSWORD:-defaultpassword}
      POSTGRES_INITDB_ARGS: "--encoding=UTF-8"
    volumes:
      - postgres_data:/var/lib/postgresql/data
      - ./init-scripts:/docker-entrypoint-initdb.d:ro
    ports:
      - "5432:5432"
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres -d jobportal"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s

  # Redis for Caching and Celery
  redis:
    image: redis:7-alpine
    container_name: jobportal_redis
    restart: unless-stopped
    command: redis-server --appendonly yes --requirepass ${REDIS_PASSWORD:-}
    volumes:
      - redis_data:/data
    ports:
      - "6379:6379"
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 3s
      retries: 3

  # Django Web Application
  web:
    build: 
      context: .
      dockerfile: Dockerfile
      target: ${BUILD_TARGET:-production}
    container_name: jobportal_web
    restart: unless-stopped
    environment:
      - DEBUG=${DEBUG:-0}
      - SECRET_KEY=${SECRET_KEY}
      - DATABASE_URL=postgres://postgres:${DB_PASSWORD:-defaultpassword}@db:5432/jobportal
      - REDIS_URL=redis://:${REDIS_PASSWORD:-}@redis:6379/0
      - CELERY_BROKER_URL=redis://:${REDIS_PASSWORD:-}@redis:6379/1
    volumes:
      - static_volume:/app/staticfiles
      - media_volume:/app/media
      - ./logs:/app/logs
    ports:
      - "8000:8000"
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_healthy
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health/"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s

  # Celery Worker
  celery:
    build: 
      context: .
      dockerfile: Dockerfile
      target: ${BUILD_TARGET:-production}
    container_name: jobportal_celery
    restart: unless-stopped
    command: celery -A jobportal worker --loglevel=info --concurrency=4
    environment:
      - DEBUG=${DEBUG:-0}
      - SECRET_KEY=${SECRET_KEY}
      - DATABASE_URL=postgres://postgres:${DB_PASSWORD:-defaultpassword}@db:5432/jobportal
      - REDIS_URL=redis://:${REDIS_PASSWORD:-}@redis:6379/0
      - CELERY_BROKER_URL=redis://:${REDIS_PASSWORD:-}@redis:6379/1
    volumes:
      - media_volume:/app/media
      - ./logs:/app/logs
    depends_on:
      - db
      - redis
      - web

  # Celery Beat Scheduler
  celery-beat:
    build: 
      context: .
      dockerfile: Dockerfile
      target: ${BUILD_TARGET:-production}
    container_name: jobportal_celery_beat
    restart: unless-stopped
    command: celery -A jobportal beat --loglevel=info --scheduler django_celery_beat.schedulers:DatabaseScheduler
    environment:
      - DEBUG=${DEBUG:-0}
      - SECRET_KEY=${SECRET_KEY}
      - DATABASE_URL=postgres://postgres:${DB_PASSWORD:-defaultpassword}@db:5432/jobportal
      - REDIS_URL=redis://:${REDIS_PASSWORD:-}@redis:6379/0
      - CELERY_BROKER_URL=redis://:${REDIS_PASSWORD:-}@redis:6379/1
