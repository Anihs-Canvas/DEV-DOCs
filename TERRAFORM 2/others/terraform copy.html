1.1.1 Definition and Core Principles

📖 Core Definition

Infrastructure as Code (IaC) is built on several fundamental principles that distinguish it from traditional infrastructure management and enable modern DevOps practices.

📝 1. Codification

All infrastructure is defined in text files that can be read, edited, and understood by both humans and machines. This enables automation and reduces human error.

🔄 2. Reproducibility

The same configuration always produces the same infrastructure, regardless of when or where it's applied. This ensures consistency across environments.

📊 3. Versionability

Infrastructure changes are tracked in version control, providing complete history, rollback capability, and comprehensive audit trails.

🧪 4. Testability

Infrastructure code can be validated, tested, and reviewed before deployment using automated tools like Terratest, OPA, and Sentinel.

💡
💡 Key Insight

IaC treats infrastructure the same way software developers treat application code - with version control, code review, testing, and continuous integration. This alignment enables true DevOps practices.

1.1.2 Declarative vs Imperative IaC

🎯 Understanding Approaches

Understanding the difference between declarative and imperative approaches is fundamental to mastering Infrastructure as Code.

🎯
Declarative

"What" you want

⚡

Imperative

"How" to get there

🎯 Declarative (Terraform)

Definition: You describe the desired end state, and the tool figures out how to achieve it.

main.tf 
# "I want 3 web servers"
resource "aws_instance" "web" {
  count         = 3
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
  
  tags = {
    Name = "web-server-${count.index + 1}"
  }
}
  • ✅
    Simpler to Write

    Focus on desired state only

  • ✅
    Built-in Idempotency

    Safe to run multiple times

⚙️ Imperative (Scripts)

Definition: You specify each step needed to reach the desired state.

deploy.sh 
#!/bin/bash
# "Create servers one by one"
for i in 1 2 3; do
  aws ec2 run-instances \
    --image-id ami-0c55b159cbfafe1f0 \
    --instance-type t2.micro \
    --tag-specifications \
    "ResourceType=instance,Tags=[{Key=Name,Value=web-server-$i}]"
done
  • ⚠️
    Manual Error Handling

    Must handle failures yourself

  • ⚠️
    No Idempotency

    Creates duplicates on re-run

1.1.3 Idempotency in Infrastructure

🔐 Safe, Repeatable Operations

Idempotency means that applying the same operation multiple times produces the same result. This is crucial for reliable infrastructure management and enables safe automation.

🔄

Why Idempotency Matters

In infrastructure management, you need confidence that running the same operation won't cause unexpected side effects. Terraform's idempotent nature means:

  • Running terraform apply multiple times is always safe
  • Failed deployments can be retried without cleanup
  • CI/CD pipelines can run on every commit
  • Drift is automatically corrected

1.3.2 Provider Plugin System

Terraform uses a plugin-based architecture where providers are separate binaries that communicate with the core via gRPC:

Provider Plugin Lifecycle:
  • Discovery: terraform init reads required_providers block
  • Download: Fetches provider from Terraform Registry or custom source
  • Verification: Validates checksums and GPG signatures
  • Caching: Stores in .terraform/providers/ directory
  • Execution: Core spawns provider process and communicates via gRPC
📄 Provider Configuration
# versions.tf - Declare required providers
terraform {
  required_version = ">= 1.0.0"
  
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
    random = {
      source  = "hashicorp/random"
      version = "~> 3.5"
    }
  }
}

# providers.tf - Configure providers
provider "aws" {
  region = "us-east-1"
  
  default_tags {
    tags = {
      Environment = "production"
      ManagedBy   = "terraform"
    }
  }
}

1.3.4 State Backend Architecture

State backends determine where and how Terraform stores state. Choosing the right backend is crucial for team workflows.

💻 Local Backend (Default)
  • State stored in terraform.tfstate
  • Good for learning and solo projects
  • ❌ No locking, no team collaboration
☁️ Remote Backends
  • State stored in S3, Azure Blob, GCS
  • ✅ State locking with DynamoDB/native
  • ✅ Team collaboration enabled
📄 S3 Backend Configuration
# Configure S3 backend for state storage
terraform {
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "prod/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-locks"  # For state locking
  }
}
Backend Features:
  • State Locking: Prevents concurrent modifications
  • Encryption: Protects sensitive data at rest
  • Versioning: Enables rollback to previous states
  • Access Control: IAM policies restrict who can modify state

3.2.5 Variable Precedence

⬆️ Order of Precedence (Lowest to Highest)

When the same variable is set in multiple places, Terraform uses the value with highest precedence.

📊 Precedence Order
📊 Variable Precedence (Lowest → Highest)
Priority (wins)
    ↑
    │  7. -var flag or -var-file on CLI        ← HIGHEST
    │
    │  6. *.auto.tfvars.json
    │
    │  5. *.auto.tfvars (alphabetical order)
    │
    │  4. terraform.tfvars.json
    │
    │  3. terraform.tfvars
    │
    │  2. TF_VAR_name environment variables
    │
    │  1. Default value in variable block       ← LOWEST
    ↓
Bash Precedence Examples 
# 1. Default in variables.tf: region = "us-east-1"

# 2. Environment variable overrides default

# 3. terraform.tfvars overrides env var
# Inside terraform.tfvars: region = "us-west-2"

# 4. *.auto.tfvars overrides terraform.tfvars
# Inside dev.auto.tfvars: region = "eu-west-1"

# 5. CLI -var flag overrides everything
terraform apply -var="region=ap-south-1"

3.6.4 For Expressions

🔄 Transform Collections

For expressions transform one collection into another, similar to list comprehensions in Python.

🔄 For Expression Syntax
HCL For Expression Examples 
# For expression creating a list: [for item in collection : expression]
# For expression creating a map: {for item in collection : key => value}

locals {
  # Transform list to uppercase
  names = ["alice", "bob", "charlie"]
  upper_names = [for name in local.names : upper(name)]
  
  # Transform list to map
  name_map = {for name in local.names : name => upper(name)}
  
  # Filter with if clause
  long_names = [for name in local.names : name if length(name) > 3]
  
  # With index
  indexed = [for i, name in local.names : "${i}: ${name}"]
  
  # Transform map
  instance_types = {
    web = "t3.micro"
    api = "t3.small"
    db  = "t3.medium"
  }
  
  # Map keys
  instance_names = [for k, v in local.instance_types : k]
  
  # Transform map values
  instance_tags = {for k, v in local.instance_types : k => "Instance: ${v}"}
}

# Practical: Create subnet map for for_each
variable "subnets" {
  default = [
    { name = "public-1", cidr = "10.0.1.0/24", az = "a" },
    { name = "public-2", cidr = "10.0.2.0/24", az = "b" }
  ]
}

locals {
  subnet_map = {for subnet in var.subnets : subnet.name => subnet}
}

resource "aws_subnet" "public" {
  for_each = local.subnet_map
  
  cidr_block        = each.value.cidr
  availability_zone = "${var.region}${each.value.az}"
  
  tags = { Name = each.key }
}

3.6.6 Dynamic Blocks

⚡ Generate Nested Blocks Dynamically
HCL Dynamic Block Example 
# Dynamic blocks generate repeated nested blocks

variable "ingress_rules" {
  default = [
    { port = 80,  cidr = "0.0.0.0/0", description = "HTTP" },
    { port = 443, cidr = "0.0.0.0/0", description = "HTTPS" },
    { port = 22,  cidr = "10.0.0.0/8", description = "SSH internal" }
  ]
}

resource "aws_security_group" "web" {
  name        = "web-sg"
  description = "Web security group"
  vpc_id      = aws_vpc.main.id
  
  # Dynamic block generates multiple ingress blocks
  dynamic "ingress" {
    for_each = var.ingress_rules
    
    content {
      from_port   = ingress.value.port
      to_port     = ingress.value.port
      protocol    = "tcp"
      cidr_blocks = [ingress.value.cidr]
      description = ingress.value.description
    }
  }
  
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# The above is equivalent to writing:
# ingress {
#   from_port = 80
#   ...
# }
# ingress {
#   from_port = 443
#   ...
# }
# ingress {
#   from_port = 22
#   ...
# }
Dynamic Block Components:
  • for_each: Collection to iterate over
  • iterator: Optional custom name (default: block name)
  • content: The nested block to generate
  • ingress.key: Map key or list index
  • ingress.value: The current element

3.7.2 Collection Functions

📋 List, Map, and Set Functions
HCL Collection Function Examples 
locals {
  # length - count elements
  list_len = length(["a", "b", "c"])  # 3
  map_len  = length({ a = 1, b = 2 })  # 2
  
  # element - get list element by index
  elem = element(["a", "b", "c"], 1)  # "b"
  
  # lookup - get map value with default
  value = lookup({ a = 1, b = 2 }, "a", 0)  # 1
  default_val = lookup({ a = 1 }, "c", 0)  # 0 (default)
  
  # concat - merge lists
  merged_list = concat(["a"], ["b", "c"])  # ["a", "b", "c"]
  
  # merge - merge maps (later wins)
  merged_map = merge({ a = 1 }, { b = 2, a = 3 })
  
  # flatten - flatten nested lists
  flat = flatten([["a", "b"], ["c"]])  # ["a", "b", "c"]
  
  # keys / values - extract from map
  map_keys   = keys({ a = 1, b = 2 })    # ["a", "b"]
  map_values = values({ a = 1, b = 2 })  # [1, 2]
  
  # contains - check if list contains value
  has_item = contains(["a", "b", "c"], "b")  # true
  
  # distinct - remove duplicates
  unique = distinct(["a", "b", "a", "c"])  # ["a", "b", "c"]
  
  # sort - sort list
  sorted = sort(["c", "a", "b"])  # ["a", "b", "c"]
  
  # reverse - reverse list
  reversed = reverse(["a", "b", "c"])  # ["c", "b", "a"]
  
  # slice - get sublist
  sliced = slice(["a", "b", "c", "d"], 1, 3)  # ["b", "c"]
  
  # zipmap - create map from two lists
  zipped = zipmap(["a", "b"], [1, 2])  # { a = 1, b = 2 }
}
